NW-RB-01: Analyse Network Traffic & Indicators

Repeatable, structured procedure to analyse suspicious network traffic and indicators — determining whether observed activity is benign noise, misconfiguration, or active malicious behaviour.

🌐 Network threats Tier 1 / 2 Investigation
📋 ID: NW-RB-01

📅 Published: 23 December 2025
·
🏷️ Category: Network Threat
·
Active
Section 1
Purpose

Provide a repeatable, structured procedure to analyze suspicious network traffic and indicators, enabling analysts to determine whether observed activity represents benign network behaviour, misconfiguration or operational noise, or malicious activity (C2, reconnaissance, exfiltration, DoS).

This runbook supports all Network playbooks (NW-PB-01 → NW-PB-05) and acts as the primary network investigation execution layer.

This runbook produces one of three verdicts

✅ Benign

Close — document and monitor

⚠️ Suspicious

Escalate — continue investigation

🔴 Malicious

Contain — invoke blocking runbooks

Section 2
Required Access
🔥 Firewall logs (perimeter and internal) 📡 NDR / IDS / IPS platform 🌐 Proxy / Secure Web Gateway logs 🔍 DNS logs (if available) 📊 Flow logs (NetFlow / VPC Flow) 🎫 SOC ticketing system
Section 3
Inputs Required
📤 Source IP address(es) 📥 Destination IP / domain / URL 🔢 Port(s) and protocol(s) 🕐 Timeframe of activity 🎫 Incident reference ID
Section 4
Step-by-Step Procedure
1
Validate Indicator Context
1
Review the alert or detection source
2
Identify the following:
  • Detection type — IDS, NDR, firewall, or proxy
  • Trigger reason — signature, anomaly, or threshold
3
Confirm indicator format is complete and valid
2
Identify Traffic Direction & Scope
1
Determine traffic direction:
  • Inbound (external → internal)
  • Outbound (internal → external)
  • East-west (internal → internal)
2
Identify source system role (user endpoint, server, appliance) and destination type (internet, internal subnet, cloud service)
3
Record all affected systems
3
Analyse Traffic Characteristics
1
Review network telemetry for the following characteristics:
  • Connection frequency — one-off vs periodic
  • Session duration
  • Payload size — small beacons vs large transfers
  • Protocol usage consistency for the system role
2
Flag the following anomalies:
  • Regular beaconing intervals — strong C2 indicator
  • Unexpected protocols for the system role
  • Abnormal ports or tunnelling behaviour
4
Enrich Network Indicators
1
Perform reputation lookups for all IPs, domains, and URLs involved
2
Check the following for each indicator:
  • Domain age — newly registered domains are higher risk
  • ASN and hosting provider
  • Known malicious infrastructure associations
3
Record all enrichment results in the incident ticket
5
Correlate with Historical Traffic
1
Search historical logs for:
  • Prior communication with the same destination
  • Other systems communicating with the same indicator
2
Identify first-seen timestamp and whether activity has expanded across hosts
3
Determine whether activity is isolated to one host or systemic across the environment
6
Correlate with Endpoint & Identity Telemetry
1
Pivot to endpoint telemetry:
  • Identify the process initiating the network traffic
  • Check command-line arguments if available
2
Pivot to identity telemetry:
  • Identify the user associated with the source IP
  • Review recent sign-ins or anomalies for that account
3
Record all correlations in the incident ticket
7
Determine Malicious Likelihood
1
Based on all evidence gathered, classify traffic as one of:
  • Benign / Legitimate — close with documentation
  • Suspicious — continue monitoring and investigation
  • Malicious — invoke containment runbooks (NW-RB-02 / NW-RB-03)
2
Record the justification for the classification clearly in the ticket
8
Document Findings
1
Summarise traffic pattern, indicators involved, and correlated endpoint / identity context
2
Attach to the ticket: log excerpts, screenshots, and enrichment results
3
Update the SOC ticket with verdict and recommended next steps
Section 5
Expected Outcome

✅ Clear understanding of network activity intent
✅ Malicious vs benign determination with supporting evidence
✅ Actionable intelligence to invoke containment runbooks if required

Section 6
Validation Checks
  • Indicators verified across multiple data sources
  • Historical correlation performed
  • Endpoint and identity context reviewed
  • Verdict supported by documented evidence
Section 7
Common Errors & Tips
A single connection does not equal compromise — look for patterns, periodicity, and context before escalating
CDN and cloud providers often host both benign and malicious traffic — always check the specific domain, not just the IP
Regular beaconing periodicity is a strong C2 indicator — even low-volume beaconing at fixed intervals warrants investigation
Always validate the source system's role before classifying traffic — a web server making DNS queries is different from a user workstation doing the same
Section 8
Output Artefacts
Network analysis summary
Indicator enrichment results
Affected systems list
Verdict classification
