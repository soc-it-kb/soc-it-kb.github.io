EP-PB-02: Malware Execution on Endpoint

Response framework for confirmed or suspected malware execution — invoked when execution has occurred, as opposed to mere suspicion. Focused on rapid containment, preventing spread, and evidence preservation.

Section 1
Purpose

Provide a standardised decision and response framework for detecting, investigating, and responding to confirmed or suspected malware execution on an endpoint, ensuring rapid containment, prevention of spread, and preservation of evidence.

This playbook is invoked when execution has occurred (or is highly likely), as opposed to mere suspicion handled under EP-PB-01.

Section 2
Scope

✅ In scope

  • Malware executed via email attachments or links
  • Drive-by downloads
  • Script execution (PowerShell, CMD, WMI)
  • User-launched malicious binaries
  • Endpoints with EDR coverage — workstations and servers
  • MDR escalations confirming execution
Common execution vectors
PowerShell WMI CMD Macro LNK ISO/IMG

❌ Out of scope

  • Pure delivery without execution → EM playbooks
  • Identity-only compromise with no host activity → ID playbooks
  • Confirmed ransomware encryption → EP-PB-05
Section 3
Trigger Conditions
  • EDR confirms malware execution on an endpoint
  • MDR escalates confirmed malicious process execution
  • SOC observes malicious child processes or script execution
  • Endpoint telemetry confirms payload execution
  • EP-PB-01 investigation escalates to confirmed execution
Section 4
Initial Validation (Decision Logic)

Confirm that execution has occurred:

  • Was a malicious or suspicious process launched on the endpoint?
  • Did the process spawn child processes?
  • Was a file written to disk or injected into memory?
  • Did execution result in outbound network communication?

If NO to all → Route back to EP-PB-01 (Suspected Compromise)
If YES to any → Proceed with malware execution assessment (Section 5)

Section 5
Malware Execution Assessment

Assess the type and impact of execution. Presence of any indicator = Confirmed Malware Execution:

Known malware family identified by EDR or MDR
Script interpreter used (PowerShell, WMI, CMD)
Obfuscation or packing observed in the payload
Outbound network communication initiated post-execution
Persistence mechanisms created on the endpoint
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Malware communicates with external C2 infrastructure
🔴 Persistence mechanisms established
🔴 Privileged user endpoint involved
🔴 Malware executed on a server or shared system
🔴 MDR confirms hands-on-keyboard attacker activity

Notify immediately

Section 9
Containment Decision Logic

🔴 Immediate isolation

  • Active malware still running on the endpoint
  • External C2 communication confirmed
  • Lateral movement suspected or confirmed

🟡 Controlled containment

  • Malware execution has completed
  • Business-critical system — isolation has high impact
  • Forensic capture required before isolation
Section 10
Scope Expansion Logic

Scope expansion is mandatory for all EP-PB-02 incidents:

  • Search for the same hash across all endpoints via EDR telemetry
  • Identify similar process trees across the tenant
  • Correlate network indicators across all hosts
  • Identify other endpoints accessed by the same user
  • Review recent email and identity activity to identify the delivery vector
Section 11
Communication & Coordination
  • Coordinate containment timing carefully with IT operations
  • Do not alert the end user prior to containment being executed
  • Align all actions with the MDR escalation workflow
  • Escalate to Incident Command if multiple endpoints are affected
Section 12
Closure Criteria
  • Malware fully removed or endpoint rebuilt from clean baseline
  • All persistence mechanisms eliminated
  • IOCs blocked across the environment
  • Credentials reset if exposure was suspected
  • No further malicious activity observed
Section 13
Evidence to Capture
Malware hashes and filenames
Process trees and command lines
Network indicators (IPs, domains, URLs)
Persistence artefacts
Timeline of execution and response
Section 14
Post-Incident Actions
  • Determine the initial infection vector
  • Tune endpoint prevention and detection rules
  • Feed IOCs into threat hunting and SIEM
  • Review user privileges and endpoint security posture
  • Update awareness materials if execution was user-initiated
