IT-SI-WI-04: PAM Platform Hardening

Vendor-agnostic implementation guidance for hardening Privileged Access Management platforms: vault encryption with HSM, session broker hardening, recording integrity, just-in-time access, identity-source integration, HA, vault-independent break-glass procedure. Operationalises §4.4 of IT-SI-SOP-01.

Prerequisites

Before applying this WI:

  • WI-01 management-plane hardening complete on the PAM platform — AAA + MFA (AAL3 for admins), management network restriction, syslog, NTP, certificate hygiene, change control all in place
  • Identity-source integration confirmed: central directory (Entra ID / AD) reachable; PAM is provisioned in the directory; MFA factors enrolled for all PAM administrators
  • HSM availability confirmed where the PAM platform supports HSM integration; partition keys available for vault-encryption use
  • Network architecture: dedicated management network for PAM exists and is logically separated from data plane; brokering interfaces' source networks defined
  • Backup destination configured (the backup target itself is independent of the PAM platform — not on a system whose access depends on PAM)
  • Break-glass procedure draft prepared with named approvers, vault-independent credential storage location, and recovery steps
  • HA pair specifications known (active/standby, sync link, geographic placement)
  • Change window approved per WI-01 §10; coordinate with Identity & Access Engineering and IT Operations Lead
This is the highest-value-target hardening in the security infrastructure domain. A compromised PAM is a credential supply for everything else. Treat every step here with the discipline that consequence implies. The break-glass procedure (Step 7) is the single most important step in this WI. Without it, an unreachable PAM is an organisational outage; with it tested, an unreachable PAM is an inconvenience.
Step-by-step instructions

The steps below cover the PAM-specific controls required by IT-SI-SOP-01 §4.4. They assume WI-01 management-plane hardening is in place.

1
Vault encryption at rest — HSM-backed keys not stored on the appliance
Vault contents protected by cryptography whose keys are not extractable from a compromise of the PAM appliance itself.

Vault encryption configuration:

  • At-rest encryption using vendor-approved cryptography (typically AES-256 or vendor equivalent). Confirm the encryption is engaged; some PAM platforms have it as a configuration toggle that's off by default in older versions.
  • HSM integration where supported — the master encryption key (or key-encryption key) lives in an HSM, not on the PAM appliance. PAM platforms that support PKCS#11 or KMIP can integrate with on-prem HSMs (Thales, Entrust nShield, Utimaco) or cloud KMS (Azure Key Vault HSM, AWS CloudHSM).
  • If HSM is not supported by the platform, document the limitation as an exception, deploy with the strongest in-platform key protection available, and flag the platform for HSM-capable replacement on the asset roadmap. This is a real gap, not just a paper one.
  • Key rotation. Master encryption key rotation procedure documented and tested annually; vendor-recommended rotation cadence followed
  • Backup key handling. Backup of encryption keys (where required for disaster recovery) stored in a separate location with separate access control — physical safe, separate HSM, or escrow service. Backup-key recovery procedure is part of the break-glass workflow (Step 7).
Verify. Show the vault encryption configuration; confirm at-rest encryption enabled. For HSM-integrated deployments, confirm the platform's connection to the HSM is healthy; force a cryptographic operation and verify it's served by the HSM. For non-HSM deployments, confirm the exception is documented in GRC-REG-EXCEPTIONS.
PAM: "vault encryption"
2
Session broker hardening — treat as Tier 0
The session broker (jump host) is at the same trust level as the highest-tier target it brokers to.

The PAM session broker (jump host, "PSM" in some vendor terminology) brokers privileged sessions to target systems. Compromise of the broker compromises every session it brokers and credentials those sessions use.

  • Broker is a Tier 0 system regardless of what tier of system it brokers to. If the broker can reach a Tier 0 server, the broker is Tier 0. Apply the Tier 0 server baseline to it.
  • Network exposure. Broker accepts inbound connections only from administrative networks (defined source list); outbound to targets passes through documented paths only
  • OS hardening per the Tier 0 server baseline — minimal services, latest patches, EDR enrolled, no shared accounts, MFA on access
  • Broker-side controls — clipboard, drive mapping, file transfer disabled or controlled per session policy (a session that can copy a file off the target defeats much of the recording integrity)
  • Process isolation — one session per broker instance where the platform supports it; ephemeral broker instances destroyed after session end (the "throwaway jumphost" model)
  • Patching — broker OS and PAM agent patched on the Tier 0 patch cycle
Verify. Broker host inventoried as Tier 0 in GRC-REG-ASSET. EDR enrolled and reporting. Patch level current. Initiate a brokered session; verify session-level controls (clipboard, drive map) behave per policy.
PAM: "session broker hardening"
3
Session recording integrity — write-once, tamper-detected
Privileged session recordings are an audit artefact whose integrity must survive an attacker with platform access.

Session recording controls:

  • Recording scope. Define which session types are recorded: typically all interactive privileged sessions to Tier 0/1 targets; selective for Tier 2/3 based on risk. Documented in the access policy per target class.
  • Write-once storage. Recordings written to immutable storage where supported — WORM disk, object storage with retention lock (Azure immutable blob, AWS Object Lock), or write-once tape. Where true write-once is not available, use integrity-protected storage with cryptographic chaining.
  • Hash chaining or signing. Recording files are hashed at write time; hashes are chained so that any modification of a previous recording invalidates the chain. The hash chain itself is signed or stored separately.
  • Tamper detection. Integrity verification runs on a schedule (daily); failures alert SOC immediately. Tampered or missing recordings are an incident, not a maintenance item.
  • Retention. Recordings retained per organisation retention policy (typically 1 year minimum for Tier 0/1 access, longer per regulatory requirement). Retention enforced by the storage layer, not by the PAM application alone (so an attacker with PAM admin cannot shorten retention).
  • Access to recordings. Read access controlled separately from PAM administration; viewing a recording is itself a logged event.
Verify. Record a test session; the file appears in the recording store with hash. Modify the file (in a controlled lab); verify the integrity check detects the modification and alerts. Confirm retention lock prevents deletion before policy retention expires.
PAM: "session recording integrity"
4
Just-in-time access enforcement — standing access is the exception
Privileged access is granted for a bounded time on documented approval; standing privileged accounts are minimised.

JIT model:

  • Default posture: users have no standing privileged access; they request access via the PAM platform's access-request workflow when they need it
  • Approval workflow — documented per target system or system class. Tier 0 targets typically require approval from two named approvers (one operational, one security); Tier 1 typically requires one approver; Tier 2/3 may have automatic approval against pre-defined eligibility
  • Time-bounded grant — access granted for a documented duration (typically 1-4 hours for break-fix, 8 hours for planned change windows); auto-expires; no manual extension without re-approval
  • Standing-access exceptions — some operational reality requires standing access (24/7 monitoring service accounts, automation, on-call engineers who can't wait for approval at 2am). Each standing-access exception is documented in GRC-REG-EXCEPTIONS with justification and review date.
  • Account provisioning model — preferred: PAM provisions ephemeral target accounts at grant time, deprovisions at expiry. Alternative: persistent target account, PAM rotates password before and after each grant.
  • Eligibility lists per target — not every user is eligible to even request access to every target; eligibility maintained per role with quarterly review
Verify. Sample a JIT grant: request submitted, approval recorded, access granted for documented duration, access auto-revoked at expiry. Sample standing-access exceptions; each has a documented justification and a current review date.
PAM: "just-in-time access"
5
Identity-source integration — no local user database except break-glass
PAM uses the central directory; user lifecycle is managed centrally; no parallel user store.

Identity integration:

  • Authentication source: central directory (Entra ID, AD) for all PAM users. SAML, OIDC, or LDAPS integration depending on platform support.
  • MFA enforcement: per WI-01 step 3 — AAL2 minimum for general access; AAL3 for access that brokers to Tier 0 systems. Enforced at the identity provider, not just at the PAM platform.
  • Group-based authorisation: central directory groups map to PAM roles (PAM admin, PAM operator, target-system administrator, auditor). Group memberships are the authorisation grant; PAM-local group memberships are not maintained.
  • Lifecycle integration: joiner / mover / leaver process drives PAM access. A leaver in HR is removed from directory groups; access to PAM disappears the same day. No PAM-side cleanup needed.
  • One exception — the break-glass local account. A single named local account exists for break-glass use. Password stored externally (see Step 7). All other authentication via the directory.
  • Service accounts and automation. Where automation needs PAM access, use platform-specific service principals (e.g. Entra application registrations, AD service accounts) with certificate-based authentication; not human-style passwords.
Verify. List local PAM user database; only the break-glass account exists. Authenticate as a directory user; succeeds; appears in audit. Test the leaver workflow with a non-production directory user; PAM access removed on directory-group removal.
PAM: "identity-source integration"
6
High availability with annual failover test
Single-node failures don't deny privileged access; HA itself is tested.

HA configuration:

  • HA pair (active/standby or active/active per vendor capability) for the PAM platform
  • Geographic placement appropriate to the DR strategy — for organisations with multi-site DR, the second PAM node is in the DR site
  • HA sync link on a dedicated path; replication health monitored
  • Replication includes vault contents, configuration, and session-recording metadata
  • Annual failover test under controlled conditions — confirm the standby actually takes over, sessions remain functional, vault contents readable from the standby. Test the inverse failback too. Results logged in the asset record.
  • DR (separate from HA). Backups (Step 8) and the break-glass procedure (Step 7) cover the case where both HA nodes are unreachable.

For PAM, HA is not a Tier 0 / Tier 1 distinction — every production PAM platform is HA. There is no "Tier 3 PAM"; if it manages credentials in production, it's high-value.

Verify. Trigger controlled failover; standby takes over; existing sessions either continue (active/active) or reconnect cleanly (active/standby). Run a privileged-access request end-to-end on the failed-over node. Failback test logged.
PAM: "high availability"
7
Break-glass procedure — documented, vault-independent, tested annually
When PAM is unreachable, recovery does not require credentials retrievable only from PAM. This is the most important step in this WI.
The break-glass procedure is the single most-emphasised control in IT-SI-SOP-01. The single most common failure pattern in PAM deployment is a non-existent or untested break-glass. Treat this step as you'd treat the PAM platform itself: design it carefully, test it, document it, review it.

What "break-glass" means in PAM context: PAM is unreachable for any reason (HA failure, network isolation, ransomware, vendor outage); the organisation needs to recover access to one or more target systems whose credentials are held in PAM.

Components:

  • Break-glass credential set — a small documented set of "last resort" credentials for the most critical target systems (Tier 0 domain admin, core network device emergency accounts, virtualisation platform emergency accounts, the PAM platform's own local break-glass account)
  • Storage location independent of PAM — physical safe (sealed envelopes with check-in/check-out log), separate password manager unrelated to the PAM platform, or a documented split-secret arrangement among named individuals. The storage must be reachable when the PAM platform is not.
  • Access control to break-glass storage — defined approvers (typically IT Operations Lead, Head of IAE, with a senior leadership escalation path); a check-out logged in writing (the physical safe approach makes this easy)
  • Documented recovery procedure — the steps to follow: who to call, how to access the break-glass store, what credentials are available, how to restore PAM service afterwards, how to rotate all credentials that were exposed during break-glass
  • Post-break-glass response — every credential exposed during break-glass is rotated within 24 hours of PAM recovery. Every action taken during break-glass is logged manually and reconciled with normal audit when PAM returns.
  • Annual test — under controlled conditions (production PAM remains running; the test simulates unavailability), the team executes the break-glass procedure end-to-end:
    • Approvers contacted in the documented escalation chain
    • Break-glass storage accessed and credentials retrieved
    • Recovery commands executed on a test target system
    • Manual log of actions captured
    • All credentials exposed during the test are rotated within 24 hours
    • Results documented; gaps remediated

Common pitfalls:

  • Break-glass credentials that have themselves expired — the procedure that hasn't been tested in two years discovers the emergency credentials no longer work (account locked, password rotated by automation that nobody flagged). Annual testing catches this.
  • Approvers unreachable — the named approver left the company eighteen months ago. Quarterly review of the approver list against current HR data.
  • Single-person break-glass — if only one person knows the safe combination, illness or unavailability is a problem. Split-knowledge or multiple approvers.
  • Break-glass that depends on PAM — e.g. the break-glass credentials are stored "securely" in a network share whose access requires the AD admin password from PAM. Defeats the purpose entirely.
Verify. Break-glass procedure document is current. Storage location is reachable and the credentials are valid (annual test). Approver list reconciles with current HR. Last annual test completed within 12 months and documented.
PAM: "break-glass procedure"
8
Vault backup with annual restore test
Vault data can be restored in a disaster scenario; restore is tested, not assumed.

Backup discipline:

  • Vault data backed up per the organisation backup policy — encrypted; off-platform; ideally off-site
  • Backup encryption keys not stored exclusively on the PAM platform (otherwise a destroyed PAM means the backup is unreadable). Keys stored in HSM, separate vault, or escrow per Step 1's backup-key handling.
  • Backup frequency appropriate to recovery point objective — typically daily, with the most recent N days retained on-line
  • Backup integrity verified on write (vendor checksum or signature)
  • Restore test annually — restore the backup to a non-production environment; verify vault contents readable; verify session-recording metadata intact; verify configuration restorable. Log the test results.
  • Restore time objective documented for the PAM platform; the recovery procedure (full restore from backup, not break-glass) is timed during the annual test
  • Backup destination security — the backup target inherits high-value classification; its access controls match the PAM platform's
Verify. Backups completing on schedule. Last restore test within 12 months and documented. Backup encryption keys location confirmed independent of PAM.
PAM: "backup & restore"
9
Network exposure — dedicated management network; brokering interfaces from admin networks only
PAM's attack surface is minimised by network architecture; not all interfaces are reachable from all networks.

Network placement:

  • Management plane on a dedicated management network (per WI-01 step 4); unreachable from data-plane VLANs
  • Brokering interface (the interface end-users connect to when they request a privileged session) reachable only from defined administrative networks: NOC, jump-net, named administrator subnets. Not internet-facing. Not reachable from general corporate user VLANs.
  • Target-side interface (the interface that connects to the target systems being brokered) reachable to target networks only; firewall rules restrict to documented paths and target-system management ports
  • API interface (where automation calls PAM) similarly restricted to known automation sources; certificate-authenticated where possible
  • Vendor cloud connectivity — if the PAM platform connects to vendor cloud (for updates, telemetry, support tunnel), the connection is documented, restricted to vendor-published IP ranges, and explicitly logged. Vendor support-tunnel access requires named on-platform approval at the time of need (not standing-on).
  • For cloud-hosted PAM (PAM appliance running in IaaS or vendor-cloud SaaS), the equivalent controls are at the cloud network-security layer (NSG, security groups, private endpoints, vendor-published IP allow-lists)
Verify. From a general corporate VLAN, attempt connection to the PAM brokering interface; refused. From a data-plane VLAN, attempt connection to the management interface; refused. Document network paths in the architecture diagram; archived in the asset record.
PAM: "network exposure"
10
Privileged session events to SIEM with extended retention
Audit trail of who accessed what privileged target when, retained per policy, anomalies surfaced.

Beyond the standard syslog from WI-01 step 7, PAM-specific event coverage:

  • Privileged session lifecycle: start (with user, target, approver, justification reference), end (with duration), failed authentication, denied access requests
  • Command execution where captured — high-value commands on Tier 0 targets (user-add, sudo, registry edits, GPO changes) logged separately from session-recording for easier correlation
  • JIT grant lifecycle: request submitted, approver action, grant active, grant expired, grant manually revoked
  • Vault access: credential retrievals (who, what credential, when); credential rotations (scheduled vs manual); credential resets
  • Integrity events: session-recording integrity check failures; tampered-record alerts; vault integrity warnings
  • Admin actions: any change to PAM configuration, access policies, eligibility lists, target system definitions
  • Anomaly detection in SIEM — off-hours privileged sessions; access to seldom-used targets; high-volume credential retrievals; geographic anomalies (where applicable). These are SOC playbook items.
  • Retention matches session-recording retention (typically 1 year minimum, longer per regulation); enforced at the SIEM tier in addition to the platform
Verify. Trigger a sample of each PAM event type; confirm SIEM ingestion. SOC has playbooks for the anomaly classes. Retention configured per policy.
PAM: "privileged session events"
Validation & verification

After applying these steps, validate the PAM platform as a whole. Validation here is particularly important — gaps in PAM hardening are gaps in every privileged access path.

Functional validation

  • Vault encryption at rest confirmed; HSM integration healthy where applicable
  • Session broker hardened to Tier 0 server baseline; EDR enrolled
  • Recording integrity verified (controlled tamper test triggers alert)
  • JIT access workflow functional; sample grant completed end-to-end
  • Identity-source integration: only break-glass account in local DB; leaver workflow removes access
  • HA failover tested annually; both members healthy
  • Break-glass procedure tested annually; results logged; gaps remediated
  • Vault backup completing; restore tested annually
  • Network exposure verified by attempted-connection tests from unauthorised networks (refused)
  • Privileged session events flowing to SIEM with extended retention

Rollback & troubleshooting

PAM rollback is particularly sensitive: rollback that breaks identity-source integration or HA can leave administrators locked out of the platform that protects every other administrator account.

Common scenarios

  • Identity-source integration lost; nobody can log in — cause: directory configuration error, certificate expired on the SAML trust, group mapping broken after a directory change. Use the break-glass local account. This is what it's for. Restore directory integration; rotate any credentials exposed during break-glass.
  • HSM unreachable; vault operations failing — cause: HSM connectivity issue, partition key issue, HSM firmware update. The PAM platform may continue serving cached credentials briefly but will degrade. Engage HSM vendor; restore HSM connectivity; verify vault operations resume. If HSM is unrecoverable: backup-key recovery procedure from Step 1.
  • HA failover left the platform in a split-brain state — cause: replication link failed during the failover. Identify the authoritative member (which one served traffic); demote the other; force re-sync. Engage vendor TAC for severe splits.
  • JIT enforcement broke a scheduled automation — cause: an automation that relied on standing access now needs JIT, but no JIT workflow was set up for it. Fix: emergency standing-access exception with sunset date; properly engineer JIT-compatible automation before sunset.
  • Recording integrity alert turned out to be a false positive — cause: clock skew, replication-induced re-hash. Verify the actual integrity (manual hash check); fix the underlying cause; document. Treat with suspicion: in PAM, "false positive" is the default explanation, but verify rather than dismiss.
  • Break-glass test revealed credentials don't work — cause: the break-glass credentials were rotated by an automated job that nobody flagged for break-glass exclusion. Fix: re-establish working break-glass credentials immediately (this is now a real gap, not a hypothetical); investigate the rotation job; exclude break-glass from automatic rotation explicitly.
  • Backup restore test revealed missing data — cause: vendor backup quirk that didn't include a particular configuration scope. Fix: engage vendor; expand backup scope; re-test. Treat as a baseline finding.

Rollback procedure

  1. Identify the failure mode — can administrators still access PAM? If no, break-glass.
  2. If PAM is reachable: identify the breaking change from change record and config diff
  3. Revert the specific change via standard change-control (emergency change if needed)
  4. Verify each function (auth, JIT request, vault access, brokering, recording, replication)
  5. For changes affecting HA: confirm both members healthy before declaring rollback complete
  6. For changes affecting recording integrity: re-establish hash chain; confirm new recordings validate
  7. If break-glass was used: rotate every credential exposed within 24 hours; reconcile manual log with PAM audit
  8. Update change record; document root cause; consider whether the change needs a more cautious re-approach
Common errors & tips
  • Untested break-glass is the single biggest implementation gap. If the procedure isn't tested annually, it's not real. Test it. Schedule the next test before closing this WI.
  • Break-glass credentials excluded from automated rotation. Document this exclusion explicitly in the rotation job; otherwise routine rotation breaks the break-glass.
  • Don't store break-glass credentials in a system protected by PAM. Circular dependency that defeats the entire purpose.
  • HSM is genuinely worth it for Tier 0 PAM. The encryption key on the appliance is the single most valuable secret in the organisation. If a compromise of the PAM appliance compromises the key, the vault encryption is theatre.
  • Session broker is Tier 0 regardless of what it brokers to. If it can reach a Tier 0 target, it's Tier 0. Patch it accordingly; harden it accordingly.
  • Recording integrity needs storage that defeats PAM admins. If PAM admins can edit or delete recordings, the audit trail isn't trustworthy. Write-once storage with retention lock at the storage tier (not the application tier) is the right control.
  • JIT enforcement breaks legacy automation. Expect to discover automations that "always had access" but were never properly engineered. Build the JIT-compatible automation; sunset the standing-access exception.
  • AAL3 requires hardware authenticators. Plan the provisioning before enforcing; otherwise Tier 0 access is impossible from the moment AAL3 is enforced.
  • HA replication health is not a "set and forget". Monitor it; alert on degraded sync; investigate within hours, not days. A PAM HA pair that has silently desynced is a single point of failure with a comforting illusion.
  • The post-break-glass credential-rotation step is often skipped under stress. Build it into the procedure as a non-skippable step; assign explicit responsibility; track to completion.
  • For SaaS-hosted PAM, many of these controls are vendor-administered. The tenant-administered controls (MFA, JIT policy, recording-retention configuration, network access lists) still apply. Confirm via vendor SOC 2 or equivalent attestation what's vendor's vs tenant's; document the boundary.
  • Identity governance for PAM — rotation cadences, JIT approval policy, recording retention — live in the future Identity domain SOP. This WI covers platform hardening, not governance. Where the boundary is unclear (e.g. "what's the right JIT duration"), defer to that future SOP; in the interim, default to the conservative end of the typical range.
