IT-NET-WI-01: Management Plane Hardening

Vendor-agnostic implementation guidance for hardening the management plane of network devices: how administrators connect, authenticate, and configure the device. Operationalises §4.1 of IT-NET-SOP-01.

Prerequisites

Before applying this WI:

  • The device is in Onboarding status in GRC-REG-ASSET (or, for existing devices, an in-place hardening change is approved)
  • Out-of-band access to the device is available in case in-band access is lost during configuration
  • Current configuration is backed up to the central config repository
  • A change window is approved per IT-NET-WI-06 change-control procedures
  • The central AAA service (TACACS+, RADIUS, or Entra ID via cloud-managed equivalent) is reachable from the device and the device is provisioned in the AAA service
  • The central syslog destination, NTP servers, and authorised-use banner text are known
  • The management network (or admin allow-list) for this device is defined
OOB access matters. Several steps in this WI restrict in-band management access. If you do not have working out-of-band access (console, dedicated management port, vendor cloud-management), do not start. The most common cause of an outage during management-plane hardening is the engineer locking themselves out without a recovery path.
Step-by-step instructions

The steps below cover the management-plane controls required by IT-NET-SOP-01 §4.1. Apply them in order; later steps assume earlier ones are in place.

1
Disable plaintext management protocols
Eliminate cleartext credential and configuration exposure on the wire.

On every in-scope device, disable:

  • Telnet (TCP/23) — disable inbound; remove any "transport input telnet" or equivalent
  • HTTP web management (TCP/80) — disable the HTTP server feature
  • SNMPv1 and SNMPv2c — remove community strings; disable the protocol versions
  • FTP and TFTP for configuration transfer — disable inbound; remove any "ip ftp" / "ip tftp" service config
  • HTTP-based vendor "cloud connect" if not in use; if in use, ensure it uses TLS
Verify. From a non-management host, attempt connections to TCP/23, TCP/80, UDP/161 with snmpwalk. All should fail (connection refused, timeout, or unsupported version). On the device, the running configuration should show no "transport input telnet", no "ip http server" (HTTPS only), and no SNMPv1/v2 community strings.
CIS: Mgmt-plane "disable insecure services"
2
Enable secure management protocols with strong configuration
Replace plaintext with cryptographically strong management protocols.

Enable and configure:

  • SSH version 2 only — disable v1 fallback; restrict cipher suites to current-generation only (drop CBC-mode ciphers, drop SHA1 MACs where successor algorithms are available)
  • HTTPS web management — TLS 1.2 minimum, prefer TLS 1.3 where supported; the device certificate is issued by the organisation's CA, not a vendor-default self-signed certificate
  • SNMPv3 with authPriv if SNMP is required — authentication: SHA-256 minimum (SHA-1 only if the device cannot do better); privacy: AES-128 minimum, AES-256 preferred; per-user views restrict OIDs to monitoring needs only
  • SCP or SFTP for file transfer (configuration export, image upload)
Verify. Confirm SSH negotiates v2 only and rejects v1 ("ssh -1" should fail). HTTPS responds with a valid certificate from the organisation's CA. SNMPv3 walks succeed with authentication; SNMPv2 walks are rejected.
CIS: Mgmt-plane "secure remote access"
3
Configure central authentication (AAA) with MFA
All administrative authentication via central directory, attributable to a named individual.

Configure the device to authenticate administrators against the central AAA service:

  • Primary: TACACS+ or RADIUS to the central AAA platform (or vendor-cloud-mediated equivalent for cloud-managed devices)
  • Fallback: a secondary AAA server in a different location for resilience
  • Last-resort local account: one named "break-glass" local account, password vaulted in PAM, used only with approval and rotated after each use
  • MFA: required wherever the AAA platform and device combination supports it; for devices that don't support inline MFA, require MFA on the bastion/jump host from which the device is reached
  • Authorisation: use AAA groups to apply privilege levels — read-only, configuration, full administrative — rather than granting full privilege by default
  • Accounting: log all authentication events and command executions to the AAA platform and to syslog
Verify. Log in as a named administrator via SSH; confirm the AAA platform shows the authentication event and the device syslog records the login. Attempt login with a deliberately invalid password; confirm rejection and that the failed attempt is logged. Confirm the break-glass local account works when AAA is unreachable (test in a controlled manner).
CIS: Mgmt-plane "AAA configuration"
4
Restrict management access to the management network
Limit administrative reach to a known, controlled source set.

Configure a management-source allow-list:

  • Identify the management-network range(s) and any specific administrative hosts (bastion/jump hosts, NOC workstations)
  • Apply an inbound access control list to the management protocols (SSH, HTTPS, SNMP, NETCONF) restricting them to those sources
  • For devices that support it, bind management services to a dedicated management interface and disable management on data-plane interfaces
  • For cloud-managed devices, restrict the vendor cloud-management dashboard access via the vendor's IP allow-list or SSO conditional-access policy
Verify. Attempt SSH from a non-management host; the connection should be refused at the device's ACL layer (or by the platform's management-plane policer). Attempt SSH from the management network; the connection should proceed. Confirm the ACL counters increment on attempted denied connections.
CIS: Mgmt-plane "restrict access by source"
5
Configure session timeouts and banners
Limit exposure of unattended sessions; display the organisation's authorised-use notice.

Configure:

  • Idle session timeout — 15 minutes maximum for privileged sessions; 30 minutes for read-only sessions if used. Configure on each management plane (SSH, HTTPS, console)
  • Login banner — pre-authentication banner stating the device is for authorised use only, that activity is monitored, and that unauthorised use may be prosecuted. Use the organisation's approved banner text (Legal-approved version, available from IT Operations)
  • MOTD banner — post-authentication banner, optional, may include device classification (Tier 0/1/2/3) and key operational notes
Verify. Connect via SSH; confirm the pre-login banner appears before the password prompt. Authenticate; confirm the MOTD if configured. Leave the session idle past the timeout; confirm the session terminates.
CIS: Mgmt-plane "session controls and banners"
6
Configure NTP with authentication
Accurate, trusted time on every device — log timestamps are only useful if they're correct.

Configure NTP:

  • Two or more authoritative NTP servers (the organisation's central NTP infrastructure)
  • NTP authentication enabled with a shared key (or NTPv4 autokey where supported)
  • Time zone set to the organisation's standard for log correlation (UTC is strongly recommended for all infrastructure regardless of site time zone)
  • NTP source interface bound to the management interface where applicable, so NTP traffic uses the management path
Verify. Show NTP status; confirm the device is synchronised (stratum less than 15, offset less than 100ms). Confirm the authentication key is configured and the NTP server is marked authenticated.
CIS: Mgmt-plane "time synchronisation"
7
Configure centralised logging (syslog)
All security-relevant events forwarded to the central SIEM.

Configure syslog forwarding:

  • Two or more syslog destinations (primary and secondary SIEM ingest endpoints)
  • Use TCP or TLS-encrypted syslog where the device and SIEM both support it; fall back to UDP only where neither does
  • Severity level: informational (level 6) minimum for security-relevant events; debug levels reserved for active troubleshooting only
  • Source-interface bound to the management interface where applicable
  • Events to include: all authentication events (success and failure), configuration changes, interface state changes, AAA events, link-flap on critical interfaces, hardware alarms
Verify. Trigger a known event (failed login, configuration change). Confirm the event appears in the central SIEM within expected latency (typically less than 60 seconds). Confirm timestamps are correct and the source IP is the management interface.
CIS: Mgmt-plane "centralised logging"
8
Console & auxiliary port hardening
Physical access does not bypass the controls applied to remote management.

For physical console and auxiliary ports:

  • Console port is physically secured (locked rack, controlled access room)
  • Console login uses the AAA configuration, not a local-only authentication path
  • Console session timeout configured (same as remote management)
  • Auxiliary port disabled where not in use (modem ports, secondary serial); if required for OOB modem, secure with strong authentication and call-back where supported
  • Vendor-default debug interfaces (e.g. "service password-recovery" semantics) reviewed; where the platform allows disabling password-recovery, this is a design decision with operational implications — document accordingly
Verify. Connect via console; confirm the AAA-mediated login flow (not a local fall-through). Confirm idle timeout terminates the console session. Confirm the auxiliary port is disabled or appropriately secured.
CIS: Mgmt-plane "physical access"
Validation & verification

After applying all steps, validate the device against the management-plane requirements as a whole. This is more than the per-step verifications; it's a sanity check that the configuration as deployed actually delivers the intended baseline.

Functional validation

  • Log in as a named administrator from the management network — should succeed
  • Attempt to log in from a non-management host — should be refused at the ACL or platform layer
  • Attempt SSHv1, Telnet, HTTP, SNMPv2 from anywhere — all should fail
  • Generate a configuration change and confirm it is logged to the central SIEM with the right timestamp and source IP
  • Generate a failed authentication and confirm it is logged to both AAA and SIEM
  • Leave the session idle past the configured timeout and confirm it terminates
  • Confirm NTP is synchronised (stratum <15, offset <100ms)

Document the verification. Capture the validation output (show running-config, show ntp status, syslog confirmation) and attach to the change record. This is the audit evidence; without it the baseline can be claimed but not proved.
Rollback & troubleshooting

If applying these changes causes loss of management access or unexpected operational impact, rollback is via the OOB console using the pre-change backup configuration.

Common loss-of-access scenarios

  • Locked out via SSH — most common cause: the management ACL was applied before the AAA configuration was complete, and the administrator's source IP is outside the allow-list. Recovery: via console, remove the ACL or add the source to it; complete AAA configuration; reapply ACL.
  • AAA unreachable, local fallback failed — cause: AAA primary and secondary both unreachable, no break-glass account, or break-glass credentials lost. Recovery: console + password-recovery procedure (where the platform allows); if password-recovery is disabled, this becomes a vendor-support call. Always test break-glass before relying on it.
  • NTP not synchronising — cause: management ACL blocking NTP traffic, or NTP servers unreachable via the management path. Recovery: confirm NTP server reachability from the device; ensure NTP traffic is permitted in the management ACL.
  • Syslog not arriving at SIEM — cause: ACL blocking outbound syslog, wrong destination address, severity threshold too high. Recovery: capture syslog packets at the device interface or via SPAN to confirm they are emitted; validate the SIEM ingest configuration.
  • SSH negotiation failing — cause: cipher mismatch (client too old, or device too strict). Recovery: confirm client supports the configured ciphers; do not weaken the device's cipher list — instead upgrade the client.

Rollback procedure

  1. Connect via OOB console
  2. Verify the device is responsive and authentication path works (use break-glass if AAA is the failed component)
  3. Compare running configuration to pre-change backup; identify the changes that caused the issue
  4. Restore the relevant configuration from backup; do not perform a wholesale restore unless the entire change is to be reverted
  5. Confirm management access is restored from the management network
  6. Update the change record with the rollback action and the root-cause finding for post-incident review
Common errors & tips
  • Don't apply the management ACL before AAA works. The ACL will silently lock you out if AAA is the path you also use to administer the device. Order: AAA first, validate from management network, then ACL.
  • Always have working OOB. If you cannot reach a console or vendor-cloud OOB, do not start this WI on a production device.
  • SNMP v3 user creation is fiddly. Each vendor handles the SNMPv3 user/group/view model slightly differently. Test from a known-good NMS before declaring complete.
  • NTP synchronisation can lag. A newly-configured device may take 5–15 minutes to fully synchronise. Don't conclude NTP is broken on the first "show ntp status" check after configuration.
  • Banner text formatting differs by vendor. Newline handling and delimiter characters vary; verify the banner displays correctly via SSH after configuration.
  • HTTPS certificates need a CA-issued cert, not a self-signed one. Self-signed certificates create training problems (administrators learn to click through warnings) and don't meet the baseline.
  • Don't disable password-recovery casually. Some vendors offer a "service password-recovery disable" option. This is a strong control, but if applied without considering the recovery process for a forgotten password or lost device, recovery becomes a hardware swap. Treat this as a design decision.
  • Cloud-managed devices need a slightly different framing. Many of these controls map to vendor-cloud-portal settings rather than CLI configuration. Apply the equivalent setting; the requirement (e.g. "MFA on admin access") doesn't change.
