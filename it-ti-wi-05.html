IT-TI-WI-05: TI-Driven Threat Hunting Triggers

Step-by-step instructions for translating threat intelligence into proactive threat hunting hypotheses — enabling the SOC to search for threats that automated detections may have missed.

Prerequisites
Validated TI — threat actor profile, campaign report, or IOC set Access to SIEM, EDR, and relevant log sources MITRE ATT&CK reference for TTP mapping
Step-by-step instructions
1
Identify the TI trigger — new threat actor profile, active campaign report, high-profile vulnerability with known exploitation, or anomaly observed in internal telemetry
2
Extract the key TTPs from the TI — what initial access method, persistence mechanism, or lateral movement technique does this threat use?
3
Map TTPs to MITRE ATT&CK technique IDs — use ATT&CK Navigator to visualise coverage gaps
4
Formulate a hunting hypothesis: 'If this threat actor is present in our environment, what evidence would we expect to see in our logs?' — be specific about the data source and expected artefact
5
Identify the data sources needed — SIEM logs, EDR telemetry, DNS logs, proxy logs, authentication logs. Confirm the data is available and adequately retained
6
Build the hunt query — start broad to understand baseline, then narrow to anomalous behaviour. Document the query logic and expected output
7
Execute the hunt across the relevant time window. Note: hunting for a campaign that started 3 months ago requires log retention that covers that period
8
Triage results — distinguish between true positives, benign anomalies, and false positives. Document findings regardless of outcome
9
If compromise is found — invoke the relevant incident response playbook immediately
10
If no compromise found — document the hunt as completed, note the baseline established, and record as evidence of proactive security activity
11
If hunt reveals a detection gap — raise a detection improvement task to create or tune a SIEM rule covering the TTP
Validation & verification
Hypothesis clearly documented before hunting begins
Data sources confirmed available and adequately retained
Hunt query documented with logic and expected output
All results triaged and documented — including no-compromise outcomes
Detection gaps identified and raised as tasks
Common errors & tips
A hunt that finds nothing is still valuable — it either confirms you are not compromised, or reveals a data visibility gap that needs fixing
Always document your hypothesis before you start hunting — it keeps the hunt focused and makes the outcome meaningful
Short log retention is a hunting killer — if your SIEM only retains 7 days of logs and a campaign started 4 weeks ago, you cannot hunt for it. Use TI findings to justify longer retention
Start with the highest-confidence TI — a CISA advisory or confirmed active campaign is a better hunt trigger than a low-confidence feed indicator
Involve Tier-2 and Tier-3 analysts in hunting — it builds skills and improves detection coverage over time
NIST CSF 2.0 alignment
DE.AE — Adverse Event Analysis: TI-driven hunting DE.CM — Continuous Monitoring: proactive threat detection RS.AN — Incident Analysis: hunting-driven discovery
