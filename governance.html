Governance principles
  • All content changes must be reviewed and approved before publication
  • SOPs are mandatory — deviations require a documented exception approved by the SOC Manager
  • Content is reviewed at minimum annually, and following any major incident or significant tooling change
  • All playbooks, runbooks, and SOPs reference the NIST CSF 2.0 function they support
  • All Work Instructions are tool-agnostic — platform-specific procedures are maintained separately
  • ASM SOPs and WIs are implementation-agnostic — they define what should be done, not which tool to use
  • This knowledge base is publicly accessible and intended as a reference for SOC teams, NGOs, not-for-profit organisations, and SMBs
  • Feedback and improvement suggestions should follow the contribution process below
Contribution & change process
1

Identify the change

Raise a change request — gap identified during an incident, lessons learned action, new threat type, tooling update, or baseline change.

2

Draft the content

Draft following the existing format and naming convention. Use published content as a template.

3

Peer review

Review by at least one other SOC analyst or subject matter expert. Feedback incorporated before escalating for approval.

4

SOC Manager / IT Operations approval

SOC Manager approves SOC content. IT Operations or Cyber Security Governance Board approves ASM SOPs. CISO-level sign-off for significant procedural changes.

5

Publish & log

Published to the knowledge base via GitHub Pages with a signed commit. Change log entry created below.
