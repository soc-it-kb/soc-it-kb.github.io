Governance principles
  • All content changes must be reviewed and approved before publication
  • SOPs are mandatory — deviations require a documented exception approved by the SOC Manager or GRC owner
  • Content is reviewed at minimum annually, and following any major incident or significant change
  • All playbooks, runbooks, and SOPs reference the NIST CSF 2.0 function they support
  • All Work Instructions are tool-agnostic — platform-specific procedures are maintained separately
  • ASM SOPs and WIs define what should be done, not which tool to use
  • GRC documents are governed by GRC-POL-01 and the Document Review Register
  • This knowledge base is publicly accessible and intended as a reference for SOC teams, NGOs, not-for-profits, and SMBs
Contribution & change process
1

Identify the change

Raise a change request — gap from incident, lessons learned, new threat, tooling update, baseline change, or GRC requirement.

2

Draft the content

Draft following the existing format and naming convention. Use published content as a template.

3

Peer review

Review by at least one other SOC analyst or subject matter expert. Feedback incorporated before escalating.

4

Approval

SOC Manager approves SOC content. IT Operations / Cyber Security Governance Board approves ASM SOPs. GRC owner approves GRC policies, SOPs, and forms.

5

Publish & log

Published via GitHub Pages with a signed commit. Change log entry created below.
