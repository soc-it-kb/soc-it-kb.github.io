ID-PB-07: Password Spray / Credential Stuffing Attack

Standardised response framework for password spray and credential stuffing attacks — high-volume, low-noise identity attacks that often precede successful account takeover if left undetected.

🔐 Identity threats Tier 1 / 2 MDR integrated

Section 1
Purpose

Provide a standardised decision and response framework for detecting, investigating, and responding to password spray and credential stuffing attacks, ensuring rapid containment, user protection, and prevention of widespread account compromise.

This playbook addresses high-volume, low-noise identity attacks that often precede successful account takeover.

Attack type reference

🔫 Password spray

One or a small number of common passwords tried across a large number of user accounts — designed to avoid account lockout thresholds

🗂️ Credential stuffing

Known username/password pairs from previous data breaches reused against a target organisation — exploits password reuse behaviour

Section 2
Scope

✅ In scope

  • Password spray attacks (one/few passwords across many users)
  • Credential stuffing attacks (known breached credentials reused)
  • Identity platform alerts for abnormal authentication failure patterns
  • MDR or SOC detections of distributed login attempts

❌ Out of scope

  • MFA fatigue or abuse → ID-PB-03
  • Single-user brute force attacks → ID-PB-01
  • OAuth-only abuse without sign-ins → ID-PB-04
Section 3
Trigger Conditions
  • Identity platform flags password spray or credential stuffing behaviour
  • MDR escalates distributed authentication attempts
  • SOC detects multiple failed logins distributed across many users
  • Spike in authentication failures from a single IP or ASN
  • Successful sign-ins following a pattern of widespread failures
Section 4
Initial Validation (Decision Logic)

Validate whether the activity matches an attack pattern:

  • Are failures distributed across many users rather than concentrated on one?
  • Is the same password or a small password set being observed across attempts?
  • Is traffic sourced from a single IP, ASN, or botnet infrastructure?
  • Are attempts using legacy authentication protocols?

If NO → Route to appropriate identity playbook
If YES to any → Proceed with attack pattern assessment (Section 5)

Section 5
Attack Pattern Assessment

Assess the nature of the attack. Presence of any indicator confirms spray / stuffing attempt:

Multiple users targeted within a short timeframe
Low failure rate per account (below lockout threshold)
Same password reused across multiple accounts
Source IP or ASN reused across all attempts
Legacy authentication protocol attempts observed
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Successful authentication achieved for any account
🔴 Privileged or VIP accounts targeted
🔴 MFA not enforced on affected users
🔴 Legacy authentication is currently enabled
🔴 MDR confirms an active and ongoing campaign

Notify immediately

Identity Team
Lockout and MFA enforcement
👔
SOC Leadership

🤝
MDR Partner
Campaign analysis
Section 7
Response Strategy

Execute the following actions based on attack severity and success:

Authentication log analysis — mandatory in all casesRequired
MFA enforcement across affected populationRequired
Legacy authentication review and restrictionRequired
Scope expansion — identify all targeted usersRequired
Source IP / ASN blocking or rate limitingConditional
Credential reset for confirmed affected usersConditional

If successful authentication is confirmed for any account → escalate immediately to ID-PB-01 (Account Compromise) and execute in parallel

Section 9
Attack Containment Logic

During response, validate and apply the following controls where applicable:

IP / ASN blocking

Temporarily block or rate-limit identified source IPs and ASNs

Conditional Access

Apply sign-in restrictions for high-risk locations or authentication patterns

Smart lockout

Review and tune smart lockout thresholds in the identity platform

Password reset

Prioritise resets for all users confirmed in the targeted population

⚠️ If existing controls are insufficient or the attack bypasses protections → escalate to Identity Engineering immediately

Section 10
Scope Expansion Logic

Scope expansion is mandatory for all ID-PB-07 incidents:

Identification actions

  • Identify all users targeted within the attack timeframe
  • Identify all source IPs, ASNs, and geolocations involved
  • Check for recurrence of the same pattern across multiple days

Correlation actions

  • Search for successful sign-ins following failure patterns
  • Correlate with breach intelligence and known credential leaks
  • Cross-reference with open MDR incidents and threat campaigns
Section 11
Communication & Coordination
  • Avoid alerting users prematurely if no compromise is confirmed — may tip off the attacker
  • Notify affected users if password resets are being enforced
  • Coordinate reset timing carefully to minimise business disruption
  • Align all response actions with MDR and incident response workflows
Section 12
Closure Criteria

The incident may be closed when one of the following outcomes is reached:

✅ No compromise confirmed

  • Attack source mitigated or blocked
  • No further abnormal authentication attempts observed
  • MFA enforced on affected population
  • Legacy authentication disabled or restricted

🔒 Compromise confirmed & remediated

  • All compromised accounts remediated under ID-PB-01
  • Sessions revoked and credentials reset
  • MFA enforced across affected population
  • No residual unauthorised access observed
Section 13
Evidence to Capture
Authentication failure and success logs
Targeted usernames list
Source IPs and ASNs
Password spray patterns observed
Conditional Access outcomes
Timeline of attack activity
Section 14
Post-Incident Actions
  • Review password policy strength and smart lockout settings
  • Expand MFA coverage and enforcement across all user populations
  • Disable or restrict legacy authentication protocols
  • Tune SOC and MDR detections for spray / stuffing patterns
  • Feed attacker IPs, ASNs, and patterns into threat hunting queries
  • Update identity security awareness guidance for affected users
