CL-PB-02: Public Exposure / Cloud Misconfiguration

Response workflow for accidental or malicious exposure of cloud resources, management interfaces, databases, object storage, SaaS sharing, or inbound network paths.

Section 1
Purpose

Response workflow for accidental or malicious exposure of cloud resources, management interfaces, databases, object storage, SaaS sharing, or inbound network paths.

This playbook provides a consistent incident lifecycle for cloud detection, containment, evidence preservation, recovery, and lessons learned.

Section 2
Scope

✅ In scope

  • Public storage bucket, object container, file share, or backup repository
  • Exposed database, management interface, admin console, or cloud service endpoint
  • Public IP or inbound network path added to restricted workload
  • Unauthorised SaaS sharing link or anonymous access
  • Security baseline drift that creates external reachability

❌ Out of scope

  • Confirmed sensitive data exposure → CL-PB-04
  • Compromised workload behaviour → CL-PB-03
  • Control-plane compromise or privileged abuse → CL-PB-01
Section 3
Trigger Conditions

Initiate this playbook when any of the following occur:

  • Public storage bucket or object container detected
  • Exposed database, management port, or admin interface
  • Public IP added to restricted workload
  • Unauthorised sharing link or anonymous access
  • External scan confirms unexpected reachability
Section 4
Initial Validation (Decision Logic)

Validate whether the activity is expected, authorised, and business justified:

  • Is the resource intended to be public and documented as an accepted exception?
  • Is authentication required and properly enforced?
  • When did exposure start and what changed?
  • Do access logs show external access, downloads, list operations, or admin attempts?

If a legitimate change or accepted exposure is confirmed, document the evidence and close as benign or approved exception. If unauthorised, suspicious, or unclear, continue with the assessment and escalation sections below.

Section 5
Public Exposure Assessment
  • Confirm what is exposed, from where, since when, and whether authentication is required.
  • Identify resource owner, data classification, business criticality, and baseline expectation.
  • Review access logs for external source IPs, user agents, principals, downloads, list operations, or admin access.
  • Determine whether exposed access involved data, credentials, management control, or business service availability.
Section 6
Severity & Escalation Criteria

Treat as High when internet exposure involves sensitive data, management interfaces, databases, privileged paths, or unknown access. Treat as Medium when exposure is confirmed but access is authenticated, non-sensitive, and quickly contained.

Section 7
Response Strategy
Restrict public access and preserve previous configuration evidence.Required
Remove risky sharing links, anonymous access, broad ACLs, or inbound network rules.Required
Apply network controls and least privilege.Required
Validate business exceptions and document residual exposure if required.Required
Create remediation actions for baseline drift, IaC correction, tagging, ownership, and preventive policy deployment.Required

Section 9
Containment Decision Logic

Containment actions

  • Disable anonymous or public access.
  • Remove unauthorised sharing links and public network paths.
  • Restrict inbound rules to approved sources.
  • Preserve before/after configuration and access evidence.

Approval boundary

  • Use read-only validation before changing production state where possible.
  • Obtain incident lead or authorised owner approval for disruptive changes.
  • Record every containment action in the incident or change ticket.
Section 10
Scope Expansion Logic

Expand investigation to

  • Search for similar exposed resources in the same subscription, tenant, project, and resource group.
  • Review baseline drift, IaC templates, tagging, and ownership metadata.
  • Check whether access logs indicate download, list, delete, or admin activity.
  • Validate policy guardrails prevent reintroduction.

Stop expansion when

  • No related risky configuration, identity, workload, or data access remains.
  • Affected owner confirms business context and residual risk is accepted or removed.
  • Monitoring validates no recurrence after containment.
Section 11
Communication & Coordination
Incident Lead
Owns severity, coordination, and closure
Cloud / Platform Owner
Approves production-impacting actions
Business Owner
Confirms service and data impact
GRC / Legal / Privacy
Engage if data, regulatory, or notification risk exists
Section 12
Closure Criteria
  • Root cause and affected resources are documented.
  • Containment, recovery, and validation actions are complete or tracked as accepted risk.
  • Evidence package is attached to the incident record.
  • Preventive controls or remediation actions are assigned with owners and due dates.
Section 13
Evidence to Capture

Section 14
Post-Incident Actions
  • Update cloud security baselines, guardrails, and monitoring rules.
  • Correct IaC, policy, ownership, tagging, or review gaps.
  • Feed lessons learned into awareness, engineering backlog, and governance registers.
  • Review whether related runbooks or work instructions require updates.
