IT-CL-WI-05: Defender Suite Configuration

Tier-aware implementation guidance for the Microsoft Defender suite: Defender for Endpoint, Defender for Office 365, Defender for Identity, Defender for Cloud Apps, Defender for Cloud, plus TVM integration and central SIEM ingestion. Operationalises §4.5 of IT-CL-SOP-01. Tenant-side configuration only; SOC operational use under SOC SOPs.

Prerequisites

Before applying this WI:

  • WI-01 through WI-04 complete — the Defender suite consumes signals from configurations established in the earlier WIs (Conditional Access, M365 audit log, Azure activity log, subscription Defender plans)
  • Tenant licensing tiers confirmed per agency in GRC-REG-ASSET — each Defender product requires specific licensing (A3/E3 for some, E5 / Plan 2 for others, A5/E5 for the most advanced)
  • For Defender for Endpoint: list of devices to onboard (endpoints, servers, mobile where licensed); device-group taxonomy decided (criticality tier, agency, environment)
  • For Defender for Identity: list of domain controllers where the agency operates AD; service-account ready for sensor installation
  • For Defender for Cloud Apps: tenants to connect (M365, Azure, and any third-party SaaS where DfCA connectors exist)
  • SOC playbook owner identified for the Defender alert types — severity-to-priority mapping documented (see WI-03 Step 6)
  • Central SIEM ingestion paths for Defender alerts confirmed — either Microsoft 365 Defender connector, Graph Security API, or Defender for Cloud continuous export per product
Defender is a suite, not one product. The Defender brand covers six distinct products with different licensing, different scopes, and different operational interfaces. The SOP (§4.5) treats them as a unified set because the suite's value comes from cross-product signals (an Identity alert + an Endpoint alert = a different priority than either alone). This WI configures each product so the suite hangs together.
Current-state note. Defender for Cloud is deployed inconsistently across tenants today. The baseline target is universal enablement at the appropriate licence tier; current gaps are tracked as time-bounded exceptions per IT-CL-SOP-01 §7.4.
Step-by-step instructions

The steps below cover the Defender suite configuration required by IT-CL-SOP-01 §4.5. Steps are licence-tier-aware and target the configuration that the SOC actually consumes; SOC operational use (triage, response) remains under SOC SOPs.

1
Defender for Endpoint deployment A3 / E3+
EDR / XDR on every managed endpoint and server; ASR rules enforcing; baseline assigned uniformly.

Defender for Endpoint (MDE) is the EDR product. Licensing varies: P1 (anti-malware + ASR), P2 (full EDR + investigation), and Server SKU. Most A3/E3 includes P1; E5 includes P2.

  • Onboarding — via Intune device configuration for Intune-managed devices; via Group Policy / configuration management for unmanaged; via Defender for Cloud auto-provisioning for Azure VMs (WI-03 Step 2)
  • Coverage — Windows endpoints, Windows Server (Azure and on-prem), macOS (where licensed), Linux servers (where licensed), iOS/Android (Defender for Endpoint mobile; E5 typically)
  • Device-group taxonomy — documented and applied:
    • By criticality (Tier 0 / 1 / 2 / 3 from asset register)
    • By agency (which operating unit owns the device)
    • By environment (production / non-production / sandbox)
    • By platform (Windows / macOS / Linux / mobile)
  • Security baseline — the Microsoft Defender for Endpoint security baseline (in Intune or via direct policy) assigned to every device group; this is separate from the Defender for Endpoint policy but ensures the underlying device-hardening is in place
  • Attack Surface Reduction (ASR) rules — the 16 ASR rules deployed in Audit mode, monitored 2–4 weeks per rule, then promoted to Block. Some rules have known false-positive patterns; tune narrowly. CIS-recommended ASR set is the starting reference.
  • EDR in block mode — on for all Windows endpoints; ensures EDR can take action even when the primary AV is non-Microsoft
  • Tamper protection — on by default in newer tenant configurations; verify; prevents local admin or malware from disabling Defender
  • Network protection — enabled in Block mode; blocks outbound connections to known-malicious destinations from the endpoint itself
  • Controlled folder access — ransomware mitigation; Audit mode first; deploy to high-risk users (finance, HR) before broad rollout
  • Web content filtering — enabled where licensed; blocks categories per organisation policy
  • Defender for Endpoint alerts forwarded to central SIEM via M365 Defender connector or Graph Security API; severity mapping documented
Verify. Sample devices show "Onboarded" status in Defender portal. ASR rule deployment confirmed per device group. EDR-in-block-mode active. Test detection (EICAR file): alert generated, Defender takes action, alert reaches SIEM.
CIS M365 (A3/E3+) · "Defender for Endpoint" · MCSB ES-x
2
Defender for Office 365 (cross-reference WI-02 Step 8) E5
Email-borne threats handled at the tenant gateway; URL and attachment detonation cover what signature engines miss.

Defender for Office 365 (Plan 1 and Plan 2) was configured in WI-02 Step 8 (M365 Tenant Hardening). This step cross-references that configuration and adds operational integration with the wider Defender suite.

  • Configuration — per WI-02 Step 8 (Safe Attachments Dynamic Delivery, Safe Links, anti-phishing impersonation protection, preset Strict policies for VIPs, Attack Simulator quarterly)
  • Alert integration with the suite:
    • A Defender for Office 365 incident automatically correlates with Defender for Endpoint alerts on the recipient's device (did they click? did they execute the attachment?) in the M365 Defender unified portal
    • Sign-in risk from Identity Protection cross-correlates with email-delivery events (was the credential phished?)
    • Cross-product incidents in M365 Defender appear as a single incident with multiple alerts attached
  • Threat Explorer access — SOC analyst named roles; queries documented in SOC SOP-EM-01 (Email Threat Response)
  • Quarantine review — periodic review of held messages for false-positive patterns; user-quarantine notifications already configured (WI-02 Step 1)
  • Attack Simulator results integrated with user training tracking; high-failure users assigned remediation training
Verify. WI-02 Step 8 controls in place (already validated). Cross-product correlation visible in M365 Defender portal (sample incident covering both email-deliver and endpoint-detection events). SOC playbook reference up to date.
CIS M365 (E5) · cross-reference WI-02 §8
3
Defender for Identity sensor on domain controllers A3 / E3+
On-prem AD activity visible in the cloud Defender suite; lateral-movement and credential-theft patterns detected.

Defender for Identity (MDI; formerly Azure ATP) is the on-prem AD detection product. It runs as a sensor on every domain controller, reading AD network traffic and event logs to detect attack patterns (Kerberoasting, Golden Ticket, DCSync, suspicious group membership changes, etc.).

  • Sensor coverage — one sensor per domain controller in every domain the agency operates; including read-only DCs and DCs in branch sites
  • Service account — gMSA (group Managed Service Account) preferred over a standard service account; minimum permissions for AD reads; documented in identity governance
  • Sensor health — monitor MDI sensor health metrics; sensor offline = blind spot. Alert on health degradation.
  • Honeytoken accounts — a few decoy accounts placed in AD that legitimate users never touch; any auth attempt on a honeytoken is high-fidelity alert; document and configure them
  • Entity tags — sensitive accounts and groups tagged in MDI so alerts on them are prioritised
  • Integration with central SIEM — MDI alerts forwarded via M365 Defender connector or Graph Security API; alert detail visible in SIEM without portal-pivot
  • VPN log integration — MDI can correlate sign-ins with VPN sessions where the VPN appliance forwards syslog to MDI
Defender for Identity is on-prem AD only. Cloud-only tenants (no AD) do not need MDI; tenants with hybrid identity (Entra Connect synchronising on-prem AD) benefit significantly. The product is sometimes overlooked in cloud-first thinking; for any agency operating AD, it is essential.
Verify. Sensor deployed on every DC in scope; health green. Test detection: simulate Kerberoasting in lab (controlled), alert fires. Honeytoken alerts test-fired. SIEM ingestion confirmed.
"Defender for Identity" · MCSB IM-x
4
Defender for Cloud Apps E5
Cloud-app usage visible; sanctioned vs unsanctioned apps; OAuth app governance; anomaly detection on cloud activity.

Defender for Cloud Apps (DfCA; formerly MCAS) is the CASB component. It provides cloud-app visibility (which SaaS apps are users using?), anomaly detection on M365 / Azure activity, OAuth app governance, and information-protection workflows.

  • App connectors — connect to M365 (mandatory), Entra ID, Azure (for activity log), and any third-party SaaS with a DfCA connector (Salesforce, ServiceNow, AWS, GCP, Workday, etc.). Each connector unlocks API-based monitoring and policy enforcement.
  • Cloud Discovery — ingests firewall / proxy logs to enumerate cloud apps users access; produces a risk-scored inventory (sanctioned / unsanctioned / under review). Integration with on-prem proxy logs is the typical setup.
  • Anomaly detection policies — enable the default set (impossible travel, mass download, mass deletion, unusual administrative activity, ransomware detection); tune over 4–6 weeks; review false-positive feedback
  • OAuth app governance — review OAuth applications consented in the tenant; revoke unused or high-risk apps; require admin consent for high-permission apps (Conditional Access policy: "require admin consent for apps requesting high-permission scopes")
  • Information-protection policies — DfCA can apply sensitivity labels and access controls in connected SaaS apps; ties into Purview Information Protection (WI-02 Step 6) for consistent classification across the estate
  • Session policies — for selected scenarios (e.g. unmanaged-device access to a sensitive app), DfCA can proxy the session and apply per-action controls (block download, watermark, etc.); requires Conditional Access integration; use deliberately, not by default
  • Alert integration — DfCA alerts to central SIEM via M365 Defender connector or Graph Security API
Verify. M365 and Azure connectors active; activity visible in DfCA portal. Anomaly policies enabled. OAuth app review completed for top-permission apps. Cloud Discovery ingesting (where proxy log integration in place).
CIS M365 (E5) · "Defender for Cloud Apps"
5
Defender for Cloud (cross-reference WI-03 Step 2)
Azure subscription-level CSPM and workload-protection plans configured per subscription tier (covered in WI-03 Step 2); cross-product correlation in the unified Defender portal.

Defender for Cloud (DfC) was configured per subscription in WI-03 Step 2 (Azure Subscription Baseline). This step cross-references and ensures DfC's signals integrate with the wider Defender suite.

  • Subscription coverage — per WI-03 Step 2 (Foundational CSPM on every subscription; Defender plans per tier; Tier 0/1 = paid Defender plans, Tier 2/3 = Free acceptable with justification)
  • Unified portal integration — DfC alerts appear in the M365 Defender unified portal alongside endpoint, identity, and email alerts; cross-product incidents correlate across Azure resources and M365 entities
  • Regulatory compliance dashboards — DfC's regulatory compliance views (CIS Azure, NIST 800-53, ISO 27001) are one of the inputs to the tenant baseline conformance report (IT-CL-SOP-01 §7.5)
  • Attack-path analysis (Defender CSPM) — enabled for Tier 0/1 subscriptions; visualises chains of misconfiguration that an attacker could exploit; prioritises remediation
  • Workflow automation — Logic Apps for automated response to defined alert types (per WI-03 Step 6); documented playbooks; tested response
Verify. Configuration per WI-03 Step 2 already validated. Unified portal shows Azure incidents alongside M365 incidents. Regulatory compliance dashboards populated. Attack-path analysis (where licensed) shows current paths.

6
Threat & Vulnerability Management feed to the central programme
Defender's vulnerability data flows to the central vulnerability management programme; remediation tracked in the same system regardless of source.

Defender for Endpoint includes Threat & Vulnerability Management (TVM), which surfaces missing patches, vulnerable software versions, and configuration weaknesses on every onboarded device. TVM is a vulnerability source that feeds the central vulnerability management programme (IT-VM-SOP-01).

  • TVM integration with central VM — export of TVM findings to the central vulnerability management platform (Defender API, or via SIEM, or direct connector depending on platform)
  • Asset attribution — TVM findings tagged with device-group taxonomy (Step 1); per-device owner identification for remediation routing
  • Prioritisation — TVM's exposure score and CVSS feed into the central prioritisation per IT-VM-SOP-01
  • Defender for Cloud vulnerability scanning for Azure VMs (where Defender for Servers enabled) — also feeds the central programme
  • Defender for SQL vulnerability assessment — per WI-04 Step 6; findings to central VM
Verify. TVM data flowing to central VM platform; recent finding traceable end-to-end. Sample remediation tracked through the standard workflow.
cross-reference IT-VM-SOP-01
7
Defender alerts to central SIEM — unified ingestion
Every Defender alert reaches the central SIEM with consistent enrichment; SOC works from one queue, not five portals.

The Defender suite produces alerts in each product. The SOC consumes them in the central SIEM (vendor-neutral; whichever platform is operationally current). Configure ingestion deliberately.

  • Ingestion path — depends on the central SIEM platform:
    • Microsoft 365 Defender connector for the unified incident stream (preferred where the SIEM supports it)
    • Graph Security API as a fallback / additional path
    • Defender for Cloud continuous export (Event Hub) for the Azure-side alerts
    • Direct per-product API integration where vendor-specific connectors don't yet support the M365 unified stream
  • Severity-to-priority mapping — documented in SOC playbook:
    • Defender High → SOC priority 1 (immediate response)
    • Defender Medium → SOC priority 2 (response within working day)
    • Defender Low → SOC priority 3 (review weekly)
    • Defender Informational → ingested but not routed to active queue
  • Enrichment — SIEM enriches Defender alerts with asset register data (tier, owner, agency); SOC analysts see the business context in the alert without portal-pivot
  • Deduplication — cross-product incidents (one event, multiple Defender products alerting) deduplicated in SIEM where the unified-incident stream provides the correlation key
  • Health monitoring — ingestion pipeline monitored; broken connector = silent blind spot. Alert when the ingestion volume drops below expected.
  • SIEM platform transition — the central SIEM is under transition (per IT-CL-SOP-01 §3.7); during the migration, both platforms may run in parallel for some tenants; the integration design accommodates the transition
Verify. Sample alert generated in each Defender product (controlled test) reaches the central SIEM with correct severity, enrichment, and entity context. Ingestion health dashboard operational.

Tenant-side configuration vs SOC operational use — boundary maintained
This baseline configures the suite; SOC operating procedures consume it. The boundary is documented so neither team duplicates or misses work.

The split is operationally important:

  • This WI's scope — tenant-side configuration of Defender products: enablement, policies, baselines, integrations, alert export
  • SOC SOPs scope — operational use of Defender signals in the SIEM: triage, investigation, hunting, response, containment. Covered by SOP-IM-01, SOP-EM-01, SOP-ID-01, SOP-EP-01.

Concrete handoff points:

  • Configuration changes to Defender products — this WI's owner (Cloud Engineering / IAE / Tenant Admins); change-controlled
  • Tuning of alert thresholds where the change is tenant-wide — this WI's owner with SOC input
  • Alert suppression for known-benign patterns within an investigation — SOC analyst; documented in the incident record; suppression review monthly
  • Hunt query development — SOC; supported by Cloud Engineering for new data sources
  • Defender product roadmap (new feature evaluation) — this WI's owner with SOC input on operational value
Verify. RACI for Defender configuration vs operational use is documented and agreed between Cloud Engineering / IAE / Tenant Admins / SOC. No tickets routed to the wrong owner.

Suite-level metrics
The Defender suite has standing measurable outputs that feed governance and demonstrate baseline conformance.

The Defender suite produces several quantitative metrics that complement the binary baseline-conformance assessment.

  • Defender for Endpoint exposure score — the TVM-driven posture score per device group; target: trend monotonically downward
  • Defender for Endpoint device coverage — percentage of in-scope endpoints onboarded; target: ≥98%
  • Defender for Endpoint ASR rule coverage — percentage of in-scope endpoints with ASR rules in Block (not Audit); target: ≥95% for the CIS-recommended ASR set
  • Defender for Office 365 effective protection — Safe Attachments / Safe Links coverage; target: 100% of mailboxes
  • Defender for Cloud Secure Score per WI-03 Step 2 (≥75% T0/1, ≥60% T2/3)
  • Cross-product MTTD / MTTR for high-severity incidents — tracked in SOC SOPs; target per IRP
  • Defender alert ingestion completeness — SIEM ingestion health; target: 100% with no >5-minute gaps
  • Defender suite conformance per tenant — binary: all applicable products enabled per licence tier; target 100% for tenants with sufficient licensing
Verify. Metrics reportable per tenant and per agency. Monthly review by IT Operations; quarterly review by GRC and Risk Committee with separate cuts per platform.

Rollback & troubleshooting

Defender configuration changes can affect end-user experience (false-positive blocks, performance impact) and SOC operations (alert volume changes). Plan rollout carefully.

Common scenarios

  • ASR rule false-positives blocking legitimate software — cause: a rule blocks behaviour that a line-of-business app uses. Fix: switch the specific rule to Audit; identify the FP pattern; add per-file or per-path exclusion (narrow); re-enable Block. Do not blanket-disable the rule.
  • Defender for Endpoint performance impact on a workload — cause: aggressive real-time scanning on a high-IO server. Fix: managed exclusions (specific paths, specific processes) per Microsoft guidance for the workload type; never disable RTP wholesale.
  • MDI sensor offline — cause: service-account issue, network restriction, or DC offline. Fix: validate service account permissions; verify required ports / endpoints reachable; check sensor logs. Track sensor health as a standing metric.
  • DfCA Cloud Discovery shows huge unsanctioned list — expected on first ingestion; not a problem. Triage: which apps are actively used, by whom, for what; sanction the legitimate ones, block the high-risk, leave the rest in monitor mode.
  • OAuth app revocation broke a legitimate integration — cause: an app with legitimate use was revoked during cleanup. Fix: re-consent the app with documented justification; review what permissions it actually needs (often less than initially requested).
  • SIEM ingestion gap detected — cause: connector token expired, API rate limit, M365 Defender side outage. Fix: rotate the connector credential; check for pending Microsoft service health issue; re-test ingestion. Backfill if possible (most Defender APIs allow time-range query).
  • Tamper protection prevents an admin-initiated config change — cause: trying to change a Defender setting locally on a device. Fix: change via Intune / portal, not locally; if local change genuinely needed, use the documented Tamper-Protection-bypass procedure (time-bounded, audit-logged).
  • Alert storm after enabling a new product — cause: backlog of pre-existing patterns now visible. Fix: triage the storm; suppress the highest-volume known-benign patterns (with documented justification); narrow the rule that's over-firing. Storm subsides as the platform learns.

Rollback procedure

  1. Identify the disruption source (specific Defender product, specific rule, specific device group)
  2. For ASR / Defender behaviour: switch the rule to Audit, not disable
  3. For sensor health: investigate root cause; do not skip sensor coverage as a workaround
  4. For SIEM ingestion: check the ingestion pipeline; do not disable export at the Defender side as a workaround
  5. Update change record; document root cause; address before re-enabling Block / re-enabling cleanup
Common errors & tips
  • Defender is a suite, not a product. Each Defender product has its own licensing, portal surface, and configuration story. Configure all the applicable products in the suite, not just MDE.
  • ASR rules need Audit before Block. Going straight to Block on rules like "Block Office applications from creating child processes" causes legitimate workflows to break (some line-of-business apps legitimately spawn child processes). Audit, review, tune, Block.
  • Tamper protection is the answer to "Defender disabled by malware." If a tenant has Defender deployed but tamper protection off, a successful local privilege escalation can disable Defender before EDR detection kicks in. Tamper protection prevents this; turn it on.
  • Defender for Identity needs network access from DCs. The sensor talks to Microsoft endpoints; in air-gapped environments or proxy-restricted environments, this fails silently. Verify connectivity.
  • Honeytoken accounts are high-fidelity alerts. They never legitimately log in; any auth attempt is suspicious. Configure a handful, name them realistically, and alert on them with high priority.
  • DfCA OAuth governance catches what other controls miss. Phishing-resistant MFA doesn't help if a user is tricked into consenting to a malicious OAuth app that gets persistent access to their mailbox. Review OAuth apps; revoke unused; require admin consent for high-permission scopes.
  • Cloud Discovery is only as good as the proxy log ingestion. If the on-prem proxy logs aren't flowing to DfCA, Cloud Discovery is blank. Configure the log connector or accept that Cloud Discovery is a partial capability.
  • SIEM ingestion health is itself a standing metric. Don't assume the connector is working because it was working last month. Alert on ingestion-volume drops.
  • Unified portal vs per-product portal. The M365 Defender portal (security.microsoft.com) is the unified view; the per-product portals still exist for some configuration. Document which portal does what; train SOC accordingly.
  • TVM is a vulnerability source, not a separate workflow. Don't track TVM remediation in a separate tracker; feed it into the central VM programme so an asset has one remediation queue regardless of where the finding came from.
  • SOC operational use is not this WI's scope. Don't write hunt queries here; don't write incident response procedures here. Configure the suite; the SOC SOPs use it.
  • Licensing matters. Many Defender features require specific tiers (Defender for Cloud Apps = E5; Defender for Office 365 P2 = E5 or P2 add-on; Defender for Endpoint P2 = E5). Don't over-claim capability in baseline conformance reports for tenants that don't license the product.
