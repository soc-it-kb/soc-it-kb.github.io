EP-RB-01: Endpoint Investigation Following Email Threat

Investigate endpoints where a malicious email artefact may have been executed or interacted with — determining compromise status and collecting evidence to support containment decisions.

💻 Endpoint threats Tier 1 / 2 Investigation

Section 1
Purpose

Investigate endpoints where a malicious email artefact may have been executed or interacted with.

Possible verdicts — this runbook determines which applies

✅ Clean

No malicious activity — close with monitoring note

⚠️ Suspicious

Indicators present — escalate to EP-PB-01 for deeper investigation

🔴 Compromised

Active compromise confirmed — isolate immediately, invoke EP-PB-02

Section 2
Required Access
🛡️ EDR console 📊 Endpoint telemetry 🎫 SOC ticket
Section 3
Inputs Required
🖥️ Endpoint hostname 👤 Username 🕐 Timestamp of interaction 📎 Associated file or URL
Section 4
Step-by-Step Procedure
1
Locate the endpoint in the EDR console using hostname or username
2
Review the detection timeline — filter to the incident timeframe
Start slightly before the email delivery timestamp to catch any pre-execution activity
3
Identify process execution — look for processes spawned by the email client or browser
Always pivot from the parent process — e.g. Outlook → cmd.exe → PowerShell is a classic malicious chain
4
Check for suspicious child processes:
  • PowerShell, WScript, CScript, MSHTA spawned from email client
  • Execution from user-writable paths (Downloads, Temp, AppData)
  • Encoded or obfuscated command lines
5
Review outbound network connections from the endpoint in the timeframe
  • Connections to unknown or low-reputation domains
  • Beaconing or periodic connection patterns
  • DNS queries for unusual or algorithmic domains
6
Identify persistence indicators:
  • Scheduled task creation
  • Registry autorun modifications
  • New service or driver installation
7
Capture all suspicious artefacts — hashes, file paths, command lines, network indicators
8
Determine the compromise status — Clean, Suspicious, or Compromised — and record with supporting evidence
Section 5
Expected Outcome

✅ Endpoint status confirmed — Clean, Suspicious, or Compromised
✅ Evidence collected and attached to incident ticket
✅ Containment recommendation documented

Section 6
Validation Checks
  • No active malicious processes running on the endpoint
  • No persistence mechanisms identified or all removed
Section 7
Common Errors & Tips
Always pivot from the parent process — the parent process chain reveals the attack vector (e.g. Outlook → WScript → C2 connection)
Check for execution from user-writable paths (Downloads, %Temp%, AppData) — legitimate software rarely executes from these locations
Correlate endpoint findings with mailbox activity — if a rule was created or forwarding enabled around the same time, the compromise is likely deeper
Section 8
Output Artefacts
Compromise verdict
Process tree export
File hashes and paths
Network indicators
Containment recommendation
