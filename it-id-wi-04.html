IT-ID-WI-04: Non-Human Identity Governance

Service accounts (AD user-style, gMSA, sMSA), Entra service principals, app registrations, managed identities (system / user-assigned), workload identity federation, certificate-based identities. Inventory, named ownership, credential rotation discipline, secret expiry tracking, OAuth scope review, decommissioning workflow. Operationalises §4.4 of IT-ID-SOP-01.

Prerequisites

Before applying this WI:

  • IT-ID-WI-03 (Role Design) at least partially established — non-human identities need scoped roles, not unbounded permission grants
  • Asset register (GRC-REG-ASSET) available — service accounts and service principals link to the systems and applications they serve
  • Secret-management capability available — Azure Key Vault, HashiCorp Vault, or equivalent for service principal secrets and certificates
  • Centralised logging of authentication events — identity sign-in events feed SIEM (IT-CL-WI-01 logging configuration and IT-SRV-WI-DC-04 for AD)
  • For Azure tenants: Microsoft Entra Workload Identities (Premium) licence checked — advanced workload identity features (conditional access for workload identities) need this
  • Awareness of legacy applications that may require service accounts with interactive sign-in (LDAP-only, vendor appliances) — documented as platform-debt
  • Inventory-source identified per non-human identity class — AD service accounts via directory query, Entra service principals via Graph API, managed identities via Azure Resource Graph
Non-human identities outnumber human identities in most estates. A typical mid-size tenant has 1,000–5,000 human identities and 5,000–30,000 non-human identities (counting AD service accounts, Entra service principals, managed identities, certificates, app-only authentication contexts). They rarely rotate credentials, often hold standing privilege, frequently lack named owners, and almost never get recertified. This is the largest under-governed surface in most estates — treat it accordingly.
Step-by-step instructions

The steps below operationalise IT-ID-SOP-01 §4.4.

1
Inventory non-human identities across all platforms
A complete list of every non-human identity in the estate — service accounts, service principals, managed identities, certificates — with type, scope, and last-activity data.

Governance starts from inventory. Identities that aren't enumerated aren't governed.

  • AD service accounts: directory query for user accounts flagged as service accounts (naming convention, OU placement, attribute markers). Includes user-style service accounts (legacy pattern), gMSA (Group Managed Service Accounts — modern AD pattern with auto-rotation), sMSA (Standalone Managed Service Accounts — less used but extant).
  • Entra service principals: Microsoft Graph API enumeration. Two classes: application service principals (from app registrations) and managed identity service principals (auto-created by Azure resources).
  • Entra app registrations: the application objects backing service principals. Includes multi-tenant apps used by the organisation.
  • System-assigned managed identities: tied to Azure resources (VMs, app services, function apps, storage); lifecycle bound to the resource.
  • User-assigned managed identities: standalone identity objects assignable to multiple resources; lifecycle is independent of the resources using them.
  • Workload identity federation: federated credentials configured on app registrations / user-assigned managed identities; replace long-lived secrets with short-lived tokens from external identity providers.
  • Certificates: identity-bearing certificates (client authentication, code signing identities) maintained in PKI / certificate vault.
  • Per-identity inventory record: identity ID, type, scope (what it can access), creation date, last sign-in / last activity, credential type (password / secret / certificate / federated), credential expiry, named owner, owner-confirmed date.
  • Refresh cadence: nightly automated; manual reconciliation monthly.
Verify. Inventory query runs successfully across all platforms; identity counts plausible (no large unexplained jumps); per-identity record has required fields populated for ≥90% of inventory; remaining 10% have an in-flight remediation plan.
NIST 800-53 AC-2 · CIS Controls 5.2 · ISO 27001 A.5.16
2
Assign a named human owner to every non-human identity
Every non-human identity has an accountable named human owner; "IT" or "DevOps" generically does not count.

The most common non-human identity problem is unaccountability. "I think Jane set it up but Jane left in 2022" is the orphaned state that defeats every other control.

  • Owner is a named individual who can be contacted, who can attest to ongoing need, who is responsible for credential rotation. Backup owner (second-person) required for production identities to handle primary-owner absence.
  • Forbidden owner values: "IT," "DevOps," "Platform Team," "Cloud Operations," shared mailboxes, distribution lists, no value at all. These all mean "no one."
  • Team-level ownership permitted with caveat: a named team (with documented current membership) where the team is small enough that members all know about the identity. Larger teams must designate a named primary owner.
  • Owner confirmation on creation: identity creation workflow captures the owner; owner receives notification and explicitly confirms.
  • Owner review annually: per IT-ID-WI-02 quarterly cadence for non-human owner attestation.
  • Owner change on leaver: per IT-ID-WI-01 Step 4 — departing owner's identities transfer to their line manager pending re-assignment within 30 days.
  • Orphan remediation: identities with no named owner identified or with departed owners; investigated for true ownership; if none can be established, decommissioned per Step 8.
Verify. Sample 10 random non-human identities; for each, owner is a named individual (not generic), individual is currently with the organisation, individual confirms ownership when asked. Orphan count tracked and trending down.
NIST 800-53 AC-2 · ISO 27001 A.5.16
3
Document scope and justification at creation
Every non-human identity has a documented reason for existing, what it can access, and why.
  • Per-identity justification record: purpose statement (what the identity does), permissions / scope (what it can access), business requestor (which team or project needed it), creation justification (why it couldn't be done another way — e.g. without a managed identity, without a service principal). Captured at creation, not added retrospectively.
  • Scope discipline: same least-privilege principles as IT-ID-WI-03 — permissions match function; scope at lowest meaningful level; reader variant where applicable; resource-level rather than subscription-level wherever possible.
  • "Why not managed identity" question at service principal creation: managed identities are preferred over service principals in Azure because there's no secret to manage. Creation workflow asks the question; documented answer required.
  • "Why interactive sign-in capability" question at AD service account creation: modern designs avoid interactive sign-in for service accounts; documented answer required if the capability is needed.
  • Permission justification per OAuth scope (service principals): each delegated / application permission has a documented business reason. "Sites.ReadWrite.All" needs more justification than "User.Read."
  • Quarterly scope review: per IT-ID-WI-02; owners attest to ongoing necessity of granted scopes.
Verify. Recent service principal creations have documented justification including "why not managed identity" answer. Sample 5 long-lived service accounts (created >1 year ago); justification is present and current.
NIST 800-53 AC-6 · CIS Controls 6.8 · ISO 27001 A.5.18
4
Credential rotation discipline
Credentials for non-human identities rotate on a documented schedule; expired credentials are replaced before expiry, not after.

Credential rotation is where many non-human governance programmes fall over. The platform allows long-lived secrets; the path of least resistance is to set "expires never" or "5 years"; rotation becomes a future problem that is also a current problem.

Per-credential-type targets

Identity / credential typeRotation targetMechanism
gMSA passwordAutomatic (default 30 days)AD-managed; no manual rotation needed
AD service account (user-style)Annual minimum; 60 days target where supportedManual or scripted; coordinated with consuming application
Service principal secretMax 1-year expiry; rotate at ≤30 days before expiryGenerated in Entra; stored in Key Vault; consuming application re-reads from vault
Managed identityN/A — no credential to rotateToken issuance is platform-managed
Workload identity federationToken is short-lived (minutes/hours)Token issuance per request; no long-lived credential
Identity-bearing certificatesPer certificate policy (typically 1–2 years)Automated via PKI / certificate vault; pre-expiry alerts
  • Rotation evidence retained: when rotated, by whom (or "automatic"), credential ID old / new.
  • Pre-expiry alerts at 30 days, 14 days, 7 days, 1 day — to owner and IAE; tracked to resolution.
  • Post-expiry use detection: authentication attempts using an expired credential (typically platform rejects but logs); SOC alert.
  • Emergency rotation path: documented for compromise scenarios; coordinate with SOC IR.
  • Forbidden patterns: passwords / secrets in source code, in container images, in config files committed to repos. Scanning tools (GitHub secret scanning, equivalent) deployed; findings remediated.
Verify. Service principal secret expiry dashboard available; secrets >1-year expiry: zero or named exception. Recent rotation evidence sampled (3 identities); all have audit trail. Post-expiry use detection firing.

5
Prefer managed identities & workload federation over secrets
New non-human identity needs are met with secret-less patterns where the workload supports them.

The simplest credential is no credential. Managed identities and workload identity federation are the modern patterns.

  • Managed identities (Azure) for Azure-resident workloads — the workload requests an access token from the Azure platform; no client secret to rotate, no secret to leak, no expiry to track.
  • System-assigned vs user-assigned:
    • System-assigned for single-resource workloads where the identity lifecycle should match the resource.
    • User-assigned for workloads that span multiple resources, that need to survive a resource recreate, or where the identity is shared (with care) across multiple resources.
  • Workload identity federation for non-Azure workloads (GitHub Actions, on-prem CI/CD, AWS roles assuming into Azure): federated credentials replace long-lived service principal secrets with short-lived tokens.
  • Migration from secrets to secret-less: existing service principals with secrets reviewed for managed-identity / federation migration; planned and tracked.
  • Where managed identity / federation don't fit: documented in the justification (Step 3); typical cases — SaaS application that doesn't support Azure managed identity, legacy on-prem system without federation capability, vendor product with vendor-controlled secret management.
  • Conditional Access for workload identities (Entra ID P2 / Workload Identities Premium): apply CA policies to service principals — location restrictions, risk-based blocking. Especially valuable for high-scope service principals.
Verify. Recent service principal creations: ratio of managed-identity to secret-bearing service principal trending toward managed-identity. Migration backlog visible. Conditional Access for workload identities applied where licensed.
NIST 800-53 IA-9 · platform-modern pattern
6
Secret expiry tracking dashboard
Upcoming credential expiries are visible to owners and IAE; secrets do not expire by surprise.
  • Centralised dashboard: all credentials with their expiry dates and owner; built from the inventory (Step 1) and key vault telemetry.
  • Sortable / filterable: by expiry date (urgent first), by owner (per-team views), by scope (high-impact first).
  • Pre-expiry alerts automated: 30 days, 14 days, 7 days, 1 day; sent to named owner and IAE.
  • Post-expiry alerts: failed authentication using expired credential; SIEM rule; SOC notification.
  • Ownership of stale credentials: credentials still active past target rotation date are tracked; the owner is on the hook to rotate or document exception; failure to act escalates to owner's manager.
  • Quarterly trend report: how many credentials currently within 30 days of expiry, how many overdue, how many are managed identities (rotation N/A). Reported to GRC quarterly.
Verify. Dashboard accessible to IAE and named owners; alerts firing at the expected pre-expiry intervals; sample alert from last 30 days resulted in rotation. Quarterly trend report exists.
NIST 800-53 IA-5(1) · ISO 27001 A.5.17 · operational hygiene
7
OAuth permission & consent grant review
Entra service principal permissions and tenant-wide consent grants are reviewed quarterly; over-permissive scopes are reduced.

OAuth permissions are how third-party apps and internal apps interact with tenant data. They accumulate. Quarterly review prevents the accumulation from becoming a breach.

  • Tenant-wide admin consents: enumerate all admin-consented permissions across all service principals. Each consent has a record (who granted, when, why); old consents whose business reason is unclear are candidates for revocation.
  • User-consented permissions: per IT-CL-WI-02, user consent should be restricted to low-risk scopes only; verify the policy is enforced.
  • High-risk permissions reviewed first: Sites.ReadWrite.All, Mail.ReadWrite, Files.ReadWrite.All, Directory.ReadWrite.All, Application.ReadWrite.All, RoleManagement.ReadWrite.Directory. Application-permission variants of these are particularly high-impact (no user delegated context).
  • Permission-scope reduction: where a service principal has read-write permissions but only needs read, the application owner is engaged to scope down; tracked as a remediation item.
  • Sign-in activity per service principal reviewed: identities not used in 90 days flagged for decommissioning (Step 8); identities used but with significantly fewer scopes than they have are scope-reduction candidates.
  • Anomalous OAuth activity alerts: new consents granted by users with privileged roles, consents granted outside business hours, consent grants for newly-registered applications; SOC investigates.
Verify. Most recent quarterly OAuth review evidence; high-risk-permission service principals justified; scope-reduction backlog visible; anomalous-consent SIEM rule active.
CIS M365 1.x · ISO 27001 A.5.16 · MITRE ATT&CK T1098.003
8
Decommissioning unused or orphaned identities
Non-human identities not used in defined intervals are flagged, disabled, then deleted; orphan identities are remediated.

Stale-identity lifecycle

StatusThresholdAction
ActiveUsed within last 90 daysStandard governance applies (Steps 2–7)
StaleUnused 90–180 daysFlagged; owner consulted; legitimate ongoing need attested or moved to disabled
DisabledUnused >180 days, owner-confirmed inactive, or owner unresponsiveIdentity disabled; retained for 6 months for restoration if newly needed
DecommissionedUnused >365 days from disabled stateIdentity deleted; secret / certificate revoked; permissions removed from target systems
  • Owner approval at each transition: not silent. Each transition records owner consultation result.
  • Test-disable before decommission: disabling a "stale" service account or service principal sometimes breaks an unknown consumer (a forgotten scheduled task, a vendor integration). Disable first, observe for breakage, then proceed to delete.
  • Orphan identities (no owner can be identified): special pathway — controlled-disable test (Step 8 disable mechanic); observe for 30 days; if nothing breaks, decommission with named-approver sign-off.
  • System-assigned managed identities: lifecycle bound to the parent resource; deletion of the resource removes the identity. Orphaned system-assigned identities indicate a deleted resource where the identity wasn't cleaned up — investigate.
  • Decommission evidence retained: who approved, when disabled, when deleted, what was the test-disable observation.
Verify. Stale-identity report current; decommission backlog visible; owners engaging at lifecycle transitions; test-disable observations recorded; recent deletions have audit trail.
NIST 800-53 AC-2(3) · CIS Controls 5.3 · ISO 27001 A.5.16
9
Flag interactive sign-in capability on service accounts
Service accounts that can sign in interactively are reviewed for legitimate need; the modern default is no interactive capability.
  • Detection: AD service accounts not denied Logon Locally / Logon through Remote Desktop Services; Entra service principals where a user has signed in as the principal interactively (rare but possible with certain configurations).
  • Legitimate cases: legacy applications that require interactive sign-in for service accounts (LDAP-only middleware, vendor appliances with old credential models); documented as platform-debt with remediation roadmap.
  • Remediation default: deny interactive sign-in rights for service accounts where the consuming application doesn't require it; documented exception path for those that do.
  • Coordinated with IT-SRV-WI-DC-01 for Tier-0 enforcement: Tier-0 service accounts must have no interactive sign-in capability.
  • Monitoring: SIEM alert on interactive sign-in by service account where deny rule should apply — misconfiguration or compromise.
Verify. Service account interactive-sign-in audit; legitimate exceptions documented; SIEM alert active for unexpected interactive sign-ins.
CIS Controls 5.4 · MITRE ATT&CK T1078.002
10
Privileged non-human identities follow the eligibility pattern
Service principals and service accounts with Tier-0 / Tier-1 scope are treated with the same eligibility / activation discipline as privileged human identities, where the platform supports it.
  • Tier classification applied to non-human identities: a service principal with Global Administrator-equivalent scope is Tier-0; a service principal that can read all SharePoint sites is Tier-1.
  • Eligibility where supported: Entra PIM for Groups (with Entra Workload Identities Premium) enables PIM-style eligibility for service principals via group membership. Apply to high-tier non-human identities where licensed.
  • Standing high-scope assignments minimised: high-tier service principals reviewed quarterly (per IT-ID-WI-02); standing assignment kept only where workload doesn't support activation pattern.
  • Coordinated with IT-ID-WI-05: the privileged eligibility governance there extends to non-human identities; non-human privileged inventory feeds the privileged inventory.
  • Activity logging intensified: high-tier non-human identity actions logged to immutable destination; quarterly review of activity patterns.
Verify. Tier classification applied to non-human identities; high-tier identities under enhanced review; activity logging in place; eligibility pattern applied where licensed.
NIST 800-53 AC-6(7) · ISO 27001 A.8.2
Validation & verification

Validate the non-human identity governance programme.

Inventory & ownership

  • Inventory across all platforms current; counts plausible
  • Named owner populated for ≥95% of non-human identities (target 100% by end FY26)
  • Orphan identity count trending down; aged orphans (>30 days) in active remediation
  • Per-identity scope justification documented; recent creations have justification at creation time

Credential hygiene

  • Service principal secrets >1-year expiry: zero or named exception
  • AD service account passwords rotated within annual target; gMSA in use where applicable
  • Pre-expiry alerts firing; recent rotation evidence available
  • Post-expiry use detection active; no expired-credential usage in last 30 days
  • Secrets-in-code scanning deployed; findings remediated within SLA

Pattern modernisation

  • Managed identities preferred for new Azure workloads
  • Workload identity federation applied where workload pattern fits (GitHub Actions, external CI/CD)
  • Migration backlog of legacy service principals visible
  • Conditional Access for workload identities applied (where licensed)

Lifecycle

  • Stale-identity report current
  • Decommission backlog visible; test-disable observations recorded
  • Interactive sign-in audit current; exceptions documented
  • Quarterly OAuth permission review evidence available; high-risk scopes justified
  • High-tier non-human identities under enhanced governance (eligibility pattern where supported)
Rollback & troubleshooting

Non-human identity changes are operationally sensitive — a misjudged disable / decommission can break unknown consumers.

Common scenarios

  • Disabled service account broke a critical scheduled task — cause: stale-identity assessment was incorrect; the account was used by a quarterly job, not visible in 90-day window. Recovery: re-enable; coordinate with consuming team; revise stale threshold for the identity (some genuinely have long usage cycles).
  • Service principal secret rotation broke a vendor integration — cause: vendor didn't pick up the new secret from key vault (caching, configuration). Recovery: vendor coordinates secret update on their side; consider longer overlap period for next rotation; document vendor-specific rotation procedure.
  • Managed identity migration broke a workload — cause: migration changed the identity used by the application; downstream permissions hadn't been granted to the new identity. Recovery: grant required permissions to managed identity; verify; complete the migration. Document the dependency map for future migrations.
  • Decommissioned identity needed for a recovery / audit — cause: identity deleted before downstream needs were fully understood. Recovery: limited — deleted identities cannot be undeleted; restore from inventory documentation (purpose, scope) and re-create with new ID; address the gap with affected stakeholders.
  • OAuth permission revoked broke a third-party app — cause: app actually was using the permission; the quarterly review judged it unused based on incomplete telemetry. Recovery: re-grant with documented justification; investigate the telemetry gap.
  • Owner attested to ongoing need but identity remained stale — cause: identity is technically owned but not actively used; "we might need it" syndrome. Recovery: time-bound the attestation ("active for 6 months"); revisit if still unused at next cycle; tighten the attestation requirement to evidence of recent use.

Rollback procedure

  1. Identify the affected identity and the breaking change
  2. Re-enable / restore credentials / re-grant permissions as needed
  3. Communicate with affected owners and consumers
  4. Update inventory record with the lesson learned
  5. Adjust the relevant threshold (stale window, rotation overlap, decommission criteria) if the failure was systemic
Common errors & tips
  • Inventory is the gate. Every other control depends on knowing what exists. An incomplete inventory means undefended identities. Build the inventory first, refresh it nightly, accept that the first cycles will be discovery-heavy.
  • Ownership is the hardest part. The inventory is technically straightforward; finding named human owners for 5,000 service accounts created over 15 years is the long, painful work. Don't shortcut it. "Owner unknown" defeats every other control downstream.
  • Managed identities are a strict upgrade where they fit. No secret to rotate, no expiry to track, lifecycle tied to the resource. Default to them; deviate with justification.
  • Service principals with "expires never" are technical debt. Some platforms let you create them; doing so means the secret will eventually be a problem. Set the maximum permitted expiry; rotate before expiry; treat "never" as forbidden.
  • Quarterly OAuth review is the consent-grant safety net. Users grant consent to apps over time; admins grant tenant-wide consent over time; without review, the grants accumulate. The quarterly cycle is the deliberate housekeeping.
  • Tenant-wide admin consents are the high-leverage targets. User-consented permissions affect one user's data; admin-consented application permissions affect all tenant data. Prioritise review accordingly.
  • Interactive sign-in by service account = compromise indicator. Modern service accounts don't sign in interactively. When one does, either the configuration is wrong or someone is using the account in ways it shouldn't be used. Detect, investigate.
  • Test-disable before decommission catches surprises cheaply. 30 days of "disabled but recoverable" surfaces consumers you didn't know about. Skip this step and you'll find them via outages instead.
  • Privileged non-human identities are the highest-impact attack target. A compromised user account is bad; a compromised Global Admin service principal can rewrite tenant policy. Treat them with the same rigour as Tier-0 human accounts.
  • Workload identity federation eliminates a whole class of credential leakage. Short-lived tokens issued per-request mean no secret to leak. Adopt aggressively for CI/CD, external workloads.
  • "DevOps owns it" means no one owns it. Team-level ownership is acceptable only when the team is small enough that members actually know about each identity. Otherwise it's an unfalsifiable claim and the identity is effectively orphaned.
  • Service accounts owned by leavers are unexploded ordnance. Per IT-ID-WI-01 Step 4, leaver process re-assigns. But verify it actually happened — this is the most common reassignment-miss.
