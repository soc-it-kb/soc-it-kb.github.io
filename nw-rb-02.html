NW-RB-02: Block IP / Domain / Network Indicator

Controlled, auditable procedure to block malicious or high-risk network indicators — stopping C2 communication, preventing data exfiltration, and containing active network-borne threats.

🌐 Network threats Tier 2 / 3 Network containment

Section 1
Purpose

Provide a controlled, auditable procedure to block malicious or high-risk network indicators — immediately reducing risk by stopping C2 communication, preventing data exfiltration, blocking attacker infrastructure, and containing active network-borne threats.

This runbook executes network containment decisions made in NW-PB-01 through NW-PB-05.

Section 2
Required Access
🔥 Firewall management (read/write or request-based) 🌐 Proxy / Secure Web Gateway management 🔍 DNS filtering platform (if applicable) 📡 NDR visibility (for validation) 🎫 SOC ticketing system

If direct write access is not permitted, this runbook defines the standard escalation and change request workflow.

Section 3
Inputs Required
🔢 Indicator type (IP / Domain / URL / ASN) 📌 Indicator value(s) 🔍 Source of intelligence 🎫 Incident reference ID ⏱️ Blocking scope (temporary / permanent)
Section 4
Step-by-Step Procedure
1
Validate Indicator Accuracy
1
Confirm the indicator format is correct (valid IP, FQDN, or URL)
2
Validate the indicator was observed in at least one of:
  • Firewall or proxy logs
  • NDR alerts
  • MDR escalation or TI feed
3
Confirm the indicator is actively malicious or high-risk

🚫 Do not block indicators with: unverified single hits · known shared cloud or CDN infrastructure · legitimate business services (unless explicitly approved)

2
Determine Block Scope & Method
Select the appropriate control based on indicator type:
Indicator typePrimary control
IP addressFirewall / NDR
DomainDNS filtering / Proxy
URLProxy / Secure Web Gateway
ASNFirewall (exceptional cases only)
1
Determine the blocking scope — perimeter only, internal egress, or specific network segments
3
Assess Business Impact Risk
1
Before blocking, assess the following:
  • Is the destination used by any business applications?
  • Is the IP or domain shared hosting infrastructure?
  • Is this a temporary containment or a permanent block?
2
If potential business impact exists — coordinate with Network / IT and apply a temporary or scoped block first
Block duration options

⏱️ Temporary

Incident-scoped — review and remove after incident closure

🔒 Permanent

TI-validated indicator — reviewed and added to long-term block list

4
Implement Block
1
Add the indicator to the appropriate platform:
  • Firewall deny list (IP / ASN)
  • Proxy block list (URL / Domain)
  • DNS sinkhole (Domain)
2
Record the following at time of implementation:
  • Platform used
  • Rule name or ID
  • Exact timestamp
  • Operator who applied the block
5
Validate Block Effectiveness
1
Confirm the rule is active and propagated across relevant platforms
2
Monitor for:
  • No further successful connections to the blocked indicator
  • Block events being logged in firewall / proxy / DNS
3
If traffic persists — re-evaluate indicator accuracy and check for alternate protocols or rotated destinations
6
Correlate Post-Block Behaviour
1
Monitor affected endpoints for:
  • Connection retries to the blocked indicator
  • Failover to secondary C2 destinations
  • New or secondary indicators emerging
2
Escalate if new C2 indicators appear or endpoint behaviour changes significantly after blocking
7
Document Actions
1
Record in the SOC ticket:
  • Indicator(s) blocked and type
  • Blocking location and scope
  • Validation outcome
2
Attach screenshots, rule IDs, and log excerpts to the ticket
Section 5
Expected Outcome

✅ Malicious network communication stopped
✅ Attacker control or data loss risk reduced
✅ Clear audit trail for all containment actions

Section 6
Validation Checks
  • Block rule active and enforced across all relevant platforms
  • No successful connections to the blocked indicator post-implementation
  • Endpoint behaviour stabilised or escalated appropriately
Section 7
Common Errors & Tips
Blocking shared cloud IPs (AWS, Azure, Cloudflare) can cause widespread outages — always prefer domain-level blocking where possible
Domain blocking is generally safer and more precise than IP blocking — use IP blocking only when the IP is attacker-controlled infrastructure
Attackers rotate infrastructure quickly — after blocking, actively monitor for failover destinations and secondary indicators
Always pair blocking with endpoint investigation — blocking network traffic does not remove malware already present on the host
Section 8
Output Artefacts
Block confirmation
Rule or policy reference
Validation logs
Updated incident timeline
