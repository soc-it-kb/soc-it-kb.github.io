IT-NET-WI-05: Firewall Hardening

Vendor-agnostic implementation guidance for hardening firewalls: default-deny, rule hygiene, egress filtering, threat prevention, TLS inspection, HA. Operationalises §4.5 of IT-NET-SOP-01.

Prerequisites

Before applying this WI:

  • Management plane is hardened per IT-NET-WI-01
  • Firewall topology is documented: which interfaces face which trust zones, where the perimeter is, where internal segmentation sits
  • An existing rule base, if any, is exported and available for analysis (this WI's rule-hygiene steps require knowing what currently exists)
  • Documented business needs for permitted flows are available (knowing what should pass is the foundation of default-deny)
  • HA pair configuration is documented if this firewall is part of one
  • Logging destination (SIEM ingest) is reachable from the firewall management interface
  • Change window approved per IT-NET-WI-06
Firewall rule changes are blast-radius decisions. A single misordered rule can drop all production traffic. Test in monitor/learning mode before enforcing where the platform supports it; have a documented immediate-rollback plan.
Step-by-step instructions

The steps below cover the firewall-specific controls required by IT-NET-SOP-01 §4.5.

1
Establish default-deny posture
Anything not explicitly permitted is denied; this is the foundational firewall rule.

For each interface or trust zone:

  • Confirm the implicit-deny at the end of every rule set (this is default in most enterprise firewalls but worth verifying)
  • Where the platform allows an explicit deny-all rule with logging, configure it — this gives visibility into denied traffic
  • Identify and remove any "permit ip any any" rules at perimeter or between trust zones; these are baseline violations regardless of vendor
  • For internal-segmentation firewalls, this means inter-segment is default-deny — flat L3 between trust zones is prohibited

Where a "permit any" rule exists today and removing it without preparation would break production, the path is: enable explicit-rule logging, observe what currently flows for 7-14 days, build explicit-permit rules for the legitimate flows, then remove the permissive rule. Don't just delete it cold.

Verify. Show rule base; no any-any allow rules at the end. Capture traffic that should be denied; confirm the firewall drops it and logs the denial.
CIS Firewall: "default-deny"
2
Rule documentation and ownership
Every rule has a recorded purpose, owner, creation date, expiry/review date.

For every rule in the rule base:

  • Description / comment field populated with: business purpose, ticket / change reference, owner (named individual or team), creation date
  • Expiry / review date — either a hard expiry (rule auto-disables) or a scheduled review (rule flagged for confirmation)
  • Naming convention for rules and objects so meaning is conveyed by the rule name (e.g. "ALLOW-HR-APP-TO-SQL-PROD")
  • For long-lived "infrastructure" rules (e.g. DNS, NTP, SIEM forwarding), tag clearly and apply a "perpetual" classification with annual review rather than expiry
Verify. Show rule base; spot-check rules — each should have description, owner, dates. Run an export and audit for rules missing these fields.
CIS Firewall: "rule documentation"
3
Egress filtering on internet-facing interfaces
Outbound from the organisation is filtered to expected destinations and protocols.

On internet-egress interfaces:

  • No "any-any-outbound" rules; egress is permitted only to documented destinations and protocols
  • For general user web browsing, egress to TCP/80, TCP/443, DNS to authorised resolvers, NTP to authorised time sources, TCP/22 only from documented admin networks
  • Block outbound TCP/25, TCP/465, TCP/587 except from authorised mail servers (anti-spambot)
  • Block outbound SMB, RDP, SSH from user networks except via explicit allow (lateral exfil mitigation)
  • For servers, more restrictive egress: only what the server's business function requires
  • For IoT and OT segments, egress is to documented vendor endpoints only

This is one of the most effective baselines for limiting damage from compromised internal hosts. Many incident-response cases involve compromised endpoints that would have been contained earlier if egress filtering existed.

Verify. From a test client on a user network, attempt outbound SMB to an internet IP; should be denied. Attempt outbound HTTPS; should succeed. Confirm denials are logged.
CIS Firewall: "egress filtering"
4
Rule lifecycle: stale-rule detection and removal
Rules with no traffic accumulate as attack surface; review and remove them.

Implement rule lifecycle review:

  • Use rule-hit-counter capability (most enterprise firewalls have this) to identify rules with no recent traffic
  • Rules with no traffic for 90 days are flagged for review by their owner
  • Rules with no traffic for 180 days are removed unless explicit justification is provided (e.g. DR-only path)
  • Rules that have never had traffic since creation are flagged immediately for review — often these are mistakes
  • For HA pairs, ensure the hit-counter view aggregates both members

This is normally a quarterly rather than continuous process; the deliverable is a stale-rule report acted on by Network Engineering.

Verify. Generate the stale-rule report; confirm it lists candidates. Confirm the review-and-remove workflow is documented and assigned.
CIS Firewall: "rule lifecycle"
5
Geo-blocking where business case supports it
Reduce attack surface by blocking traffic from geographies with no legitimate business case.

Configure geo-blocking carefully:

  • Identify geographies where the organisation has no users, no offices, no partners, no customers — these are candidates for blocking inbound
  • Inbound: deny connections sourced from those geographies (where firewall supports geo-IP feeds)
  • Outbound: more selective — some users legitimately travel; consider blocking only the most malicious geographies outbound
  • Document the geo-block policy with business sign-off so future users in those geographies know to request exceptions
  • Combine with anomalous-source detection: a user account suddenly authenticating from a never-seen geography is a higher-fidelity signal

For NGOs and humanitarian organisations, geo-blocking decisions are particularly sensitive because legitimate beneficiary and partner traffic may come from many places. Apply judgement.

Verify. Test inbound from a VPN exit in a blocked geography; should be denied. Review denied-traffic logs for the past month; assess any false positives.
CIS Firewall: "geo-IP blocking"
6
IPS / IDS / antivirus engine where licensed
Layered detection and prevention beyond Layer-3/4 filtering.

Where the firewall licence includes threat-prevention engines:

  • Enable IPS in prevention mode for high-confidence signatures; alert mode for lower-confidence
  • Subscribe to vendor signature updates; verify update frequency (daily for high-confidence signatures is typical)
  • Enable antivirus scanning on flows where TLS inspection is in place (otherwise AV cannot see the content)
  • Enable URL filtering / category blocking for known-malicious categories (malware C2, exploit kits, phishing)
  • Tune the IPS to suppress false positives common in the organisation's traffic profile (every environment has some)
  • Forward IPS / AV detections to SIEM with sufficient context (source IP, destination, signature, severity)
Verify. Trigger a known IPS signature with an EICAR-equivalent test; confirm detection. Confirm signatures are current. Confirm detections flow to SIEM.
CIS Firewall: "threat prevention engines"
7
TLS inspection policy
Visibility into encrypted traffic balanced against privacy, performance, and supportability.

TLS inspection is a policy decision, not just a configuration:

  • Identify categories where TLS inspection is on (broad: most general categories), partially on (banking, healthcare excluded for compliance), or off (specific categories per policy)
  • Where TLS inspection is in place, the firewall presents an organisation-issued certificate to the client; client trust of the firewall's CA must be in place
  • Where TLS inspection is not in place, log SNI / certificate visibility metadata so encrypted traffic is at least catalogued
  • Pinned-cert applications (some mobile apps, some banking apps) will break with TLS inspection; exclude them by destination
  • Document the policy clearly; communicate to users where inspection occurs — this is a privacy-relevant control
Verify. From an inspected category, browse a test site; confirm the certificate presented is the organisation's. From an excluded category, confirm the original certificate is preserved. Confirm SNI logging where inspection is off.
CIS Firewall: "TLS inspection"
8
Application-layer (Layer-7) controls
Filter by application identity, not just port/protocol.

Where the firewall supports application identification:

  • Use application-aware rules where possible: "allow Microsoft 365" rather than "allow TCP/443 to *.microsoft.com"
  • Block applications by category where business case doesn't support them (P2P sharing, anonymisers, evasion tools)
  • Combine application identification with user/group identity (where SAML/AD integration is in place) for "Finance users can use SAP, HR users can use Workday" style rules
  • Be aware: application signatures can be evaded; this is a defence layer, not a bulletproof control
Verify. Identify a few flows in logs; confirm application classification is correct. Test blocked applications from a user network; confirm denial and logging.
CIS Firewall: "application-layer controls"
9
Anti-spoofing, fragmentation, reassembly
Hardening against malformed-traffic evasion.

Platform-specific protections to enable:

  • Anti-spoofing per interface: traffic with source addresses that shouldn't appear on that interface is dropped
  • IP fragmentation defence: tiny fragments, overlap-fragment attacks are dropped or reassembled before inspection
  • TCP-state validation: out-of-order, flag-anomaly, window-anomaly handling per vendor recommendations
  • DNS rebinding protection where the firewall has DNS-aware features
  • ICMP rate-limiting on internet-facing interfaces
Verify. Show interface anti-spoofing configuration. Run vendor-provided self-test if available. Inject a known-bad traffic pattern (lab); confirm drop.
CIS Firewall: "evasion protections"
10
High availability for Tier 0/1 firewalls
Single-firewall outages don't cause site-level outages.

For Tier 0 and Tier 1 firewalls:

  • HA pair configured (active-standby or active-active per vendor capabilities and design)
  • HA link is on a dedicated interface, not shared with data-plane traffic
  • Configuration sync verified working — changes on the primary propagate
  • Failover behaviour documented: convergence time, session preservation, traffic loss
  • Annual failover test under controlled conditions, results logged

For Tier 2 and Tier 3 firewalls, HA is desirable but not always cost-justified; the asset record documents the tier decision and acceptance criteria.

Verify. Trigger failover (planned, controlled); confirm traffic continues, sessions preserved where supported by vendor, convergence time matches expected. Confirm both pair members show "in sync" status before declaring complete.
CIS Firewall: "high availability"
Validation & verification

After applying all steps, validate the firewall holistically. Use a representative subset of traffic flows, not just synthetic tests.

Functional validation

  • Default-deny is in effect; unauthorised traffic is dropped and logged
  • Documented permitted flows pass without issue
  • Egress filtering blocks lateral-exfil protocols (SMB, RDP, SSH from user networks)
  • Stale-rule report identifies candidates for removal
  • Geo-blocked sources are denied
  • IPS / AV detections flow to SIEM
  • TLS inspection (where active) presents organisation cert; excluded categories preserve original cert
  • HA failover works under controlled conditions

Rollback & troubleshooting

Firewall rule changes can cause site-wide outages. Rollback approach must be prepared in advance.

Common scenarios

  • Mass connectivity outage after default-deny removed permissive rule — cause: a permit-all was implicitly carrying legitimate traffic that wasn't enumerated in specific rules. Fix: revert; observe traffic; build explicit rules; re-attempt.
  • Specific application broken after rule cleanup — cause: a "stale" rule was actually used by an infrequent flow (monthly batch, quarterly reporting). Fix: identify the flow from app team, restore the specific rule with proper documentation.
  • HTTPS sites won't load after TLS inspection enabled — cause: firewall CA certificate not in client trust store. Fix: deploy CA via MDM/GPO; until then exclude critical flows.
  • IPS prevention dropping legitimate application — cause: false-positive signature triggers on legitimate traffic pattern. Fix: switch the specific signature to alert mode; tune the policy; engage vendor for signature update if commonly affected.
  • Geo-block dropping legitimate user from new country — cause: user travelled, or partner came online in a new geography. Fix: add a specific exception via firewall identity-based rule, not a blanket geo-unblock.
  • HA failover doesn't complete — cause: HA-link issue, config-sync drift, vendor bug. Fix: investigate HA link health, force a re-sync, vendor TAC engagement.

Rollback procedure

  1. Have the pre-change config exported and reachable from the management station
  2. For point fixes (one bad rule), revert that specific change rather than the entire change set
  3. For broad outage caused by policy switch, revert the policy; investigate; resubmit with fixes
  4. Document the rollback action including what was reverted and why; post-incident review to identify why testing didn't catch it
Common errors & tips
  • Don't delete "permit any" rules cold. Enable logging, observe, build explicit rules, then remove. The right answer is rarely "just delete the permissive rule and see what breaks".
  • Rule descriptions are not optional. A rule base where you can't tell what each rule does is a rule base nobody can maintain. Six months later, "ALLOW-TEMP" rules become permanent.
  • Egress filtering is the single best lateral-movement control for endpoint compromise. Even imperfect egress filtering forces attackers to use noisier C2 channels.
  • Stale-rule reports need an owner. Generating reports nobody acts on is worse than not generating them — it creates the appearance of process.
  • Geo-blocking must consider humanitarian context. Beneficiaries and partners in sensitive geographies may be exactly the people you exist to serve. Apply judgement; don't blanket-block on threat-intel feeds alone.
  • TLS inspection is privacy-relevant. Some jurisdictions and employment contexts require disclosure that traffic is inspected; coordinate with legal/HR.
  • IPS in "alert" mode forever is theatre. The reason for buying IPS was to prevent; if everything is alert-only, the investment is undermined. Move signatures to prevention as confidence grows.
  • HA tested in production is not a test. Use maintenance windows; have a backout plan; coordinate with operations.
