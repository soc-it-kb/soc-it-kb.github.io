NW-PB-03: Internal Scanning & Reconnaissance Activity

Standardised framework for detecting, investigating, and responding to internal scanning and reconnaissance — where an attacker maps the internal environment to identify systems, services, and lateral movement paths.

Section 1
Purpose

Provide a standardised decision and response framework for detecting, investigating, and responding to internal scanning and reconnaissance activity, where an attacker maps the internal environment to identify systems, services, users, and paths for lateral movement.

This playbook focuses on early attacker detection — reconnaissance typically precedes lateral movement, privilege escalation, and ransomware deployment.

Section 2
Scope

✅ In scope

  • Internal port scanning or service enumeration
  • Host discovery (ICMP sweeps, SMB enumeration, LDAP queries)
  • Abnormal east-west traffic patterns
  • Reconnaissance detected by firewalls, NDR, EDR, or identity logs
  • MDR escalations for suspected enumeration
Common recon commands
net view arp -a nslookup nmap nltest ldapsearch ping sweep net user /domain

❌ Out of scope

  • External scanning against perimeter → NW-PB-01
  • Confirmed lateral movement with execution → EP-PB-03
  • Legitimate vulnerability scanning (documented tooling)
Section 3
Trigger Conditions
  • NDR flags internal scanning behaviour
  • Firewall detects abnormal east-west connection attempts
  • EDR detects discovery commands or enumeration tools on an endpoint
  • MDR escalates suspected reconnaissance activity
  • SOC identifies repeated connection attempts across multiple internal hosts
Section 4
Initial Validation (Decision Logic)

Validate whether the activity is authorised before escalating:

  • Is the source a known vulnerability scanner or IT management system?
  • Is the activity scheduled or documented in a change record?
  • Does it originate from known admin tooling or approved service accounts?
  • Is the volume and pattern consistent with normal operations?

If YES — legitimate confirmed → Document and close as Authorised Scanning
If NO or UNCLEAR → Proceed with reconnaissance assessment (Section 5)

Section 5
Reconnaissance Assessment

Assess intent and confidence. Presence of any indicator = Suspected Internal Reconnaissance:

Multiple hosts probed from a single source in a short timeframe
Sequential port or service enumeration pattern
Discovery commands observed on the source endpoint (net view, arp, nslookup, LDAP)
SMB, RPC, or LDAP probing of multiple systems
Recon activity follows a phishing event or endpoint compromise
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Recon originates from a confirmed or suspected compromised endpoint
🔴 Recon targets domain controllers, servers, or sensitive systems
🔴 Multiple network segments have been probed
🔴 Recon follows confirmed malware execution or C2 activity
🔴 MDR confirms attacker-driven enumeration

Notify immediately

Section 7
Response Strategy

The following actions must be orchestrated based on assessment findings:

Source endpoint identificationRequired
Recon traffic analysis — tools, commands, targetsRequired
Endpoint compromise assessmentRequired
Containment decision — isolate or block sourceRequired
Scope expansion — identify all probed systemsRequired
Identity exposure assessmentConditional

If recon is confirmed malicious → escalate immediately to EP-PB-03 (Lateral Movement) — the attacker may already be progressing

Section 9
Containment Decision Logic

Choose the appropriate containment approach — prioritise source control to prevent progression:

🔴 Immediate isolation

  • Recon originates from a user workstation
  • Recon follows a confirmed endpoint compromise
  • High-value assets (DCs, servers) are being targeted

🟡 Targeted blocking

  • Source is a shared server or critical system
  • Full isolation would cause significant business disruption
Section 10
Scope Expansion Logic

Scope expansion is mandatory for all NW-PB-03 incidents:

Identification actions

  • Identify all hosts probed by the source endpoint
  • Search for similar scanning behaviour across the environment
  • Identify any successful access attempts following recon

Correlation actions

  • Correlate recon timing with endpoint compromise events
  • Correlate with identity anomalies (ID-PB-01 / ID-PB-06)
  • Cross-reference with open MDR incidents
Section 11
Communication & Coordination
  • Coordinate containment actions with network and endpoint teams before execution
  • Avoid alerting the source user or system owner prematurely
  • Align actions with MDR investigation timelines
  • Escalate to Incident Command if recon is widespread across multiple segments
Section 12
Closure Criteria

The incident may be closed when all of the following are confirmed:

  • Reconnaissance activity fully stopped at the source
  • Source endpoint validated clean or fully remediated
  • No progression to lateral movement observed
  • All probed assets confirmed uncompromised
  • All actions documented and incident record completed
Section 13
Evidence to Capture
Source and destination IPs
Ports and services probed
Discovery commands or tools observed
Firewall / NDR / EDR alert exports
Correlated endpoint and identity events
Timeline of reconnaissance activity
Section 14
Post-Incident Actions
  • Review east-west network segmentation and improve where gaps exist
  • Limit unnecessary service exposure across internal network
  • Harden endpoint discovery controls and restrict enumeration tools
  • Tune NDR and EDR reconnaissance detection rules
  • Feed indicators into threat hunting use cases
  • Update internal scanning detection baselines
