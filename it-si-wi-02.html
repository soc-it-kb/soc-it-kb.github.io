IT-SI-WI-02: WAF Hardening

Vendor-agnostic implementation guidance for hardening Web Application Firewalls: rule-set selection, monitor-then-block progression, OWASP Top 10 / CRS coverage, false-positive management, TLS termination, bot management. Operationalises §4.2 of IT-SI-SOP-01.

Prerequisites

Before applying this WI:

  • WI-01 management-plane hardening complete on the WAF
  • Application inventory available: which applications the WAF protects, with each application's owner, criticality tier, public-facing or internal classification, technology stack (helps tune)
  • Application teams identified and committed to triaging false positives within agreed SLA — without this, the WAF either stays in monitor mode or loses operational support
  • The current OWASP Top 10 edition and (if applicable) the current OWASP CRS version are known
  • Baseline traffic profile available for at least one of the protected applications (peak request rate, typical request body sizes, common user-agents) so tuning is grounded in reality
  • Change window approved per WI-01 §10
Without application-team commitment, a WAF never reaches active blocking. The hardest part of WAF deployment is not technical; it is the operational discipline of false-positive triage. Confirm app-team buy-in before deploying in blocking mode; without it, the WAF will live in monitor-only for years and provide no real protection.
Step-by-step instructions

The steps below cover the WAF-specific controls required by IT-SI-SOP-01 §4.2. They assume WI-01 management-plane hardening is in place.

1
Inventory protected applications and assign WAF policy ownership
No application is silently protected or silently unprotected; every WAF policy has an owner.

For each application behind the WAF:

  • Application name, owner (named individual or team), criticality tier (T0/T1/T2/T3), public-facing or internal
  • Hosting model: on-premises, IaaS, PaaS, SaaS frontend
  • Authentication mechanism (SAML, OIDC, basic, none); the WAF needs to treat authentication endpoints differently
  • Known technology stack (e.g. Java/Spring, .NET, Node.js, PHP/WordPress) — informs which signatures matter most
  • Assigned WAF policy ID, current mode (monitor / block), policy version, last review date

This inventory is the master list against which conformance is measured. It should live in (or link to) GRC-REG-ASSET; the WAF policy ID is the cross-reference.

Verify. Pull the application inventory; cross-check against the WAF's list of configured policies. Every application has a policy; every policy has an application. Orphans get investigated.
WAF policy: "inventory and ownership"
2
Select the rule set: OWASP Top 10 coverage + OWASP CRS where supported
Baseline signature coverage that addresses the documented common attack patterns.

For each WAF policy, select rule-set coverage:

  • OWASP Top 10 coverage is the baseline; every WAF policy must cover the current Top 10 categories. Map vendor signature groups to Top 10 categories so coverage is auditable.
  • OWASP CRS (Core Rule Set) where the WAF supports it — ModSecurity-derived signatures usable across many platforms. Deploy at paranoia level 2 minimum after tuning; paranoia 3 for high-sensitivity applications (financial, health, identity). Paranoia 1 is a starting point only — leaving it there is insufficient.
  • Vendor-specific signature engines (F5 ASM signatures, FortiWeb signatures, Cloudflare Managed Rules, Azure WAF Managed Rule Set) — enable on top of OWASP coverage
  • Geo-IP rules as a complement, not as primary defence; never rely on geo-IP alone
  • Application-specific signature groups where the vendor provides them (e.g. "WordPress signatures", "SharePoint signatures") — enable for the relevant applications only

Document the chosen rule set, paranoia level, and exclusions per application policy.

Verify. For each policy, the configured rules include explicit OWASP Top 10 coverage. Run a deliberate test attack pattern (e.g. an obvious SQL injection string against a non-production endpoint of the application); confirm the WAF logs the event (in monitor mode) or blocks it (in block mode).
WAF: "rule-set selection"
3
Deploy in monitor mode, tune, transition to blocking — within 90 days for Tier 0/1
Manage the false-positive curve so the WAF reaches active blocking without breaking the application.

The deployment progression:

  • Day 0–14: monitor mode. WAF logs what it would block; application traffic unaffected. Engineering and app team review logs daily.
  • Day 14–45: tuning. Identify false positives in the monitor logs; build exclusions per legitimate traffic pattern. Tune signatures (raise threshold, exclude specific parameters from specific signatures, allowlist known administrators or partners). Tune by exclusion of false positives, not by disabling entire rule groups.
  • Day 45–75: dual-mode. WAF in transactional-block mode (block highest-confidence signatures, monitor lower-confidence) or canary blocking (block for a percentage of traffic). Watch the application closely.
  • Day 75–90: full block. WAF in active blocking mode for the full rule set as tuned. Confirm no application functionality lost; final exclusion list documented.

For Tier 0 and Tier 1 applications, this progression completes within 90 days of WAF deployment. Tier 2/3 applications have more flexibility but still aim for blocking within 6 months. Monitor-only is a transition state, not a permanent posture.

Verify. WAF policy mode is "block" for each in-scope policy after the timeline. The exclusion list per policy is documented and reviewed. False-positive triage backlog is within SLA.
WAF: "monitor-then-block transition"
4
False-positive triage workflow with documented SLA
False positives are investigated and resolved promptly; the WAF policy stays current with application changes.

Operate the FP workflow:

  • Detection. WAF logs feed the SIEM; alerts on block events for legitimate-looking traffic; user-reported issues come in via service desk
  • Triage SLA. ≤5 working days for Tier 0/1 application alerts; ≤10 working days for Tier 2/3. Track time-to-resolution per FP, not just per ticket.
  • Resolution patterns:
    • Application-side fix — application changes input handling to not look attacker-like
    • WAF exclusion — specific parameter on specific path excluded from specific signature (narrow exclusion, never global)
    • Signature tuning — threshold raised because the signature is too sensitive for this application's legitimate traffic
    • Allowlist — specific trusted source allowed to bypass specific rules (used sparingly)
  • Avoid pitfall: "disable rule for this app" is the easy answer that erodes coverage; force triage to the narrowest viable resolution
  • Quarterly review — review accumulated exclusions; are any no longer needed? Are application changes invalidating them?
Verify. FP triage queue current; oldest open item within SLA. Quarterly exclusion-review evidence in the change log.
WAF: "false-positive management"
5
Signature and rule-set update cadence
WAF coverage stays current with the vendor's threat intelligence and the wider community ruleset.

Update discipline:

  • Routine updates applied within 30 days of vendor release — include in standard change cadence
  • Emergency signatures (active exploitation, named CVE coverage) applied within 7 days; for severe (e.g. RCE in a widely-deployed framework), apply within 24 hours via emergency change
  • OWASP CRS minor versions tracked and adopted within 90 days of release; major versions tracked with a documented adoption plan (a CRS major version may include breaking changes that need re-tuning)
  • Vendor PSIRT subscription — subscribe to the WAF vendor's security advisory feed; track separately from product release notes
  • Update testing — for Tier 0/1 applications, signature updates pass through a canary policy (one application or one environment first) before broad rollout
Verify. Show signature version per policy; cross-check against vendor's current release. Confirm subscription to PSIRT feed. Sample recent updates; each has a change record and verification step.
WAF: "signature currency"
6
TLS termination policy and back-end re-encryption
TLS at the WAF tier uses approved cipher suites; back-end re-encryption for sensitive applications.

Where the WAF terminates TLS (most deployments):

  • TLS 1.2 minimum, TLS 1.3 where supported; TLS 1.0/1.1 disabled
  • Cipher suites limited to AEAD ciphers (AES-GCM, ChaCha20-Poly1305); ephemeral key exchange (ECDHE) required; CBC-mode and RC4 disabled
  • Server certificate from organisation CA or public CA per the application's needs; tracked in central certificate inventory
  • HSTS enabled on public-facing applications with documented max-age
  • Back-end re-encryption — for Tier 0/1 applications, the WAF re-encrypts before forwarding to the application server. Cleartext between WAF and application is not permitted for these tiers (the WAF can decrypt for inspection, but the wire from WAF to application server is TLS).
  • For Tier 2/3 applications, cleartext back-end may be acceptable where the WAF-to-application path is within a trusted segment with documented compensating controls

Cross-reference: TLS configuration details (cipher selection, OCSP stapling, certificate chain) live in WI-03 (Load Balancer) where they overlap; this step is about the WAF-specific aspects.

Verify. Run testssl.sh or SSL Labs against the WAF's TLS endpoint; result is A or better (no protocol downgrades, no weak ciphers). For Tier 0/1 applications, confirm back-end traffic is TLS (e.g. tcpdump on the back-end network shows TLS handshake).
WAF: "TLS termination"
7
Bot management and rate limiting on public-facing applications
Defend against automated abuse: credential stuffing, scraping, denial of service.

For public-facing applications:

  • Bot detection where the WAF supports it — JavaScript challenge, behavioural analysis, known-bot signature database, CAPTCHA for high-risk endpoints
  • Rate limiting — per source IP, per session, per endpoint. Authentication endpoints get tighter limits (typically 5–10 attempts per source per minute before challenge or block)
  • Brute-force protection — explicit detection on login endpoints with progressive response (challenge → block source for X minutes → permanent block on repeated abuse)
  • API rate limiting for any API endpoints (often missed; APIs are typically more abuse-prone than user-facing pages)
  • Allowlists for known automation (monitoring, partner integrations) so legitimate automated traffic isn't blocked — these allowlists are time-bounded and reviewed
Verify. Simulate a brute-force pattern against a non-production login endpoint; confirm the WAF challenges or blocks after threshold. Confirm legitimate automation (a known monitoring source) is not blocked.
WAF: "bot management / rate limiting"
8
Application allowlist or learning (where supported)
Move from signature-based negative-only model toward a positive security model for high-sensitivity applications.

For Tier 0 applications and high-sensitivity Tier 1 applications, where the WAF supports learning or positive-model policies:

  • Enable application learning to build a profile of legitimate URLs, parameters, expected types and ranges
  • Curate the learned profile (review, accept, reject suggestions) — automatic-accept of learning is dangerous
  • Promote curated profiles to enforce: requests matching the learned profile pass; deviations are flagged or blocked
  • Re-learn after major application releases or when traffic patterns change significantly

Application-level learning is high-value but high-effort. Reserve for genuinely high-sensitivity applications where signature-based defence alone is insufficient.

Verify. For applications under learning/positive-model: profile is documented; promotion to enforce reviewed; deviation alerts active.
WAF: "application allowlist / learning"
Validation & verification

After applying these steps, verify the WAF as a whole. Validation includes both technical checks and operational ones (the FP workflow has to actually be running, not just documented).

Functional validation

  • Every protected application has a documented WAF policy with current owner
  • Tier 0/1 applications in active blocking mode; transitioned within 90-day target
  • OWASP Top 10 coverage verified per policy; deliberate test attacks logged or blocked
  • Signature versions current (≤30 days behind vendor release)
  • TLS configuration scores A or better via testssl.sh or SSL Labs
  • Back-end re-encryption verified for Tier 0/1 applications
  • Bot/rate-limit controls effective on a simulated brute-force test
  • FP triage queue within SLA; quarterly exclusion review in evidence

Rollback & troubleshooting

WAF rollback approaches depend on which step caused the problem.

Common scenarios

  • Application broken after transition to blocking mode — cause: a signature blocks legitimate traffic that wasn't seen during monitor-mode tuning (rare path, occasional user). Fix: identify the blocking rule from WAF logs; add a targeted exclusion (narrowest scope); restore application; investigate root cause for permanent fix.
  • Mass false positives after signature update — cause: vendor update changed signature behaviour. Fix: roll back to the previous signature version; engage vendor TAC; do not stay reverted indefinitely — address with vendor and re-update.
  • WAF in blocking mode blocking legitimate API client — cause: API client uses request patterns that trigger generic signatures (e.g. tools that put JSON in URL parameters). Fix: targeted exclusion for the client's source + API path + specific rule.
  • OWASP CRS paranoia bump caused widespread FP — cause: paranoia 3 is too aggressive for this application's traffic. Fix: revert to paranoia 2 with documented justification; raise again with application changes.
  • TLS scan dropped below A grade — cause: a recent change re-enabled a deprecated cipher or protocol. Fix: re-tighten cipher policy; verify with rescan.
  • Bot challenge breaking legitimate users — cause: JavaScript challenge unsupported in some legitimate user-agent contexts (older mobile browsers, screen readers). Fix: refine the challenge population; offer accessible-friendly alternatives.
  • Learning profile incorrectly accepting attack-like traffic — cause: learning was run with an application bug present that exposed unusual parameters. Fix: discard the learned profile; relearn after the application is in a known-good state.

Rollback procedure

  1. Switch the affected policy to monitor-only (immediate operational relief; protection is now alert-only but available)
  2. Identify the specific signature or rule causing the issue from WAF logs
  3. Apply a targeted exclusion or revert the specific change
  4. Return policy to blocking; verify application functional
  5. Update change record; document the root cause for prevention
Common errors & tips
  • Don't disable rule groups to fix false positives. The right fix is a narrow exclusion (this parameter, this path, this signature). Disabling rule groups erodes coverage and accumulates technical debt.
  • Monitor-only forever is a baseline violation. A WAF in monitor mode for 18 months provides no protection. Transition to blocking on schedule, even if it means smaller scope.
  • FP triage without an SLA is FP triage that doesn't happen. The 5-working-day target for Tier 0/1 is what makes the workflow real.
  • OWASP CRS at paranoia 1 is a starting point, not a destination. Paranoia 1 misses well-known attack patterns; if you stop at 1, you'd have been better off without the WAF.
  • Application allowlists are not free. Every allowlist is a hole in WAF coverage; review them periodically; remove ones that no longer serve their purpose.
  • Don't TLS-terminate in cleartext to the back-end for Tier 0 apps. Even on a trusted segment. The cost of re-encryption is small; the value of end-to-end TLS is large.
  • Bot challenges have accessibility implications. Some users will hit them legitimately and need a path through; accessibility-friendly alternatives matter both ethically and legally.
  • Vendor emergency signatures aren't always reliable. Some emergency signatures cause more harm than good (false positives on critical traffic). For Tier 0/1, canary the emergency signature in monitor mode first if you have a few hours; for active exploitation under attack, accept the risk and deploy in blocking.
  • Cloud WAFs have different controls. Azure Front Door WAF, AWS WAF, Cloudflare each implement these requirements via their own console settings; the requirements don't change but the mechanism does. The future Cloud Baselines domain will cover the platform-integration aspects.
