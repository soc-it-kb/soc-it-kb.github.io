ID-PB-03: MFA Fatigue / MFA Bypass Attack

Standardised response framework for MFA fatigue, bypass, and abuse attacks — addressing scenarios where MFA is technically enabled but operationally bypassed through user coercion, push bombing, or protocol abuse.

Section 1
Purpose

Provide a standardised decision and response framework for detecting, investigating, and responding to Multi-Factor Authentication (MFA) fatigue, MFA bypass, or MFA abuse attacks, ensuring rapid containment and prevention of repeated compromise.

This playbook addresses attacks where MFA is technically enabled but operationally bypassed through user coercion, push bombing, or protocol abuse.

Section 2
Scope

✅ In scope

  • MFA push-notification fatigue attacks
  • MFA approval abuse
  • MFA challenge replay or bypass
  • Identity risk detections indicating MFA abuse
  • User-reported unexpected MFA prompts

❌ Out of scope

  • Password spray without MFA interaction → ID-PB-07
  • OAuth token abuse → ID-PB-04
  • Hardware token compromise (handled separately)
Section 3
Trigger Conditions
  • MDR escalates suspected MFA fatigue or bypass
  • Identity platform flags anomalous MFA behaviour
  • User reports repeated or unexpected MFA prompts
  • SOC detects successful sign-in following multiple MFA denials
  • Sign-ins approved from suspicious or unexpected locations
Section 4
Initial Validation (Decision Logic)

Confirm the event is MFA-related by checking for any of the following:

  • Multiple MFA challenges generated within a short time window?
  • MFA approvals occurring without a user-initiated login event?
  • MFA challenge followed immediately by a successful sign-in?
  • Use of legacy or weak authentication paths bypassing MFA?

If NO to all → Route to the appropriate identity playbook
If YES to any → Proceed with MFA abuse assessment (Section 5)

Section 5
MFA Abuse Assessment

Assess the attack pattern. Presence of any indicator confirms MFA abuse:

High volume of MFA prompts in a short window
User reports being told "Approve to stop alerts"
Successful sign-in recorded after MFA spam
MFA approved from an unfamiliar location or device
Pattern of repeated MFA failures followed by success
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Successful sign-in achieved via MFA abuse
🔴 Privileged or VIP account targeted
🔴 MFA abuse repeated across multiple users simultaneously
🔴 MDR confirms active attacker persistence following bypass

Notify immediately
🔐
Identity Team
MFA reset and re-registration
👔
SOC Leadership
High severity awareness
🤝
MDR Partner
If applicable
Section 7
Response Strategy

Execute the following actions immediately and in parallel where possible:

Immediate session revocation — revoke all active tokensRequired
Credential resetRequired
MFA method review — assess method strengthRequired
MFA re-registration with stronger methodRequired
Sign-in log reviewRequired
Persistence check (mailbox rules, OAuth)Conditional
Endpoint validation (if sign-in succeeded)Conditional

Section 9
MFA Hardening & Containment Logic

During response, validate and strengthen the MFA posture of the affected account:

MFA method

Confirm method used and upgrade if weak (SMS → authenticator app → FIDO2)

Number matching

Is MFA number matching enabled? Prevents blind push-approval attacks

Legacy auth

Is legacy authentication still enabled? Block via Conditional Access if so

Conditional Access

Are Conditional Access policies enforced for this account and application?

⚠️ If MFA posture is weak or misconfigured → escalate to Identity Engineering and record as a formal control gap

Section 10
Scope Expansion Logic

Triggers for expansion

  • Same IP targeting multiple users
  • Similar MFA abuse patterns across the tenant
  • MFA fatigue detected in parallel incidents
  • MDR indicates campaign-level activity

Expansion actions

  • Search for users with repeated MFA prompts
  • Correlate attacking IPs and ASNs across accounts
  • Review MFA logs tenant-wide for pattern matching
Section 11
Communication & Coordination

Inform the affected user of the following:

  • That an MFA abuse attack was detected and what occurred
  • That credentials have been reset and MFA re-registration is required
  • How to recognise and respond to future MFA fatigue attempts

Coordinate MFA re-enrolment carefully — ensure the user registers on a trusted device. Align all actions with the MDR ticket and record the re-registration timestamp.

Section 12
Closure Criteria
  • Credentials reset and MFA fully re-registered with a stronger method
  • All active sessions revoked
  • MFA abuse source blocked or mitigated
  • Sign-in activity reviewed and confirmed clean
  • User confirms no further unexpected MFA prompts
Section 13
Evidence to Capture
MFA event logs
Sign-in logs (pre and post reset)
Attacking IP addresses and geolocation
MFA method details at time of attack
Timeline of MFA prompts and approvals
Section 14
Post-Incident Actions
  • Recommend MFA hardening — number matching, phishing-resistant MFA (FIDO2)
  • Review and tighten Conditional Access policies
  • Tune MFA abuse detection thresholds in the identity platform
  • Feed indicators into threat hunting queries
  • Update user awareness materials to cover MFA fatigue scenarios
