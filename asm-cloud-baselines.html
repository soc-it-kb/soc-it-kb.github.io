Cloud Baselines

Tenant-level security baselines for Microsoft 365 and Microsoft Azure across the institution and its agencies. Aligned to CIS Microsoft 365 Foundations and CIS Microsoft Azure Foundations Benchmarks. The baseline is licensing-tier-aware (A1 / E1 / A3 / E3 / E5) so each tenant sees the realistic minimum for its licence and the additional controls available at higher tiers.

📌 About this domain. Cloud security baselines are different from on-premises baselines in one important way: the boundary between vendor-administered and tenant-administered controls is part of the baseline itself. The vendor (Microsoft, in our case) operates the underlying platform; we administer the tenant. The baselines in this domain define what every tenant must enforce, vendor-neutral where the equivalent control exists across platforms, and explicitly licensing-tier-aware so that agencies on lower-tier licences see realistic expectations.
Scope

This domain covers tenant-level security configuration for the two cloud platforms central to the estate:

  • Microsoft 365 (M365) — productivity tenants across all agencies, regardless of licensing tier. Includes Exchange Online, SharePoint Online, OneDrive, Teams, Purview (DLP / Information Protection / Audit), and the Defender for Office 365 controls available at each licence tier. Education tenants (A1 / A3 / A5) and Business / Enterprise tenants (E1 / E3 / E5) are covered under the same baseline, with feature availability per tier explicitly documented.
  • Microsoft Azure — IaaS and PaaS tenants where the institution or an agency operates Azure subscriptions. Includes subscription-level controls (Defender for Cloud, Azure Policy, diagnostic settings), identity and access at tenant level, common workload patterns (storage accounts, key vault, networking baseline), and Azure-native cloud-WAF on Application Gateway and Front Door.

Out of scope:

  • AWS and GCP — acknowledged but deferred. A small number of workloads exist on AWS and GCP across some agencies; these are recorded in the Asset Register and a future multi-cloud extension to this domain will provide baselines. Until then, the operating agency applies the CIS AWS Foundations or CIS GCP Foundations benchmarks directly and records the choice in GRC-REG-ASSET.
  • Identity governance — joiner / mover / leaver lifecycle, access recertification, eligibility-list curation, role design — deferred to a future Identity domain SOP. Tenant-level identity controls (Conditional Access policies, PIM enablement, security defaults, named locations, break-glass account setup, sign-in risk policies) are in scope of WI-01.
  • SOC operational use of cloud security tooling — alert triage, incident response, hunt queries — covered under SOC SOPs (SOP-IM-01, SOP-EM-01, SOP-ID-01, SOP-EP-01). The tenant-side configuration of Defender (enablement, policy, baseline settings, log integration) is in scope of WI-05.
  • Third-party cloud-WAFs — Cloudflare, AWS WAF, Akamai — deferred to a future vendor-specific extension. Azure-native cloud-WAF (Azure WAF on Application Gateway and Front Door) is in scope of WI-04.
  • Cloud architecture reference design — landing zone hierarchies, management group structure, hub-spoke network design, multi-region failover patterns — these belong in a Cloud Architecture document rather than a baseline. Azure Policy as enforcement mechanism is in scope of WI-03 because it is the backbone for enforcing everything else.
Platform Overview

Unlike the network and security-infrastructure baselines, where the control structure is largely shared across device types, M365 and Azure are operationally distinct platforms with almost no overlap beyond Entra ID (Azure AD). The baseline structure reflects this: shared identity controls live in one Work Instruction (WI-01), then each platform has its own dedicated WIs.

Tenant Identity (Entra ID)

Cross-cutting. Conditional Access, PIM, Security Defaults, named locations, sign-in risk policies, break-glass accounts. Applies to both M365 and Azure tenants. IT-CL-WI-01.

M365 Tenant Hardening

Exchange Online, SharePoint / OneDrive, Teams, Purview audit and DLP. Licensing-tier-aware: A1/E1 minimum, A3/E3 expanded, E5 advanced. IT-CL-WI-02.

☁️

Azure Subscription Baseline

Defender for Cloud enablement, Azure Policy enforcement, diagnostic settings, activity log routing, log forwarding to the central SIEM. IT-CL-WI-03.

🛠️

Azure Workload Patterns

Storage accounts, Key Vault, network baseline (NSG, Private Endpoint), Azure WAF on Application Gateway / Front Door. IT-CL-WI-04.

🛡️

Defender Suite Configuration

Tenant-side enablement and policy for Defender for Endpoint, Defender for Office 365, Defender for Identity, Defender for Cloud Apps, Defender for Cloud. Licensing-tier-aware. IT-CL-WI-05.

Why tier-aware baselines. The estate has tenants on A1, E1, A3, E3, and E5. Many advanced controls (Safe Attachments, Safe Links, Defender for Office 365 P2, PIM, sign-in risk policies, Insider Risk Management) require specific licence tiers. A baseline that demands E5 features of A1 tenants is unimplementable; one that targets only A1 features misses the advanced protections E5 tenants should be using. Each control therefore specifies the minimum licence required, and conformance is reported per tier so deviations are visible against achievable expectations.

Work Instructions
IT-CL-WI-01
Tenant Identity & Conditional Access
Entra ID hardening cross-cutting M365 and Azure: Security Defaults or Conditional Access (CA per tier), MFA enforcement, named locations, break-glass account discipline, PIM where licensed, sign-in / user risk policies where licensed, legacy authentication elimination, guest access control.
IT-CL-WI-02
M365 Tenant Hardening
Exchange Online (anti-phishing, anti-spoof, DKIM/DMARC, transport rules), SharePoint and OneDrive (external sharing policy, link expiry, sensitivity labels), Teams (external access, federation, meeting policies), Purview (unified audit, DLP, Information Protection). Tier-aware: A1/E1, A3/E3, E5.
IT-CL-WI-03
Azure Subscription Baseline
Subscription-level: Defender for Cloud enablement and plans, Azure Policy assignment at management group scope, diagnostic settings on every resource, activity log forwarding, central SIEM integration via Event Hub or Azure Monitor, secure score targets and reporting.
IT-CL-WI-04
Azure Workload Patterns
Storage accounts (no public, secure transfer, blob anonymous off, encryption with CMK for Tier 0/1), Key Vault (RBAC mode, soft-delete + purge protection, network restrictions), networking baseline (NSG, Private Endpoint over public, no Service Endpoints to data services), Azure WAF on Application Gateway / Front Door (managed rules, custom rules, log streaming).
IT-CL-WI-05
Defender Suite Configuration
Tenant-side enablement and policy: Defender for Endpoint (onboarding, ASR rules, EDR-in-block-mode), Defender for Office 365 (Safe Links, Safe Attachments — E5 / Plan 2 only), Defender for Identity (sensor deployment on DCs), Defender for Cloud Apps (connectors, anomaly detection), Defender for Cloud (per WI-03). Licensing-tier-aware.

Roles & Ownership

IT Operations Lead

Owns the baseline. Approves changes and exceptions. Sets the conformance reporting cadence. Accountable to the Risk Committee for tenant baseline state.

Cloud Engineering

Implements the baseline in central / shared-services tenants. Maintains golden Azure Policy initiatives. First-line responder for cloud-platform incidents. Owns the central SIEM integration patterns.

Identity & Access Engineering

Implements WI-01 (Conditional Access, PIM, Entra ID). Co-owner of break-glass account procedure. Coordinates with Cloud Engineering on Azure RBAC and with M365 admin teams on tenant identity policies.

M365 Tenant Administrators (per agency)

Implement WI-02 in their tenant. Maintain Exchange / SharePoint / Teams / Purview configuration. Track licensing tier in GRC-REG-ASSET. Submit exception requests for unachievable controls.

SOC

Consumes signals: Defender alerts, sign-in risk events, audit log anomalies, Defender for Cloud findings. Investigates and escalates per SOC SOPs. Does not own tenant configuration.

GRC

Owns baseline governance and audit evidence. Maintains exception register. Reports tenant conformance to Risk Committee quarterly. Tracks the SIEM-integration remediation across uncovered tenants.

Local IT (per agency)

Applies baseline in the agency's tenant. Submits exception requests via LCSO/CSLC. Coordinates change windows with central Cloud Engineering and IAE.

Architecture / Technical Authority

Adjudicates interpretation questions. Approves design-level exceptions. Reviews new tenant onboarding to confirm baseline applicability and licensing-tier mapping.

Metrics & Oversight

The metrics below measure adherence to this baseline. They are reviewed monthly by IT Operations and quarterly by GRC, separated by platform and licensing tier so conformance is reported against achievable expectations:

  • Tenant baseline conformance rate — percentage of in-scope M365 and Azure tenants fully conformant to IT-CL-SOP-01 against their licensing tier. Target: ≥95% for Tier 0/1 tenants; ≥85% for Tier 2/3.
  • Microsoft Secure Score — quantitative metric for M365 (and where applicable Azure). Targets per tier: A3/E3 ≥70%; E5 ≥80%; A1/E1 reported but not target-bound (limited features available).
  • Azure Secure Score (Defender for Cloud) — for Azure subscriptions. Target: ≥75% for Tier 0/1 subscriptions; ≥60% for Tier 2/3.
  • Central SIEM integration coverage — percentage of in-scope tenants forwarding audit, sign-in, and security logs to the central SIEM. Baseline target: 100%. Current state: one tenant (~7k accounts). Remaining tenants tracked as time-bounded exceptions until integrated, target horizon 12–18 months.
  • MFA coverage — percentage of administrative accounts and standard accounts protected by MFA. Target: 100% administrative; ≥98% standard.
  • Legacy authentication elimination — percentage of tenants with legacy auth fully blocked. Target: 100% by end of FY 2026.
  • Defender suite enablement — per Defender product, per applicable licence tier. Target: 100% of licensed tenants have the corresponding Defender product enabled and configured per WI-05.
  • Exception register volume — tracked alongside the others. Expect elevated counts on initial publication; trend down over the SIEM-integration remediation window.
Exception Handling

Where a tenant cannot meet a control (licensing tier limitation, vendor capability gap, legitimate business exception, in-flight migration), the operating agency submits a deviation request via the standard ASM exception process (IT-GOV-SOP-01). Deviations are reviewed and approved by the IT Operations Lead with GRC consultation. Approved deviations are recorded in the Baseline Exceptions Register (GRC-REG-EXCEPTIONS), linked from the tenant's asset record in GRC-REG-ASSET, and reviewed annually for currency.

Note on initial publication. The central SIEM integration baseline (universal log forwarding) means that on day one of publication, every tenant other than the currently integrated one is in a deviation state. These are batch-tracked as a single category of exception against a 12–18-month remediation window. This is deliberate: the baseline crystallises a real estate gap that was previously invisible, and the exception register becomes the management view of the remediation programme.
