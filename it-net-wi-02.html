IT-NET-WI-02: Control Plane Hardening

Vendor-agnostic implementation guidance for hardening the control plane of network devices: how the device decides where traffic goes. Operationalises §4.2 of IT-NET-SOP-01.

Prerequisites

Before applying this WI:

  • Management plane is hardened per IT-NET-WI-01 (this WI assumes secure admin access is in place)
  • Device is L3-capable for routing-related steps; L2-only switches use the steps relevant to spanning-tree, storm control, ARP inspection
  • Routing topology and protocols are documented (which protocols, which neighbours, key administrative boundaries)
  • Authentication keys for routing protocols are available (generated, vault-stored, ready for deployment)
  • A change window is approved per IT-NET-WI-06
  • Working OOB access is confirmed (some control-plane changes can disrupt routing convergence briefly)
Routing changes propagate. Control-plane changes don't just affect this device — they affect every routing neighbour. Coordinate with the wider network team before applying authentication changes or boundary filters that touch active neighbours.
Step-by-step instructions

The steps below cover the control-plane controls required by IT-NET-SOP-01 §4.2.

1
Authenticate routing-protocol adjacencies
Prevent unauthorised devices injecting routing information.

For every active routing protocol on the device, enable strong authentication:

  • OSPF — cryptographic authentication (HMAC-SHA256 or strongest available; MD5 only as fallback for legacy compatibility)
  • EIGRP — HMAC-SHA256 authentication; SHA-256 keys rotated annually
  • BGP — TCP-AO (RFC 5925) where supported, otherwise MD5 with documented rotation schedule
  • IS-IS — HMAC-SHA256 area and domain authentication
  • HSRP / VRRP / GLBP — authentication enabled, strong keys

Keys are stored in PAM, rotated per organisation policy (annual minimum), and key-rotation procedures are documented per protocol.

Verify. Show running configuration confirms authentication is configured on each routing instance. Show neighbour state confirms adjacencies are established (not stuck in init or 2-way pending auth). Attempt a deliberately mis-keyed neighbour from a lab; confirm it fails to form.
CIS: Control-plane "routing authentication"
2
Control-plane policing (CoPP)
Limit control-plane CPU exposure to known-good traffic rates.

Configure control-plane policing (or the vendor-equivalent: CPP, control-plane filter, host inbound traffic policy):

  • Identify legitimate control-plane traffic classes: routing protocols, management protocols (SSH/HTTPS/SNMP), ARP/ND, ICMP for diagnostics, BGP/OSPF/etc, NTP, syslog
  • Apply rate limits per class, scaled to expected traffic volumes plus reasonable burst headroom
  • Drop or rate-limit untrusted classes: undefined traffic to the control plane, ICMP from non-management sources, fragmented traffic targeting control plane
  • Where the device offers a default vendor template (e.g. "control-plane host" policies), use it as a starting point and tune rather than starting from scratch
Verify. Show CoPP policy and counters; under normal operation, drops should be near-zero on legitimate classes. Simulate a control-plane flood from a test source; confirm rate-limiting kicks in and the device remains responsive.
CIS: Control-plane "CoPP"
3
Spanning-tree protection (access switches)
Prevent rogue switches and loops from disrupting the L2 topology.

On all access switches:

  • BPDU Guard on every access port (where end users or unmanaged devices connect) — shuts the port down if a BPDU is received, defending against unauthorised switches
  • Root Guard on ports that should never become the root bridge path — protects topology stability
  • Loop Guard on point-to-point trunk links where unidirectional failure would otherwise create a loop
  • UDLD (UniDirectional Link Detection) or equivalent on fibre uplinks where supported
  • PortFast on access ports to speed convergence; combine with BPDU Guard, never deploy PortFast without it
Verify. Show spanning-tree details confirms BPDU Guard, Root Guard, Loop Guard active where intended. Plug a test switch into a protected port (controlled test); confirm the port shuts down and an err-disabled message logs to syslog.
CIS: Control-plane "spanning-tree hardening"
4
Storm control
Contain broadcast/multicast/unknown-unicast storms.

On all access ports:

  • Storm control for broadcast traffic at a threshold matched to normal operations (typically 1–5% of port bandwidth)
  • Storm control for multicast traffic where multicast is not legitimately used on that port
  • Storm control for unknown-unicast traffic at a similar threshold
  • Action: shutdown or trap-log (trap-log for visibility without disrupting users; shutdown for sites with stronger operational tolerance)
Verify. Show storm-control configuration on access ports. Test with a controlled broadcast generator (only in a lab); confirm the storm control action triggers and logs.
CIS: Control-plane "storm control"
5
Anti-spoofing (uRPF / BCP38)
Drop traffic with forged source addresses.

On customer-facing and uplink interfaces of routers and L3 switches:

  • Unicast Reverse Path Forwarding (uRPF) — strict mode on customer-facing interfaces (only forward if return path matches incoming interface); loose mode on multihomed interfaces where strict would drop legitimate traffic
  • Drop bogon and unallocated prefixes at the edge where the device terminates internet uplinks
  • Drop traffic with private (RFC 1918) source addresses arriving on internet-facing interfaces
  • For IPv6, equivalent uRPF or source-address verification per RFC 8704
Verify. Show uRPF configuration on relevant interfaces. Inject a packet with a spoofed source (lab test); confirm it is dropped and counters increment.
CIS: Control-plane "anti-spoofing / BCP38"
6
Dynamic ARP Inspection & IPv6 RA Guard
Prevent ARP/NDP poisoning attacks on access networks.

On access switches:

  • Dynamic ARP Inspection (DAI) enabled on user VLANs, with DHCP snooping bindings as the source of truth (DAI relies on the binding table)
  • IPv6 RA Guard on access ports of IPv6-capable networks — blocks rogue Router Advertisements from non-router sources
  • IPv6 ND Inspection on access ports for neighbour-discovery integrity
  • Trust configuration: only the legitimate gateway port carries trust; user ports are untrusted
Verify. Show ARP inspection counters; check DHCP snooping binding table is populated. From a test client, attempt a gratuitous ARP for the gateway IP; confirm DAI blocks it.
CIS: Control-plane "L2 control protection"
7
Disable unused routing protocols and apply passive-interface defaults
Reduce the control-plane attack surface by removing what isn't needed.

Routing protocol hygiene:

  • Disable any routing protocol not actively used (RIP, IGRP, legacy protocols left over from migrations)
  • Apply passive-interface default to user-facing interfaces; only enable routing on actual peer-facing interfaces
  • Disable proxy-ARP on user-facing interfaces (default-on for some vendors; should be off)
  • Disable IP directed broadcasts (smurf-attack mitigation)
  • Disable ICMP unreachables on transit interfaces if vendor recommends it for control-plane CPU protection
Verify. Show running configuration; confirm only required routing protocols are configured. Show ip protocols confirms passive-interface defaults applied. Show ip interface confirms proxy-ARP off, directed-broadcast off.
CIS: Control-plane "minimise enabled services"
8
Route filtering at administrative boundaries
Prevent route leaks and accept only expected prefixes from neighbours.

At BGP and routing-domain boundaries:

  • Configure prefix lists or route maps that explicitly enumerate accepted prefixes from each external peer
  • Configure outbound filters that explicitly enumerate prefixes advertised to each external peer
  • Maximum-prefix limits on every BGP neighbour (drops neighbour or warns if peer advertises more than expected, common attack/misconfiguration indicator)
  • Filter bogon prefixes inbound from internet peers
  • For internal routing, use route summarisation at boundaries and filter to expected ranges
Verify. Show BGP neighbours confirms max-prefix and route maps applied. Show ip bgp confirms only expected prefixes received. Test by announcing an unexpected prefix from a lab peer; confirm it's rejected.
CIS: Control-plane "route filtering at boundaries"
Validation & verification

After applying all steps, verify the control plane as a whole. Many of these checks should be performed under normal operating load, not in a quiet maintenance window, so that production traffic patterns are exercised.

Functional validation

  • All routing adjacencies are up and authenticated; show ip ospf neighbour / show ip bgp / show ip eigrp neighbour confirms expected state
  • CoPP counters show drops on untrusted classes but near-zero on legitimate classes
  • Spanning-tree topology is stable (no flaps in last 24 hours); root bridge is the intended device
  • BPDU Guard test on a non-production port (controlled): plugging a switch causes err-disable as expected
  • DAI/snooping binding tables are populated for user VLANs
  • uRPF counters show drops of spoofed traffic where present
  • Routing tables look as expected (no unexpected routes from external peers)

Rollback & troubleshooting

Control-plane changes can disrupt routing if mis-applied. Roll back via OOB with the pre-change configuration as the safety net.

Common scenarios

  • Routing adjacency drops after enabling authentication — cause: peer device not configured with matching key, or key strength mismatch. Fix: align keys on both sides simultaneously; coordinated change is essential.
  • Legitimate traffic dropped by CoPP — cause: rate limit too aggressive for legitimate burst, or a traffic class missed in policy. Fix: identify the affected class via counters, widen the limit, or add a class.
  • Network outage after BPDU Guard activation — cause: BPDU Guard applied to a port carrying legitimate switch-to-switch traffic (should be a trunk). Fix: re-classify the port; BPDU Guard belongs on access edges only.
  • Storm control firing during normal operation — cause: threshold too tight for the actual traffic profile (e.g. PXE boot environment, IP multicast video). Fix: tune the threshold; consider exempting specific VLANs.
  • uRPF dropping legitimate asymmetric traffic — cause: strict mode applied to a multihomed or asymmetric path. Fix: change to loose mode on that interface, or fix the upstream routing to be symmetric.
  • BGP max-prefix shutting neighbour down — cause: peer legitimately grew table beyond the limit, or peer leaked. Fix: investigate the peer first; if growth is legitimate, raise the limit; if a leak, the protection worked.

Rollback procedure

  1. Connect via OOB or stable management path
  2. Identify the specific change that caused the issue (compare running config against pre-change backup)
  3. Reverse the offending change only; do not wholesale-restore unless required
  4. Confirm routing convergence: show ip route, show ip ospf neighbour, show ip bgp summary as relevant
  5. Update change record and identify follow-up to address root cause
Common errors & tips
  • Coordinate routing-authentication changes across all peers simultaneously. One side configured, other side not, equals dropped adjacency. Use a controlled change window with both ends in scope.
  • CoPP starting point should be a known-working vendor template, not "deny all". Building CoPP from scratch in production is how you lock yourself out from BGP, NTP, syslog, and SSH simultaneously.
  • BPDU Guard belongs on access ports facing end-users. Putting it on trunks between switches will eventually err-disable an inter-switch link during a legitimate topology change.
  • Don't enable DAI without DHCP snooping populated first. Empty binding table + DAI = all ARP dropped = all hosts disconnected.
  • uRPF strict mode hates asymmetric paths. If your network has dual-homed sites with traffic engineering, loose mode is usually safer than strict.
  • Max-prefix on BGP should be the "expected plus headroom", not the "current". Set it 20–30% above the steady-state to avoid false positives on legitimate growth.
  • Storm control thresholds vary by site profile. A campus with heavy multicast video needs different multicast thresholds than a typical office. One size does not fit all.
  • Don't disable proxy-ARP on transit interfaces in legacy environments without checking. Some legacy applications and devices rely on proxy-ARP behaviour; investigate before flipping the switch in production.
