IT-CL-WI-03: Azure Subscription Baseline

Implementation guidance for the Azure subscription-level controls: management group hierarchy, Defender for Cloud enablement and plans, Azure Policy at management group scope, diagnostic settings, Activity Log forwarding, Defender continuous export, RBAC least-privilege, resource locks, cost-anomaly alerting, tagging enforcement. Operationalises §4.3 of IT-CL-SOP-01.

Prerequisites

Before applying this WI:

  • WI-01 (Tenant Identity & Conditional Access) complete on the parent tenant — MFA, CA, break-glass, PIM in place
  • Azure tenant licensing and Defender for Cloud plan posture decided per subscription tier (Tier 0/1 = Standard / Defender plans; Tier 2/3 = Free is acceptable with documented justification)
  • Management group structure exists or will be created — flat structure (single root) acceptable for small estates; hierarchical (root → environment → agency) preferred
  • Central Log Analytics workspace identified (or per-region workspace where data residency requires it)
  • Central SIEM integration pattern chosen: Event Hub-based ingestion, native SIEM connector (vendor-specific), or Azure Monitor agent forwarding — document the choice and the data flow
  • Azure RBAC role design exists: who can be Owner (named platform admins only), who can be Contributor (engineering teams scoped to RGs), who reads (auditors, SOC)
  • Resource tagging policy agreed: cost centre, owner, environment (prod / non-prod / dev), sensitivity classification
Apply at management group scope, not subscription scope. Azure Policy assignments at the subscription level have to be repeated for every new subscription. Assigning the baseline initiative at the management group root means every subscription, including ones created tomorrow, inherits the baseline by default. This is the entire point of management groups.
Step-by-step instructions

The steps below cover the Azure subscription-level controls required by IT-CL-SOP-01 §4.3. They apply to every Azure subscription in scope, regardless of the workload running inside it.

1
Establish management group hierarchy
Subscriptions inherit policy from a defined hierarchy rather than receiving repeated per-subscription assignments.

Management groups are the policy-inheritance backbone. Azure tenants come with a root management group ("Tenant Root Group") and a single default subscription parent. Build a hierarchy beneath:

  • Minimum useful structure — Tenant Root → Platform → Landing Zones → (Prod / NonProd / Sandbox). Subscriptions sit under the appropriate environment management group.
  • For agency-segmented tenants — add an agency level (Tenant Root → Agency → Environment). This allows agency-specific policy overlays while keeping the institution baseline at the root.
  • Avoid deep nesting — more than 4–5 levels makes RBAC and policy debugging painful; the platform supports 6 but operational reality favours fewer
  • Naming convention — documented and consistent: mg-<institution>-<scope> e.g. mg-akdn-platform-prod, mg-aga-landingzone-prod
  • Subscription movement requires Owner role on both source and destination management group; document the change-control process
  • Tenant Root Group access is restricted to a very small named group (platform-admin role); subscription Owners do not have access
Verify. Management group hierarchy documented in architecture pages. Subscriptions placed in the correct environment management group. Tenant Root Group access list reviewed quarterly.
MCSB GS-1 · "Management group hierarchy"
2
Enable Defender for Cloud on every subscription
Subscription-level CSPM and workload-protection signal active; the central security posture view exists.

Defender for Cloud (DfC) is the Azure-native cloud security posture management (CSPM) platform. The Foundational CSPM tier is free; the Defender plans (per resource type) are paid. Both are required to varying extent.

  • Foundational CSPM — enabled on every subscription by default; gives Secure Score, regulatory compliance dashboards (CIS, NIST, ISO mappings), basic recommendations. Free.
  • Defender CSPM (Cloud Security Posture Management) — enhanced posture features including attack-path analysis, agentless scanning, governance rules. Paid; per resource per month. Required for Tier 0/1 subscriptions.
  • Defender for <resource type> plans — per-resource-type workload protection. Each plan is independently priced. Required minimums per subscription tier:
    • Tier 0 / 1: Servers (P2), App Service, Storage, SQL, Containers, Key Vault, ARM, DNS, Open-Source Relational DB — all relevant plans enabled
    • Tier 2 / 3: at minimum Servers, Storage, Key Vault, ARM; others as workload type justifies
    • Free tier: only Foundational CSPM — acceptable only with documented justification (e.g. genuinely non-production, no sensitive data, time-bounded)
  • Auto-provisioning — Log Analytics agent / Azure Monitor agent on VMs auto-deployed via Defender; vulnerability assessment auto-deployed for SQL and VMs where applicable
  • Email notifications — high-severity alerts emailed to a named distribution list (typically SOC + cloud platform engineering)
  • Continuous export — configure Defender alerts and recommendations to flow to Event Hub (or directly to SIEM connector) for central SIEM ingestion
Verify. Defender plans enabled per subscription tier (visible in DfC → Environment settings). Auto-provisioning operational; agents deployed on existing VMs. Continuous export functioning; sample alert visible in central SIEM.
CIS Azure 2.x · MCSB LT-x
3
Assign the core Azure Policy baseline at management group scope
Configuration drift detected and (where appropriate) prevented at create-time; baseline conformance measurable.

Azure Policy is the enforcement mechanism. Built-in initiatives are the starting point; custom initiatives are added where the built-ins don't cover an organisation-specific requirement.

  • Starting initiative: assign Microsoft Cloud Security Benchmark (MCSB) at the Tenant Root management group scope. MCSB is Microsoft's reference and is the Defender for Cloud default; assigning it gives a regulatory-compliance view in DfC.
  • Effect choice per policy — Azure Policy effects are Audit, Deny, Modify, DeployIfNotExists, Disabled. Start with Audit across the board for visibility; promote specific policies to Deny after operational review.
  • Core deny set for baseline (assign these as a custom initiative or override the MCSB defaults):
    • Storage account public network access (deny unless exception)
    • Storage account secure transfer (deny non-HTTPS)
    • Storage account allow blob anonymous access (deny)
    • Key Vault soft-delete and purge protection (deny disable)
    • Public IP creation in production subscriptions (deny without justification tag)
    • Resource creation in non-approved regions (deny)
  • Diagnostic settings enforcement — DeployIfNotExists policies that configure diagnostic settings on every supported resource type, routing logs to the central Log Analytics workspace and an Event Hub. Without this, log coverage drifts as new resources are created.
  • Exemptions are time-bounded and documented — Azure Policy supports policy exemptions natively; exemption justification stored in the exemption record; reviewed quarterly
  • Compliance reporting — Azure Policy compliance dashboard reports the overall conformance state. This is one of the inputs to the tenant baseline conformance metric.
Effects evolve. A new policy goes through Audit → Audit-with-alerting → Deny over weeks. Going straight to Deny on a busy estate triggers unexpected blocks at create-time and causes outages. Audit first, deny when the false-positive rate is known.
Verify. MCSB initiative assigned at root. Custom deny initiative assigned. DeployIfNotExists diagnostic settings deploying on test resource. Compliance dashboard populated. Exemption process documented.
CIS Azure 5.x · MCSB GS-x
4
Configure diagnostic settings on every resource type
Resource-level activity (data plane and management plane) flows to Log Analytics and the central SIEM.

Diagnostic settings are the per-resource log-routing configuration. Without them, resource-level events live only inside the resource itself and are invisible to monitoring and SIEM.

  • Per resource type — each Azure resource type has its own diagnostic-settings surface (e.g. Storage has StorageRead/Write/Delete log categories; Key Vault has AuditEvent; Network Security Groups have NSG flow logs as a separate path)
  • Destinations — for each resource:
    • Log Analytics workspace (for KQL queries, dashboards, Azure Workbooks) — required
    • Event Hub (for streaming to central SIEM) — required
    • Storage account archive (optional; for long retention beyond Log Analytics)
  • Enforce via Azure Policy — DeployIfNotExists policies per resource type (built-in policies exist for the major resource types). Without policy enforcement, diagnostic settings drift the moment a new resource type is added.
  • Workspace placement — regional Log Analytics workspaces where data residency requires; a central workspace where it doesn't. Document the workspace topology.
  • Retention — Log Analytics workspace retention 90–365 days typical; archive to Storage Account for longer retention (lower cost, slower retrieval); document the retention by data class
  • Log volume monitoring — sudden drops in log volume from a resource indicate a problem (resource compromise, diagnostic setting deleted, agent broken); alert on it
Verify. Sample resources of each in-scope type have diagnostic settings configured. Logs reaching Log Analytics workspace (KQL query confirms recent events). Event Hub ingestion functioning; central SIEM has the events.
CIS Azure 5.x · "Diagnostic settings"
5
Forward Azure Activity Log to the central SIEM
Subscription-level management-plane activity (resource creates, deletes, role assignments, policy changes) flows to the SIEM.

The Activity Log is the subscription-level audit log: who did what, when, in the Azure control plane. It is distinct from the per-resource diagnostic settings and from Entra sign-in logs.

  • Per subscription — configure the Activity Log diagnostic setting to send to Log Analytics workspace and Event Hub
  • Enforced via Azure Policy — built-in policy "Activity log should be archived to Storage Account" (or equivalent Log Analytics / Event Hub variants)
  • High-value Activity Log events alerted in SIEM:
    • Role assignment Owner or User Access Administrator created
    • Diagnostic setting deleted
    • Network Security Group rule allowing 0.0.0.0/0 to a sensitive port
    • Storage account public access enabled
    • Key Vault soft-delete or purge protection disabled (this should be impossible if Step 3 deny is in place, but alert anyway)
    • Azure Policy initiative removed or modified at high-level scope
    • Defender for Cloud plan downgraded to Free
Verify. Activity log entries from test subscription change visible in central SIEM within ~5 minutes. Sample alert rules firing on test events. Coverage gap analysis: every in-scope subscription is forwarding.
CIS Azure 5.1.x · "Activity log"
6
Forward Defender for Cloud findings to the central SIEM
Defender alerts (security events, anomalies, recommendations) are visible to SOC in the central SIEM, not only in the Defender portal.

Defender for Cloud's continuous export configures Defender alerts and recommendations to flow to Event Hub or directly to the Log Analytics workspace. From there they integrate with central SIEM.

  • Continuous export — configure per subscription or via Azure Policy DeployIfNotExists; export to Event Hub for SIEM streaming and Log Analytics for query / dashboarding
  • Export scope — all security alerts (always); recommendations of High and Medium severity; secure score updates (weekly)
  • SIEM mapping — Defender alert severity (Informational / Low / Medium / High) mapped to central SIEM severity / priority per documented mapping; tested with sample alert
  • Defender for Cloud workflow automation — for the highest-severity alerts, Logic App triggers can automate immediate response (e.g. quarantine a VM, disable a public IP). Document the automation and its trigger conditions; SOC reviews the playbook.
Verify. Continuous export configured on every in-scope subscription. Sample High-severity test alert appears in central SIEM with correct severity mapping. SOC playbook references Defender alerts.
CIS Azure 2.x · MCSB IR-x
7
Subscription RBAC least-privilege
No standing Owner outside named platform admins; Contributor and Reader scoped to RG, not subscription.

Azure RBAC is granular but trivially over-scoped. The default pattern of "give everyone Contributor at subscription" creates a wide blast radius for any compromised account.

  • Owner role at subscription scope — only named platform administrators (typically 2–4 people per agency); managed via PIM (WI-01 Step 7); no standing assignments outside break-glass
  • Contributor role — scoped to resource group, not subscription. Each application / workload team gets Contributor on their resource groups only.
  • Reader role — broader; auditors, SOC, GRC can have Reader at subscription scope for visibility
  • Custom roles for common scoped duties (e.g. "Network Operations" with rights on NSGs and route tables only; "Storage Operations" with rights on Storage and Backup only); avoid granting Contributor where a custom role would do
  • Group-based assignment — assign roles to Entra groups, not individuals; group membership is the access record; group reviews via Identity Governance (E5)
  • Service principals have least-privilege scoped to the resource group(s) they actually need; documented owner per service principal; secret expiry <1 year; rotated
  • Managed identities preferred over service principals for in-Azure workloads (no secret to manage; lifecycle tied to the resource)
  • Quarterly access review — named approver per scope; expired / unjustified assignments removed
Verify. Owner count per subscription matches the named platform admin list. Sample resource group shows Contributor scoped to RG. Service principals enumerated with owners. Quarterly review scheduled.
CIS Azure 1.x · "Azure RBAC"
8
Apply resource locks on critical resource groups
Accidental deletion of production data, key vaults, and shared-services hubs prevented at platform level.

Resource locks are an in-platform safeguard against accidental delete or modify. They are removed only by Owner or User Access Administrator at the scope — trivial to reverse for a malicious actor with rights, but very effective against accidents.

  • CanNotDelete — the resource can be modified but not deleted. Apply to: production resource groups, shared-services hubs (networking, identity), key vaults holding production keys, storage accounts with compliance data
  • ReadOnly — the resource cannot be modified or deleted. Apply to: completed projects / archived workloads, golden images, audited compliance evidence
  • Lock at resource-group scope typically — child resources inherit; locking at subscription scope is too coarse
  • Lock by Azure Policy — DeployIfNotExists policies that ensure named-critical RGs have CanNotDelete; avoids drift when RGs are recreated
  • Lock removal goes through change control; sudden lock removal on a critical RG followed by destructive action is a SIEM-alert pattern
Verify. CanNotDelete locks on production resource groups. Test: attempt to delete a locked resource as a Contributor; refused. Lock removal logged in Activity Log; alerts fire in SIEM.
MCSB GS-x · "Resource locks"
9
Configure cost-management alerting
Anomalous spend is a leading indicator of resource compromise (cryptomining, data exfiltration egress); detect it.

Cost is a security signal. A compromised subscription frequently shows as unusual spend before any conventional security alert — cryptominer VMs spin up at large scale, data egress bills jump, public-access storage accounts incur large blob-read traffic.

  • Budgets per subscription — monthly budget with thresholds at 50%, 75%, 90%, 100% of expected spend; alerts to subscription owner + cloud engineering + GRC at thresholds
  • Cost anomaly detection — Cost Management's anomaly detection feature for AI-detected unusual patterns; weekly digest to subscription owners
  • Resource-type spike alerts — Log Analytics queries on resource creation rate (VMs spun up beyond normal, public IPs created in non-expected RGs); cross-reference in SIEM with cost data
  • Cryptomining-specific signals — large GPU VM SKUs, NCv-series, NDv-series — alert on any creation in a non-expected RG / non-expected subscription
  • Egress bandwidth from a single storage account suddenly >10× normal — potential exfiltration signal
Verify. Budget configured per in-scope subscription; threshold alerts deliver. Cost Anomaly detection active. Resource-creation-rate query operational in Log Analytics / SIEM. Test alert chain fires correctly.
MCSB IM-x · "Cost anomaly detection"
10
Enforce tagging policy
Every resource is attributable to a cost centre, owner, environment, and sensitivity classification.

Tags are the cross-cutting metadata used by cost management, RBAC scoping, automation, and incident response (whose resource is this, who do I call, how sensitive is it?). Without enforcement, tag coverage drifts to single digits.

  • Required tags:
    • CostCentre — finance code; mandatory for billing reconciliation
    • Owner — named individual or team distribution list; the operational owner
    • Environment — prod / nonprod / dev / sandbox
    • SensitivityClassification — aligned to the organisation data classification taxonomy (e.g. Public / Internal / Confidential / Restricted)
    • Agency — which operating agency the resource serves
  • Enforcement via Azure Policy — Modify effect to inherit tags from parent resource group; Deny effect to refuse resource creation without required tags; built-in policies exist for these patterns
  • Tag value lists — for tags with a finite vocabulary (Environment, SensitivityClassification), constrain values via Azure Policy allowedValues
  • Reporting — tag-coverage dashboard; missing-tag report to owners weekly
  • Incident response value — the Owner tag is the on-call escalation in a Defender alert; the SensitivityClassification tag drives incident priority; the Environment tag distinguishes outages from dev experiments
Verify. Tag policy assigned at management group root. Resource creation without required tags refused (test in sandbox). Tag-coverage report shows ≥95% on production subscriptions.
MCSB GS-x · "Resource tagging"
Validation & verification

After applying these steps, validate the subscription baseline is in place.

Functional validation

  • Management group hierarchy populated; subscriptions in the correct environment node
  • Defender for Cloud enabled per subscription tier; Secure Score visible and at or above target (≥75% T0/1, ≥60% T2/3)
  • Azure Policy compliance dashboard shows the MCSB and custom-deny initiatives assigned; compliance >90% at policy level
  • Test: create a storage account with public network access — refused by policy (Deny)
  • Test: create a Key Vault with purge protection disabled — refused by policy
  • Test: create a resource without required tags — refused by policy
  • Diagnostic settings active on test resources; logs in Log Analytics workspace; events in central SIEM
  • Activity Log entries flowing to SIEM within ~5 minutes
  • Defender continuous export operational; sample alert in SIEM
  • Subscription Owner count matches platform admin list; Contributor scoped to RGs
  • CanNotDelete locks present on production RGs
  • Cost budget alerts deliver; cost-anomaly detection active

Rollback & troubleshooting

Subscription-baseline rollout can be disruptive if Deny policies are enabled before the estate is conformant. Plan the rollout carefully.

Common scenarios

  • Azure Policy Deny is blocking legitimate resource creation — cause: a policy was promoted from Audit to Deny before the estate was conformant; resources that previously existed cannot be recreated identically. Fix: switch the policy back to Audit; identify the non-conformant pattern; either remediate the resource or document the exemption.
  • Diagnostic settings policy is over-provisioning Log Analytics ingestion — cause: too many log categories enabled at once on too many resources; Log Analytics ingestion bill spikes. Fix: review log categories per resource type; some are extremely verbose (e.g. some Storage data-plane categories); deselect what isn't needed.
  • Defender for Cloud agent installation broken — cause: VM image without the required prerequisites, network restrictions blocking agent communication. Fix: install agent manually; for image issue, update the golden image; for network issue, confirm required endpoints are reachable.
  • Tagging policy refused critical hot-fix resource creation — cause: an emergency resource needed creation without all required tags. Fix: targeted exemption with time-bound and follow-up to apply tags within 24 hours; do not blanket-disable the policy.
  • Subscription Owner permission required for activity that should be Contributor-scope — cause: misunderstanding of what requires Owner. Resource-level RBAC change requires Owner or User Access Administrator at the resource scope. Fix: clarify in the agency operating model; use User Access Administrator at RG scope where the only need is role assignment.
  • Resource lock blocking a legitimate change — cause: lock at too-broad scope. Fix: temporarily remove lock for the change window; restore immediately after; document in change record. If pattern repeats, narrow the lock scope.

Rollback procedure

  1. Identify the blocking change (Azure Policy compliance state, Activity Log, change record)
  2. For policy: revert effect from Deny to Audit at the relevant scope
  3. For diagnostic settings: narrow log categories; switch routing temporarily if SIEM ingestion is the issue
  4. For RBAC: targeted role assignment; do not broaden default model
  5. For lock: temporary removal with logged justification; restore
  6. Update change record; document root cause; remediate the underlying gap before re-enabling Deny
Common errors & tips
  • Apply policy at management group, not subscription. Per-subscription policy assignment scales poorly and drifts. Management group inheritance is why management groups exist.
  • Audit before Deny, always. A policy promoted directly to Deny causes outages on any resource that doesn't already conform. Audit for at least 2 weeks; review compliance state; remediate or exempt; then Deny.
  • Diagnostic-settings policies are the unsung hero. Without them, log coverage degrades silently as new resource types appear in the estate. They are the single most important set of policies in the baseline.
  • Defender plans are per resource type, not per subscription. Enabling "Defender for Servers P2" doesn't enable "Defender for Storage." Each plan has its own configuration and pricing. Review the full Defender Plans surface per subscription tier.
  • Free tier Defender for Tier 0/1 subscriptions is not acceptable. The CSPM is fine; the workload-protection plans are not. Tier 0/1 = paid plans, no exceptions outside a documented and time-bounded migration.
  • Activity Log retention defaults to 90 days. Forensic investigations frequently need longer. Either Log Analytics retention extension or Storage Account archive.
  • Subscription Owner is not "use this everyday." If standing Owner is needed for routine work, the RBAC design is wrong. Custom roles + PIM eligibility for the rare cases.
  • Service principals proliferate. Old apps leave behind orphaned SPs; secrets expire without rotation; tracking gets lost. Quarterly review of all SPs in scope; deprecate aggressively; prefer managed identities.
  • Resource locks are not RBAC. A user with Owner rights can remove a lock then delete the resource. The lock prevents accidents, not deliberate actions. Alert on lock removal.
  • Cost spikes are security signals. A 10× spike on egress, a sudden GPU-VM creation, a public-blob being downloaded at TBs — these are incidents, not finance issues. The cost alert often beats the security alert by hours.
  • Tagging without enforcement = 30% coverage. Self-attested tagging discipline drifts. Modify-and-Deny policies make it real.
  • Continuous export to SIEM must be tested. Configuring it isn't enough; verify alerts actually reach the SIEM and are correctly mapped. The day you need the alert is the wrong day to discover the pipeline broke.
