GRC-DR-WI-03 — Core IT Services Recovery (Worked Example)

An illustrative DR plan populated for the five most common core IT services: identity, email, file services, network, and endpoint security (EDR). Serves as a reference for agency IT teams populating their own runbooks using the template (GRC-FRM-DR-01). Not authoritative for any specific agency — agency teams populate their own runbooks against their actual estate.

5 core services Illustrative Tier-1 / Tier-2 examples
Purpose & how to read this

This page is a worked example, not an authoritative plan. It shows what a populated DR plan looks like, at the level of detail expected for a Tier-1 or Tier-2 IT service, so that agency IT teams have a concrete model to follow when populating their own runbooks.

The five services covered are deliberately the ones most agencies operate in some form: identity, email, file services, network, and EDR. Each is documented at illustrative depth: tier classification, RTO/RPO, dependencies, backup & replication design, recovery procedure summary, validation steps, and where the full runbook lives.

What this is not. It is not a substitute for an agency populating its own runbook against its actual configuration. Vendor versions, named individuals, network details, and tenant identifiers are illustrative and must not be used as-is. It is not an attempt to be comprehensive about every variation; agencies on different cloud configurations, with different vendors, or with different recovery designs will deviate from this and that is expected.

How to use this. When populating your own runbook using GRC-FRM-DR-01, look at the corresponding service in this worked example to see the level of detail expected and the structure of the operational thinking.

Service 1 — Identity service (Entra ID + hybrid AD)
Identity service TIER 1 Owner: Identity & Access Engineering Lead
RTO
≤ 4 hours
RPO
≤ 15 minutes
Hosting
Hybrid — Entra + on-prem AD
Last exercise
Q4 2025 (tabletop)
Why this tier

Identity is the foundation that virtually every dependent service relies on. Authentication failure causes cascading failure of any tenant-resident service, including Tier-1 clinical and educational systems. Tier-1 with stringent RPO is essential because identity data (group memberships, role assignments, conditional access state) drives security posture and inconsistency between regions is a security risk.

Critical dependencies
  • Entra ID tenant (Microsoft-operated; multi-region by design)
  • On-prem AD domain controllers (per agency; 2× per site minimum)
  • Entra Connect synchronisation servers (2× for HA, active/staging pattern)
  • Federation infrastructure where deployed (AD FS or third-party); pass-through authentication agents on member servers
  • FIDO2 security keys for break-glass and high-privilege accounts
Backup & replication design
  • Entra ID: Microsoft operates with cross-region replication. No tenant-side backup is required for Entra directory data; Microsoft soft-delete retains deleted users 30 days. Caveat: tenant-level configuration (Conditional Access policies, application registrations, custom roles) is the tenant’s responsibility — configuration export per IT-CL-WI-01 on a weekly cadence to immutable storage.
  • On-prem AD: per IT-SRV-WI-DC-01: system state backup nightly with 30-day retention; weekly full server image; quarterly authoritative restore test on isolated test forest; AD recycle bin enabled.
  • Entra Connect: configuration backed up weekly to immutable storage; staging server maintained in standby; staging server promotion procedure documented.
  • FIDO2 keys for break-glass: 2 keys per break-glass account, held in physical vaults at separate locations.
Recovery procedure summary
  • If Entra tenant lockout → follow GRC-DR-WI-02 Scenario 3. Break-glass account engaged from vault, lockout cause remediated, break-glass stood down with credential rotation.
  • If on-prem AD failure at a site → promote remaining DC at the site; if all site DCs lost, point clients at DCs at next-nearest site; if site is isolated, follow IT-SRV-WI-DC-01 AD recovery procedure for the site.
  • If forest-level AD failure → authoritative restore from system-state backup to a clean DC; sysvol re-initialised; replication to other DCs allowed to flow.
  • If Entra Connect failure → promote staging server using documented promotion procedure; verify sync direction matches design; verify selected attributes flow.
  • If tenant compromise → follow GRC-DR-WI-02 Scenario 4; SOC owns response; DR coordinates clean-up or clean-tenant decision.
Validation
  • Authentication: sign in to Azure portal, M365 portal, a tenant-resident application; sign in with on-prem account where federated.
  • Sync health: Entra Connect Health dashboard clean; sample new on-prem account appears in Entra within sync window.
  • Privileged operations: PIM activation works; Conditional Access enforces as expected; break-glass excluded from CA verified.
  • SOC alerting: identity-related SIEM events arrive; honeytoken sign-ins generate alerts.
Full runbook

The agency-specific identity runbook held in the IT operations document store, classified as Tier-0 sensitive. Offline copy held by the IT DR Coordinator and the Identity & Access Engineering Lead per DR policy §3.3.

Service 2 — Email (Exchange Online)
Email service TIER 2 Owner: M365 Tenant Administrator (per agency)
RTO
≤ 24 hours
RPO
≤ 4 hours
Hosting
Exchange Online (M365)
Last exercise
Q3 2025 (tabletop)
Why this tier

Email is business-critical for most agency communication and approval workflows, but not safety-critical. Most processes have a documented tolerance window for delayed email of up to a working day. Outlook cached mode provides continuity of cached content during outages of moderate duration.

Critical dependencies
  • Exchange Online (Microsoft-operated; multi-region by design)
  • Identity service (Tier 1 dependency; see Service 1)
  • DNS for MX records, SPF, DKIM, DMARC
  • Outlook clients in cached mode on managed endpoints
  • Third-party email security gateway where deployed in front of Exchange Online
Backup & replication design
  • Mailbox data: Exchange Online native retention with Litigation Hold or In-Place Hold on Tier-0/1 user mailboxes (executive, legal, HR, finance) — retains all mail including deleted, indefinitely or per policy.
  • Independent backup: third-party mailbox backup product (e.g. Veeam for M365, AvePoint, Acronis) for ransomware / catastrophic data scenarios — daily incrementals, immutable retention 30 days; one full export quarterly to immutable storage.
  • DNS records: zone configuration backed up weekly; DNS provider supports point-in-time recovery for short window.
  • Tenant configuration: Exchange Online configuration export (transport rules, anti-spam policies, mailflow connectors) weekly to immutable storage per IT-CL-WI-02.
Recovery procedure summary
  • If Exchange Online outage → follow GRC-DR-WI-02 Scenario 2. Communication fallback activated (SMS or alternative platform per agency BCP). Cached mailbox content remains available to users with Outlook cached mode.
  • If transport configuration broken → restore from configuration backup; if recent change is suspected, roll back; mailflow validation via test mail flow.
  • If mailbox-level loss (deleted user mailbox, ransomware-corrupted mailbox) → restore from retention hold within Microsoft retention window; or restore from independent backup if outside Microsoft retention; or restore from quarterly export for catastrophic case.
  • If DNS record loss / hijack → restore MX and authentication records from backup; coordinate with DNS provider for cache flush; SPF / DKIM / DMARC alignment validated post-restore.
  • If tenant-wide event (lockout, compromise) → identity recovery (Service 1) is the foundation; email recovery follows.
Validation
  • Send and receive internal mail; send and receive external mail (to / from a known external mailbox under our control).
  • Verify mail headers show expected authentication results (SPF pass, DKIM signed, DMARC aligned).
  • Verify mailbox content present; sample older mail accessible; deleted-items recovery works.
  • Verify SOC alerting on suspicious mailflow (test alert).
Full runbook

The agency-specific email runbook held in the IT operations document store. References IT-CL-WI-02 M365 tenant hardening for baseline configuration.

Service 3 — File services (SharePoint / OneDrive)
File services TIER 2 Owner: M365 Tenant Administrator (per agency)
RTO
≤ 24 hours
RPO
≤ 4 hours
Hosting
SharePoint / OneDrive (M365)
Last exercise
Q1 2026 (walkthrough)
Why this tier

File services support document-based business processes across most agencies. Most processes tolerate short outages through OneDrive sync cached content and local caches. Exception: where a specific SharePoint site supports a Tier-1 process (e.g. a clinical reference repository), that site inherits the higher tier and is handled separately.

Critical dependencies
  • SharePoint Online + OneDrive for Business (Microsoft-operated)
  • Identity service (Tier 1 dependency)
  • OneDrive sync client on managed endpoints (cached content during outage)
  • Sensitivity labels and DLP policies (per IT-CL-WI-02)
Backup & replication design
  • SharePoint sites and OneDrive content: Microsoft native retention (recycle bin 93 days; version history default 500 versions); preservation hold on Tier-0/1 content as needed.
  • Independent backup: third-party M365 backup product covering SharePoint and OneDrive — daily, immutable 30-day retention; one full export quarterly to immutable storage.
  • Site configuration: site collection structure and permissions inventory exported weekly to immutable storage.
  • Highly sensitive Tier-1 content (where present): synchronised to an alternate location (separate tenant or on-prem) on a per-site basis where business case warrants.
Recovery procedure summary
  • If SharePoint / OneDrive outage → follow GRC-DR-WI-02 Scenario 2. OneDrive cache provides continuity for synced content; users continue working locally; sync resolves when service returns.
  • If site-level data loss (deletion, ransomware on cloud-resident files) → restore from version history first (smallest blast radius); from recycle bin if within window; from third-party backup if outside Microsoft retention; from quarterly export for catastrophic case.
  • If site configuration loss (permissions corruption, accidental site deletion) → restore from configuration backup; deleted-site recovery within Microsoft retention window.
  • If sync conflicts after extended outage → OneDrive conflict resolution UI; communicate to users that conflicting copies need manual reconciliation.
Validation
  • Open and edit a SharePoint document; save; verify version increments.
  • Upload to OneDrive; verify sync to a second device.
  • Permissions sample: verify a known restricted document is inaccessible to a non-member account.
  • Sensitivity labels apply on save (where mandatory labelling enabled at A3/E3+).

The agency-specific file services runbook held in the IT operations document store. Tier-1 site-level runbooks (where a specific site supports a Tier-1 process) maintained separately by the site owner.

Service 4 — Network connectivity
Network connectivity (LAN, WAN, internet egress) TIER 1 Owner: Network Engineering Lead (per region)
RTO
≤ 4 hours
RPO
N/A (config-based)
Hosting
On-prem + ISP / cloud
Last exercise
Q4 2025 (live failover)
Why this tier

Network connectivity is foundational. Loss of network is loss of access to all dependent services regardless of where they are hosted. The RTO mirrors the highest-tier dependent service per the inheritance rule in WI-01 step 3. Sites supporting Tier-1 clinical or beneficiary-safety processes may require tighter RTOs than 4 hours and are treated per site.

RPO is not the principal metric for network; configuration backup ensures rapid rebuild but the data flowing through the network is the responsibility of the dependent services. The relevant measure is mean-time-to-restore.

Critical dependencies
  • Core switches, distribution, and access switches per site
  • Routers and WAN edge devices
  • Internet egress firewalls (primary + HA pair)
  • ISP / WAN provider (primary + secondary where dual-homed)
  • Wireless controllers and access points where wireless is in scope
  • DNS internal resolvers
  • Out-of-band management infrastructure (for recovery)
Backup & replication design
  • Configuration backups: daily automated config backup for Tier-0/1 devices, weekly for Tier-2/3, per IT-NET-WI-06. 90-day retention for Tier-0/1, 30-day for Tier-2/3. Annual restore test.
  • Golden configurations: per device class, versioned, stored in immutable repository.
  • Firmware / OS images: current-and-previous images held locally for rapid recovery without external dependency.
  • HA pairs: firewalls, core switches in HA pairs with annual failover test.
  • Hot spares: critical low-volume devices (core switches, edge firewalls) held as hot spare at the site or at a regional spare location.
  • WAN redundancy: dual ISP / dual-path where business case warrants (typically site-level decision based on dependent processes).
Recovery procedure summary
  • If single device failure with HA partner healthy → HA partner takes over; replace failed device using hot spare or RMA; restore from config backup; rejoin HA pair; failover-back validated.
  • If device failure without HA (Tier-2/3 only) → replace device; restore configuration from backup to the new hardware; validate connectivity; restart dependent services.
  • If site-level network failure → assess scope (single device cascade vs broader); engage hot-spare hardware; out-of-band management used to restore basic connectivity to staff at site; full restore per device-class procedure.
  • If WAN failure → secondary ISP / path activated where designed; ISP escalation for restoration; BCP for site activates if outage extends beyond business tolerance.
  • If ransomware or destructive event affecting network management → re-establish from golden configs to known-clean hardware; do not restore configs that may be tampered; SOC engaged for compromise scope assessment.
Validation
  • Internal connectivity: site-to-site reachability where designed; sample VLAN reachability; default-gateway responsiveness.
  • Egress: external connectivity from a sample endpoint; DNS resolution; HTTPS to a known internet endpoint.
  • Wireless: client association; authentication via 802.1X; expected VLAN assignment.
  • Firewall: known-good test traffic permitted; known-bad blocked; logs flowing to SIEM.
  • HA pair: confirm failover state correct; planned failover test if change scope warrants.
Full runbook

The agency-specific network runbooks held per site or per region in the IT operations document store. Device-class procedures reference IT-NET-WI-01 through IT-NET-WI-06. Offline copy of the WAN topology and emergency contact list maintained by the regional Network Lead per DR policy §3.3.

Service 5 — Endpoint Detection & Response
EDR / MDR (CrowdStrike + Obrela) TIER 2 Owner: SOC Lead (CISO function)
RTO
≤ 24 hours
RPO
≤ 4 hours (telemetry gap)
Hosting
SaaS (vendor-operated)
Last exercise
Q4 2025 (simulation)
Why this tier

EDR is not directly user-facing but loss of EDR creates a window of reduced visibility that increases security risk. Most agencies can tolerate up to a day of degraded detection capability with compensating measures (heightened SOC manual monitoring, restricted high-risk activity); longer outages elevate risk materially.

The MDR partnership with Obrela means SOC capability degrades but does not stop entirely on EDR-side outage — Obrela retains visibility into prior telemetry and can advise on compensating measures.

Critical dependencies
  • CrowdStrike Falcon platform (vendor-operated SaaS)
  • Falcon sensor deployed on all in-scope endpoints (Windows servers, Windows workstations, macOS, Linux per scope)
  • Obrela MDR tenant integration with CrowdStrike
  • Identity service (Tier 1 dependency for SOC analyst access to platforms)
  • Internet egress from endpoints for telemetry forwarding
  • SIEM platform receiving Falcon alerts
Backup & replication design
  • Telemetry: vendor-operated; vendor retains per contract (Falcon retains 30–90 days depending on subscription); Obrela retains under their MDR contract.
  • Configuration: prevention policies, sensor groups, detection rules exported weekly to immutable storage per IT-CL-WI-05.
  • API credentials: rotated annually per IT-ID-WI-04; backup credentials available for emergency restoration.
  • Custom detections / hunts: versioned in the SOC content repository; restorable to the platform.
Recovery procedure summary
  • If CrowdStrike platform outage → CrowdStrike status page checked; impact assessed (telemetry gap vs detection gap); Obrela notified; compensating measures invoked (heightened SOC manual monitoring of available logs; restrict high-risk activity for the duration).
  • If sensor-level issue affecting fleet (e.g. bad sensor version) → rollback via Falcon console using prior-version channel; if console inaccessible, escalate to CrowdStrike support; document gap window.
  • If individual sensor failure → per SOP-EP-01 SOC procedure for sensor health remediation.
  • If MDR partner outage → in-house SOC takes full ownership of triage; Obrela escalation; assessment of duration and impact on response capability.
  • If tenant compromise of CrowdStrike → CrowdStrike-side incident; coordination with CrowdStrike security team; consider sensor-level isolation pending assurance of platform integrity.
Validation
  • Telemetry flow: sample endpoint shows recent check-in (< 1 hour); telemetry events visible in console.
  • Detection: known-good test detection (test file or controlled action) generates an alert; SIEM receives the alert; Obrela receives the alert.
  • Response capability: isolate-host action works on a test endpoint; release-isolation works.
  • SOC console access: SOC analyst sign-in works with MFA; expected role permissions enforced.
Full runbook

Operational EDR/MDR procedures live in SOP-EP-01 (SOC Endpoint operations). The DR-specific runbook focuses on platform-level recovery and is maintained by the SOC Lead.

What this worked example illustrates
  • Tier inheritance. Identity and network inherit Tier-1 because of dependent processes; email, file services, and EDR are Tier-2 by inherited dependency.
  • RTO/RPO are concrete, not aspirational. Each service has named numbers, traceable back to BIA and to the standard tier bands.
  • Dependencies are named. Each service has an explicit list of what it depends on; the dependency map drives the recovery sequence in real events.
  • Backup is more than “does it run?”. Each service has independent backup for cyber resilience (not relying solely on vendor retention); immutable copies for ransomware protection; tested restores at the tier cadence.
  • Recovery procedure summary, not a step-by-step rebuild. The full runbook lives separately. The DR plan references it; the document inventory tracks both.
  • Validation is explicit. Each service has named validation steps; “the service is up” means the validation passes, not just that the platform health endpoint is green.
  • Cross-references to baseline documents. Each service points at the relevant baseline WI (cloud, network, identity, server). DR uses those baselines; doesn’t re-author.
