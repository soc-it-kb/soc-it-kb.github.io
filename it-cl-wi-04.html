IT-CL-WI-04: Azure Workload Patterns

Implementation guidance for the most frequent Azure workload patterns: storage accounts, Key Vault, virtual networks and NSGs, Private Endpoints, Azure WAF on Application Gateway and Front Door, SQL / Cosmos protection, virtual machines, App Service and Functions, backup and DDoS posture. Operationalises §4.4 of IT-CL-SOP-01. Tier 0 / 1 additions called out per pattern.

Prerequisites

Before applying this WI:

  • WI-01 (Tenant Identity), WI-02 (M365), and WI-03 (Azure Subscription) complete — the subscription baseline policy framework needs to be in place before tightening individual workload patterns
  • The workload's tier (Tier 0 / 1 / 2 / 3) recorded in GRC-REG-ASSET; this determines several controls in this WI (CMK encryption, soft-delete retention, HSM-backed keys, Defender for SQL/Cosmos)
  • Network design decided — Private Endpoint topology, hub-spoke or VWAN, how on-prem connectivity reaches Azure (ExpressRoute, VPN), how Azure egress is filtered (Azure Firewall, third-party NVA)
  • For Tier 0 / 1 workloads: customer-managed key (CMK) strategy decided; the Key Vault hosting CMKs is itself baselined to the highest standard in this WI
  • For workloads with public-facing web applications: decision on Azure WAF placement (Application Gateway vs Front Door); SOC awareness of the WAF logging endpoint
This WI addresses common Azure workload patterns, not every possible service. The patterns covered here are the most frequent attack surfaces in Azure: storage, key vault, networking, public-facing apps, databases, virtual machines, and App Service. Workloads using less common services (AKS, Service Fabric, HDInsight, Synapse) follow the same principles but require service-specific baselines that are deferred to future extensions.
Step-by-step instructions

The steps below cover the Azure workload patterns required by IT-CL-SOP-01 §4.4. They apply to every Azure workload within an in-scope subscription, with Tier 0/1 additions called out per step.

1
Storage account baseline
Public exposure eliminated by default; encryption strong; data immutable or soft-deletable where compliance requires.

Storage accounts are the most common cloud exposure in any estate. Default permissive settings have historically caused major data leaks. Tighten universally:

  • Secure transfer required — HTTPS only; storageAccountProperties.supportsHttpsTrafficOnly = true; enforced via Azure Policy (WI-03 Step 3)
  • Public blob anonymous access disabled at the storage account level (allowBlobPublicAccess = false) — this prevents containers being set to public-anonymous regardless of container-level setting
  • Minimum TLS 1.2 — minimumTlsVersion = TLS1_2; TLS 1.0 / 1.1 disabled
  • Network access controls:
    • Default action Deny
    • Allow access only via Private Endpoint (preferred) or specific service endpoint / firewall allow-list
    • "Allow trusted Microsoft services" enabled where the storage account is read by other Azure services (Backup, Monitor, Defender)
  • Encryption at rest — AES-256 always (platform default). For Tier 0 / 1: customer-managed keys (CMK) using a key in Key Vault Premium or Managed HSM
  • Soft-delete for blobs and containers — ≥7 days; for Tier 0/1: 30–90 days
  • Immutable storage for compliance-relevant data — legal-hold or time-based policies prevent deletion / modification within retention window
  • Cross-tenant replication disabled unless explicitly required (default-off in newer Azure regions; verify)
  • Shared Key authorization — consider disabling and using Entra-only authentication for storage account access (limits the blast radius if a storage account key is leaked)
  • Diagnostic logging — StorageRead, StorageWrite, StorageDelete to Log Analytics + Event Hub (per WI-03 Step 4); without these, blob-level access is invisible
Verify. Test: create blob in test storage account, attempt anonymous public read; refused. Test: HTTP request to storage endpoint; rejected. Network restrictions confirmed via Defender for Cloud recommendations.
CIS Azure 3.x · MCSB DP-x
2
Key Vault baseline
Key Vault is treated as a Tier 0 asset everywhere; soft-delete and purge protection are non-negotiable; network restricted.

Key Vault holds secrets, keys, and certificates — everything else's trust ultimately depends on it. Treat every Key Vault as Tier 0 even when the workload it supports is lower-tier.

  • RBAC mode (preferred over access policies) — Key Vault has two access models; RBAC is the modern model and is easier to audit and integrate with PIM
  • Soft-delete enabled — non-disablable in new Key Vaults (Microsoft enforces it); retention 7–90 days configurable
  • Purge protection enabled — once enabled, cannot be disabled; soft-deleted vaults / keys cannot be purged before retention expiry. This is the single most important Key Vault setting.
  • Network access — default-deny with Private Endpoint preferred; trusted Microsoft services allowed only where the dependency is genuine
  • Diagnostic logging — AuditEvent category to Log Analytics + Event Hub; without this, key access is invisible
  • Key types — software-protected keys for Tier 2/3 workloads; HSM-backed keys (Premium tier or Managed HSM) for Tier 0/1 production cryptography
  • Key rotation — automatic rotation policies for all keys (typical: 1–2 years); rotation tested annually; previous key version retained for grace period
  • Key Vault firewall — if Private Endpoint is not used, restrict to specific virtual networks and IP ranges; never "Allow access from all networks" in production
  • Backup — Key Vault backup procedure documented; cross-region or cross-subscription depending on DR requirement
  • Access by managed identities preferred over service principals or app-secret patterns; ties access to a resource lifecycle
If you do nothing else in this step, enable Purge Protection on every Key Vault. Without it, a malicious or accidental "delete and purge" can permanently lose keys, which can permanently lose the data they encrypt. With purge protection, every deletion waits out the retention window and can be recovered.
Verify. Soft-delete and purge protection enabled on every Key Vault. Network restrictions in place. Test: attempt to disable purge protection; refused. Diagnostic logs visible in Log Analytics.
CIS Azure 8.x · MCSB DP-x · "Key Vault"
3
Virtual network baseline (NSG, flow logs)
Every subnet has explicit allow rules with default-deny inbound; flow logs feed central SIEM for network forensics.

Network Security Groups (NSGs) are subnet-level (and NIC-level) firewalls. They are the primary segmentation control inside an Azure subscription.

  • NSG on every subnet — never leave a subnet without an NSG. The platform allows it; the baseline does not.
  • Default-deny inbound at the subnet NSG. Azure's default rules allow VirtualNetwork-to-VirtualNetwork; tighten with explicit allows for the services that actually need cross-subnet communication.
  • No 0.0.0.0/0 to sensitive ports — RDP (3389), SSH (22), WinRM, database ports (1433, 3306, 5432, etc.), management endpoints. Public access to these requires explicit JIT VM access (Defender for Servers) or documented exception.
  • Application Security Groups (ASGs) for workload-internal segmentation — reference VMs by role tag rather than IP; NSG rules use ASG references; surviving IP changes
  • NSG flow logs — enabled on every NSG (Network Watcher); Version 2 (with metadata); routed to Log Analytics workspace; analytics enabled for Traffic Analytics (or equivalent in central SIEM)
  • Diagnostic settings on NSGs — NetworkSecurityGroupEvent and NetworkSecurityGroupRuleCounter categories
  • Service Tags — use Microsoft-managed service tags (e.g. AzureCloud, Sql, Storage) for outbound rules instead of hardcoded IP ranges; the tag updates automatically as Microsoft IP ranges evolve
  • Network Watcher — enabled per region (it is by default in newer subscriptions; verify); supports connectivity diagnostics
  • Azure Firewall or third-party NVA for egress filtering on Tier 0/1 subscriptions — not a strict baseline requirement, but recommended; documented in cloud architecture rather than this baseline
Verify. Every subnet has an NSG. Sample NSG shows explicit allow rules + default-deny inbound. Flow logs flowing to Log Analytics; Traffic Analytics dashboard populated.
CIS Azure 6.x · "Network Security Groups"
4
Prefer Private Endpoints over public exposure for PaaS data services
Data-plane access to PaaS services happens via private network paths; public endpoints disabled wherever feasible.

Most PaaS data services (Storage, SQL Database, Cosmos DB, Key Vault, Service Bus, Event Hubs) default to public endpoint exposure. Private Endpoint gives each instance a private IP inside the virtual network and disables the public endpoint.

  • Default pattern: PaaS data services accessed via Private Endpoint; public endpoint disabled. Network access on the resource set to "Disabled" or restricted-to-Private-Endpoint.
  • Service Endpoints as fallback — older mechanism; the PaaS endpoint stays public but only VNet-routed traffic is allowed. Less secure than Private Endpoint (the public endpoint still exists). Acceptable only where Private Endpoint is not supported.
  • Private DNS zones — required for Private Endpoint to work correctly; the private DNS zone for the PaaS resource type (e.g. privatelink.blob.core.windows.net) is linked to the VNet; without DNS resolution, the Private Endpoint is invisible to consumers
  • DNS architecture — centralised in a hub VNet for hub-spoke topologies; each Private Endpoint registers in the central zones; spokes inherit via VNet link
  • Network access on the PaaS resource — even with Private Endpoint, the resource may still allow public access. Disable explicitly.
  • Enforced via Azure Policy — built-in policies "Storage accounts should restrict network access," "Key Vault should use a Private Endpoint," etc., assigned at management group scope; effect: Audit initially, Deny after estate remediated
Verify. Sample Storage / Key Vault / SQL has Private Endpoint configured; public endpoint disabled. Test resolution: from a VM in the VNet, nslookup resolves to the private IP. Defender for Cloud recommendation "should use private link" cleared.
CIS Azure 3.x, 4.x, 8.x · MCSB NS-x · "Private Endpoint"
5
Azure WAF on Application Gateway or Front Door Tier 0 / 1 apps
Public-facing web apps protected by managed rule set in Prevention mode; logs visible to SOC.

This step absorbs the Azure-native cloud-WAF scope that was deferred from IT-SI-WI-02 (the on-prem / vendor WAF baseline). Azure WAF is offered on Application Gateway (regional, scoped to one app per gateway typically) and Front Door (global, multi-app, anycast); both share the same managed rule set engine.

  • Placement — in front of every public-facing web application for Tier 0/1 apps; for Tier 2/3 apps where the workload type justifies it
  • WAF policy attached — policy lives separately from the gateway/Front Door, referenced from the listener / route; one policy can apply across multiple endpoints
  • Mode:
    • Initial deployment: Detection mode for 2–4 weeks; review logs for false positives
    • After tuning: Prevention mode within 90 days of go-live for Tier 0/1 apps
    • Detection-mode forever is a non-control; track the Detection → Prevention transition explicitly
  • Managed rule set — Azure's managed rule set (DRS — Default Rule Set; current version) covers the OWASP Top 10 patterns: SQL injection, XSS, command injection, LFI / RFI, protocol anomalies, scanner detection. Enable as the baseline.
  • Bot Manager rules — Azure WAF bot protection rule set; enable to filter known-bad bot traffic; review false positives carefully on apps with legitimate scraper / monitoring traffic
  • Custom rules for app-specific patterns:
    • Geo-restriction (block sign-in attempts from non-business regions)
    • Rate limiting per source IP (mitigate credential stuffing, scraping)
    • Path-specific bypass for legitimate scenarios where the managed rule set is over-triggering
  • Logging — WAF access log and firewall log to Log Analytics + Event Hub; sample dashboards (top blocked source IPs, top triggered rules, false-positive candidates); SIEM correlation with sign-in events
  • Tuning discipline — named owner per WAF policy; documented exception process; quarterly review of triggered rule trends; do not blanket-disable rules — tune narrowly
  • TLS — minimum TLS 1.2 on the listener; HTTPS-only redirects from HTTP; managed certificates (App Gateway / Front Door integration with Key Vault) for automatic renewal
Verify. WAF policy attached to every public-facing Tier 0/1 web app. Mode: Detection initially → Prevention after tuning. Logs flowing to SIEM. Test: send a known-malicious pattern (controlled test like an SQL-injection-shaped query string) — blocked in Prevention mode, logged in Detection mode. Tuning ownership documented.
CIS Azure 7.x · MCSB NS-x · "Azure WAF"
6
SQL Database and Cosmos DB protection
Database services run with native security features enabled; sensitive data encrypted with CMK where tier justifies.

Database services are high-value targets. Several built-in security features need to be enabled rather than left at default-off.

  • Azure SQL Database / Managed Instance:
    • Entra authentication enabled; SQL authentication used only where unavoidable; Entra admin assigned (group preferred over individual)
    • Transparent Data Encryption (TDE) always-on; CMK with Key Vault for Tier 0/1
    • Defender for SQL enabled (vulnerability assessment + advanced threat protection) — for Tier 0/1; recommended for any sensitive data
    • Auditing to Log Analytics + Event Hub; retention per data classification
    • Server-level firewall: deny by default; named IPs only; Private Endpoint preferred per Step 4
    • Vulnerability assessment baseline run quarterly; findings tracked
    • Always Encrypted for highly sensitive columns (specific data subject to extra controls)
  • Cosmos DB:
    • Entra authentication preferred over key-based; data-plane RBAC enabled
    • Defender for Cosmos DB enabled for Tier 0/1
    • Network restrictions: Private Endpoint or VNet service endpoint; public access disabled
    • Diagnostic logging: DataPlaneRequests + control plane to Log Analytics + Event Hub
    • Continuous backup with PITR (Point-in-Time Restore) for Tier 0/1; 7–30 day window
    • CMK encryption for Tier 0/1
  • Other database services (Azure Database for PostgreSQL / MySQL / MariaDB) follow analogous patterns: Entra auth where supported, SSL enforcement, Defender plan for the database type, network restrictions, diagnostic logging
Verify. Sample SQL Database has TDE on with CMK (Tier 0/1); Defender for SQL enabled. Sample Cosmos has Entra auth and Defender enabled. Test: SQL injection pattern caught by Defender; alert visible.
CIS Azure 4.x · MCSB DP-x · "Defender for SQL"
7
Virtual machine baseline
VMs onboarded to Defender for Servers; managed identities preferred over service principals; JIT VM access where exposed.

Azure VMs combine all the usual server-hardening concerns (covered by the endpoint / server baseline domain in asm-security-baselines.html) with cloud-specific patterns. This step covers the Azure-specific additions; server-internal hardening remains under the server baseline domain.

  • Defender for Servers — P2 plan for Tier 0/1, P1 acceptable for Tier 2/3 with justification; onboarding auto-provisioned via DfC (WI-03 Step 2); EDR integration with Defender for Endpoint (WI-05)
  • Just-in-Time (JIT) VM access — for VMs that require occasional inbound management access (RDP/SSH), JIT replaces standing inbound NSG rules with time-bounded, source-IP-bounded approval-based access; default deny inbound on management ports otherwise
  • Managed identities — system-assigned or user-assigned, used for VM-to-Azure-service authentication; eliminates secrets-in-VM-config
  • Disk encryption — Azure Disk Encryption (BitLocker / dm-crypt) or Server-Side Encryption with CMK; the latter is simpler and equivalent for most use cases; Tier 0/1 = CMK
  • Trusted Launch (Gen 2 VMs) — vTPM, secure boot, boot integrity monitoring; enable for new VMs where the image supports it
  • VM extension control — Azure Policy restricting which VM extensions can be installed; an attacker with Contributor on a VM can install arbitrary extensions, so restrict at policy level
  • Guest configuration / Azure Automanage — assign the baseline guest-configuration policies to verify in-guest hardening continuously; integrates with the server baseline domain
  • Update Management — Azure Update Manager for patch deployment per IT-PM-SOP-01; non-conformant VMs flagged in DfC
Verify. Sample VMs show Defender for Servers active, JIT configured for management access, managed identity assigned, CMK on disk. Defender recommendations cleared.
CIS Azure 7.x · "Defender for Servers" · "JIT"
8
App Service and Functions baseline
PaaS app hosting follows secure defaults: HTTPS-only, managed identity, authentication where the app doesn't handle its own, diagnostic logging on.

Azure App Service and Azure Functions are popular workload hosts. They have several "off by default" settings that need to be on by default for any production workload.

  • HTTPS only — httpsOnly = true; HTTP redirected to HTTPS
  • Minimum TLS version — TLS 1.2 minimum
  • FTP / FTPS deployment — disabled unless deliberately needed; deployment via Azure DevOps / GitHub Actions / Bicep + service principal preferred
  • Authentication / authorisation — for apps that don't implement their own, enable Easy Auth (Entra ID integrated identity provider); for apps with their own, ensure they validate tokens correctly
  • Managed identity for connections to Key Vault, Storage, SQL — no connection strings with secrets in app config
  • Key Vault reference syntax for any app-config setting that holds a secret — @Microsoft.KeyVault(...) — centralises secret management and rotation
  • Network restrictions — access restrictions to specific IP ranges / VNets; or VNet integration with private endpoint for inbound
  • Defender for App Service — enabled for Tier 0/1 subscriptions per WI-03 Step 2
  • Diagnostic logging — AppServiceHTTPLogs, AppServiceConsoleLogs, AppServiceAppLogs to Log Analytics + Event Hub
  • Slot deployment — production slot protected by Azure Policy from direct deployment; deployments go to staging slot then swap; audit-trail and rollback in one step
  • For Functions specifically — host keys rotated periodically; consider Function authentication via Entra integration rather than function keys for higher-security functions
Verify. Sample App Service shows HTTPS-only, managed identity, Easy Auth where applicable, diagnostic logging on. Defender for App Service recommendations cleared.
CIS Azure 9.x · "App Service" · MCSB DP-x
9
Backup and recovery posture
Production workloads have tested backups; recovery scope and RPO/RTO match the asset tier.

Backup is operational hygiene that is also a security control — ransomware recovery depends entirely on backup integrity.

  • Azure Backup or service-native backup per workload type: VMs → Azure Backup; SQL → SQL automated backups + LTR (long-term retention); Cosmos → continuous backup; Storage → soft-delete + versioning + (where critical) Azure Backup for Blob
  • Recovery Services Vault — in a different region or different subscription from the workload it protects; cross-region for DR scenarios
  • Soft-delete on the vault — recovery points cannot be deleted within retention (14-day minimum); aligns with "immutable backups" pattern
  • Multi-user authorisation (MUA) on the vault — high-risk operations (vault delete, retention reduction, soft-delete disable) require a second approver
  • RPO/RTO per tier — documented in the asset record; Tier 0/1 typically RPO ≤ 24h, RTO ≤ 4h; Tier 2/3 longer windows acceptable
  • Restore testing — quarterly for Tier 0/1; annual for Tier 2/3; documented test results retained as audit evidence
  • Backup integration with SIEM — backup failures and unusual restore activity (especially restore to non-original RG) alerted
Verify. Backup policies configured per tier; vault soft-delete and MUA enabled. Restore test completed within the cadence per tier; evidence retained.
MCSB BR-x · "Azure Backup"
10
DDoS protection on public-facing workloads
Public IPs serving Tier 0/1 workloads are behind Azure DDoS Protection; defence-in-depth with WAF.

Azure provides DDoS protection at two levels — basic (free, network-layer, on every public IP) and standard (paid, with application-layer telemetry, mitigation tuning, attack notifications).

  • DDoS Protection Standard on the VNets containing Tier 0/1 public-facing resources — covers all public IPs in the VNet for a flat-rate per protected VNet
  • DDoS Protection Plans reusable across VNets in the same subscription / scope — the cost is per plan, not per VNet, when plan is attached to multiple
  • Telemetry — DDoS metrics in Log Analytics; alerts on mitigation events; SOC playbook for DDoS response (correlation with WAF logs and app availability)
  • Coverage — Application Gateway with WAF, Front Door, exposed VMs, Public IPs on load balancers, Bastion external endpoints
  • Tier 2/3 — DDoS Basic acceptable; consider Standard where DDoS-risk profile justifies the cost
Verify. DDoS Protection Standard attached to VNets containing Tier 0/1 public IPs. Telemetry visible. DDoS-event alert chain tested.
MCSB NS-x · "DDoS Protection"
Validation & verification

After applying these steps to a workload, validate.

Functional validation

  • Storage accounts deny public anonymous access; HTTPS-only; minimum TLS 1.2; CMK on Tier 0/1; soft-delete configured
  • Key Vault: soft-delete + purge protection enabled and not disablable; Private Endpoint; RBAC mode; diagnostic logging on
  • Subnets all have NSGs; default-deny inbound; flow logs to Log Analytics
  • Private Endpoints in place for PaaS data services; private DNS resolves correctly; public endpoints disabled
  • Azure WAF on public-facing Tier 0/1 apps; Prevention mode after tuning; logs in SIEM
  • SQL: TDE on with CMK (T0/1); Defender for SQL; Entra auth; auditing on
  • Cosmos: Entra auth where supported; Defender for Cosmos (T0/1); Private Endpoint
  • VMs: Defender for Servers; JIT for management; managed identity; disk CMK (T0/1)
  • App Service: HTTPS-only; managed identity; Easy Auth or app-validated tokens; diagnostic logging
  • Backup: configured per tier; vault soft-delete and MUA on; restore tested per cadence
  • DDoS Protection Standard on Tier 0/1 public VNets

Rollback & troubleshooting

Workload-level changes are typically isolated to the workload but can have surprising blast-radius (a network restriction can break a dependency two services away). Plan rollback per change.

Common scenarios

  • Private Endpoint deployed without DNS update — consumers still resolve to the public endpoint, get network-denied. Fix: confirm private DNS zone exists, VNet link configured, A record created. Test resolution from a VM in the VNet.
  • WAF Prevention mode blocking legitimate traffic — cause: a managed rule triggers on legitimate app behaviour. Fix: switch back to Detection; tune the specific rule (exclusion by URI pattern or argument); re-enable Prevention. Do not blanket-disable the rule.
  • NSG default-deny breaking an inter-subnet flow — cause: a previously-implicit VNet-to-VNet allow was relied on; explicit allow not added. Fix: add the explicit allow rule with documented justification; verify minimum necessary scope.
  • Key Vault purge protection blocks restoring a deleted vault to a new name — this is the intended behaviour. The vault name is reserved during soft-delete period. Fix: wait out soft-delete period, OR recover the soft-deleted vault to original name, OR use a new vault name.
  • JIT VM access requesting failed — cause: NSG rules don't reach the JIT-managed rules due to user-defined rule precedence. Fix: review NSG rule priorities; JIT-managed rules use specific priority range (typically 100–110); leave that range free.
  • App Service managed identity not authorising to Key Vault — cause: RBAC role assignment missing on the Key Vault for the managed identity object ID. Fix: assign "Key Vault Secrets User" (or appropriate role) to the managed identity at the vault scope; allow a few minutes for replication.
  • SQL Defender vulnerability scan failing — cause: storage account for scan results not configured or not accessible. Fix: configure the SQL Defender storage account; ensure SQL Managed Identity has Storage Blob Data Contributor on it.
  • Backup operation failed — cause: VSS errors on Windows VM, agent issue, network connectivity to vault. Fix: per the Azure Backup error guide; verify agent health; retry; if pattern persists, snapshot-consistent backup may fail back to crash-consistent (acceptable for many workloads with documentation).

Rollback procedure

  1. Identify the breaking change (recent change record, Activity Log, application telemetry)
  2. Revert the specific config — do not blanket-roll-back the whole step
  3. Test connectivity / function from the affected consumer
  4. For WAF / Defender alerts: switch to Detection / suppress alerts narrowly; tune; re-enable
  5. Update change record; document root cause; re-attempt with tuning
Common errors & tips
  • Public blob anonymous is disabled at storage-account level for a reason. Even one container set to public-anonymous becomes a long-term data-leak risk. Disable at the account level so per-container settings can't surprise you.
  • Key Vault purge protection cannot be turned off. Enable it deliberately, knowing the consequence: deleted resources cannot be permanently destroyed within the retention window. This is the desired behaviour for security; sometimes inconvenient for operations.
  • Private Endpoint without DNS is broken Private Endpoint. The most common Private Endpoint failure mode is forgetting the private DNS zone. The PE works, but consumers resolve to the public endpoint and get blocked.
  • WAF Detection mode forever is no WAF. The whole point is Prevention. The Detection → Prevention transition is a tracked milestone, not a "we'll get to it."
  • WAF managed rules need tuning. Out of the box they will block some legitimate traffic. Plan the tuning effort; budget the time. Rule exclusions are surgical, not "disable rule N."
  • NSGs and Azure Firewall are not redundant. NSGs are stateful per-subnet/NIC firewalls; Azure Firewall is a centralised L4-L7 firewall with FQDN filtering. Use both: NSG for east-west, Firewall for north-south.
  • Service tags update automatically. Don't hard-code Microsoft IP ranges in NSG rules; use service tags (AzureCloud, Storage.<region>, Sql.<region>). Microsoft re-allocates IPs; your hard-coded list will drift.
  • Managed identity is the preferred pattern, everywhere. If a workload uses a service principal with a secret, ask why; the answer should be "managed identity is not supported for this case." If the answer is "we did it this way historically," migrate.
  • App Service slot swap is a security feature too. Direct deploy to production = no audit trail of pre-swap state. Slot deploy + swap = the previous code is one swap away if rollback is needed.
  • Easy Auth doesn't validate inside the app. Easy Auth provides the identity to the app; the app still has to authorise what the identity can do. Identity != authorisation.
  • Backup soft-delete is mandatory. Ransomware that compromises Azure RBAC can delete the backups too — unless the vault has soft-delete and MUA. These convert the backup from "deletable in one action" to "deletable in two actions across two principals across two weeks."
  • DDoS Standard pricing is per-VNet, not per-IP. Plans-mode lets one paid plan cover many VNets in scope; check pricing model before assuming "this costs per public IP."
  • Private DNS zones are often the trickiest part. Hub-spoke topology + Private Endpoint = a private DNS zone in the hub, linked to each spoke VNet, with a Private DNS Resolver if on-prem also needs to resolve the names. Diagram before implementing.
