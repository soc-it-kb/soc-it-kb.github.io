EP-PB-01: Suspected Endpoint Compromise

Entry-point playbook for all endpoint-centric incidents. Serves as the escalation target from Email and Identity playbooks when host-level impact is suspected — covering investigation, containment, and evidence preservation.

Section 1
Purpose

Provide a standardised decision and response framework for identifying, investigating, and responding to suspected endpoint compromise, ensuring rapid containment, minimal business impact, and preservation of evidence.

This playbook serves as the entry point for all endpoint-centric incidents and acts as the escalation target from Email (EM-xx) and Identity (ID-xx) playbooks when host-level impact is suspected.

Section 2
Scope

✅ In scope

  • Workstations and laptops (corporate-managed)
  • Servers where EDR coverage exists
  • Endpoints implicated in phishing, credential compromise, or malware execution
  • MDR escalations and EDR detection alerts

❌ Out of scope

  • Pure identity incidents with no endpoint interaction → ID-PB-01
  • Network-only incidents → Network playbooks
  • Confirmed ransomware events → EP-PB-05
Section 3
Trigger Conditions
  • EDR raises a high-confidence malicious detection
  • MDR escalates suspected endpoint compromise
  • Endpoint involved in phishing where execution is suspected
  • Identity playbook confirms attacker sign-in with endpoint activity
  • SOC analyst observes anomalous process, network, or file behaviour
Section 4
Initial Validation (Decision Logic)

Confirm the event is endpoint-related:

  • Is there evidence of process execution on the host?
  • Is there file creation or modification?
  • Are there outbound network connections from the host?
  • Is the endpoint associated with a known or suspicious user action?

If NO to all → Route to appropriate EM or ID playbook
If YES to any → Proceed with compromise assessment (Section 5)

Section 5
Compromise Assessment

Assess the likelihood and depth of compromise. Presence of any indicator = Suspected Endpoint Compromise:

Malicious process execution detected
Suspicious child processes spawned
Unexpected outbound network connections from the host
Persistence mechanisms observed
Endpoint involved in prior phishing, MFA, or OAuth incident
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Active malicious process confirmed on the endpoint
🔴 Lateral movement suspected or confirmed
🔴 Privileged user endpoint involved
🔴 Endpoint hosts sensitive or critical data
🔴 MDR confirms active attacker presence

Notify immediately
👔
SOC Leadership
High severity awareness
💻
Endpoint / IT Ops
Containment support
🤝
MDR Partner
If applicable
Section 7
Response Strategy

Execute the following actions based on assessment findings:

Endpoint telemetry review — processes, files, networkRequired
Containment decision — isolate vs monitorRequired
Malware and process investigationRequired
Persistence checkRequired
Credential exposure assessmentConditional
Scope expansionConditional

⚠️ If encryption activity or mass file modification is detected → escalate immediately to EP-PB-05 (Ransomware Activity)

Section 9
Containment Decision Logic

🔴 Immediate isolation

  • Active malware still running on the endpoint
  • C2 communication confirmed
  • Privileged credentials in active use

🟡 Monitored containment

  • Behaviour is suspicious but unconfirmed
  • Business-critical system where isolation causes impact

All containment actions must be logged and approved per IR governance before execution.

Section 10
Scope Expansion Logic

Triggers for expansion

  • Same file hash observed on multiple endpoints
  • Network indicators shared across hosts
  • User accessed multiple systems
  • MDR indicates campaign-level activity

Expansion actions

  • Search for related hashes across EDR telemetry
  • Identify endpoints with similar process trees
  • Correlate network indicators tenant-wide
Section 11
Communication & Coordination
  • Coordinate containment with IT to minimise operational disruption
  • Inform the affected user after containment is complete, not before
  • Align all actions and timelines with the MDR partner
  • Escalate to Incident Command if scope expands beyond single endpoint
Section 12
Closure Criteria
  • Endpoint confirmed clean or fully rebuilt
  • All malicious artefacts removed
  • All persistence mechanisms eliminated
  • Credentials reset if exposure was suspected
  • No further malicious activity observed
Section 13
Evidence to Capture
EDR detection details
Process trees and command lines
File hashes and paths
Network indicators (IPs, domains)
Timeline of all actions taken
Section 14
Post-Incident Actions
  • Identify root cause (phishing, exploit, credential abuse)
  • Update endpoint detections and prevention rules
  • Feed IOCs into threat hunting queries
  • Recommend endpoint hardening actions
  • Update awareness or training materials if user-driven
