Section 1
Infrastructure Tier Model

The Infrastructure Tier Model separates the environment into distinct trust boundaries to reduce privilege escalation, lateral movement, and the blast radius during security incidents. It is the foundational concept referenced across all SOPs, WIs, checklists, runbooks, and playbooks.

📌 Core rule: Access must never flow from a lower tier to a higher tier. A Tier-2 device cannot administer Tier-0 or Tier-1 systems under any circumstances.

Tier 0Identity & Security Control Plane
Highest trust level. Systems that define identity, authentication, and trust. Total environment compromise if breached.
Includes:
Active Directory Domain Controllers
Entra ID / Azure AD tenant
ADFS, PKI, Certificate Authorities
Privileged Access Management systems
Security platforms with identity control
Expectations: Dedicated Tier-0 admin accounts only · Access from hardened workstations only · MFA enforced · No email, browsing, or productivity tools · Strict monitoring
Tier 1Server & Application Infrastructure
High business impact. Depends on Tier-0 for identity. Separate administration from Tier-0 and Tier-2.
Includes:
Application and database servers
File, print, and virtualisation platforms
Backup and recovery systems
Internal business applications
Expectations: Tier-1 admin accounts only · No admin access from Tier-2 devices · Restricted internet access · Hardened baseline · Strict patching
Tier 2End-User Computing
Largest attack surface. Most exposed to phishing and malware. Lowest internal trust level.
Includes:
User laptops and desktops
VDI user sessions
Standard user accounts
Productivity and collaboration tools
Expectations: No admin access to Tier-0 or Tier-1 · Endpoint protection (AV/EDR) · Least privilege · Regular patching and hardening
Tier 3External / Untrusted (Conceptual)
Treated as untrusted by default. Includes internet, third-party systems, BYOD, and partner environments.
Includes:
Internet and external services
Third-party and partner systems
BYOD devices
Expectations: Access only via secure gateways · Strong authentication and conditional access · Assume untrusted by default

⚠️ All operational procedures assume you know which tier you are operating in, you are using the correct account, and you are working from an approved device. If unsure — stop and re-read this page before proceeding.

Section 2
Administrative Account Types

Administrative access is tier-specific and role-specific. Accounts must never be reused across tiers. Misuse of administrative accounts is one of the most common causes of major security incidents.

Account TypeDescriptionTier scope
Standard User AccountDay-to-day user activity — email, browsing, collaboration, business applicationsTier 2 only
Privileged Admin AccountAdministrative tasks only — system administration, configuration changes, security operationsTier-specific
Tier-Scoped Admin AccountDedicated admin account per tier — Tier-0, Tier-1, or Tier-2 adminOne tier only
Service / System AccountNon-interactive automated operations by applications or servicesLeast privilege

👤 Standard User Account

Used for email, browsing, collaboration, and business applications only. No administrative privileges. No access to Tier-0 or Tier-1 systems. High exposure to phishing and malware.

🔐 Privileged Admin Account

Used exclusively for system administration, configuration changes, and security operations. Separate from standard user account. No email or internet browsing. MFA enforced. Used only when required.

🏛️ Tier-Scoped Admin Account

Dedicated admin accounts per tier. Never reused across tiers. Never logged in from lower-tier devices. Strong authentication enforced. A Tier-0 admin account must never be used on a Tier-2 device.

⚙️ Service / System Account

Used by applications or services for automated tasks. Least privilege. No interactive login. Password rotation or managed identity. Regular review with defined ownership.

Using a standard user account for admin tasks
Reusing the same admin account across tiers
Logging into Tier-0 with a Tier-1 or Tier-2 device
Using service accounts interactively

📌 Many SOPs and WIs explicitly state "use the appropriate tier-scoped admin account." If unsure which account to use — stop and consult this page.

Section 3
Privileged Access Workstations (PAW)

A PAW is a dedicated, hardened workstation used exclusively for privileged administration. PAWs reduce the risk of credential theft and lateral movement by enforcing clean separation between admin and standard user activity.

Most major breaches involve admin credentials stolen from compromised user devices, or privileged sessions running on Tier-2 systems. PAWs directly eliminate this attack path.

ActivityPAW Required
Tier-0 administration (Domain Controllers, identity systems)Yes — mandatory
Tier-1 server administrationYes
Security platform administrationYes
Standard user tasksNo

✅ Allowed on a PAW

Administrative tools and management consoles
Security platforms and monitoring tools
Required system utilities
Approved admin portals (web access)

🚫 Not allowed on a PAW

Email clients
Web browsing (except approved admin portals)
Office and productivity applications
Messaging or collaboration tools
Personal use of any kind

⚠️ Tier-0 PAWs must never access Tier-1 or Tier-2 user services. PAWs are treated as high-trust assets and must follow stricter patching and monitoring standards than standard endpoints. See IT-EP-SOP-PAW-01 and IT-EP-WI-PAW-01 through PAW-07 for implementation details.

Section 4
Tier-Based Access Rules & Boundaries

These rules define what access is permitted and prohibited between infrastructure tiers. They are enforced through technical controls, SOPs and WIs, and monitoring and audits.

📌 Core rule: Access can only flow from higher trust tiers to lower trust tiers — never the reverse.

⚠️ Tier boundaries affect patch management, incident response, change activities, and remote access solutions. If access feels "blocked" — it is usually by design.

Section 5
Tier Alignment with SOPs, WIs & Checklists

All SOPs, Work Instructions, and Checklists in this knowledge base assume tier awareness by default. Understanding how tiers shape procedures prevents security control bypass, audit findings, and increased breach impact.

📋 SOPs define what must be done

Assume correct tier, account, and device usage. Reference tier-specific constraints without prescribing which tool to use.

🔧 Work Instructions define how

Explicitly reference tier-specific constraints at each step. Call out when a Tier-0 account or PAW is required before proceeding.

✅ Checklists validate compliance

Confirm correct access paths and tools were used. Validate tier compliance as part of operational assurance.

📖 Runbooks & Playbooks reference tiers

Tier-0 incidents escalate immediately. Containment steps and recovery paths vary by tier. Tier-2 infections are treated as assumed breach.

🩹 Patch Management example

Tier-0: Controlled windows, dedicated admin access, PAW required
Tier-1: Scheduled maintenance windows, PAW usage, pre-patch validation
Tier-2: Automated patching, endpoint controls, standard admin account

🚨 Incident Response example

Tier-0 incidents escalate immediately to SOC and Identity Services
Tier-2 infections treated as assumed breach of user credentials
Containment steps, isolation methods, and rebuild procedures vary by tier

📌 If an SOP or WI feels restrictive or unclear: (1) identify the tier involved, (2) confirm the account type required, (3) verify the workstation being used, (4) re-read the relevant section on this page.

Section 6
Security Principles Used Across the Knowledge Base

These principles are assumed by default across all SOPs, WIs, runbooks, and playbooks. They drive how procedures are written and why certain controls exist.

🔐 Least Privilege

Every account and process has only the minimum access required to perform its function — nothing more

👥 Separation of Duties

No single person should have unchecked ability to perform high-impact actions without oversight or approval

💥 Assume Breach

Design controls assuming attackers are already inside the environment — limit lateral movement and blast radius

🛡️ Defence in Depth

Multiple overlapping controls — no single control is relied upon exclusively to prevent or detect an attack

🚧 Explicit Boundaries

Trust boundaries between tiers are explicit and enforced — not implicit or assumed based on network location
