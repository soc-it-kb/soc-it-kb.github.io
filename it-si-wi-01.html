IT-SI-WI-01: Common Management Plane & Logging

Vendor-agnostic implementation guidance for the shared management-plane controls that apply to WAFs, load balancers, and PAM platforms alike. Operationalises §4.1 of IT-SI-SOP-01. Referenced by WI-02, WI-03, and WI-04 as the foundation each builds on.

Prerequisites

Before applying this WI:

  • The platform (WAF / load balancer / PAM) is in Onboarding status in GRC-REG-ASSET (or, for existing platforms, an in-place hardening change is approved)
  • Out-of-band access to the platform is available in case in-band access is lost during configuration (vendor console, dedicated OOB interface, or platform-cloud-management)
  • Current configuration is backed up to the central config repository
  • A change window is approved per organisation change-control procedures
  • The central AAA service (TACACS+, RADIUS, SAML provider, or Entra ID) is reachable and the platform is provisioned in the AAA service
  • The central syslog destinations, NTP servers, and authorised-use banner text are known
  • The management network (or admin allow-list) for the platform is defined; for PAM, a dedicated management network separated from data-plane is mandatory
  • The organisation CA can issue a certificate for the platform's management interface
This WI is the shared foundation for WI-02, WI-03, and WI-04. Each device-type WI assumes management-plane hardening from this WI is in place before its device-specific controls are applied. Apply this WI first, then proceed to the relevant device-type WI.
Step-by-step instructions

The steps below cover the shared management-plane controls required by IT-SI-SOP-01 §4.1. They apply to WAFs, load balancers, and PAM platforms alike, with platform-specific notes called out where they matter.

1
Disable plaintext management protocols
Eliminate cleartext credential and configuration exposure on the wire.

On every in-scope platform, disable:

  • Telnet (TCP/23)
  • HTTP web management (TCP/80) — on appliances that default to HTTP-redirect-to-HTTPS, ensure HTTP is fully disabled, not just redirecting
  • SNMPv1 and SNMPv2c — remove community strings; disable the protocol versions
  • FTP and TFTP for configuration transfer or log export
  • Vendor-specific debug or "easy management" services (e.g. specific platform debug daemons left on by default)

Platform notes. Load balancers often have a separate "management partition" or "config sync" interface — these need the same treatment. PAM platforms sometimes have a "support/diagnostic" service running on the appliance that bypasses the documented authentication path — confirm with the vendor's hardening guide and disable.

Verify. From a non-management host, attempt connections to TCP/23, TCP/80, UDP/161 with snmpwalk. All should fail. On the platform, the running configuration should show no Telnet listener, no HTTP listener (HTTPS only), and no SNMPv1/v2 community strings.
Sec-Infra Mgmt: "disable plaintext"
2
Enable secure management protocols with strong configuration
Replace plaintext with cryptographically strong management protocols, with the appliance's own certificate hygiene up to baseline.

Enable and configure:

  • SSH version 2 only — restrict cipher suites to modern AEAD ciphers; drop CBC-mode and SHA1 MACs where successor algorithms exist; key-based authentication preferred over password where the platform supports both
  • HTTPS web management — TLS 1.2 minimum, prefer TLS 1.3 where supported. The platform's management certificate is organisation-CA-issued, never a vendor-default self-signed cert — training users to click through certificate warnings on the management UI is a baseline-defeating habit
  • SNMPv3 with authPriv if SNMP is needed — SHA-256 authentication minimum, AES-128 privacy minimum (AES-256 preferred); per-user views restrict OIDs to monitoring needs only
  • SCP or SFTP for file transfer (config export, signature/policy import, log export)
Verify. SSH negotiates v2 only. HTTPS responds with a valid certificate from the organisation's CA — confirm via browser or openssl s_client. SNMPv3 walks succeed with authentication; SNMPv2 walks rejected.
Sec-Infra Mgmt: "secure remote access"
3
Configure central authentication with MFA — AAL3 for Tier 0 PAM
All administrative authentication via central directory, attributable to a named individual, with authentication strength matched to what the platform protects.

Configure the platform to authenticate administrators against the central AAA service:

  • Primary: TACACS+, RADIUS, SAML, or Entra ID per the central identity architecture (the choice depends on what the platform supports natively)
  • Secondary AAA in a different location for resilience
  • MFA: required for all administrative access. For WAF and load balancer administration, AAL2 (typical TOTP or push MFA) is the minimum. For PAM administration where the PAM brokers access to Tier 0 systems, AAL3 is required (hardware-backed authenticators, FIDO2 security keys, or smartcard). This is a meaningful step-up from general administration and may require coordination with the identity team.
  • Last-resort local account: one named "break-glass" local account per platform, password vaulted in PAM (or in a vault independent of this platform if it is the PAM — see step 8 of WI-04)
  • Authorisation: AAA groups apply privilege levels (read-only, configuration, full administrative). Default grants are read-only; full administrative is exception-by-exception.
  • Accounting: all authentication events and command executions logged to AAA platform and to syslog
Verify. Log in as a named administrator; confirm AAA event in the identity platform and syslog event on the platform. Attempt with a deliberately invalid password; confirm rejection logged. For PAM, verify that AAL3 enforcement actually blocks AAL2-only sessions (try authenticating without the hardware factor — should fail).
Sec-Infra Mgmt: "AAA + MFA"
4
Restrict management access to a dedicated management network
Limit administrative reach to a known controlled source set.

Configure management-source restriction:

  • Identify the management network and specific administrative sources (jump hosts, NOC workstations)
  • Apply inbound access control on management protocols (SSH, HTTPS, SNMP, NETCONF) restricting them to those sources
  • Dedicated management interface — bind management services to a separate physical or logical interface where the platform supports it. Disable management on data-plane interfaces.
  • For PAM specifically: the management plane is on a dedicated VLAN logically separated from data plane; the management network is unreachable from any data-plane VLAN; the management interface does not share an IP namespace with anything an end-user (or attacker via end-user) can reach
  • For cloud-managed WAFs, restrict the vendor cloud-portal access via the vendor's IP allow-list or SSO conditional-access policy
Verify. Attempt SSH from a non-management host; refused at the platform's ACL layer. From the management network, succeed. For PAM, attempt to reach the management plane from a data-plane VLAN; should be unreachable (no route, or firewall-dropped). Confirm ACL counters increment on attempted denied connections.
Sec-Infra Mgmt: "restrict by source"
5
Session timeouts and authorised-use banner
Limit exposure of unattended sessions; display the authorised-use notice.

Configure:

  • Idle session timeout — 15 minutes maximum for privileged sessions; 30 minutes for read-only sessions if used. Configure on each management plane (SSH, HTTPS, console)
  • PAM-specific: tighter timeout (typically 10 minutes) on PAM administrator sessions; absolute session lifetime (e.g. 4 hours regardless of activity)
  • Pre-authentication banner — the organisation's approved authorised-use notice (Legal-approved version, available from IT Operations)
  • Post-authentication MOTD — optional; can include platform classification (Tier 0/1/2/3), pointer to the relevant playbook for incidents involving this platform
Verify. Connect; banner appears pre-login. Authenticate; MOTD if configured. Leave session idle past timeout; session terminates. For PAM, verify both idle and absolute timeouts.
Sec-Infra Mgmt: "session controls"
6
NTP with authentication
Accurate trusted time; correlation of cross-platform events; non-repudiation in audit logs.

Configure NTP:

  • Two or more authoritative NTP servers (the organisation's central NTP infrastructure)
  • NTP authentication enabled (shared key or NTPv4 autokey where supported)
  • Time zone set to UTC for log correlation; do not use local time on infrastructure
  • NTP source interface bound to the management interface where applicable
  • PAM-specific: session recording timestamps depend on accurate time; check that the recording subsystem uses the same NTP-synchronised clock as the audit log subsystem (some platforms maintain separate clocks)
Verify. Show NTP status; synchronised (stratum <15, offset <100ms). Authentication key configured. PAM: a brief test session's recording timestamp matches the audit log timestamp.
Sec-Infra Mgmt: "time synchronisation"
7
Centralised logging with platform-specific event coverage
All security-relevant events forwarded to central SIEM with sufficient context for SOC investigation.

Configure syslog forwarding to two SIEM destinations. Beyond the standard events (authentication, configuration changes, interface state, AAA, hardware alarms), each platform type adds its own event categories:

  • WAF — block/alert events with rule ID, request metadata, source IP, URI, user-agent; signature update events; learning-mode transitions; mode changes (monitor↔block)
  • Load balancer — TLS handshake failures (often indicative of scanning or downgrade attempts), pool member health-state changes, virtual server enable/disable, certificate expiry warnings
  • PAM — privileged session start/end with target, brokered command execution where captured, JIT access grant/revoke, vault access events, integrity-check failures, anomaly detections

Use TCP or TLS-encrypted syslog where the platform supports it. Source-interface bound to the management interface.

Verify. Trigger a known event per platform type (failed login + a WAF rule trigger / TLS handshake failure / PAM session). Events arrive at central SIEM within expected latency (typically <60s). Confirm timestamps correct and source IP is the management interface.
Sec-Infra Mgmt: "centralised logging"
8
Certificate hygiene for the platform itself
The appliance's own certificates are managed with the same discipline as production workload certificates.

Certificate management for the platform:

  • Management interface certificate — organisation-CA-issued; tracked in the central certificate inventory; expiry alerted ≥30 days before expiry; auto-renewal where possible (ACME or vendor-cloud-managed)
  • SNMP/syslog client certificates where the platform uses certificate-based authentication to those services — same discipline
  • API client certificates if the platform's API is consumed by automation — vaulted; rotation procedure documented
  • Vendor support certificates (vendor cloud / call-home) — track and confirm; treat any vendor-side certificate as a third-party dependency that requires its own monitoring
  • WAF-specific: certificates used for TLS termination on protected applications are tracked under WI-02 (WAF) and WI-03 (Load Balancer), not here — this step is the platform's own management certificate only
Verify. Pull the platform's management certificate via openssl s_client; confirm issuer is the organisation's CA, expiry >30 days, key strength meets policy (RSA 2048+ or ECDSA P-256+). Confirm tracking in the central certificate inventory.
Sec-Infra Mgmt: "certificate hygiene"
9
Console and out-of-band access hardening
Physical or OOB access does not bypass the controls applied to remote management.

For physical console and OOB ports:

  • Console port physically secured (locked rack, controlled-access room)
  • Console login uses the AAA configuration, not a local-only path
  • Console session timeout configured to match remote management
  • OOB management network (where the platform has dedicated OOB) on a separate physical or logical infrastructure; OOB never carries data-plane traffic
  • Vendor cloud-OOB or call-home features reviewed and either disabled or configured per the vendor hardening guide; document the choice
  • PAM-specific: console access to a PAM platform is a privileged action equivalent to PAM-admin login; treat console session with the same controls (AAL3 where applicable, recording where supported)
Verify. Console login flows through AAA. Idle timeout terminates console session. OOB network confirmed separated. Vendor cloud-management state documented.
Sec-Infra Mgmt: "physical/OOB access"
10
Change control discipline at the management plane
Every configuration change is traceable to an approved change record.

Establish change-control integration at the platform:

  • Every configuration change is recorded in the change-management system with the standard fields (what, why, who, when, verification, rollback)
  • Standard changes — pre-approved, low-risk (e.g. add a documented user to AAA, refresh a signature ruleset, add a known new IP to an existing allow-list)
  • Normal changes — reviewed by change advisory board; covers rule additions, virtual server changes, certificate replacement, firmware updates
  • Emergency changes — expedited path for security-related fixes (close an exposed management port, block a malicious source, deploy an emergency WAF signature); post-hoc review within 5 working days
  • For each change, capture pre/post configuration diff; this is the audit artefact
  • Maintenance windows defined by platform tier (Tier 0/1 have tighter windows and coordinated cross-team approval)
Verify. Sample recent platform changes; each has a corresponding change record with the standard fields. Sample emergency changes; each has a post-hoc review entry within SLA.
Sec-Infra Mgmt: "change control"
Validation & verification

After applying all steps, verify the management plane is hardened to baseline. Many of these checks are identical across WAF/LB/PAM; the platform-specific aspects of the validation are minor.

Functional validation

  • Log in as a named administrator from the management network — succeeds
  • Attempt login from a non-management host — refused
  • Attempt Telnet, HTTP, SNMPv2 — all fail
  • Trigger a configuration change; confirm it appears in central SIEM with correct timestamp and source IP
  • Trigger a failed authentication; confirm logged to both AAA and SIEM
  • Session idle past timeout terminates; for PAM, both idle and absolute timeouts confirmed
  • NTP synchronised (stratum <15, offset <100ms)
  • For PAM with AAL3: AAL2-only authentication fails as expected

Document the verification. Capture the validation evidence (configuration export, certificate verification, syslog confirmation, AAA test screenshots) and attach to the change record. This is the baseline conformance evidence for audit.
Rollback & troubleshooting

Management-plane changes can lock administrators out of the platform. The general rollback approach: OOB or console access, restore the offending change from the pre-change backup, validate.

Common loss-of-access scenarios

  • Locked out via management ACL — same as the network-WI pattern; almost always because the ACL was applied before AAA was confirmed working from the management network. Recovery via console: remove the ACL or add the missing source; complete AAA validation; reapply.
  • AAA unreachable and break-glass not tested — if the break-glass local account was never tested, this is the moment you discover it doesn't work. Vendor support call. Test break-glass before relying on it.
  • AAL3 step-up fails for everyone — cause: AAL3 enforcement enabled before all administrators have hardware authenticators provisioned. Fix: revert to AAL2; coordinate with identity team to provision hardware factors; re-enable AAL3 when ready.
  • HTTPS management certificate not trusted — cause: CA chain not deployed to administrator workstations. Fix: deploy CA via MDM/GPO; do not work around with cert-warning click-through.
  • Syslog flooding the SIEM — cause: severity threshold set too low (debug); fix: raise to informational; investigate the actual events later.
  • NTP not synchronising — cause: management ACL blocking NTP, or wrong source-interface binding. Fix: verify NTP server reachability via management path.

Rollback procedure

  1. Connect via OOB console (or vendor cloud-OOB)
  2. Verify the platform responds; use break-glass if AAA is the failed component
  3. Compare running config to pre-change backup; identify the breaking change
  4. Reverse only the breaking change — not a wholesale restore unless the entire change is to be aborted
  5. Confirm management access restored from the management network
  6. Update change record with rollback action and root-cause finding; schedule the resubmission with the fix
Common errors & tips
  • Don't apply management ACL before AAA is verified working. Same lesson as the network WI: order matters; AAA first then ACL.
  • Always have working OOB. For PAM particularly, do not start this WI on a production platform without OOB; locking yourself out of PAM has cascading consequences.
  • Test break-glass before you need it. The break-glass account is the recovery path. If it doesn't work, recovery is a vendor support case. Do an annual controlled test, document the result.
  • AAL3 requires hardware factor provisioning across all admins. Coordinate with identity team; ensure every PAM administrator has their hardware factor before flipping AAL3 enforcement on.
  • Self-signed certs on management UI train bad habits. Once admins learn to click through the warning, they'll do it on a phished imitation interface too. Deploy a CA-issued certificate even if it's just a few admins; the discipline matters.
  • Cloud-managed platforms have a different control surface. Many controls in this WI map to cloud-portal settings rather than CLI; the requirement (e.g. "MFA on admin access") doesn't change but the mechanism does. Confirm via the vendor's hardening guide.
  • Don't forget the call-home and vendor support backdoors. Many appliances have a vendor-cloud-management service running by default that effectively bypasses your management ACL. Disable it or harden it explicitly; document the choice.
  • For PAM, management plane separation is harder than it sounds. Some PAM platforms host their administrator interface on the same network as the end-user-facing brokering service. If yours does, treat that as a known limitation and apply compensating controls (jump host + strict authentication) while raising with the vendor.
  • Syslog volume from a WAF can dwarf everything else combined. When configuring WAF syslog forwarding, anticipate the volume; the SIEM and the platform's local buffer both need headroom. Start with sampling/tuning before unleashing full event flow.
