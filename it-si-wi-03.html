IT-SI-WI-03: Load Balancer Hardening

Vendor-agnostic implementation guidance for hardening load balancers and application delivery controllers: TLS, certificate lifecycle, HSTS, virtual server hygiene, health-check security, rate limiting, source IP preservation, HA. Operationalises §4.3 of IT-SI-SOP-01.

Prerequisites

Before applying this WI:

  • WI-01 management-plane hardening complete on the load balancer
  • Virtual server inventory available — what's published, on which IP, on which ports, with which back-end pool
  • Application owners identified per virtual server
  • Current certificate inventory available; auto-renewal mechanism (ACME, vendor-cloud, vault-integrated) in place where possible
  • Cipher policy decision made — TLS 1.2 minimum, modern AEAD ciphers (the approved cipher list from the central security policy)
  • HA pair configuration known (active/standby, active/active, sync-link configuration)
  • Change window approved per WI-01 §10
TLS configuration changes are visible immediately to every client. Disabling TLS 1.0 or removing CBC ciphers will break clients that don't support newer alternatives. Audit the client base first; understand the legitimate-old-client population before tightening.
Step-by-step instructions

The steps below cover the load-balancer-specific controls required by IT-SI-SOP-01 §4.3.

1
TLS protocol and cipher suite hardening
Modern TLS only; AEAD ciphers; ephemeral key exchange; verifiable security.

Configure TLS on every virtual server:

  • Protocols: TLS 1.2 minimum, TLS 1.3 enabled and preferred where back-end and clients support it. TLS 1.0 and 1.1 disabled. SSLv2 and SSLv3 disabled (these should not exist on any modern load balancer but verify).
  • Cipher suites: AEAD only (AES-GCM, ChaCha20-Poly1305). CBC-mode ciphers disabled. RC4 disabled. 3DES disabled. DES and export ciphers disabled.
  • Key exchange: ECDHE (elliptic curve ephemeral) preferred; DHE acceptable with ≥2048-bit groups; static RSA key exchange disabled (no forward secrecy)
  • Curves: X25519 and P-256 minimum; P-384 acceptable; deprecated curves disabled
  • Cipher order: server-preferred (the server picks from its prioritised list, not the client's preference)
  • Renegotiation: secure renegotiation (RFC 5746) only; client-initiated renegotiation disabled

For TLS 1.3, the cipher question is simpler (only AEAD ciphers exist in the spec); the focus is enabling TLS 1.3 itself and removing legacy TLS 1.2 ciphers from the offered list.

Verify. Run testssl.sh against every internet-facing virtual server. Result: A or A+ score (Mozilla "modern" or "intermediate" profile compliance). No protocol downgrades succeed. No legacy cipher accepted.
LB: "TLS protocols & ciphers"
2
Certificate lifecycle management
No expired certificates in production; renewal happens before expiry; CT logging awareness.

Certificate management discipline:

  • Inventory. Every certificate on every virtual server tracked in the central certificate inventory: CN, SAN list, issuer, expiry, key type and length, where deployed, owner
  • Expiry alerting. Alert ≥30 days before expiry to the certificate owner; ≥14 days escalate to IT Operations; ≥7 days escalate to leadership. Aim for renewal completed ≥10 days before expiry.
  • Auto-renewal. Where possible (ACME for Let's Encrypt, ZeroSSL; vendor-cloud-managed for cloud WAFs; vault-integrated for enterprise CA) — auto-renewal eliminates the most common cause of unexpected outage.
  • Key strength. RSA 2048-bit minimum, 3072 or ECDSA P-256 preferred for new certificates. RSA 1024 retired immediately if any remain.
  • Certificate Transparency. Public certificates are CT-logged automatically (by the issuing CA). Monitor CT logs for unexpected certificates issued for organisation domains — these can indicate misissuance or compromise.
  • Wildcard discipline. Wildcard certificates are convenient but distribute risk (one private key protects many services). For Tier 0 applications, use SAN certificates over wildcards. Where wildcards are used, the private key is treated as a high-value secret.
  • Private key handling. Private keys never leave the load balancer's secure store (HSM where supported, secure key storage otherwise). Export only with documented approval; rotation when exported.
Verify. Pull the certificate inventory; cross-reference against virtual servers — every cert tracked. Sample certificates: expiry >30 days; key strength meets policy; alerting in place. Test the renewal automation (force a renewal in a non-production environment).
LB: "certificate lifecycle"
3
HSTS on internet-facing virtual servers
Browsers enforce HTTPS-only access; protocol-downgrade attacks blocked client-side.

HSTS configuration:

  • Enable HSTS on all internet-facing virtual servers serving HTTPS
  • max-age — 1 year (31536000 seconds) minimum for production; shorter (1 month) during initial rollout to allow recovery from misconfiguration
  • includeSubDomains — enable for production domains where all subdomains support HTTPS
  • preload — only after policy approval; preload is a near-permanent commitment (removal takes months and requires browser-vendor coordination)
  • HTTP-to-HTTPS redirect — every HTTP virtual server redirects to HTTPS; no plaintext content served on production domains
Verify. curl -I https://[application] shows Strict-Transport-Security header with appropriate max-age. curl -I http://[application] shows 301/308 redirect to HTTPS.
LB: "HSTS & HTTP→HTTPS"
4
Virtual server hygiene — purpose, ownership, lifecycle
No orphan virtual servers; every published endpoint has a documented purpose.

Virtual server hygiene rules:

  • Every virtual server has: documented purpose, owner, application reference, creation date, last-review date
  • Traffic monitoring. Virtual servers with no traffic in the last 30 days are flagged for review; 90 days with no traffic → removed unless documented justification (DR-only, scheduled batch)
  • Deprecated services. When an application is retired, the corresponding virtual server is removed (not left as "just in case"). Asset register update triggers virtual-server cleanup.
  • IP allocation. Internet-facing IP addresses on load balancers tracked centrally; reclamation of decommissioned IPs is part of the cleanup workflow
  • Naming convention. Virtual servers follow an organisation naming convention (e.g. vs-app-env-region) so meaning is conveyed by the name
Verify. Pull virtual server list; cross-reference against application inventory and recent traffic metrics. No-traffic VSes have documented justification or are removed. Naming convention applied.
LB: "virtual server hygiene"
5
Health-check security
Health checks do not leak application metadata or provide an alternative attack surface.

Health check configuration:

  • Dedicated health endpoint — applications expose a dedicated path (e.g. /healthz) for load-balancer probes; not a real application path that returns useful data to general callers
  • No application metadata leaked — health response is minimal (HTTP 200 with no body, or a short "OK"); not a full application status page revealing component versions, database state, or internals
  • Limit health endpoint reachability — the health endpoint accepts requests only from the load balancer's IP range; external callers get a 404 or a generic error
  • TCP-only health checks on services where HTTP health would expose information — sometimes a TCP connect check is the right control
  • Authenticate health checks where the platform supports it (HTTP-basic from the load balancer to the application, or mTLS) so the application can validate the source
  • Health-check frequency — frequent enough for fast failover (typically 5–15 second intervals) but not so frequent it impacts the application
Verify. From the public internet, request the health endpoint of a Tier 0/1 application; should fail or return generic response (not full health detail). From the load balancer's network position, the health endpoint works normally.
LB: "health checks"
6
Connection rate limiting and SYN-flood protection
Load balancer absorbs and shapes anomalous traffic before it reaches application servers.

Edge protection:

  • Per-source connection rate limiting on internet-facing virtual servers — typical baseline 100 new connections per second per source IP before throttling; tune per application
  • SYN-flood protection — SYN cookies or platform equivalent enabled; configured thresholds tuned to legitimate burst patterns
  • Per-virtual-server connection limit — prevents one popular VS from starving connection tables shared with other VSes
  • HTTP slowloris protection — slow-request detection where the platform supports it; close idle connections aggressively
  • Geo-IP rate limiting for traffic from geographies with no business case (complement, not replacement for, full geo-blocking decision)
  • Coordinate with upstream DDoS protection (where deployed) so the load balancer's protection is a second tier, not the only tier
Verify. Simulate connection-rate burst from a test source (controlled lab); confirm rate limit triggers and logs. SYN flood test (controlled); SYN cookies engage as expected.
LB: "anti-DDoS / rate limiting"
7
Source IP preservation
Back-end logs reflect real client IPs, not the load balancer's IP.

Two main approaches, each with its trade-offs:

  • X-Forwarded-For (and X-Real-IP) headers — load balancer adds the real client IP to a header; application reads and logs from the header. Risk: applications must trust the header only when it comes from the load balancer (otherwise attackers spoof IPs by injecting the header). Configure the application to accept these headers only from the load balancer's source IPs.
  • PROXY protocol (haproxy's PROXY v1/v2) — pre-pends an IP marker to the TCP connection; back-end parses it. Cleaner than HTTP headers (works for non-HTTP protocols) but requires application support.
  • Direct Server Return / transparent mode — load balancer forwards the original client IP at L3; application sees the real source IP directly. Most accurate but harder to architect and limits some load-balancer features.

For Tier 0/1 applications where audit traceability matters, source IP preservation is mandatory (one of the three approaches). Document the chosen approach per application.

Verify. Generate a known-source request; confirm the application logs the real client IP, not the load balancer's. For X-Forwarded-For approach, attempt to spoof the header from a non-LB source; confirm the application ignores the spoofed header.
LB: "source IP preservation"
8
High availability with annual failover test
Single load-balancer outages don't cause site-level outages.

For Tier 0 and Tier 1 load balancers:

  • HA pair configured (active/standby or active/active per vendor capabilities)
  • HA sync link on a dedicated interface; not shared with data plane
  • Configuration sync verified working — changes on primary propagate to secondary
  • Failover behaviour documented: convergence time, session preservation (for stateful apps), traffic loss budget
  • Annual failover test under controlled conditions; results logged in the asset record
  • For internet-facing virtual servers, DNS or upstream-routing failover where the LB-pair-level HA isn't sufficient (multi-site)

For Tier 2/3, HA is desirable but not always cost-justified; asset record documents the decision and acceptance criteria.

Verify. Trigger failover under controlled conditions; traffic continues; sessions preserved where supported; convergence within budget. Both members show "in sync" before declaring complete.
LB: "high availability"
Validation & verification

After applying these steps, validate the load balancer as a whole.

Functional validation

  • TLS scan score A or A+ on every internet-facing virtual server
  • Certificate inventory complete; alerting in place; auto-renewal tested where applicable
  • HSTS active on internet-facing servers; HTTP→HTTPS redirect working
  • Virtual server inventory current; no orphans
  • Health endpoints don't leak metadata; restricted to load-balancer source
  • Rate-limiting and SYN-flood protection engage under simulated load
  • Source IP preservation verified at the application back-end
  • HA failover test completed annually with results logged

Rollback & troubleshooting

Load balancer changes are visible immediately; have rollback prepared.

Common scenarios

  • Clients can't connect after TLS tightening — cause: a population of legitimate clients only supports TLS 1.0/1.1 or older ciphers (legacy embedded devices, old enterprise software). Fix: identify the affected clients; either move them to a separate virtual server with relaxed policy (and a sunset date for the legacy) or accept and replace the clients.
  • Certificate renewed but new cert not picked up — cause: cert installed but virtual server not reloaded; or cert installed in wrong store. Fix: reload the virtual server; verify cert via testssl.sh.
  • HSTS deployed too aggressively; users can't reach the site — cause: a backend issue forced HTTP downgrade but HSTS prevents the browser from accepting it. Fix: resolve the backend issue; HSTS itself cannot be undone for clients that already received it — you wait for max-age, or accept the lockout period.
  • Health-check fails after tightening — cause: load balancer source IP changed (added a node) and the application's allow-list isn't updated. Fix: update the application allow-list; verify health checks pass.
  • Rate limit blocks legitimate burst — cause: rate limit set too tight for legitimate end-of-month batch traffic. Fix: tune the limit; consider per-account rather than per-IP for known partners.
  • Failover doesn't complete — cause: HA-link issue, config-sync drift, vendor bug. Fix: investigate HA link health; force re-sync; vendor TAC engagement.
  • Source IP preservation broken — cause: X-Forwarded-For misconfigured at the application end (application uses the wrong header field, or doesn't trust the LB source). Fix: align LB and application configuration.

Rollback procedure

  1. OOB or management access
  2. Identify the breaking change from pre/post config diff
  3. Revert the specific change; not the entire change set unless required
  4. For TLS changes affecting a population, consider a per-VS revert rather than global
  5. Update change record; document root cause
Common errors & tips
  • Test TLS configurations before deploying to internet-facing. A staging environment with the same cipher policy lets you catch broken legitimate clients before they reach production.
  • HSTS preload is nearly permanent. Removal from preload lists takes months and browser-vendor coordination. Don't enable preload until you're certain.
  • Certificate expiry alerting that nobody acts on is a non-control. Alerts must reach someone with authority and incentive to act. If the cert owner is no longer in the role, the alert routes to a vacant inbox.
  • Don't trust X-Forwarded-For from arbitrary sources. Only trust the header when the connection came from a load-balancer source IP; otherwise an attacker spoofs origins by setting the header themselves.
  • HSM key storage is meaningful for high-value certs. The single biggest improvement to certificate security for Tier 0 apps is moving the private key to an HSM.
  • Wildcard certificates accumulate risk. A single compromised wildcard private key affects every service it protects. For T0, prefer SAN certs.
  • Health checks that "always pass" are not health checks. If the health endpoint always returns 200, it's not detecting anything. Verify the endpoint actually exercises the application's critical path.
  • Annual failover test is not optional. An untested HA pair is a single point of failure with a comforting illusion of redundancy.
  • SYN-cookies have a small impact on TCP options. Some application performance tuning relies on TCP options that aren't preserved when SYN-cookies are engaged. The trade-off is rare and the protection is worth more.
