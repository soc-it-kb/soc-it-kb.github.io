GRC-DR-WI-02 — Cloud Platform Disaster Recovery

Procedures for cloud platform disaster recovery scenarios: Azure region failure, Microsoft 365 sustained outage, Entra tenant lockout, Entra tenant compromise, and SaaS provider disruption. Operationalises the cloud DR principles in GRC-DR-SOP-01 §6. The estate is hybrid; cloud DR is a first-class concern rather than an afterthought.

5 scenarios Azure · M365 · Entra Tenant lockout Tenant compromise
Purpose & positioning

Cloud platforms reduce some traditional DR concerns (hardware loss, single-site outage, rack-level fire) and introduce others (region-level provider outage, tenant-level compromise, certificate-driven lockout, sustained service degradation that no failover within the same tenant resolves). This WI addresses the credible failure modes.

It is the operational complement to IT-CL-WI-03 and IT-CL-WI-04, which define how cloud workloads are designed for resilience. This WI describes what to do when those designs are tested for real, and what to do in scenarios that workload-level resilience cannot solve (notably tenant-level events).

Boundaries:

  • This WI handles recovery to operations. Active investigation and containment of cloud incidents (tenant compromise, identity attacks) live in SOC playbooks (SOP-ID-01 and identity playbooks / runbooks). The two run in parallel during a real event; this WI describes the DR side.
  • This WI does not re-author resilience design. Paired-region setup, GZRS configuration, AZ deployment patterns are governed by IT-CL-WI-03/04 and the Cloud Baselines domain. This WI uses those patterns; recovery presumes they exist.
  • This WI does not author break-glass governance. Eligibility, ownership, and rotation of break-glass and emergency access accounts are governed by the Identity Governance domain (ASM-ID, particularly IT-ID-WI-05). This WI describes break-glass use during a recovery event.

Scenario 1 — Azure region failure
1.1Confirm region-scope event

Before invoking region-failover procedures, confirm the event is genuinely region-scope and not a workload-specific failure that resembles one.

  • Check Azure service health portal and Microsoft service health communications. A region-scope event is named by Microsoft (e.g. “impacted region: West Europe”) with multiple affected service categories.
  • Check whether workloads in the same region but on different resource types are also affected. If the impact is limited to one service category, this may be a service-level event not a region event.
  • Confirm with Microsoft support if scope is unclear before initiating cross-region failover. Premature failover risks split-state across regions.
1.2Invoke per-workload failover per design

Each Tier-1 workload should have a paired-region failover design per IT-CL-WI-04. Invocation is per-workload following the workload’s DR runbook.

  • Identify the in-scope workloads from the agency’s tier inventory (per WI-01) where primary region matches the impacted region.
  • For each: confirm the secondary region is healthy; follow the workload runbook for failover (typically: DNS / Front Door endpoint change; database failover-group manual failover or wait for automated; storage account read access from secondary if RA-GRS).
  • Sequence: foundation services first (DNS, identity routing endpoints), data layer second, application layer third. Application layer that depends on a still-failing data layer will not be healthy.
  • Validate each workload before declaring recovered: synthetic transactions per the runbook, not just “health endpoint returns 200”.
1.3Coordinate cross-workload dependencies

Many workloads share dependencies. Coordinating failover across them is the principal work in a real region failure event.

  • Identity dependency: nearly every workload depends on Entra ID. Entra is multi-region by Microsoft design and does not need explicit failover, but federated identity systems (on-prem AD via Entra Connect) may.
  • Cross-workload data dependencies: workload A reads from workload B’s database. Failing over A without B leaves A pointing at the unhealthy region.
  • External-facing endpoints: Azure Front Door, Application Gateway, public IP DNS records all need to point at the secondary region. Where Front Door is in front, automatic failover handles this; where DNS-based, manual update.
  • Crisis Management Team coordinates across IT Service Owners for the failover sequence; the Cloud Platform team executes for platform-level resources.
1.4Plan and execute failback

Failback is itself a controlled change, not an automatic reversal.

  • Wait for Microsoft formal confirmation of region recovery, including post-incident report on extent of data continuity.
  • Plan failback at a scheduled window. Failback during business hours doubles the operational exposure window; failback to a region that may regress (immediate post-event days) is high-risk.
  • Resynchronise data layer (e.g. failover groups reverse role). Validate before redirecting traffic.
  • Per-workload failback following the runbook in reverse. Application layer first if it has been operating on read-only secondary; data layer when synchronisation completes.
  • Post-event AAR per WI-02 within 14 days. A real region event is a Tier-1 exercise event and counts against the cadence.
Scenario 2 — Microsoft 365 sustained outage
2.1Confirm the outage is sustained, not transient

M365 transient issues resolve within an hour. A sustained outage (> 4 hours) is rare and may trigger continuity invocation. Confirm before invoking communications fallback.

  • Check Microsoft service health for an active incident on Exchange Online, SharePoint Online, or Teams. Microsoft posts incident IDs (e.g. EX12345) for the affected service.
  • Check whether the impact is global or regional. Tenant region may matter for some services.
  • Distinguish from auth-only outage (covered by Scenario 3 if Entra-side; covered here if M365 service-side).
  • Estimate likely duration from the Microsoft communication. Microsoft typically posts an ETA range; treat the upper bound as the planning horizon.
2.2Activate communications fallback

Critical communications during a sustained M365 outage must not rely on M365.

  • Pre-staged fallback channel. Per the DR policy (GRC-DR-SOP-01 §6), the agency operates a communication fallback that does not depend on M365 — typically SMS-to-leadership distribution lists, a non-M365 group messaging platform (Signal / WhatsApp where culturally appropriate), or out-of-band telephone bridges.
  • Crisis Management Team activated. CMT convenes on the fallback channel and coordinates from there until M365 returns.
  • Stakeholder notification. Material outage exceeding 2 hours is communicated to agency leadership and to other agencies; outage exceeding 8 hours is communicated to beneficiaries / external stakeholders where impact warrants.
2.3Manage data continuity during the outage

M365 sustained outage does not typically cause data loss — data is held server-side, returns when the service returns. But certain operations have continuity implications.

  • Cached access remains. Outlook cached mode, OneDrive sync, Teams cached files all continue to work for the cache contents during the outage. Discourage cache clears and sign-outs.
  • New mail sent during the outage queues on the client and may bounce externally. Communicate via fallback channel that external mail is delayed.
  • Document edits in cached OneDrive sync persist on the local machine and sync when the service returns. Risk: conflicting edits if multiple users edit cached copies of the same document; the conflict resolution will require manual reconciliation.
  • Discourage workarounds that create parallel state (e.g. emailing documents around through alternative channels); the documents will return when the service returns, and parallel copies cause divergence.
2.4Validate return-to-service and conduct AAR

Microsoft will declare the incident resolved. Local validation follows.

  • Sample test: send and receive external email; open and edit a SharePoint document; participate in a Teams call.
  • Monitor for residual issues (slow performance, intermittent failures) for 24 hours post-resolution.
  • Capture impact: estimated user-hours lost; processes affected; whether business workarounds were invoked.
  • AAR per WI-02: focus on whether the fallback channel performed as designed, whether the CMT received the right information at the right time, whether external stakeholders received notification.
Scenario 3 — Entra tenant lockout
3.1Identify the lockout cause

“Tenant lockout” covers several distinct events with different recovery paths.

  • Credential lockout: all known Global Admin accounts unable to sign in. May be: MFA cascade failure (lost devices), forgotten passwords combined with locked self-service reset, accidental disable of Global Admin accounts.
  • Conditional Access lockout: a CA policy was deployed that locks out administrators (e.g. requires a control none of the admin accounts can satisfy). Most common cause of accidental lockout.
  • Certificate expiry: federation certificate expired without renewal, or token signing certificate expired, breaking auth.
  • Authentication policy lockout: a new authentication method policy is misconfigured and blocks the admins as well as users.
  • Each cause has a different recovery path. Diagnosis first, action second.
3.2Engage break-glass / emergency access accounts

Per the tenant identity baseline (IT-CL-WI-01 and IT-ID-WI-05), two break-glass accounts exist that are: cloud-only, vaulted off-tenant, excluded from every Conditional Access policy, monitored by SOC for any sign-in (every sign-in is an alert).

  • Confirm break-glass health before use. If quarterly validation has been performed, this should be known. If not, the first action is to verify the credentials still work; if break-glass is also broken, escalate to Microsoft Support immediately (Scenario 3.6).
  • Document the use. Break-glass use is a contemporaneously documented event — time of decision, decision-maker, justification, time of access, actions taken. Logs supplement but do not replace the contemporaneous record.
  • SOC will alert on the sign-in. Coordinate with SOC so the alert is acknowledged as legitimate; SOC remains on watch for any anomalous activity during the recovery window.
  • Break-glass operates from a clean device. A PAW per IT-EP-SOP-PAW-01 if available; otherwise a freshly-built device on a controlled network.
3.3Remediate the cause

Once break-glass access is established, fix the cause.

  • Conditional Access lockout: identify the offending policy, set to report-only or remove the locking condition, verify normal admin sign-in restored. Common offenders: location-based restrictions excluding admin home networks; device compliance requirements where admin devices aren’t compliant; require approved-app where admins use unmanaged tools.
  • Credential lockout: reset admin passwords, re-enrol MFA methods, restore Global Admin assignments where accidentally removed.
  • Certificate expiry: renew federation or token-signing cert per the platform’s renewal procedure; verify trust chain; coordinate with on-prem AD FS team if federated.
  • Authentication policy lockout: roll back the policy change; verify auth works for both admins and a sample user.
  • Validate end-to-end: sign in as a regular admin (not break-glass), perform a sample admin action, sign out, sign back in.
3.4Stand down break-glass and rotate credentials

Break-glass is for the emergency; standing down is part of the procedure.

  • Sign out of break-glass account. Verify no residual sessions.
  • Rotate the break-glass credentials. Even if there is no indication of compromise, used credentials are rotated per the identity baseline.
  • Restore the break-glass to its quiescent state (in vault, off-device, SOC-monitored).
  • If credential change requires re-establishing MFA, do so per the baseline (FIDO2 keys, vaulted; not phone-based factors).
3.5Conduct AAR and treat root cause

A tenant lockout is a high-severity event regardless of duration. AAR within 14 days; root cause treated.

  • Conditional Access lockouts typically reflect missing report-only-then-enable discipline. Treatment: enforce report-only deployment for any new CA policy; require “admin not affected” check as a deployment criterion.
  • Credential lockouts typically reflect inadequate admin account hygiene: too few admins, no documented MFA recovery, single device for MFA. Treatment: increase admin count to at least three plus break-glass; require multi-device MFA; document MFA recovery procedure.
  • Certificate expiry reflects monitoring gap. Treatment: certificate expiry monitoring with multi-week lead time alerts; renewal procedure documented and rehearsed.
  • Policy lockouts reflect change control gaps. Treatment: change control for authentication policies as a Tier-0 change category.
3.6Microsoft Support escalation if break-glass fails

The catastrophic case: break-glass accounts are also locked out. Recovery requires Microsoft engagement.

  • Open a Severity A support case via the tenant’s Microsoft account team or, if access to the support portal itself is impacted, via account team direct contact.
  • Be prepared to provide: tenant ID, tenant primary domain, evidence of identity (organisation contract, named admin verification).
  • Microsoft tenant-recovery procedures are out-of-tenant and may take days. This is one of the strongest arguments for break-glass hygiene; treat the procedure as a possibility but plan to never need it.
  • During the wait: BCP activates for affected processes; SOC monitors for indicators of related compromise; business workarounds operate per BCP plans.
Scenario 4 — Entra tenant compromise

Tenant compromise is the most severe cloud DR scenario. SOC owns the response (containment, investigation, eradication) under SOP-ID-01 and the identity playbooks / runbooks. This WI describes the DR side: recovery to operations, which may take weeks and in the worst case requires migration to a clean tenant.

4.1Coordinate with SOC on recovery threshold

DR does not start until SOC confirms the response is sufficiently contained that recovery actions will not interfere.

  • SOC owns the containment decisions (suspend privileged accounts, revoke sessions, block compromised app registrations, isolate affected resources).
  • DR engages in parallel from the start for planning; execution waits on SOC’s recovery threshold call.
  • The recovery threshold is not “incident closed” — it is “safe to start recovery without re-introducing the attacker’s persistence”. Recovery may begin while investigation continues for related artefacts.
4.2Assess the recovery path: clean-up vs clean-tenant

The decision between in-tenant clean-up and migration to a clean tenant is the most consequential one in this scenario.

  • In-tenant clean-up is the default for events where: persistence is identified and removable; the compromise scope is bounded; SOC has high confidence in identification of all attacker artefacts. Recovery is on the order of days to weeks.
  • Clean tenant migration is the path for events where: persistence is suspected but not confidently bounded; the attacker had Global Admin level access for sustained time; cross-tenant federation was abused; or the agency’s assurance bar (regulatory, beneficiary safety) requires a clean restart. Recovery is on the order of weeks to months.
  • The decision is the BCDR Steering Committee’s. Brief on: SOC’s confidence assessment, scope of impact, regulatory implications, business cost of each path. Document the decision and basis.
  • Clean-up is irreversible to clean-tenant. Once clean-up begins, evidence is altered; migration to clean tenant from that point is far harder. The decision must be made before recovery actions begin.
4.3In-tenant clean-up recovery

Where in-tenant clean-up is chosen, the recovery sequence is:

  • Rotate every credential the attacker may have observed or accessed: privileged passwords (per IT-ID-WI-04 for service accounts), service principal secrets, certificate-based credentials, federation certificates, refresh tokens (force re-authentication tenant-wide), API keys held in compromised vaults.
  • Review and reset privileged role assignments. Anyone holding standing privilege during the compromise window has eligibility reassessed; PIM activations during the window are reviewed; any assignment not re-justified is removed.
  • Hunt residual persistence. SOC continues hunting; DR coordinates to ensure changes to tenant state during recovery do not destroy evidence of persistence yet to be found.
  • Rebuild identities suspected of compromise. Disable, force re-enrol MFA, validate group memberships, sign back in from clean device.
  • Validate end-to-end across critical services. Each Tier-1 service is exercised to confirm functionality is restored and no attacker-modified configuration persists.
4.4Clean tenant migration recovery

Where clean tenant migration is chosen, the recovery is a major project. Outline only; the specific runbook is developed at event time with Microsoft and the agency’s SI partner.

  • Stand up a new tenant. New Azure AD / Entra tenant; new domain registration or domain transfer; new identity baseline applied per IT-CL-WI-01; baseline Conditional Access deployed before any user identities migrate.
  • Migrate identities. Trusted attestation of which identities to bring over (not all — potentially compromised accounts may be left behind for forensic preservation). Re-enrolment of MFA. Re-issuance of devices where compromise scope warrants.
  • Migrate data. Mail, files, SharePoint sites in priority order driven by tier inventory. Validate data integrity at migration; re-apply sensitivity labels and DLP per IT-CL-WI-02.
  • Migrate services. Azure subscriptions, application registrations, third-party SaaS integrations all re-established against the new tenant. Federation re-configured.
  • Operate in parallel during cutover. Old tenant remains for forensic access; new tenant is operational; users are migrated in cohorts.
  • Decommission old tenant when forensic, legal, and audit holds permit. Coordination with regulators and legal counsel before decommission.
  • Throughout: BCP runs for the duration; business processes operate on continuity plans for the weeks to months involved.
4.5Document and learn

Tenant compromise events produce structural lessons.

  • Joint AAR with SOC, DR, Identity, and BCP — not parallel separate AARs. The interaction between disciplines is the principal source of lessons.
  • Joint findings feed: the identity baseline (IT-CL-WI-01 / IT-ID-WI-05), the cloud baseline (IT-CL-SOP-01), the IRP, this WI.
  • Regulatory and contractual notifications are coordinated through Legal and Communications.
Scenario 5 — SaaS provider disruption
5.1Confirm and characterise the disruption

SaaS disruption covers a range from provider-side outage to provider compromise to provider business failure. Characterisation drives the response.

  • Outage: provider service unavailable; typically transient but may be sustained. Provider communicates status. Recovery is wait-plus-business-workaround.
  • Provider compromise: provider announces a breach affecting customer data. Response includes assessing exposure (what data was held by the provider, what was actually accessed), engaging legal / DPO, communicating to affected stakeholders.
  • Provider sustained failure: provider unable to operate for extended period (e.g. cyber attack on the provider, business failure, regulatory shutdown). Recovery is migration to alternative provider where feasible.
  • Data exfiltration risk: regardless of provider state, organisation data with the provider is the organisation’s liability. Independent backup or export capability for Tier-1 SaaS systems is a baseline requirement (per DR policy §5).
5.2Activate dependent BCP workarounds

SaaS disruption affects business processes that depend on the SaaS. BCP for those processes activates.

  • Identify dependent processes from the BIA dependency map.
  • Each dependent process has a continuity plan (per BCP policy) describing the workaround for SaaS unavailability.
  • The Business Process Owner invokes the workaround; the IT side coordinates technical aspects (e.g. providing read-only access to the most recent SaaS data export).
5.3Engage alternative-access pattern if designed

For Tier-1 SaaS systems, the DR policy requires either an alternative-access pattern or a documented limit-of-recovery acknowledgement.

  • Alternative-access patterns may include: read access to the most recent independent export of the SaaS data; a parallel provisioning with an alternative provider; manual / spreadsheet-based continuity for the duration; documented “business operates without this system for up to N days” with no technical alternative.
  • Where no alternative is designed (the SaaS is genuinely irreplaceable in the short term), the BIA should have surfaced this as a risk. The risk register entry guides expectation-setting during the disruption.
5.4Recovery and lessons-learned

Recovery from a SaaS disruption follows the disruption type.

  • Outage: provider returns; validate access; resume normal operations; AAR focuses on workaround effectiveness.
  • Compromise: assess data exposure; complete legal / regulatory notifications; possibly migrate from the provider depending on confidence in their remediation; AAR feeds supplier assurance per GRC-POL-01.
  • Sustained failure: migrate; this is itself a multi-week to multi-month project. The BCP plan handles continuity during migration.
  • All paths produce a finding for the supplier assurance programme — the risk that materialised, the controls that worked, the controls that did not, the implication for future supplier selection.
Common pitfalls
  • Untested break-glass. Quarterly validation is mandatory in the cloud baseline for a reason — the one time it matters, it has to work. Failure pattern: created during initial baseline, never tested, password long forgotten, MFA device decommissioned.
  • Failover not coordinated across workloads. Workload A fails over independently while still pointing at workload B in the original region. Sequence and dependency matter.
  • Communication fallback that depends on the failing service. SMS fallback to a distribution list that lives in Exchange Online does not work during an M365 outage. Fallback must be genuinely independent.
  • Recovering before SOC says it’s safe. In a compromise scenario, premature recovery re-introduces persistence. SOC’s recovery threshold call is non-negotiable.
  • Clean-up when clean-tenant was warranted. Where attacker had sustained Global Admin access, clean-up provides imperfect assurance. The decision criteria in 4.2 must be applied honestly.
  • SaaS in scope but not in DR plan. Critical SaaS systems are sometimes treated as “provider problem” and excluded from DR planning. Provider unavailability is a DR scenario regardless of root cause.
  • Failback without a plan. Failback during business hours, or to a still-flaky region, doubles the recovery exposure.
