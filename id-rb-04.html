ID-RB-04: Review OAuth Application & Permissions

Identify, review, and assess OAuth applications and consent grants associated with a user or tenant — detecting malicious, excessive, or unauthorised access that persists independently of credentials.

Section 1
Purpose

Provide a repeatable, structured procedure to identify, review, and assess OAuth applications and consent grants associated with a user or tenant, in order to detect malicious, excessive, or unauthorised access.

This runbook supports ID-PB-04 (OAuth App Abuse) investigations.

Section 2
Required Access
🔑 Entra ID / Azure AD directory read access 📱 Application registrations visibility 🏢 Enterprise applications visibility 📊 Audit and sign-in logs 🎫 SOC ticketing system
Section 3
Inputs Required
👤 Username or User ID 🎫 Incident reference ID 🕐 Approximate compromise timeframe
Section 4
Step-by-Step Procedure
1
Identify User-Consented Applications
1
Open the Entra ID portal and locate the affected user account
2
Review all applications, permissions, and consent grants associated with the user
3
Export or record all apps that have been granted any of the following:
  • Delegated permissions
  • Offline access (refresh tokens)
  • Mail, Files, or Microsoft Graph scopes
2
Review Enterprise Applications
1
Navigate to Enterprise Applications in the Entra ID portal
2
Filter by recently created apps and apps with user consent
Sort by creation date — apps created around the time of compromise are highest priority
3
Flag any of the following:
  • Unknown or unrecognised publisher names
  • Unverified publishers (no blue verified badge)
  • Apps consented to near the time of the incident
3
Assess Permission Scope Risk

For each identified app, assess the permissions granted against the following high-risk scopes:

Mail.Read Mail.ReadWrite Files.ReadWrite.All offline_access User.Read.All Calendars.ReadWrite
1
For each app, determine: user consent vs admin consent, and whether refresh tokens are enabled
2
Flag as high risk if the app can access data without an interactive sign-in (offline_access) or permissions exceed business need
Risk rating guide

✅ Legitimate

Known app, minimal permissions, business-justified

⚠️ Suspicious

Unknown publisher, excessive scope, created near incident

🔴 Malicious

Unverified, offline access, data scopes, no business justification

4
Correlate with Audit & Sign-In Logs
1
Review audit logs for consent grant events and app creation events
2
Correlate app consent timestamps with phishing events, MFA abuse, and anomalous sign-ins
3
Record source IPs and the actor who performed the consent grant
5
Document Findings
1
For each suspicious or malicious app, record: app name, app ID, publisher, permissions granted, and consent timestamp
2
Attach all evidence to the SOC incident ticket
3
Mark each app as Legitimate, Suspicious, or Malicious — and document the rationale
Section 5
Expected Outcome

✅ Complete inventory of OAuth apps related to the incident
✅ Risk-rated assessment of permissions for each app
✅ Clear recommendation for revocation or retention per app

Section 6
Validation Checks
  • All apps reviewed for the affected user — no gaps
  • Audit trail captured for all consent events
  • Permissions understood and documented for each app
Section 7
Common Errors & Tips
OAuth access persists after password resets — always check for OAuth apps independently of credential remediation
"Verified publisher" does not imply the app is legitimate or safe — attackers can obtain verification
offline_access is a major persistence risk — it allows the app to access data without any further sign-in by the user
Always review Enterprise Apps, not only App Registrations — malicious apps may appear only in Enterprise Apps
Section 8
Output Artefacts
OAuth application inventory
Risk assessment notes per app
Audit log extracts
Recommendation summary (revoke / retain)
