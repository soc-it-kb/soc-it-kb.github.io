EP-PB-04: Endpoint Persistence Mechanisms

Framework for detecting, investigating, and eradicating endpoint persistence mechanisms — ensuring that endpoint compromise is fully remediated, not merely contained. Persistence must be removed before any incident can be closed.

💻 Endpoint threats Tier 2 / 3 Full eradication required MDR integrated

Section 1
Purpose

Provide a standardised decision and response framework for detecting, investigating, and eradicating endpoint persistence mechanisms used by attackers to maintain access after initial compromise.

This playbook ensures that endpoint compromise is fully remediated, not merely contained. Persistence must always be removed before any endpoint incident is closed.

Section 2
Scope

✅ In scope

  • Scheduled tasks executing unknown or suspicious binaries
  • Registry run keys and startup folder entries
  • Malicious services and drivers
  • WMI event subscriptions
  • Local user or admin account creation
  • Persistence discovered during EP-PB-01/02/03 or MDR escalations
Persistence types

Scheduled tasks

Tasks executing unknown binaries

Registry autorun

Run keys with obfuscated commands

Services

Running from user-writable paths

WMI subscriptions

Encoded payload triggers

Local accounts

New local user or admin created

❌ Out of scope

  • Identity-only persistence (mailbox rules, OAuth) → ID-PB-02 / ID-PB-04
  • Network device persistence → Network playbooks
Section 3
Trigger Conditions
  • EDR detects persistence-related behaviour (task creation, registry modification, service installation)
  • MDR escalates suspected persistence activity
  • SOC identifies startup, service, or scheduled task anomalies
  • Endpoint compromise response requires persistence validation before closure
  • Post-containment verification requires persistence assurance
Section 4
Initial Validation (Decision Logic)

Validate whether the observed mechanism is legitimate:

  • Is the persistence mechanism documented and approved?
  • Is it associated with known software or IT management tooling?
  • Was it created during an approved change window?
  • Does it align with the endpoint's role and the user's function?

If YES — legitimate → Document and close as Authorised Configuration
If NO or UNCLEAR → Proceed with persistence assessment (Section 5)

Section 5
Persistence Assessment

Assess type, intent, and risk. Presence of any indicator = Confirmed Malicious Persistence:

Scheduled task executing an unknown or unsigned binary
Registry autorun entry with obfuscated or encoded command
Service running from a user-writable or temp path
WMI event subscription with an encoded payload
New local user or admin account created without authorisation
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Persistence survives a system reboot
🔴 Persistence tied to external C2 communication
🔴 Privileged account or system involved
🔴 Persistence exists on multiple endpoints
🔴 MDR confirms hands-on-keyboard attacker activity

Section 9
Containment Decision Logic

🔴 Immediate isolation

  • Persistence is executing active payloads right now
  • External C2 communication observed
  • Multiple persistence methods present simultaneously

🟡 Controlled containment

  • Persistence is currently dormant
  • Business-critical system — isolation has high impact
  • Forensic capture required before removal

Containment must be coordinated to avoid tipping off the attacker prematurely — especially when persistence is dormant and the attacker may still have situational awareness.

Section 10
Scope Expansion Logic

Scope expansion is mandatory for all EP-PB-04 incidents:

  • Search for identical persistence artefacts (task names, registry keys, file paths) across all endpoints
  • Correlate file paths, registry keys, and task names tenant-wide
  • Identify other endpoints accessed or used by the same compromised user
  • Review lateral movement indicators (EP-PB-03) for connected compromise
  • Validate no persistence exists on related or adjacent systems
Section 11
Communication & Coordination
  • Coordinate removal timing carefully with IT operations
  • Avoid notifying the affected user until persistence is fully removed
  • Align all remediation actions with the MDR team
  • Escalate to Incident Command if persistence is widespread across multiple systems
Section 12
Closure Criteria

The incident may be closed when all of the following are confirmed:

All persistence mechanisms identified and removed
Endpoint rebooted and re-validated post-removal
No re-creation of persistence observed after removal
Credentials reset if any exposure was suspected
Monitoring confirms endpoint stability and clean state
Section 13
Evidence to Capture
Type and location of persistence mechanism
Creation timestamps
Associated binaries or scripts
User or process that created persistence
Timeline of removal and validation
Section 14
Post-Incident Actions
  • Review and update endpoint hardening baselines
  • Reduce user write permissions where unnecessarily broad
  • Enforce application control to prevent unknown binary execution
  • Tune EDR persistence detection rules
  • Feed artefacts into threat hunting use cases
  • Update endpoint recovery procedures if gaps were identified
