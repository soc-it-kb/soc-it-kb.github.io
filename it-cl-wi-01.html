IT-CL-WI-01: Tenant Identity & Conditional Access

Vendor-neutral implementation guidance for the Entra ID tenant-identity controls shared between M365 and Azure: MFA, Conditional Access, named locations, legacy authentication elimination, break-glass accounts, PIM, Identity Protection. Operationalises §4.1 of IT-CL-SOP-01. Licensing-tier-aware. Referenced as the foundation by WI-02, WI-03, WI-04, WI-05.

Prerequisites

Before applying this WI:

  • The tenant is in the Onboarding or Active state in GRC-REG-ASSET with the licensing tier recorded (A1 / E1 / A3 / E3 / A5 / E5)
  • Two break-glass / emergency-access account candidates identified (named individuals, documented justification, vaulting plan in place)
  • For tenants on A3 / E3 or above: at least two named Global Administrators with FIDO2 hardware keys provisioned, ready for AAL3-grade authentication
  • Existing Conditional Access policies (if any) reviewed and documented; the rollout plan accounts for current users with active sessions
  • Communication plan in place for users — MFA enrolment campaigns, legacy-auth deprecation notices, named-location rollouts
  • Change windows approved per organisation change-control procedures
WI-01 is the foundation for everything else in the cloud baseline. The platform-specific WIs (WI-02 M365, WI-03/04 Azure, WI-05 Defender) assume the identity controls in this WI are in place. Apply this WI first, then proceed to the relevant platform WIs.
Step-by-step instructions

The steps below cover the tenant-identity controls required by IT-CL-SOP-01 §4.1. They apply to both M365 and Azure tenants — the Entra ID layer is shared. Where tier matters, each step calls out the achievable scope at each licence tier.

1
Establish break-glass / emergency-access accounts All
Tenant recovery is possible even if Conditional Access, MFA, or directory integration fail catastrophically.

The break-glass accounts are the first thing configured because every later control could lock administrators out if misapplied. Standard pattern (two accounts per tenant):

  • Cloud-only accounts (not synchronised from on-prem AD) — recovery cannot depend on AD connectivity
  • Named after the role, not an individual (e.g. breakglass-01@tenant.onmicrosoft.com) — accountability via the check-out log, not the account itself
  • Strong, unique passwords (minimum 30 characters), stored in a vault that is itself not in this tenant (offline, separate password manager, or physical safe)
  • Permanent Global Administrator role — not PIM-eligible — so PIM unavailability does not block use
  • Excluded from every Conditional Access policy so a policy misconfiguration cannot lock them out (see Step 4)
  • MFA configuration: per organisation choice — either MFA-exempt (the simplest recovery path) with strong compensating controls, or MFA-enrolled with a separate fallback factor; document the choice and the rationale
  • Sign-in monitoring: any sign-in by a break-glass account triggers an immediate high-severity alert to SOC; expected usage is annual test or genuine emergency only
  • Annual test (controlled): sign in with each break-glass account, confirm administrative access works, rotate password, log the test
The break-glass / emergency-access pattern is the single most important step in this WI. Tenants without break-glass accounts that lose Global Administrator access have no recovery path other than vendor support escalation, which is slow and unreliable. Document, test, monitor.
Verify. Both break-glass accounts created. Password vault location confirmed independent of tenant. Annual test scheduled. SOC monitoring rule active. Each account excluded from every CA policy (re-verify after Step 4).
CIS M365 1.1.1 · CIS Azure 1.x · NIST 800-63B
2
Eliminate legacy authentication at tenant level All
Block basic-auth protocols that bypass MFA — single largest source of credential-stuffing risk in M365.

Legacy authentication (basic auth, IMAP, POP, MAPI/HTTP, SMTP AUTH, older Exchange ActiveSync, EWS in basic mode) bypasses modern authentication and therefore MFA. The vast majority of credential-stuffing attacks against M365 use legacy auth.

  • Audit first. Use the Entra sign-in logs filtered by Client App = "Other clients" / "IMAP" / "POP" / "SMTP" / "Authenticated SMTP" / "MAPI" / "Exchange ActiveSync (basic)" to see what's currently using legacy auth. Look back ≥30 days.
  • Identify legitimate users / applications. Common cases: shared mailboxes used by scripts, line-of-business apps still on basic SMTP, older mobile clients. Each gets a migration plan or a documented exception.
  • Block via Conditional Access (A3 / E3+): CA policy "Block legacy authentication" targeting all users, all cloud apps, client app = legacy auth clients only, grant = block. Exclude break-glass accounts.
  • Block via Security Defaults (A1 / E1): Security Defaults block legacy auth by default; ensure Security Defaults is enabled at tenant level (Properties → Manage Security Defaults).
  • Exchange Online additional hardening. Disable basic auth at organisation level for Exchange protocols (Set-AuthenticationPolicy); apply the policy to all mailboxes; SMTP AUTH disabled by default at tenant level for new mailboxes.
Verify. Entra sign-in logs filtered by Client App = legacy show only break-glass or documented exceptions in the previous 7 days. Exchange Online Get-AuthenticationPolicy shows the policy applied. Sample test: attempt IMAP authentication with valid credentials; rejected at tenant.
CIS M365 1.2.x · "Block legacy authentication"
3
Enforce MFA universally — Security Defaults or Conditional Access
All sign-ins (administrative and standard) protected by a second factor, with AAL3-grade for Global Administrators.

Two paths depending on tier:

  • A1 / E1 Security Defaults — Entra Properties → Manage Security Defaults → Enabled. This enforces MFA on all users (with a 14-day registration window per user), blocks legacy auth, and protects privileged actions. It is binary — on or off — but is the right control for tenants without Conditional Access licensing.
  • A3 / E3+ Conditional Access — granular MFA enforcement (see Step 4). Disable Security Defaults when CA is deployed; the two are mutually exclusive.

AAL3-grade authentication for Global Administrators

  • Global Administrators authenticate with phishing-resistant factors: FIDO2 security keys, Windows Hello for Business (where supported on managed devices), or certificate-based authentication
  • SMS, voice-call, and basic Authenticator-push MFA are not sufficient for Global Administrator (these are AAL2 at best and have known interception risks)
  • Conditional Access policy: "Require phishing-resistant MFA for admin roles" targeting Entra directory roles = Global Administrator + Privileged Role Administrator + Security Administrator + Conditional Access Administrator

Authentication methods configuration

  • Entra Authentication Methods policy: enable Microsoft Authenticator (push, with number-matching), FIDO2 keys, Windows Hello for Business, certificate-based
  • Disable: SMS as primary factor (acceptable as fallback only), voice call, secondary email as recovery
  • Enable number-matching for Microsoft Authenticator push (eliminates "MFA fatigue" attacks)
Verify. All users have at least one strong factor registered (Entra Authentication Methods report). Global Administrators show a phishing-resistant factor as primary. Sign-in logs show MFA requirement on every interactive sign-in.
CIS M365 1.1.x · NIST 800-63B AAL2/AAL3
4
Deploy Conditional Access baseline policies A3 / E3+
Granular, context-aware access control beyond simple MFA enforcement.

The Conditional Access baseline consists of a small set of policies that every A3/E3+ tenant deploys. Each policy follows the report-only → enable pattern: deploy in report-only mode for 7–14 days, review sign-in impact, then enable.

Baseline CA policies

  • CA-001 Require MFA for all users — all cloud apps, all users (including guests), grant = require MFA; exclude break-glass accounts
  • CA-002 Block legacy authentication — per Step 2
  • CA-003 Require phishing-resistant MFA for admins — per Step 3, targeting directory roles
  • CA-004 Require compliant device for admins — administrative sign-ins from Intune-compliant or hybrid-joined devices only
  • CA-005 Block sign-in from high-risk countries — define the country set with the security and compliance teams; block or require additional verification
  • CA-006 Session lifetime — sign-in frequency 8–24 hours for standard users, 4–8 hours for administrators; persistent browser session disabled
  • CA-007 Block guest access to sensitive apps — guests cannot access named applications without justification; default-deny pattern

Exclusions discipline

  • Every CA policy excludes the break-glass accounts (verify after each policy creation)
  • Service-account exclusions documented and time-bounded; reviewed quarterly
  • "Emergency access" exclusion lists are auditable; additions logged
Verify. Each CA policy created in report-only first, reviewed for unexpected impact, then enabled. Break-glass accounts excluded from all of them. Sign-in logs show CA result for every sign-in.
CIS M365 1.x · MCSB IM-x
5
Configure named locations A3 / E3+
Defined trust boundaries usable in Conditional Access policies; visibility of geo-anomalous sign-ins.

Named locations are reusable IP ranges or country lists in Conditional Access. They enable policies like "block sign-in from outside expected geographies" or "skip MFA inside corporate networks" (the latter is discouraged but the capability exists).

  • Trusted IPs — known corporate egress IP ranges (HQ networks, agency office egress, VPN egress). Used for visibility, not for MFA-bypass.
  • Country sets — define an expected-country list per tenant (where the agency operates); define a blocked-country list per the organisation's geo-risk policy.
  • Use in CA — block sign-in from blocked countries (CA-005 above); flag sign-in from unexpected countries for additional verification.
  • Do not use named locations to skip MFA — even on corporate networks, MFA still applies. This is a common configuration drift to push back on.
Verify. Named locations defined and used in CA policies. Sign-in logs show location attribution. Geo-anomaly alerts visible in Entra Identity Protection (where licensed) or via SOC SIEM queries.
CIS M365 1.x · "Named locations"
6
Enable Identity Protection risk policies E5
Real-time risk-based response to compromised credentials and anomalous sign-in behaviour.

Entra Identity Protection (Entra ID P2; available in E5) detects sign-in risk (geo-velocity, anonymous IP, unfamiliar sign-in, leaked credentials) and user risk (cumulative). Risk policies act on these signals.

  • Sign-in risk policy — risk level High = block; risk level Medium = require MFA + additional verification. Implement as Conditional Access policy using risk as the condition.
  • User risk policy — risk level High = require password change (with MFA); risk level Medium = require MFA. Same CA pattern.
  • MFA registration policy — enforce MFA registration for all users (Identity Protection feature, not CA)
  • Risk feedback loop — administrators confirm or dismiss risk events; the platform learns from feedback over time. Document the triage workflow.
  • SOC integration — risk events forwarded to central SIEM (sign-in risk, user risk, alerts). SOC has playbooks for the higher-severity events.
Verify. Risk policies enabled at the targeted levels. Sample test: simulate an anonymous IP sign-in (Tor or known anonymising proxy on a controlled test account); risk policy fires. Risk events flowing to SIEM.
CIS M365 (E5) · MCSB IM-x
7
Configure Privileged Identity Management (PIM) E5
Time-bounded eligibility for privileged roles; just-in-time activation; approval workflows for the highest-impact roles.

PIM (Entra ID P2; E5) moves privileged roles from standing assignments to time-bounded eligible-then-active activations. The principle is the same as IT-SI-WI-04 (PAM) Step 4: standing access is the exception, not the default.

  • Roles in scope: Global Administrator, Privileged Role Administrator, Security Administrator, Conditional Access Administrator, Exchange Administrator, SharePoint Administrator, User Administrator, Application Administrator — start with these; expand as the model matures
  • Eligibility, not assignment. Named individuals are eligible for each role with a defined assignment duration (typically 12 months); they activate when needed via a documented process
  • Activation requirements: MFA required at activation; justification text mandatory; activation duration max 8 hours for most roles, 1 hour for Global Administrator; ticket/change reference where applicable
  • Approval workflow for Global Administrator: activation requires approval from another named approver (not the requestor). For other roles, activation may be self-approval with auditing depending on the agency's risk appetite.
  • PIM access reviews quarterly for all eligible assignments; expired eligibility is removed; new assignments require justification
  • Alerts — PIM alerts (suspicious activation patterns, role assignments outside PIM, role activation without MFA) flow to SOC
  • Exclude break-glass. Break-glass accounts hold permanent Global Administrator assignment (Step 1); they are not PIM-eligible.
Verify. Privileged roles show eligible assignments (not active assignments) for named individuals; only break-glass and approved exceptions hold permanent assignments. Activation workflow tested. Quarterly access review scheduled.
CIS M365 (E5) 1.x · CIS Azure 1.x
8
Govern guest and external access All
External collaboration is enabled deliberately, with auditable access control, not as a default-permissive state.

External Entra users (B2B guests) have access to tenant resources through invitations. The default tenant settings are permissive; tighten them:

  • Entra External Identities settings. Guest invite settings: only administrators and users with named role assignments can invite; not "anyone." Collaboration restrictions: allow invitations only to specified domains where the organisation has explicit business relationships, OR deny invitations to specified high-risk domains.
  • Guest user permissions. "Guests have limited access" — they cannot read other users' or groups' directory information by default. CIS M365 baseline setting.
  • Guest MFA. Conditional Access policy CA-001 includes guests, so MFA is required for guest sign-in.
  • Guest access review quarterly — list guests, validate ongoing business relationship, remove stale guests. Access reviews can be automated via Entra Identity Governance (E5) or run manually.
  • Cross-tenant access settings (B2B Direct Connect, Teams shared channels) — opt-in per partner tenant; default-deny for organisations not explicitly approved.
  • Sensitive applications. CA-007 blocks guest access to a named list of sensitive applications without explicit justification.
Verify. Guest invite settings restricted. Cross-tenant access default-deny with allow-list. Guest user count auditable (Entra report). Recent guest sign-ins show MFA applied.
CIS M365 1.x · "External collaboration settings"
9
Enable self-service password reset with MFA All
Users recover from forgotten passwords without help-desk involvement; recovery itself is protected.

Self-Service Password Reset (SSPR) reduces help-desk load and improves user experience. Configure it once, correctly:

  • SSPR enabled for all licensed users
  • Authentication methods for SSPR aligned with sign-in MFA — same factor set, no weaker recovery path
  • Number of methods required to reset: 2 (the platform default of 1 weakens the recovery path)
  • Combined registration enabled — users register their MFA factor and SSPR methods in one flow, not two
  • SSPR write-back to on-prem AD (where hybrid identity is in use) — Entra Connect SSPR write-back enabled
  • Administrator accounts use the same SSPR settings, not a relaxed admin-recovery path
Verify. SSPR enabled and tested by a non-privileged user. Combined registration in effect. Administrators show MFA methods registered. SSPR audit logs flow to central SIEM.
CIS M365 1.x · "Self-service password reset"
Validation & verification

After applying all steps, validate the tenant-identity baseline is in place. Many of these checks combine cross-WI conformance signals.

Functional validation

  • Sign in as a standard user from an untrusted network — MFA prompted, succeeds
  • Sign in as a Global Administrator — phishing-resistant factor required; SMS / push alone insufficient
  • Attempt sign-in with a legacy auth client (IMAP, basic SMTP) — rejected at tenant
  • Sign in from a blocked country (test source via approved channel) — blocked with CA-005
  • Sign-in event appears in Entra logs with full context (location, client, CA result)
  • Break-glass account sign-in triggers immediate high-severity SOC alert (test under controlled conditions)
  • (E5) Risk policies fire on simulated anonymous-IP sign-in
  • (E5) PIM activation flow works end-to-end including approval where required
  • (E5) Identity Protection feedback loop tested (confirm-compromised, dismiss-as-false-positive)
  • Guest invitation requires named-role approver; new guest receives MFA challenge on first sign-in

Rollback & troubleshooting

Identity changes are visible to every user immediately; a misconfiguration can lock out the entire tenant. Have OOB recovery prepared.

Common scenarios

  • Locked out by Conditional Access misconfiguration — cause: a CA policy was deployed enabled (not report-only), or break-glass exclusion was missed. Recovery: sign in with break-glass account (which is excluded from all CA), revert the policy, re-deploy in report-only. If break-glass also locked out: Microsoft support escalation; this is the failure mode break-glass exists to prevent.
  • Phishing-resistant MFA enforced before all admins enrolled — cause: CA-003 deployed before all Global Administrators had FIDO2 keys. Fix via break-glass: temporarily disable CA-003; coordinate hardware-key issuance for remaining admins; re-enable.
  • Mass legacy-auth disruption after Step 2 — cause: legitimate line-of-business apps were using basic SMTP / IMAP. Fix: re-enable for the specific service-account / mailbox via tenant-level auth policy override; migrate the application to modern auth on a defined timeline; re-disable for that account afterwards.
  • Security Defaults disabled accidentally and CA not deployed — cause: switching tenant from A1 to A3 mid-rollout; Security Defaults was disabled (it must be when CA is used) but the CA-001 policy was not yet active. Fix: re-enable Security Defaults until CA-001 is in place, or expedite CA-001 deployment.
  • PIM activation broken — cause: approver group misconfigured or PIM service issue. Fix via break-glass admin access; correct approver configuration; for service issues, vendor support; in the interim, document-justified temporary assignment is acceptable but tracked.
  • Guest access lost after CA-007 — cause: a legitimate external collaborator is in a blocked guest set. Fix: targeted exception (named guest, named application, time-bounded); not a global CA-007 disablement.
  • Identity Protection blocking false positives — cause: legitimate user behaviour resembles risk signals (VPN exit nodes, frequent travel). Fix: dismiss as false positive (Identity Protection learns over time); for persistent patterns, named-user exclusion from the risk policy with documented justification and quarterly review.

Rollback procedure

  1. If administrators are locked out: use break-glass account; this is its purpose
  2. Identify the breaking change from CA policy state, sign-in logs, and recent changes
  3. Revert the specific change — not the whole change set unless required
  4. For CA: switch policy to report-only or disabled; review and re-test before re-enabling
  5. For PIM: extend specific eligibility temporarily; do not wholesale disable PIM
  6. Update change record; document root cause; if break-glass was used, rotate the break-glass password and reconcile sign-in audit with the manual log
Common errors & tips
  • Break-glass first, everything else after. Configuring CA before break-glass exists is how organisations lock themselves out of their own tenant. Pattern: break-glass → legacy auth block → MFA → CA → advanced.
  • Break-glass passwords in the tenant they protect. Don't store the break-glass password in OneDrive, Teams, or SharePoint of the same tenant. If the tenant is the failure, the recovery credential is unreachable.
  • Security Defaults vs Conditional Access. Mutually exclusive; if you enable CA, Security Defaults must be off. Forgetting this leaves users unprotected during transition.
  • Phishing-resistant MFA requires hardware factor procurement. FIDO2 keys cost money and time to distribute. Order them before the rollout deadline, not during.
  • Report-only mode is not optional. Every CA policy goes through report-only for at least a week before enabling. Skipping this is how policies break sign-in for real users you didn't anticipate.
  • SMS as primary MFA is AAL2, not AAL3. For Global Administrators it is insufficient. The CIS M365 baseline calls this out specifically.
  • Trusted IPs don't bypass MFA — they shouldn't. A common drift is "skip MFA inside corporate network." Network location is no longer a trust signal; CA-001 should require MFA regardless of location.
  • Guest access defaults are permissive. Out-of-the-box, any user can invite a guest. Tighten this in Step 8; otherwise external collaboration is uncontrolled.
  • PIM eligibility is not PIM activation. An admin being "eligible" for Global Administrator does not mean they have the role; they have to activate it. This is the whole point. Configure access reviews so eligibility itself is reviewed.
  • Identity Protection is E5-only. For A3/E3 tenants, the risk-based controls are unavailable; baseline conformance is reported "feature unavailable at this licence tier" not "exception." Mark these clearly in the per-tenant conformance report.
  • Combined registration matters. Without it, users register MFA in one place and SSPR in another, see two separate prompts, and often miss one. Single unified flow.
  • SOC sign-in monitoring for break-glass is non-optional. A break-glass account that signs in unexpectedly is an incident, not a routine event. Alert on it in the SIEM with high severity.
