EP-PB-05: Ransomware Activity

Critical incident response playbook for suspected or confirmed ransomware activity, including encryption behaviour, mass file modification, shadow copy deletion, ransom notes, destructive commands, and coordinated containment across endpoint, identity, network, backup, and business recovery teams.

💻 Endpoint threats Critical severity Tier 3 / Incident Command Containment before recovery

Section 1
Purpose

Provide a structured response framework for suspected or confirmed ransomware activity. The objective is to stop propagation, preserve evidence, protect recoverability, and coordinate business recovery without increasing attacker visibility or destroying forensic value.

Critical rule: do not reboot, wipe, restore, or reconnect affected systems until containment, evidence preservation, and recovery approval have been completed by Incident Command.

Section 2
Activation Triggers
  • EDR/XDR alert for ransomware, encryption activity, suspicious file renaming, or destructive tooling
  • Mass file modification, high-volume write activity, ransom note creation, or unusual archive/exfiltration staging
  • Shadow copy deletion, backup agent tampering, service stopping, or security tool impairment
  • Multiple endpoints or servers reporting similar encryption, deletion, or impact behaviour
  • User, IT, MDR, or business report of inaccessible files, ransom note, or suspicious remote access activity

Section 3
Immediate Triage Questions

Impact validation

  • Which systems show encryption, deletion, or service disruption?
  • Are shared drives, file servers, databases, or virtual infrastructure affected?
  • Is there evidence of backup tampering or shadow copy deletion?

Spread validation

  • Are there multiple hosts with the same indicators?
  • Is lateral movement visible through RDP, SMB, WinRM, SSH, VPN, or remote admin tools?
  • Is the initiating user, host, or privileged account still active?
Section 4
Immediate Containment Actions
  • Declare a critical incident and activate Incident Command
  • Isolate affected endpoints using EP-WI-02
  • Block lateral movement paths using network segmentation controls and NW-WI-03
  • Disable or contain suspected compromised accounts and privileged sessions
  • Preserve affected systems for evidence collection before rebuild or restore
  • Protect backup repositories, recovery platforms, hypervisors, and domain controllers from attacker access

Where encryption is active, containment takes priority over normal change windows. Use emergency change procedures and record all actions in the incident ticket.

Section 6
Scope Expansion Logic

Scope expansion is mandatory. Search across endpoints, identity, network, email, servers, backup infrastructure, and remote access logs.

Endpoint scope

  • Same file hashes, process names, command lines, parent processes, ransom note names
  • Hosts accessed by the same user or source system
  • Hosts showing security tool disablement or shadow copy deletion

Identity and network scope

  • Privileged account use, failed logons, new admin membership, remote access sessions
  • RDP, SMB, WinRM, SSH, VPN, remote admin tools, and outbound staging destinations
  • Unusual access to backup, hypervisor, file server, and domain controller assets
Section 7
Recovery Decision Gates
  • Containment confirmed and no active encryption or lateral movement observed
  • Evidence collected and approved for preservation by Incident Command
  • Root cause or likely initial access path sufficiently understood to prevent reinfection
  • Backup integrity validated and recovery point approved by business owner
  • Rebuild, restore, and reconnect actions approved through emergency change control
Section 8
Communication and Coordination
  • Notify MDR, CISO / cyber leadership, IT operations, Legal, Communications, Data Protection, and affected business owners
  • Use approved incident communications only; do not speculate on attribution, data loss, or payment decisions
  • Coordinate with backup, infrastructure, identity, network, and application owners before restoration
  • Preserve attacker communications, ransom notes, wallet addresses, and contact channels as evidence
Section 9
Evidence to Capture
Ransom note filename, content, and timestamp
Affected host list and encryption timeline
Suspicious process tree and command lines
User and privileged account activity
File hashes, IPs, domains, and remote access artefacts
Backup and recovery integrity evidence
Section 10
Closure Criteria
  • All impacted systems isolated, rebuilt, restored, or formally risk-accepted
  • No active ransomware behaviour or attacker persistence observed after monitoring period
  • Compromised credentials reset and privileged access paths reviewed
  • Recovery evidence, business validation, and legal/compliance obligations documented
  • Lessons learned completed and corrective actions assigned to owners
