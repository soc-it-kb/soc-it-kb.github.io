NW-WI-02: Block IP / Domain Indicators

Tool-agnostic procedure for applying network blocks against malicious IP addresses, domains, and URLs — stopping C2 communication, preventing exfiltration, and containing active network threats once approved.

Section 1
Purpose

Provide a consistent, tool-agnostic procedure to block malicious network indicators (IP addresses, domains, URLs) to stop C2 communication, prevent data exfiltration, contain active network threats, and reduce attacker reach and persistence.

📌 Scope boundary: This WI does not authorise blocking decisions. It defines how to apply blocks safely once approved by Tier-2 or via SOAR. Analysis is performed via NW-WI-01 first.

Section 5
Pre-Checks & Safety Notes
🚫 Mandatory constraints
Do NOT block shared cloud or CDN infrastructure without explicit approval
Do NOT expand scope beyond what is approved
Do NOT apply permanent blocks without documented approval
Prefer domain blocking over IP blocking where possible

⚠️ If the indicator is a widely shared CDN, cloud provider, or partner IP — pause and escalate. Broad blocks can cause significant business disruption.

Section 6
Step-by-Step Actions
1
Validate the Indicator
1
Review the indicator value carefully — confirm exact IP, domain, or URL. Verify it matches what was observed in network logs, NDR alerts, or MDR escalation and that the indicator type aligns with the approved action.
2
Confirm Blocking Method & Scope
Indicator typeTypical blocking control
IP addressFirewall deny rule / NDR block policy
DomainDNS filtering / Proxy / Secure Web Gateway
URLProxy / Secure Web Gateway

⏱️ Temporary block

Incident-scoped — set review date at closure

🔒 Permanent block

Threat-intel validated — requires documented approval

3
Apply the Block
1
Add the indicator to the approved control — deny/block list or sinkhole for domains
2
Record: rule or policy name, timestamp, operator, scope, and duration
4
Validate Block Effectiveness
1
Confirm:
  • Connections to the indicator are now blocked
  • Block events are appearing in logs
  • No successful connections persist
2
If traffic continues — do not widen scope automatically. Escalate to Tier-2 for reassessment.
5
Monitor for Secondary Indicators
1
After blocking, watch for attacker adaptation:
  • Alternate IPs or domains used by the same threat actor
  • Protocol switching to bypass the block
  • Resumed beaconing from a different source host
2
Report any new indicators immediately — rapid infrastructure rotation signals an active and capable attacker
6
Preserve Evidence & Document
1
Do not remove logs or telemetry. Maintain the block until reviewed by Tier-2.
2
Update the incident record with: indicator(s) blocked, control and scope, timestamp, validation outcome, and any observed side effects. Attach screenshots or log excerpts.
Section 7
Expected Outcome

✅ Malicious network communication blocked at the approved enforcement point
✅ C2, exfiltration, or malicious inbound traffic stopped
✅ Clear audit trail with rule reference, scope, and timestamp

Section 8
Common Mistakes to Avoid
Blocking shared cloud or CDN infrastructure without validating that only the malicious use is affected
Blocking IPs where domain-level blocking is safer and more precise
Applying permanent blocks without documented Tier-2 approval
Expanding scope without reassessment — one indicator does not justify blocking related infrastructure
Failing to document the rule details — blocks without records cannot be reviewed or removed safely
Section 9
Escalation Conditions
Blocking causes unexpected business disruption
Indicator appears to be shared or widely used infrastructure
Multiple systems are affected and scope needs reassessment
New indicators emerge rapidly after blocking — active attacker infrastructure rotation
Blocking fails to stop the activity
