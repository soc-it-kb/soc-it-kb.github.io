ID-RB-02: Review and Remove Malicious Mailbox Rules

Identify and remove malicious or unauthorised mailbox rules — a common post-phishing persistence technique used to hide activity, exfiltrate data, or maintain attacker visibility.

🔐 Identity threats Tier 1 / 2 Persistence removal

🏷️ Category: Identity Threat

Active
Section 1
Purpose

Identify and remove malicious or unauthorized mailbox rules commonly used for persistence after phishing or account compromise.

Suspicious rule types to look for

Auto-delete

Silently moves or deletes messages to hide attacker activity

Auto-forward

Redirects email to an external address for data exfiltration

Keyword-based

Targets financial, security, or HR terms — selectively hides content

Hidden / obfuscated

Non-descriptive names like "." or blank — designed to avoid detection

Section 2
Required Access
🔑 Exchange Admin access 📊 Audit log access
Section 3
Inputs Required
📬 User mailbox 🎫 Incident reference
Section 4
Step-by-Step Procedure
1
Access the mailbox rule configuration via Exchange Admin Center or PowerShell
2
Review all inbox rules — including those with non-descriptive or blank names
Use PowerShell (Get-InboxRule) to enumerate rules that may not appear in the UI
3
Identify suspicious rules using the following indicators:
  • Rules that auto-delete or move messages to Deleted Items
  • Rules that auto-forward to external addresses
  • Rules with obfuscated, blank, or non-descriptive names
  • Rules targeting keywords like "invoice", "payment", "password", "security"
4
Disable and remove all identified malicious rules
Document each rule before removal — name, conditions, actions, and creation timestamp are all evidence
5
Check and disable external forwarding settings on the mailbox (see EM-RB-07)
6
Review rule creation timestamps — correlate with the suspected compromise timeframe
7
Document all findings, removed rules, and timestamps in the incident ticket
Section 5
Expected Outcome

✅ All malicious or unauthorised rules removed
✅ Mailbox persistence fully eliminated
✅ External forwarding disabled

Section 6
Validation Checks
  • No unauthorised inbox rules remain (confirmed via PowerShell and UI)
  • External forwarding confirmed disabled
Section 7
Common Errors & Tips
Attackers frequently use benign-sounding or blank rule names — never assume a rule is legitimate based on its name alone
Check both inbox rules (user-level) and transport rules (tenant-level) — malicious actors may have created rules at either layer
Use PowerShell (Get-InboxRule -Mailbox user@domain) to enumerate all rules including those hidden from the standard UI
Section 8
Output Artefacts
Removed rule details (name, conditions, actions)
Rule creation timestamps
Forwarding configuration record
