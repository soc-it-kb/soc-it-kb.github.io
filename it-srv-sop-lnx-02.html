IT-SRV-SOP-LNX-02: Linux Identity & SSH Hardening Addendum

Identity, authentication, and SSH hardening requirements for Linux servers — protecting privileged access and supporting SOC detection and response.

Section 1
Purpose

This addendum defines highly recommended identity, authentication, and SSH hardening requirements for Linux servers. It strengthens the Linux Server Security Baseline by introducing identity-first controls that protect privileged access, reduce credential abuse, and support SOC detection and response.

⚠️ This addendum supplements IT-SRV-SOP-LNX-01 — it does not replace it. The full Linux Server Baseline must be in place before applying these additional controls.

Section 2
Threat Model

This addendum mitigates credential theft via weak SSH configurations, abuse of shared accounts and unmanaged keys, privilege escalation through misconfigured sudo, persistence via unauthorized SSH keys, and lateral movement using service accounts. Controls assume breach elsewhere and focus on protecting administrative access.

Section 3
Identity & Access Principles

🚫 No shared identities

No shared accounts for human access — every user uniquely identified

🔑 Key-based authentication

Default for all SSH access — password authentication disabled

🎯 Least privilege via sudo

No blanket sudo access — scoped per role and logged

📊 Full auditability

All access and privilege use logged and forwarded to SOC

Section 4
Mandatory Controls

🔒 SSH Authentication Hardening

  • Password-based SSH authentication disabled
  • Root login via SSH disabled
  • SSH access restricted to approved users/groups
  • Strong cryptographic algorithms enforced
  • Idle session timeout configured

🔑 SSH Key Management

  • Unique key per user with passphrase where feasible
  • Keys rotated periodically
  • Unauthorized or orphaned keys removed
  • Centralised key management preferred

⚡ sudo Configuration

  • Direct root login prohibited
  • Escalation via sudo only — least privilege scoped
  • No blanket ALL access
  • sudo usage logged and monitored

🤖 Service Accounts

  • Non-interactive shells where possible
  • Restricted to required hosts and commands
  • Key-based authentication only
  • No unrestricted sudo access

🚫 Explicitly prohibited: shared SSH keys between users, password-based SSH for administrators, direct root access for routine administration, storing private keys in scripts or repositories. Violations are treated as security incidents.

Section 5
Governance & Review

Owner: Cyber Security / Platform Engineering  ·  Review Cycle: Annual or post-incident  ·  Triggers: Identity-related incidents, CIS benchmark updates, authentication architecture changes
