ID-WI-04: Review OAuth Applications

Tool-agnostic procedure for reviewing OAuth applications and delegated permissions to identify malicious consent grants, excessive permissions, and token-based persistence — investigation only, no revocation actions.

🔐 Identity threats Tier 1 / 2 Investigation only No revocation actions
🔧 WI ID: ID-WI-04
·
👤 Author: Paulo Ferreira
·
📅 Published: 7 January 2026
·
🔍 Investigation only
·
Active
Section 1
Purpose

Provide a consistent, tool-agnostic procedure to review OAuth applications and delegated permissions associated with a user or tenant to identify malicious or suspicious apps, excessive permissions, persistence mechanisms via OAuth tokens, and indicators of token abuse following phishing or BEC.

📌 Scope boundary: Investigation and evidence collection only. OAuth revocation is performed via ID-WI-05 once this review is complete and Tier-2 has approved action.

⚠️ OAuth tokens persist after password reset. An attacker with a valid OAuth token retains access even if the user's password is changed — until the token is explicitly revoked.

Section 5
Pre-Checks & Safety Notes
🚫 Mandatory constraints — this WI is read-only
Do NOT revoke OAuth permissions in this WI
Do NOT notify the user during review
Assume tokens may still be valid and actively used
OAuth abuse may persist after password reset — treat as live
Section 6
Step-by-Step Actions
1
Identify OAuth Applications in Scope
1
Locate all OAuth applications associated with the target user (or tenant if instructed) — include both user-consented and admin-consented apps
Note admin-consented apps separately — revocation for these requires elevated approval.
2
Ensure both active and recently added apps are visible — do not limit view to current sessions only
2
Review Application Metadata
1
For each app, record: application name, application ID, publisher/developer, consent type (user vs admin), and consent timestamp. Flag:
  • Generic, misleading, or random-looking app name
  • Unknown or unverifiable publisher
  • Consent timestamp aligns with phishing or incident timing
3
Review Granted Permissions
1
For each app, identify permission type (delegated vs application-level) and full permission scope. Flag any permissions enabling:
  • Reading or sending email
  • Accessing files or cloud storage
  • Directory or user profile access
  • Offline or long-lived access (offline_access)
See the quick reference card above for the full list of high-risk permission strings.
4
Review Consent Context
1
Assess how consent was granted and flag anomalies:
  • User consented to high-privilege permissions without IT approval
  • Consent granted during or immediately after a phishing event
  • Consent occurred outside normal business hours
  • No MFA challenge recorded at time of consent
5
Correlate Token Usage with Sign-In Activity
1
Where available, correlate OAuth app activity with sign-in logs. Flag:
  • App activity continues after password reset
  • Access from unusual locations or IPs via OAuth
  • Non-interactive (token-based) access without user sign-in events
  • Access patterns inconsistent with the user's normal behaviour
6
Assess Risk & Summarise Findings

⚪ No action

Known app, low permissions, consent timing is normal

⚠️ Review by Tier-2

Uncertain publisher, elevated permissions, or timing overlap

🔴 Revoke

Confirmed malicious — invoke ID-WI-05 after Tier-2 approval

📌 Prepare a summary for the incident ticket including: number of apps reviewed, suspicious apps identified, permissions of concern, timeline correlation, and recommendation.

Section 7
Expected Outcome

✅ Full visibility of OAuth applications associated with the user or tenant
✅ Suspicious or high-risk apps identified with permission detail
✅ Evidence ready to support revocation or closure decisions via ID-WI-05

Section 8
Common Mistakes to Avoid
Ignoring "legitimate-looking" app names without checking publisher and permissions — attackers use convincing names
Overlooking user-consented apps — admin-consented apps are not the only risk
Assuming password reset invalidates OAuth tokens — it does not
Failing to review the depth of permission scope — offline_access combined with Mail.Read is extremely high risk
Confusing admin-consented apps with lower-risk apps — admin consent means broader scope, not safer
Section 9
Escalation Conditions
App grants mail, file, or directory access — especially with offline_access
Consent timestamp aligns with confirmed phishing or BEC activity
App remains active and accessing data post password reset
Publisher is unknown, unverifiable, or suspicious
Multiple users affected by the same application
