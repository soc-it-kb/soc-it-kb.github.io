ID-PB-04: OAuth App Abuse / Malicious Consent Grant

Standardised framework for detecting, investigating, and remediating OAuth application abuse — including malicious or overly permissive consent grants that allow attackers to maintain persistent access without credentials.

Section 1
Purpose

Provide a standardised decision and response framework for detecting, investigating, and remediating OAuth application abuse, including malicious or overly permissive consent grants that allow attackers to maintain access without credentials.

This playbook addresses one of the most persistent post-phishing attack paths, where password resets alone do not evict the attacker.

Section 2
Scope

✅ In scope

  • Malicious OAuth app registrations
  • User-granted OAuth consent with excessive permissions
  • Token-based access abuse (Graph, mail, files, calendar)
  • MDR or SOC detections of suspicious app activity
  • OAuth artefacts discovered during ID-PB-01 / ID-PB-03 investigations

❌ Out of scope

  • Legitimate enterprise app onboarding (IAM governance)
  • Service principal compromise without user consent (handled separately)
Section 3
Trigger Conditions
  • MDR escalates suspected OAuth abuse
  • SOC identifies a new or suspicious OAuth app consent
  • Identity platform flags risky consent activity
  • OAuth artefacts discovered during account compromise response
  • User reports unusual mailbox or file access without visible sign-ins
Section 4
Initial Validation (Decision Logic)

Confirm the activity is OAuth-related by checking for any of the following:

  • New app consent granted recently without IT approval?
  • Permissions include mail, files, or offline access?
  • App not recognised or not on the business-approved list?
  • Data access observed without corresponding interactive sign-in events?

If NO to all → Route to appropriate identity playbook
If YES to any → Proceed with OAuth abuse assessment (Section 5)

Section 5
OAuth Abuse Assessment

Assess risk and intent. Presence of any indicator confirms OAuth risk:

App not on approved / known list
Excessive permissions granted (see high-risk list below)
Consent granted at or near the time of a phishing event
App publisher is unverified or unknown
Token activity detected without corresponding sign-in events
High-risk OAuth permission scopes
Mail.Read Mail.ReadWrite Files.ReadWrite.All offline_access Contacts.ReadWrite Calendars.ReadWrite User.Read.All Directory.ReadWrite.All
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Mailbox or file access permissions granted to the app
🔴 Offline access or refresh tokens enabled
🔴 Privileged or VIP user consented to the app
🔴 OAuth artefact persists after a password reset has been performed
🔴 MDR confirms active data access via the OAuth token

Notify immediately

Section 7
Response Strategy

The following actions must be orchestrated across the response team:

OAuth app inventory review — enumerate all consented appsRequired
Permission assessment — evaluate scope of access grantedRequired
Consent revocation — remove malicious app authorisationRequired
Token invalidation — revoke all issued tokens and refresh tokensRequired
Sign-in and audit log reviewRequired
Account containment (if sign-in compromise also present)Conditional
Scope expansion to other affected usersConditional

Section 9
Persistence & Access Validation Logic

After remediation, confirm all of the following before proceeding to closure:

OAuth app is no longer authorised on the account
Access tokens and refresh tokens have been revoked and invalidated
No continued data access without a corresponding interactive sign-in
No additional OAuth artefacts exist for the same user

⚠️ If access persists after revocation → escalate to ID-PB-05 (Privileged Account Compromise) — the attacker may have deeper persistence

Section 10
Scope Expansion Logic

Triggers for expansion

  • Same OAuth app consented by multiple users
  • Consent granted from the same IP or ASN
  • Similar permission scopes seen tenant-wide
  • MDR identifies campaign-level behaviour

Expansion actions

  • Tenant-wide OAuth app inventory review
  • Search for users with similar consent patterns
  • Correlate consent timestamps with phishing events
Section 11
Communication & Coordination

Inform the affected user of the following:

  • That a third-party application was accessing their data
  • What specific permissions were granted to the app
  • That access has been fully revoked and what they need to do next

Coordinate with Identity Engineering to address governance gaps — specifically whether user consent should be restricted to admin-approved apps only. Align all actions with the MDR ticket workflow.

Section 12
Closure Criteria
  • Malicious OAuth app fully revoked from all affected accounts
  • All tokens and refresh tokens invalidated
  • No residual data access observed
  • Sign-in and audit logs reviewed and confirmed clean
  • User confirms no further abnormal behaviour
Section 13
Evidence to Capture
OAuth app name, ID, and publisher
Permissions granted (full scope list)
Consent timestamp and source IP
Token revocation confirmation
Audit and sign-in log excerpts
Section 14
Post-Incident Actions
  • Recommend consent governance controls — admin-only consent or restricted scope policies
  • Review Conditional Access policies for OAuth app access
  • Tune MDR and SOC detections for consent abuse patterns
  • Update phishing playbooks to include OAuth artefact checks as a standard step
  • Feed app indicators and publisher details into threat intelligence
