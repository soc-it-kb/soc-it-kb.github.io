Security Infrastructure Baselines

Vendor-agnostic security baselines for Web Application Firewalls (WAFs), load balancers, and Privileged Access Management (PAM) platforms. These devices and platforms sit at high-value trust boundaries between users, applications, and credentials — a weakness on any of them undermines protections everywhere else. Aligned to CIS Benchmarks where vendor-specific benchmarks exist, with controls organised by device type and a shared management-plane WI common to all three.

📌 About this domain. WAFs, load balancers, and PAM platforms are easy to overlook in baseline programmes because they're often deployed once and then mostly left alone, treated as infrastructure rather than as security controls. They are security controls — a WAF that's never tuned drifts into "alert-only-then-ignored" mode within a year, a load balancer with default cipher suites enables TLS downgrades, and a PAM platform with weak session-broker hardening becomes the single highest-value target in the estate. These baselines define the minimum security state for each.
Scope

This domain covers three classes of security-infrastructure platform:

  • Web Application Firewalls (WAFs) — both on-premises appliances (F5 ASM, Imperva, FortiWeb, Barracuda) and cloud-managed WAFs at the application or CDN tier (Azure Front Door / Application Gateway WAF, Cloudflare, AWS WAF). Cloud-WAF tenant-platform integration specifics defer to the future Cloud Baselines domain; the controls in this domain cover what every WAF must enforce regardless of where it runs.
  • Load balancers and application delivery controllers — F5 LTM, Citrix ADC (NetScaler), HAProxy Enterprise, NGINX Plus, A10, AVI Networks, and equivalent. Includes both hardware appliances and virtual/software instances.
  • Privileged Access Management (PAM) platforms — CyberArk PAM, BeyondTrust Privileged Remote Access / Password Safe, Delinea (Thycotic) Secret Server, HashiCorp Vault (as a credential broker), and equivalent. Includes the appliance-hardening layer; credential-vault and session-brokering policy detail defer to the future Identity domain SOP.

Out of scope: Reverse proxies and API gateways (NGINX as RP, Kong, Apigee, Azure APIM, AWS API Gateway) — deferred to a future Application Delivery sub-domain. End-user devices, servers, and network devices — covered under their respective ASM domains. SaaS-hosted PAM or WAF where the vendor controls the security configuration entirely — these still appear in the Asset Register; the baseline applies to anything tenant-administered.

Device-type Overview

Unlike the network baselines, where switches, routers, firewalls, and APs share a common conceptual structure (management / control / data plane), WAFs, load balancers, and PAM platforms have distinct architectures and threat models. The baseline groups shared controls (management plane, logging, change control, certificate hygiene) into one Work Instruction common to all three, then has device-type-specific WIs for the unique controls of each.

🛡️

Common Mgmt & Logging

Management plane, AAA + MFA, secure protocols, NTP, syslog, certificate lifecycle, change control. Applies to all three device types. IT-SI-WI-01.

🌐

WAF Hardening

Rule-set selection, blocking versus monitor mode, signature/learning tuning, false-positive management, TLS termination, OWASP Top 10 coverage. IT-SI-WI-02.

⚖️

Load Balancer Hardening

SSL/TLS termination policy, cipher suites, certificate management, virtual-server configuration, health checks, persistence, anti-DDoS. IT-SI-WI-03.

🔐

PAM Platform Hardening

Vault hardening, session broker, recording integrity, network exposure, HA pair, break-glass recovery, integration with identity sources. IT-SI-WI-04.

Why a shared common-controls WI. All three device types need the same management-plane and logging controls (AAA, MFA, syslog, NTP, change control). Putting these in a shared WI keeps the device-type-specific WIs focused on what's actually unique, avoids three near-identical copies of the same content, and means a change to the shared controls updates one document instead of three.
Core SOP
IT-SI-SOP-01
Security Infrastructure Baseline
Authoritative policy-level baseline for WAFs, load balancers, and PAM platforms. Defines shared principles, minimum security requirements per device type, framework alignment, roles, and governance. Operationalised by IT-SI-WI-01 through IT-SI-WI-04.
Work Instructions
IT-SI-WI-01
Common Management Plane & Logging
Shared management-plane controls applicable to all three device types: AAA + MFA, SSH/HTTPS only, management network restriction, NTP, syslog to SIEM, certificate hygiene, change control discipline. Referenced by WI-02/03/04.
IT-SI-WI-02
WAF Hardening
WAF-specific controls: rule-set selection, monitor-then-block progression, OWASP Top 10 / OWASP CRS coverage, signature engine and learning tuning, false-positive management, TLS termination policy, bot management, application allowlisting.
IT-SI-WI-03
Load Balancer Hardening
Load-balancer-specific controls: TLS termination with strong cipher suites, certificate lifecycle (auto-renew, CT logging, key strength), virtual-server hygiene, health-check security, session persistence, anti-DDoS / connection-rate limiting, source-IP preservation.
IT-SI-WI-04
PAM Platform Hardening
PAM-specific controls: vault encryption and HSM integration, session-broker hardening, session recording integrity, just-in-time access enforcement, identity-source integration, HA pair and disaster recovery, break-glass procedure with vault-independent recovery.

For WAFs and PAM platforms, no single CIS-style benchmark covers the major vendors comprehensively. The baselines therefore lean more heavily on OWASP (for WAFs) and NIST 800-63B + vendor hardening guides (for PAM). Where the device is an F5 product, the CIS F5 BIG-IP benchmark provides primary alignment. Where the device is something else, the vendor's published hardening guide plus the NIST 800-53 families above apply.

Roles & Ownership

Network Engineering

Owns implementation of WAF and load-balancer hardening. Maintains golden configurations. Executes change windows. First-line responder for WAF/LB incidents.

Identity & Access Engineering

Owns implementation of PAM platform hardening. Maintains PAM configuration. Coordinates with Network Engineering on appliance-hardening aspects.

IT Operations Lead

Owns the baseline document and accepts deviations. Approves change windows. Owns the patch and firmware lifecycle.

Application Teams

Co-own WAF policy for their applications (rule-set selection, false-positive triage, allowlisting). Engage with Network Engineering on WAF tuning.

SOC

Consumes logs from WAF (block events, anomalies), load balancers (anomalous traffic patterns), PAM (privileged session events). Investigates and escalates.

GRC

Owns governance of the baseline domain. Validates compliance periodically. Maintains audit trail. Records exceptions and deviations.

Metrics & Oversight

The following metrics measure adherence to this baseline. They are reviewed monthly by IT Operations and quarterly by GRC:

  • Baseline conformance rate — percentage of in-scope WAFs, load balancers, and PAM platforms fully conformant to IT-SI-SOP-01. Target: ≥95% for Tier 0/1 assets; ≥85% for Tier 2/3.
  • WAF coverage — percentage of in-scope public-facing web applications protected by a WAF in active blocking mode (not monitor-only). Target: 100% for Tier 0/1 applications; ≥80% for Tier 2.
  • WAF rule-set currency — percentage of WAFs running current signature/rule-set versions (≤30 days behind vendor release). Target: ≥95%.
  • TLS cipher conformance — percentage of load-balancer virtual servers exposing only approved cipher suites (TLS 1.2 minimum, modern AEAD ciphers). Target: 100% for internet-facing; ≥95% for internal.
  • Certificate currency — percentage of in-scope certificates expiring within the next 30 days that have a renewal in progress. Target: 100%.
  • PAM coverage — percentage of in-scope privileged accounts and credentials managed through PAM (not stored elsewhere). Target: ≥95% for Tier 0/1 systems; ≥85% for Tier 2/3.
  • PAM session-recording completeness — percentage of privileged sessions in scope of recording that have an intact recording in the retention store. Target: ≥99%.
  • Open exceptions — count of approved deviations from baseline in force. Reviewed annually for currency.
Exception Handling

Where a platform cannot meet the baseline (vendor limitation, operational constraint, integration requirement, in-flight migration), the operating unit submits a deviation request via the standard ASM exception process (IT-GOV-SOP-01). Deviations are reviewed and approved by the IT Operations Lead with GRC consultation. Approved deviations are recorded in the Baseline Exceptions Register (GRC-REG-EXCEPTIONS), linked from the affected asset records in GRC-REG-ASSET, and reviewed annually for currency.
