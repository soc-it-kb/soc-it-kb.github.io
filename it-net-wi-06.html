IT-NET-WI-06: Configuration Management & Change Control

Vendor-agnostic implementation guidance for keeping the network baseline real over time: golden configurations, backups, change control, drift detection, firmware lifecycle, secure decommissioning. Operationalises §4.6 of IT-NET-SOP-01.

Prerequisites

Before applying this WI:

  • The configuration management tool (or vendor-cloud-management equivalent) is in place and reachable from devices
  • A central config repository (Git, vendor-cloud-config-storage, or equivalent) is available
  • Change control process is documented per organisation policy (often this WI is the operationalisation of that process)
  • Firmware/OS upgrade procedures are documented per vendor
  • The asset register (GRC-REG-ASSET) is current; each device has an asset record
  • This WI applies continuously to every device throughout its lifecycle, not as a one-time activity
This WI is foundational. Without configuration management discipline, the other WIs' controls drift away over time. WI-06 is what keeps the baseline real after the initial hardening project completes.
Step-by-step instructions

The steps below cover the configuration management and change control controls required by IT-NET-SOP-01 §4.6.

1
Establish golden configuration per device class
A documented "this is what good looks like" reference for each device role.

Define golden configurations:

  • Per device class — access switch, distribution switch, core switch, branch router, edge firewall, internal firewall, wireless controller, wireless AP
  • Per agency where needed — if agencies have different conventions (VLAN IDs, IP ranges), document agency variations as named overlays on the base golden config
  • Template format — configuration as code where possible (Jinja2 templates, vendor templating systems, IaC tools)
  • Storage — version-controlled repository (Git), reviewed by Network Engineering before changes are accepted
  • Variables — hostname, IP, VLAN IDs are variables; the structural configuration is templated

The golden config is not the entire running configuration of any one device — it's the security-relevant structure that should be the same across the class.

Verify. A golden config exists for each device class in scope. It is in version control. It is documented (purpose, last updated, owner). Spot-check: render the template for a real device; compare to expected.
CIS Configuration management: "golden config"
2
Automated configuration backups
Every device's current configuration is recoverable without manual intervention.

Backup configuration:

  • Frequency — daily for Tier 0/1 devices, weekly for Tier 2/3; on every configuration change for any tier where the tooling supports it
  • Destination — central repository, not on the device itself; not vendor-cloud-only (vendor outage shouldn't lock you out of recovery)
  • Retention — 90 days minimum of historical backups for Tier 0/1; 30 days for Tier 2/3
  • Encryption — backups encrypted at rest; passwords and shared secrets in backups treated as Confidential L3 minimum
  • Tested restore — annual restoration test (controlled lab device) confirms the backups are usable
Verify. Backup system shows recent successful backups for all in-scope devices. Pull a backup from a week ago; confirm it is intact and contains expected content. Run an annual restore test and document results.
CIS Configuration management: "backup & recovery"
3
Change control workflow
Every configuration change is requested, reviewed, approved, applied, and recorded.

Change control elements:

  • Standard change — pre-approved, low-risk, repeatable (e.g. add a documented user to AAA, enable a port from the unused pool). Logged, not individually approved.
  • Normal change — reviewed and approved via change advisory board or equivalent. Most rule changes, firmware updates, topology changes.
  • Emergency change — rapid path for security-related fixes (close an open management port, block a malicious source). Expedited approval, post-hoc review within 5 working days.
  • Change record — what was changed, why, who reviewed, who applied, when, how it was verified, rollback path
  • Maintenance windows — defined by tier and operational impact; coordinated across operating units for shared infrastructure
Verify. Sample recent changes; each has a change record with the required fields. Sample emergency changes; each has a post-hoc review entry.
CIS Configuration management: "change control"
4
Configuration drift detection
Configuration drift from golden config is detected and triaged.

Drift detection:

  • Automated daily or near-real-time comparison of each device's running config against its expected (golden + agency overlay + device variables) state
  • Drift report flags differences; categorised: structural (template changed) vs incidental (uptime counters, sequence numbers, etc., suppressed from report)
  • Each drift item has an owner (Network Engineering for the device) and a target close date
  • Recurring drift on a device or class indicates an underlying issue (manual intervention, malware, broken automation) and warrants investigation, not just remediation
  • Drift detection feeds the GRC baseline-conformance metric
Verify. Drift reports are produced on schedule. Test by manually adding a non-golden setting to a lab device; confirm it appears in the next drift report. Confirm closure workflow is followed (review, decide, remediate, document).
CIS Configuration management: "drift detection"
5
Firmware/OS version tracking and update discipline
Devices run vendor-supported versions; security advisories are tracked.

Firmware lifecycle:

  • Maintain a per-device record of current firmware/OS version (this lives in GRC-REG-ASSET field 17 "OS / platform")
  • Track vendor lifecycle: end-of-sale, end-of-software-maintenance, end-of-support — act before each milestone
  • Subscribe to vendor PSIRT / security advisory feeds; cross-reference advisories against the version inventory
  • Critical security advisories trigger emergency change (subject to vendor-recommended workarounds in the interim)
  • Major firmware upgrades follow normal change process with adequate testing in lab or canary deployment
  • Decommission devices that fall out of vendor support — running unsupported network kit is a baseline violation requiring exception
Verify. Pull firmware-version report; cross-check against vendor lifecycle pages. Confirm advisory-subscription is in place. Confirm any unsupported devices have current exception records or active replacement plans.
CIS Configuration management: "firmware lifecycle"
6
Secure decommissioning
Retired devices don't leak credentials, configurations, or data.

Decommissioning per GRC-SOP-05 §12:

  • Capture final backup of device configuration (kept for audit history, classified L3)
  • Revoke any certificates issued to the device (PKI revocation, RADIUS shared-secret rotation)
  • Remove the device's accounts from AAA
  • Factory-reset the device to clear all configuration including PSKs, certificates, SSH host keys
  • Physical disposal per data classification: L3 data on a device requires NIST SP 800-88 Purge minimum; physical destruction for L4
  • Update GRC-REG-ASSET to Decommissioned status with decommissioning date and method
  • Remove the device from monitoring, log forwarding, NMS, asset trackers
Verify. For decommissioned devices, audit trail shows: final backup, certificate revoked, accounts removed, reset performed, asset record updated.
CIS Configuration management: "decommissioning"
7
Test/lab device isolation and discipline
Test environments don't compromise production.

Lab and test environment rules:

  • Test and lab devices are on isolated networks — no L3 routing to production VLANs except through documented filtered paths
  • Test devices are clearly identifiable (hostname convention, asset record marks them as Dev/Test environment)
  • No production credentials, certificates, or shared secrets on test devices — lab uses lab AAA, lab CA, lab keys
  • Test SSIDs do not advertise the production corporate SSID name
  • Test devices are still in scope of this baseline at a proportionate level — less stringent monitoring, but management plane and basic data plane still hardened
Verify. Lab network topology documented and isolated. Audit credentials: no production AAA users have lab access; no lab keys appear in production. Lab devices appear in GRC-REG-ASSET with Dev/Test tags.
CIS Configuration management: "test isolation"
8
Out-of-band management infrastructure
OOB doesn't become a soft target.

OOB management is in-scope of this entire baseline:

  • Console servers, OOB switches, dedicated management networks are network devices and must meet the baseline
  • Management plane (WI-01) applies fully: AAA, MFA where supported, restricted access, logging
  • OOB networks are physically separate from production where feasible, or strongly logically separated (separate VLAN, separate VRF, dedicated firewall path)
  • OOB authentication path is independent of production AAA in case production AAA is unreachable — documented break-glass procedure
  • OOB session recording where the platform supports it (typing entered into a console session is captured for audit)
Verify. Audit OOB devices: in asset register, in scope of monitoring, in scope of backups. Test break-glass procedure in controlled conditions. Confirm physical/logical separation from production.
CIS Configuration management: "OOB management"
Validation & verification

Configuration management is continuous; validation is the ongoing assurance that the controls are operating.

Functional validation

  • Golden configs exist for every device class; in version control; reviewed
  • Backups produced on schedule for every device; restore tested annually
  • Change records exist for every configuration change in the audit period
  • Drift reports produced regularly; drift items have owners and close dates
  • Firmware versions tracked; advisories reviewed; unsupported versions flagged
  • Decommissioned devices have complete decommissioning evidence
  • Lab devices isolated; no production credentials on lab
  • OOB infrastructure in baseline scope

Rollback & troubleshooting

WI-06 changes are mostly process and tooling; rollback applies to configuration changes made via the change-control workflow rather than to the WI itself.

Common scenarios

  • Backup tool fails for a class of devices — cause: credential change, network path change, vendor protocol change. Fix: investigate, restore the tool, verify backups resume; meanwhile take manual configuration snapshots.
  • Drift detection generating excessive false positives — cause: golden config doesn't accurately reflect legitimate variability (timestamp counters, dynamic state). Fix: tune the drift comparator to suppress incidental fields.
  • Change-control workflow blocking emergency fix — cause: emergency-change path not properly documented or not followed. Fix: invoke the emergency path; conduct post-hoc review; refine the process so the next emergency is faster.
  • Firmware update broke production — cause: insufficient pre-prod testing or unknown bug. Fix: rollback firmware on affected device using vendor-supported downgrade path; investigate; engage vendor TAC; pause further rollout.
  • Decommissioned device data discovered after disposal — cause: configuration backup missed before reset, or backup discarded before retention met. Fix: investigate; if data is recoverable from other sources do so; if not, document the loss and update the decommissioning procedure.
Common errors & tips
  • "Golden config" is not the device's running config snapshot. It's the security-relevant structure; device-specific values (IPs, hostnames) are variables. Confusing these wastes time on differences that don't matter.
  • Backup to a vendor cloud is not enough. Vendor outages happen; backups should be retrievable independent of vendor service availability. Keep a local copy.
  • Change records that say "fixed it" are not records. Capture what was changed, the rationale, the verification steps; future-you (and the auditor) will need them.
  • Drift detection without an owner is theatre. Generate the report, but the report only matters if drift items are closed; otherwise you're cataloguing the absence of discipline.
  • "Latest firmware" isn't always the safest. New releases sometimes introduce bugs; vendor recommendations (and your own canary deployments) inform the right version to be on. Stay current, but stay informed.
  • Lab device drift is not "lab problems". A lab device with production credentials is a production exposure. Treat lab isolation as a real control.
  • OOB is not "the network we don't have to harden". If anything, OOB needs more discipline because it bypasses production monitoring and incident response.
  • Maintain the golden config under the same change discipline as the devices. Otherwise the golden becomes a fiction that nothing actually matches.
