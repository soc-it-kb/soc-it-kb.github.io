IT-NET-WI-03: Data Plane Hardening

Vendor-agnostic implementation guidance for hardening the data plane of network devices: how the device forwards user and application traffic. Operationalises §4.3 of IT-NET-SOP-01.

Prerequisites

Before applying this WI:

  • Management plane is hardened per IT-NET-WI-01
  • Control plane is hardened per IT-NET-WI-02 (DHCP snooping is needed for DAI which is referenced from this WI's port-security context)
  • Network topology is documented: which VLANs exist, which segments connect where, which ports terminate end-users versus servers versus other switches
  • The agency's VLAN numbering convention is known (which IDs are user, server, management, DMZ, OT, etc.)
  • An inventory of unused ports is available (or generated as part of step 1)
  • Change window approved per IT-NET-WI-06
Step-by-step instructions

The steps below cover the data-plane controls required by IT-NET-SOP-01 §4.3.

1
Identify and disable unused ports
Eliminate the easiest L2 attack vector: an open, unconfigured port.

Identify and remediate unused ports:

  • Generate the port-utilisation inventory: which ports have shown link-up activity in the last 30 days, which have not
  • For ports with no recent activity, confirm with operations that they are genuinely unused (not for occasional users or printers)
  • Administratively shut down unused ports (configuration: shutdown / disabled)
  • Assign unused ports to an "unused/parking" VLAN that has no L3 gateway and no inter-VLAN routing
  • Document the unused-port policy so future port activations follow a controlled enable process
Verify. Show interface status / show port-channel summary confirms unused ports are administratively down. Show vlan brief confirms unused ports are in the parking VLAN. Spot-check a few: physically plug a cable into a disabled port; it should remain down.
CIS: Data-plane "unused ports"
2
Port security on access ports
Restrict which MAC addresses can use each access port.

On access ports terminating end-user devices, IP phones, printers:

  • Enable port security with a sensible MAC limit: 1 for ports expected to have one device; 2 for IP-phone-with-PC daisy chains; tuned upward only where the use case justifies
  • Sticky MAC learning — the first MAC seen is bound to the port; protects against simple cable swaps
  • Violation action — "restrict" (allow learned, drop new with logging) for less disruption; "shutdown" (err-disable the port) where stricter posture is required
  • Aging — time-based aging on sticky entries (e.g. 8 hours) so that legitimate device changes don't permanently brick a port

Where 802.1X is deployed, port security MAC learning can be combined with 802.1X for stronger access control, with the MAC limit as a backstop.

Verify. Show port-security shows configured limits and learned MACs. Plug a device with a different MAC into a port with sticky-learnt MAC; confirm the violation action triggers and logs.
CIS: Data-plane "port security"
3
VLAN hygiene: native VLAN and trunk discipline
Prevent VLAN hopping and reduce blast radius from misconfiguration.

Apply VLAN hygiene rules:

  • Native VLAN on trunks — never the default (VLAN 1); never a VLAN that carries actual user or production traffic; use a dedicated "trunk-native-only" VLAN ID per agency convention, with no L3 interface and no DHCP scope
  • Trunk allow-list — every trunk port allows only the specific VLANs it needs to carry (switchport trunk allowed vlan x,y,z), never "all"
  • Negotiation — trunks are configured manually (switchport mode trunk + nonegotiate); DTP negotiation is disabled to prevent unauthorised trunk formation
  • VLAN 1 hygiene — VLAN 1 carries no production traffic; ports are not assigned to VLAN 1; the management interface is not in VLAN 1
Verify. Show interfaces trunk confirms native VLAN per design, allowed list is restrictive. Show vlan brief confirms VLAN 1 has no user-facing ports. Test inter-VLAN access via a double-tagged frame from a lab (controlled test); confirm VLAN hopping is not possible.
CIS: Data-plane "VLAN configuration"
4
DHCP snooping
Bind clients to their legitimate DHCP-assigned addresses; populate the table that DAI relies on.

On all access switches carrying client VLANs:

  • Enable DHCP snooping globally and on the relevant VLANs
  • Designate as trusted only the ports facing legitimate DHCP servers (or trusted relay interfaces)
  • All access ports are untrusted; only DHCP responses arriving on trusted ports are forwarded
  • Enable rate-limiting on DHCP messages from untrusted ports (defends against starvation attacks)
  • Verify the binding table populates as clients lease addresses; this table is the source of truth for Dynamic ARP Inspection
Verify. Show ip dhcp snooping binding shows leases for active clients. Show ip dhcp snooping shows trusted/untrusted port classification. Attempt to run a rogue DHCP server on an untrusted port; confirm clients do not receive its responses.
CIS: Data-plane "DHCP snooping"
5
Inter-VLAN routing filtered, not flat
Inter-segment traffic passes through filtering; no implicit any-to-any.

Wherever the device performs inter-VLAN routing:

  • Apply ACLs on Layer 3 VLAN interfaces (SVIs) limiting inter-VLAN traffic to permitted flows
  • Or, where a firewall sits between segments, route inter-VLAN traffic via the firewall rather than the L3 switch
  • Default-deny applies: anything not explicitly permitted is dropped
  • Document the inter-VLAN matrix: which segment can talk to which, on which ports/protocols, with which authentication
  • Avoid "permit ip any any" except where genuinely required between two segments at the same trust level
Verify. Show ip access-list on each SVI matches the documented matrix. Test inter-VLAN connectivity from a client; only permitted flows succeed.
CIS: Data-plane "inter-VLAN filtering"
6
Segment isolation for guest, IoT, OT
High-risk segments are isolated from the corporate network and from each other.

Apply segment isolation:

  • Guest networks — separate VLAN, separate L3 segment, no reachability to corporate resources, internet egress via firewall only
  • IoT devices — separate VLAN, allowed only to documented cloud or local management endpoints, no lateral reach
  • OT segments — isolated VLAN with strict firewall between IT and OT; only documented protocols pass; monitoring via data diode or one-way replication where the OT environment requires it
  • Cross-segment traffic — passes through a firewall, not a router or L3 switch ACL alone, for segments that warrant stateful inspection
  • Client isolation — on guest VLANs, clients cannot reach each other (where the switch/AP supports the option)
Verify. From a guest network host, attempt to reach a corporate server — should fail. From the corporate network, attempt to reach a guest client — should fail. From a guest client, attempt to reach another guest client — should fail (client isolation).
CIS: Data-plane "network segmentation"
7
Disable unnecessary L2 services on access ports
Reduce L2 attack surface.

On access ports:

  • Disable CDP / LLDP outbound on user-facing ports (information disclosure); leave inbound for management visibility only if needed
  • Disable VTP on access switches; use transparent or off mode to prevent unauthorised VLAN database changes
  • Disable PAgP/LACP negotiation on user-facing ports (no legitimate user device negotiates EtherChannel)
  • Disable IPv6 router advertisements being honoured on user ports (RA Guard from WI-02 handles this from a different angle)
Verify. Show CDP neighbours from a user port returns nothing useful. Show VTP status shows transparent or off. Show etherchannel summary shows no negotiated channels on access ports.
CIS: Data-plane "minimise L2 services"
8
Private VLANs where appropriate
Layer-2 isolation within a single subnet where peer-to-peer reachability is undesired.

Where appropriate (server farms, DMZ segments, guest networks not handled by AP client isolation):

  • Configure private VLANs with isolated ports (no peer-to-peer within the same subnet)
  • Use community VLANs only when a defined group of hosts must talk to each other (rare; usually isolated is correct)
  • Promiscuous ports for the upstream router/firewall
  • Document the PVLAN topology — PVLANs are easy to misconfigure once the design is forgotten
Verify. From an isolated host, attempt to ping another host in the same subnet on the same switch — should fail. From the same host, reach the gateway — should succeed.
CIS: Data-plane "private VLANs (where applicable)"
Validation & verification

After applying all steps, verify the data plane as a whole.

Functional validation

  • Unused ports remain administratively down; new cables don't bring them up
  • Port security violations log to syslog and trigger the configured action
  • Trunks carry only allowed VLANs; native VLAN is the dedicated non-default ID
  • DHCP snooping binding table is populated for active clients
  • Inter-VLAN ACLs deny unintended flows and permit intended ones (test both ways)
  • Guest, IoT, OT clients cannot reach corporate resources beyond documented exceptions
  • Client isolation on guest SSIDs prevents peer-to-peer reachability

Rollback & troubleshooting

Data-plane changes can disrupt user connectivity. Rollback through OOB or stable management.

Common scenarios

  • Legitimate user disconnected after port security activation — cause: user's MAC differs from the sticky-learned MAC (laptop swap, dock change). Fix: clear the violation, allow re-learning, or extend MAC limit if multi-device is legitimate.
  • VLAN trunk drops after restricting allowed VLANs — cause: a VLAN was carrying traffic that wasn't in the documented list. Fix: identify the missing VLAN by traffic capture, decide whether to add it to the trunk or migrate the traffic to a documented path.
  • DHCP clients fail to acquire addresses — cause: DHCP server port not marked trusted, or rate limit too aggressive. Fix: confirm trusted-port designation; check counters for rate-limit drops.
  • Inter-VLAN application breaks after ACL deployment — cause: application uses a flow that wasn't in the documented matrix (often: backend port not documented). Fix: identify the missing flow via ACL hit counters, decide whether to permit (with risk acceptance) or restructure the application.
  • Guest network has no internet — cause: guest VLAN's firewall rule misordered or default-deny too tight. Fix: confirm the egress rule on the firewall is correct and the routing path from the guest VLAN to the internet is intact.
  • Private VLAN isolation breaks application clustering — cause: PVLAN deployed where members of a cluster need to talk to each other. Fix: move them to a community VLAN, or relocate to a non-PVLAN segment.

Rollback procedure

  1. OOB access
  2. Identify the breaking change via pre/post config diff
  3. Revert minimum necessary; if a single permission is missing, add it rather than disabling the entire ACL
  4. Confirm user connectivity restored
  5. Schedule root-cause investigation rather than calling rollback "done"
Common errors & tips
  • Map ports before disabling. A "no-recent-activity" port may be for a rarely-used printer or occasional contractor; confirm before shutting down.
  • Port security default of 1 MAC is wrong for IP phones with PC daisy chain. Use 2 (or whatever the phone+PC combo needs).
  • "Restrict" violation action is friendlier than "shutdown" but harder to notice. Choose based on the operational tolerance for user disruption versus detection latency.
  • Trunk native VLAN must match on both ends. Mismatched native VLAN is a CDP error and can cause traffic leakage. Document and verify.
  • DAI without DHCP snooping is broken DAI. WI-02 references DAI; the binding table comes from WI-03's DHCP snooping. Order matters.
  • Inter-VLAN ACLs need to permit return traffic explicitly on routers that don't track state. L3 switches typically don't do reflexive ACLs by default; consider whether stateful filtering belongs at a firewall.
  • OT segmentation isn't just "another VLAN". OT systems often communicate differently from IT (older protocols, less tolerant of latency, sensitive to monitoring traffic). Coordinate with OT operations before applying IT-grade controls.
  • Don't disable LLDP entirely on all ports. Many monitoring and management tools rely on LLDP for topology discovery; disable LLDP only on user-facing ports and only outbound where possible.
