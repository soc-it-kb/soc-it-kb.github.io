Network Device Security Baselines

Vendor-agnostic security baselines for switches, routers, firewalls, wireless access points, and wireless controllers across all operating units. Aligned to CIS Level 1 benchmarks for the major network vendor families (Cisco IOS-XE, NX-OS, Juniper Junos, Palo Alto, Check Point, Wireless LAN), with controls organised by control domain (management plane, control plane, data plane, wireless, firewall, configuration management) rather than by vendor product.

CIS profile
📌 About this domain. Network devices sit at every traffic boundary between users, systems, and services. Inconsistent or weak hardening on these devices is one of the highest-impact security gaps in any organisation — a single misconfigured firewall rule, an unhardened management interface, or a flat layer-2 network can expose otherwise well-protected systems. These baselines define the minimum security state for every network device under organisational control, regardless of vendor, location, or operating unit.
Scope

This domain covers all network and security-network devices operated by or on behalf of the organisation:

  • Layer-2 / Layer-3 switches — access, distribution, and core switches at any site (data centre, headquarters, hospital, school, field office)
  • Routers — site-edge routers, branch routers, WAN routers
  • Firewalls — perimeter firewalls, internal segmentation firewalls, host-edge firewalls integrated into other devices
  • Wireless access points and controllers — campus wireless, hospital wireless, school wireless, guest networks
  • SD-WAN edge devices — where deployed for branch connectivity
  • Out-of-band management infrastructure — console servers, OOB switches, dedicated management networks

Out of scope: Web Application Firewalls (WAFs) and load balancers — covered separately under the Security Infrastructure baselines domain (future delivery). End-user devices (laptops, mobile devices) — covered under Endpoint Security Baselines. Servers (including jump hosts) — covered under Server Security Baselines.

Control Domain Overview

The baseline is organised by control domain — the conceptual layers of a network device. Each domain has a dedicated Work Instruction:

🔑

Management Plane

How administrators access and configure the device. Covers AAA, secure protocols (SSH/HTTPS only), management-network restriction, console security, banner notices, NTP, syslog forwarding. IT-NET-WI-01.

🧭

Control Plane

How the device decides where traffic goes. Covers routing-protocol authentication, control-plane policing (CoPP), BPDU/storm protection, anti-spoofing (uRPF, BCP38), ARP/NDP protection. IT-NET-WI-02.

🚦

Data Plane

How the device forwards user traffic. Covers port security, VLAN hygiene, disabled unused ports, ACLs, inter-VLAN segmentation, native-VLAN tagging, DHCP snooping. IT-NET-WI-03.

📡

Wireless

Wireless-specific hardening. Covers WPA3 / WPA2-Enterprise, management-frame protection, rogue-AP detection, guest network isolation, SSID hygiene, client isolation, RF policy. IT-NET-WI-04.

🛡️

Firewall

Firewall-specific hardening. Covers default-deny, rule hygiene, egress filtering, geo-blocking, IPS/AV (where licensed), TLS inspection policy, logging, expired-rule cleanup. IT-NET-WI-05.

⚙️

Config Management

Lifecycle controls. Covers golden configuration, drift detection, scheduled backups, version control, change windows, firmware/OS update discipline, decommissioning. IT-NET-WI-06.

Why control-domain organisation, not vendor-specific. A managed switch, router, firewall, and wireless access point all have a management plane. They all forward or filter traffic. They all need configuration management. Organising the baseline by what is being protected — rather than by which vendor's syntax to use — keeps the controls portable across the agencies' mixed estates and gives engineers a stable conceptual map regardless of what vendor they're working with.

The baselines in this domain are derived from the recognised standards below. CIS L1 is the primary source-of-truth where a vendor-specific benchmark exists. The relevant CIS benchmarks span the major network and security-network vendors:

Where a vendor product has no specific CIS benchmark (less common vendors, niche product lines), the closest analogous benchmark applies, supplemented by the vendor's hardening guide and the relevant NIST 800-53 control families. The SOP and WIs identify where this substitution is needed.

Roles & Ownership

Network Engineering

Owns implementation. Configures devices to baseline. Maintains golden configurations. Executes change windows. First-line responder for network incidents.

IT Operations Lead

Owns the baseline document and accepts deviations. Approves change windows. Owns the patch and firmware lifecycle for network devices.

SOC

Consumes logs from devices (syslog, NetFlow, NDR). Monitors for anomalous behaviour. Investigates suspicious traffic and authentication events.

GRC

Owns governance of the baseline domain. Validates compliance periodically. Maintains audit trail. Records exceptions and deviations.

Local IT (per agency)

Applies baseline to devices in their operating unit. Reports drift and deviations through their LCSO/CSLC.

Architecture / Technical Authority

Adjudicates baseline interpretation questions. Approves exceptions affecting design or topology.

Metrics & Oversight

The following metrics measure adherence to this baseline. They are reviewed monthly by IT Operations and quarterly by GRC:

  • Baseline conformance rate — percentage of in-scope network devices fully conformant to IT-NET-SOP-01. Target: ≥95% for Tier 0/1 assets; ≥85% for Tier 2/3.
  • Drift count — number of devices showing configuration drift from golden config in the most recent reconciliation. Target: zero unexplained drift.
  • Firmware currency — percentage of devices running a supported firmware or OS version. Target: 100% supported; ≥80% within one major version of current.
  • Backup currency — percentage of devices with a successful configuration backup in the last 7 days. Target: 100%.
  • Mean time to remediate deviation — time from drift detection to baseline restoration. Target: ≤5 working days for non-critical findings; ≤24 hours for critical (e.g. open management interface).
  • Open exceptions — count of approved deviations from baseline in force. Reviewed annually for currency.
Exception Handling

Where a device cannot meet the baseline (vendor limitation, operational constraint, integration requirement), the operating unit submits a deviation request via the standard ASM exception process (IT-GOV-SOP-01). Deviations are reviewed and approved by the IT Operations Lead with GRC consultation. Approved deviations are recorded in the Baseline Exceptions Register (GRC-REG-EXCEPTIONS), linked from the affected asset records in GRC-REG-ASSET, and reviewed annually for currency.
