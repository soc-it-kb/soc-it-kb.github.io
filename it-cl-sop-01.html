IT-CL-SOP-01: Cloud Security Baseline

The authoritative policy-level baseline for Microsoft 365 and Microsoft Azure tenants across operating units. Defines shared principles plus platform-specific minimum security requirements with explicit licensing-tier awareness. Operationalised by IT-CL-WI-01 through IT-CL-WI-05.

Section 1
Purpose

This SOP defines the minimum security baseline for cloud tenants across the Institution and its Agencies: Microsoft 365 (productivity tenants on any licensing tier from A1 / E1 through E5) and Microsoft Azure (IaaS / PaaS tenants where the Institution or an Agency operates subscriptions). It establishes the controls every such tenant must implement to be considered fit for production use, with controls grouped by platform and by licensing tier so that conformance is reported against achievable expectations.

Cloud tenants are different from on-premises infrastructure in one important way: the boundary between vendor-administered and tenant-administered controls is itself part of the baseline. Microsoft operates the underlying platform; we administer the tenant. This SOP and its operationalising Work Instructions cover only the tenant-administered surface — the configuration choices, policy settings, and integrations the operating Agency is responsible for. The shared-responsibility model is the implicit framing.

This SOP is operationalised by five Work Instructions: IT-CL-WI-01 (Tenant Identity & Conditional Access — cross-cutting M365 and Azure), IT-CL-WI-02 (M365 Tenant Hardening — tier-aware), IT-CL-WI-03 (Azure Subscription Baseline), IT-CL-WI-04 (Azure Workload Patterns), and IT-CL-WI-05 (Defender Suite Configuration — tier-aware). See §4 for the requirements summary and the linked WIs for implementation detail.

Section 2
Scope

2.1 In scope

  • Microsoft 365 tenants across all agencies — Education tenants (A1 / A3 / A5) and Business / Enterprise tenants (E1 / E3 / E5). Includes Exchange Online, SharePoint Online, OneDrive, Teams, Purview, and the Defender for Office 365 features available at each licence tier.
  • Microsoft Azure tenants — every subscription where the Institution or an Agency operates resources, in any region. Includes subscription-level controls (Defender for Cloud, Azure Policy, diagnostic settings, activity log), workload patterns (storage, key vault, networking, Azure WAF), and Entra ID at the tenant level.
  • Entra ID (Azure AD) at the tenant-configuration level — Conditional Access policies, PIM enablement, Security Defaults, named locations, sign-in risk policies, break-glass account setup. Both M365-only tenants and combined M365 + Azure tenants are in scope.
  • Tenants in production, pre-production, disaster-recovery, and any environment carrying Institutional data or production workloads
  • Tenants at any tier (Tier 0 / 1 / 2 / 3 per the existing tier model)

2.2 Out of scope

  • AWS and GCP tenants — acknowledged but deferred. A small number of workloads exist on AWS and GCP across some Agencies; these are recorded in GRC-REG-ASSET and a future multi-cloud extension to this domain will provide baselines. Until then, the operating agency applies the CIS AWS Foundations or CIS GCP Foundations benchmarks directly and records the choice in the asset register.
  • Identity governance — joiner / mover / leaver lifecycle, access recertification, eligibility-list curation, role design — deferred to a future Identity domain SOP. Tenant-level identity controls (Conditional Access, PIM, break-glass, sign-in risk policies) are in scope of this SOP via WI-01.
  • SOC operational use of cloud security tooling — alert triage, incident response procedures, hunt queries — covered under SOC SOPs (SOP-IM-01, SOP-EM-01, SOP-ID-01, SOP-EP-01). The tenant-side configuration of Defender (enablement, policy, baseline settings, log integration) is in scope of this SOP via WI-05.
  • Third-party cloud-WAFs — Cloudflare, AWS WAF, Akamai — deferred to a future vendor-specific extension. Azure-native cloud-WAF (Azure WAF on Application Gateway and Front Door) is in scope of WI-04.
  • Cloud architecture reference design — landing zone hierarchies, management group structure, hub-spoke network design, multi-region failover patterns — these belong in a Cloud Architecture document. Azure Policy as enforcement mechanism is in scope of WI-03 because it is the backbone for enforcing everything else.
  • End-user devices, servers, and network devices connected to cloud tenants — covered under their respective ASM domains (Security Baselines, Network Baselines, Security Infrastructure Baselines).
Section 3
Baseline Principles

The seven principles below shape every requirement in this baseline. Where an implementation choice arises, refer to these principles to resolve it consistently.

3.1 The shared-responsibility model is the framing

Microsoft operates the platform; we administer the tenant. The baseline only covers what the tenant can administer. Where a control depends on the platform behaving correctly (regional failover, vendor data protection, vendor incident response), the baseline records the dependency rather than restating the control.

3.2 Licensing-tier-aware, not tier-blind

Each requirement specifies the minimum licence tier that supports it. Controls available only at higher tiers are documented as "available at tier X" with the corresponding implementation expectation. Conformance is reported per tier so deviations are visible against achievable expectations. The baseline does not pretend an A1 tenant can implement E5 features.

3.3 Identity is the perimeter

In cloud, the perimeter is the identity layer. Strong authentication (MFA universally; AAL3-grade where the platform supports it), Conditional Access by default, legacy authentication eliminated, privileged identities behind PIM where licensed. WI-01 is the foundation; the platform-specific WIs assume it is in place.

3.4 Default-deny applies at the tenant boundary

External sharing in SharePoint and OneDrive is restricted by default, not permissive. External collaboration in Teams is governed by federation policy. Tenant-to-tenant access is opt-in, not opt-out. New Azure subscriptions inherit Azure Policy denies for non-conformant configurations rather than relying on after-the-fact detection.

3.5 Centralised logging is non-negotiable

Every in-scope tenant forwards security-relevant logs to the central SIEM: M365 audit log (Exchange / SharePoint / Teams / Entra), Defender alerts, sign-in logs, Azure activity log, Defender for Cloud findings. This requirement is the most common gap in the estate today and is the largest single category of expected exceptions on initial publication of this baseline. See §3.7 for the transition treatment.

3.6 Microsoft Secure Score as quantitative complement

The Microsoft Secure Score and Azure Secure Score (Defender for Cloud) provide tenant-specific numerical conformance metrics that complement the binary baseline-conformance assessment. Targets are set per tier (see §7). They are not a replacement for the baseline; they are a leading indicator of trajectory and a useful conversation tool with tenant administrators.

3.7 The central SIEM platform is under transition

The institution's central SIEM is in transition during the lifetime of this SOP. Tenants integrate with the SIEM platform that is operationally current at the time of onboarding; the baseline requires the integration, not a specific vendor. The 12–18-month SIEM migration roadmap is the natural sunset window for the bulk of the initial exception register entries that this baseline crystallises.

Section 4
Minimum Security Requirements

The requirements below are mandatory for every in-scope tenant. Each requirement domain is operationalised by a dedicated Work Instruction (linked at the right of each domain header). The tier tags indicate the minimum licence at which each requirement is achievable: All = required at every tier; A3 / E3+ = required at A3 / E3 and above; E5 = required where E5 licensing is in force. Where a tenant cannot meet a requirement at its licensing tier, the baseline considers this an exception only if the corresponding upgrade is feasible; otherwise it is a feature unavailability and not a deviation.

🔐

4.1 Tenant Identity & Conditional Access (cross-cutting Entra)

IT-CL-WI-01 →
  • AllMFA enforced for all administrative accounts; AAL3-grade (FIDO2 / Windows Hello for Business / certificate-based) for tenant Global Administrators
  • AllMFA enforced for all standard user accounts via Security Defaults (free) or Conditional Access (A3 / E3+)
  • AllLegacy authentication (basic auth, IMAP, POP, SMTP AUTH, older Exchange protocols) fully blocked at tenant level
  • AllBreak-glass / emergency-access accounts: two per tenant, documented, MFA-bypass-exempt where required by recovery procedure, passwords vaulted independently of the tenant, monthly sign-in monitoring
  • A3 / E3+Conditional Access deployed in preference to Security Defaults; named locations defined; baseline policies (require MFA for admins, block legacy auth, require compliant device for admins, geo-fence high-risk regions)
  • A3 / E3+Guest access governed by an explicit policy: invitation-only, MFA on first sign-in, conditional access applied to guests, periodic guest review
  • E5Sign-in risk and user risk policies enabled; risky sign-ins block or require step-up authentication; risky users require password change
  • E5Privileged Identity Management (PIM) enabled for Entra and Azure roles; time-bounded eligibility for all Global Administrator and equivalent roles; PIM reviews quarterly
  • AllSelf-service password reset enabled and combined with MFA registration
📨

4.2 M365 Tenant Hardening

IT-CL-WI-02 →
  • AllExchange Online: anti-spam, anti-malware, anti-phishing baseline policies enabled; mailbox audit logging on by default
  • AllSPF, DKIM, and DMARC configured for every accepted domain; DMARC policy at p=quarantine minimum (p=reject target)
  • AllExternal email tagging (warning banner on external messages) enabled
  • AllSharePoint and OneDrive: external sharing restricted by default; anonymous (anyone) links disabled or capped to specific scenarios with expiry
  • AllTeams: external access (federation) governed by allow-list; meeting policies prevent anonymous join for sensitive meetings; lobby on by default for external attendees
  • AllUnified Audit Log enabled and retention configured to maximum available at the tenant's licence tier
  • A3 / E3+Purview Information Protection sensitivity labels deployed for at least the high-confidentiality classification; auto-labelling for regulated data classes where supported
  • A3 / E3+Purview DLP policies for the data classes the organisation processes (PII, financial, health where applicable); test mode then enforce
  • E5Defender for Office 365 Plan 2: Safe Attachments, Safe Links, Attack Simulator deployed
  • E5Insider Risk Management policies for at least the high-risk indicator classes; Communication Compliance where regulatory requirements apply
  • E5Customer Lockbox enabled; Privileged Access Management for Exchange Online enabled

4.3 Azure Subscription Baseline

IT-CL-WI-03 →
  • Defender for Cloud enabled on every subscription; standard tier (Defender plans) on Tier 0 / 1 subscriptions; free tier acceptable only for Tier 3 with documented justification
  • Azure Policy assigned at management group scope (not per subscription) for the core baseline initiative: deny non-conformant resource types, enforce diagnostic settings, deny public network exposure where not justified
  • Diagnostic settings configured on every resource type that supports them; routed to a Log Analytics workspace and to the central SIEM via Event Hub or equivalent
  • Activity log forwarded to the central SIEM
  • Defender for Cloud findings forwarded to the central SIEM
  • Secure Score target ≥75% for Tier 0 / 1 subscriptions; ≥60% for Tier 2 / 3
  • Subscription RBAC follows least-privilege: no standing Owner role outside named platform administrators; Contributor and Reader scoped to resource groups, not subscriptions
  • Resource locks (CanNotDelete, ReadOnly) on critical resource groups (production data, key vaults, networking hubs)
  • Cost management alerting configured so unexpected spend (often the first signal of resource compromise) is visible
  • Tagging policy enforced: cost centre, owner, environment, sensitivity classification on every resource
🛠️

4.4 Azure Workload Patterns

IT-CL-WI-04 →
  • Storage accounts: secure transfer required (HTTPS only); public blob anonymous access disabled at the storage account level; minimum TLS 1.2; firewall rules restricting access to known networks or private endpoints
  • Storage accounts (Tier 0 / 1): customer-managed keys (CMK) for encryption at rest; soft-delete enabled with ≥7-day retention; immutable storage for compliance-relevant data
  • Key Vault: RBAC mode (preferred over access policies); soft-delete and purge protection enabled and not disablable; network access restricted (Private Endpoint preferred over service endpoints); diagnostic logging enabled
  • Key Vault (Tier 0): HSM-backed keys for production cryptography; key rotation procedure documented and tested annually
  • Virtual networks: NSGs on every subnet with explicit allow rules; default-deny inbound; flow logs enabled and forwarded to central SIEM
  • Private Endpoints preferred over public network exposure for PaaS data services (Storage, SQL, Key Vault, Cosmos); service endpoints permitted where Private Endpoint is unavailable
  • Azure WAF on Application Gateway / Front Door for public-facing web applications: managed rule set (OWASP Top 10) enabled in Prevention mode within 90 days for Tier 0 / 1 apps; custom rules documented; log streaming to central SIEM
  • SQL Database / Cosmos: Defender for SQL / Cosmos enabled where the workload tier justifies; transparent data encryption with CMK for Tier 0 / 1
  • Virtual machines: managed identities preferred over service principals; Defender for Servers integration; just-in-time VM access where exposed to management networks
  • App Service / Functions: managed identity to access dependencies; HTTPS only; minimum TLS version 1.2; authentication / authorisation enabled where the app does not handle its own; diagnostic logging on
🛡️

4.5 Defender Suite Configuration

IT-CL-WI-05 →
  • A3 / E3+Defender for Endpoint deployed to all managed endpoints (Windows, macOS, Linux servers, mobile where licensed); ASR rules in audit-then-block progression; EDR-in-block-mode enabled
  • A3 / E3+Defender for Endpoint security baseline assigned to every device group
  • E5Defender for Office 365 Plan 2: Safe Links and Safe Attachments policies enabled for all users; Attack Simulator used at least quarterly
  • A3 / E3+Defender for Identity sensor deployed on all domain controllers (where the agency operates AD); alert integration with central SIEM
  • E5Defender for Cloud Apps connected to M365 and Azure tenants; anomaly detection policies in default state minimum; OAuth app governance policies
  • Defender for Cloud (Azure): per WI-03 §4.3; the Defender plans enabled per subscription tier
  • All Defender alerts forwarded to the central SIEM; severity-to-priority mapping documented
  • Defender for Endpoint device-tagging convention (criticality tier, agency, environment) applied to every onboarded device
  • Threat & Vulnerability Management (TVM) feeds vulnerability findings to the central vulnerability management programme (IT-VM-SOP-01)
The one expectation that's worth stating twice. The single largest gap this baseline crystallises is central SIEM integration. Today, one tenant (~7k accounts) is integrated; the rest are not. This baseline requires universal integration. On initial publication, every uncovered tenant is in a deviation state. These deviations are batch-tracked against the 12–18-month SIEM migration roadmap and are the natural way to make the gap visible to leadership.
Section 5
Roles & Responsibilities

IT Operations Lead

Owns this SOP. Approves baseline changes and exceptions. Accountable to the Risk Committee for tenant conformance. Sets reporting cadence and oversees the SIEM-integration remediation programme.

Cloud Engineering

Implements the baseline in central / shared-services tenants. Maintains golden Azure Policy initiatives. Owns the central SIEM integration patterns (Event Hub, Azure Monitor agent, Graph API export). First-line responder for cloud-platform incidents.

Identity & Access Engineering

Implements WI-01 (Conditional Access, PIM, Entra ID). Co-owner of break-glass account procedure. Coordinates with Cloud Engineering on Azure RBAC and with M365 admins on tenant identity policies.

M365 Tenant Administrators (per agency)

Implement WI-02 in their tenant per the agency's licence tier. Maintain Exchange / SharePoint / Teams / Purview configuration. Maintain the tenant's licence-tier record in GRC-REG-ASSET. Submit exception requests for unachievable controls.

Azure Subscription Owners (per agency)

Implement WI-03 and WI-04 in their subscriptions. Maintain Defender for Cloud configuration and Azure Policy compliance. Coordinate with Cloud Engineering on SIEM integration.

SOC

Consumes signals from Defender suite, sign-in logs, audit logs, Defender for Cloud findings. Investigates and escalates per SOC SOPs. Does not own tenant configuration; raises baseline-gap observations as input to GRC.

GRC

Owns governance of the baseline domain. Maintains exception register. Reports tenant conformance to the Risk Committee quarterly with separate views for SIEM-integration remediation. Reviews baseline annually.

Architecture / Technical Authority

Adjudicates interpretation questions. Approves design-level exceptions. Reviews new tenant onboarding to confirm baseline applicability and licensing-tier mapping.

Separation of duties. The engineer who configures a Conditional Access policy, Azure Policy assignment, or tenant DLP rule is not the same person who approves its move to production. Configuration changes pass through change control. Tenant Global Administrator access is held by named individuals only and is reviewed quarterly; Global Admin is not a working role.

Section 7
Governance & Review

7.1 Baseline ownership and update authority

This SOP is owned by the IT Operations Lead with co-ownership by the Head of GRC and the Head of Identity & Access Engineering (for the WI-01 aspects). Material changes (new requirements, scope changes, tier-mapping changes, framework-alignment changes) require sign-off from all three. Minor clarifications and editorial improvements may be applied by the owner with notification to GRC.

7.2 Review cadence

  • Annual review — minimum, by the owner with GRC sign-off, on the anniversary of approval
  • Out-of-cycle review — triggered by: a CIS Foundations Benchmark version update (M365 or Azure); a Microsoft licensing-feature change that affects tier mappings; the central SIEM platform transition; a major incident involving cloud tenants; an audit finding
  • Sub-WI review — each Work Instruction is reviewed independently and may update more frequently than the SOP, particularly WI-02 (M365) and WI-05 (Defender), which track Microsoft product changes

7.3 Tier-specific Secure Score targets

Microsoft Secure Score and Azure Secure Score targets are reviewed quarterly and adjusted only with documented justification. Current targets:

  • M365 Secure Score: A1 / E1 — reported but not target-bound (limited feature surface); A3 / E3 — ≥70%; E5 — ≥80%
  • Azure Secure Score (Defender for Cloud): Tier 0 / 1 subscriptions — ≥75%; Tier 2 / 3 — ≥60%

7.4 Exception handling

Where a tenant cannot meet a requirement in §4 at its licensing tier, the operating agency follows IT-GOV-SOP-01 (Baseline Exception & Risk Acceptance). Exceptions are time-bounded, recorded in GRC-REG-EXCEPTIONS, linked from the tenant's record in GRC-REG-ASSET, and reviewed annually.

Two exception categories deserve specific treatment:

  • Licence-tier unavailability. Where a control is unavailable at the tenant's licence tier (e.g. an A1 tenant cannot implement Conditional Access; only Security Defaults), this is recorded as a feature-unavailability with the upgrade-feasibility assessment. Annual review checks whether upgrade has become feasible.
  • Central SIEM integration deferral. Tenants not yet integrated to the central SIEM are batch-tracked against the 12–18-month SIEM migration roadmap. Each tenant gets a single exception entry referencing the migration programme rather than a per-control set of entries. This is administratively necessary and audit-defensible.

7.5 Compliance & audit

Baseline conformance is measured through: Microsoft Secure Score and Azure Secure Score (quantitative); Defender for Cloud regulatory compliance dashboards (CIS, NIST, ISO mappings); Azure Policy compliance state; manual tenant-administrator attestation against the WI checklists (semi-annual). Reports are produced monthly to IT Operations and quarterly to GRC and the Risk Committee, with separate cuts per platform (M365 / Azure) and per licensing tier. Audit evidence includes: tenant Secure Score reports, Defender regulatory-compliance exports, Azure Policy compliance snapshots, exception register entries, change-control records, SIEM-integration status records.
