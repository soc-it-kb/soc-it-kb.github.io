IT-NET-SOP-01: Network Device Security Baseline

The authoritative policy-level baseline for the security state of all network devices across operating units. Defines minimum security requirements organised by control domain (management, control, data, wireless, firewall, configuration management). Operationalised by IT-NET-WI-01 through IT-NET-WI-06. CIS L1 aligned, vendor-agnostic.

Section 1
Purpose

This SOP defines the minimum security baseline for network and security-network devices operated by or on behalf of the Institution. It establishes the controls every device must implement to be considered fit to carry Institutional traffic, regardless of vendor, location, or operating unit.

Network devices sit at every boundary in the Institution's IT estate: between users and services, between segments, between sites, between the Institution and the internet. A weakness on these devices is uniquely impactful because it potentially exposes everything connected to them. This baseline reduces that risk to a defined, monitored, and audit-defensible state.

This SOP is operationalised by six Work Instructions — one per control domain — that provide implementation-level guidance for engineers. See §4 for the requirements summary and the linked WIs for execution detail.

Section 2
Scope

2.1 In scope

  • Switches — access, distribution, and core switches at any site (Layer 2 and Layer 3)
  • Routers — site-edge routers, branch routers, WAN routers, SD-WAN edge devices
  • Firewalls — perimeter, internal-segmentation, and host-edge firewalls
  • Wireless access points and controllers — campus, hospital, school, field-office wireless including guest networks
  • Out-of-band management infrastructure — console servers, OOB switches, dedicated management networks
  • Devices in production, pre-production, disaster-recovery, and any environment carrying Institutional traffic
  • Devices at any tier (Tier 0 / 1 / 2 / 3 per the existing tier model)

2.2 Out of scope

  • Web Application Firewalls (WAFs) and load balancers — covered under future Security Infrastructure baseline domain
  • End-user devices (laptops, desktops, mobile) — covered under Endpoint Security Baselines
  • Servers including jump hosts — covered under Server Security Baselines
  • OT-segment network kit dedicated to operational technology — covered under separate OT-specific guidance (where it exists); minimum management-plane requirements still apply
  • Vendor-cloud-managed devices where the vendor controls the security configuration entirely (these still appear in the Asset Register; the baseline applies to anything tenant-administered)
Section 3
Baseline Principles

The seven principles below shape every requirement in this baseline. Where an implementation choice arises, refer to these principles to resolve it consistently.

3.1 Defence in depth

No single control is sufficient. Management-plane controls assume the control plane could be attacked; control-plane controls assume the data plane could carry malicious traffic; data-plane controls assume something has bypassed the perimeter. Each layer assumes the others may fail.

3.2 Least privilege at every interface

Management interfaces accept only the protocols, sources, and accounts genuinely required. Traffic interfaces forward only the traffic genuinely required. Default-deny applies to filtering, segmentation, and administrative access alike.

3.3 Strong authentication, audited authorisation

No shared accounts. All administrative access via named accounts authenticated against a central directory with multi-factor authentication where the device supports it. Every action is attributable.

3.4 Secure protocols only

Plaintext management protocols (Telnet, HTTP, SNMPv1/v2, FTP, TFTP for config) are disabled. SSH (key-based where supported), HTTPS (TLS 1.2 minimum), SNMPv3 with authentication and privacy, SCP/SFTP for file transfer.

3.5 Segmentation by purpose

Management traffic is isolated from production traffic. Guest networks are isolated from corporate networks. OT segments are isolated from IT segments. Inter-VLAN traffic is filtered, not implicitly trusted.

3.6 Configuration as documented state

Every device has a documented golden configuration. Deviations are exceptions, recorded and time-bounded. Configuration backups are automated and recoverable. Drift from golden config is detected and reconciled.

3.7 Comprehensive, centralised logging

Every device forwards authentication events, configuration changes, and security-relevant events to the central log destination. Logs include accurate timestamps from a trusted time source. Logs are retained per the Institution's retention policy.

Section 4
Minimum Security Requirements

The requirements below are mandatory for every in-scope device. Each requirement domain is operationalised by a dedicated Work Instruction (linked at the right of each domain header). Where a device cannot meet a requirement, a documented exception is required per §7.

🔑

4.1 Management Plane

IT-NET-WI-01 →
  • Plaintext management protocols (Telnet, HTTP, SNMPv1/v2c, FTP, TFTP for configuration) are disabled
  • SSH version 2 with strong cipher suites; HTTPS with TLS 1.2 minimum; SNMPv3 with authPriv where SNMP is required
  • All administrative authentication is via a central directory (AAA: TACACS+ / RADIUS / Entra ID) with MFA where the device supports it
  • No shared accounts; the local emergency account is documented, password-vaulted, and used only with break-glass approval
  • Management-plane access is restricted to a defined management network or a small allow-list of administrative sources
  • Idle session timeout configured (≤15 minutes for privileged sessions)
  • Login banner displaying the authorised-use notice required by the Institution's policy
  • NTP configured to two or more authoritative time sources with authentication
  • Syslog forwarding to the central SIEM destination; events include authentication, configuration changes, and security-relevant alerts
  • Console access physically secured; console port disabled where remote management is the operational norm; auxiliary port disabled unless explicitly required
🧭

4.2 Control Plane

IT-NET-WI-02 →
  • Routing-protocol authentication enabled (OSPF, EIGRP, BGP, IS-IS) with strong keys; key rotation procedure documented
  • Control-plane policing (CoPP) or equivalent configured to limit control-plane CPU consumption to known-good traffic rates
  • BPDU guard, root guard, and loop guard enabled on access-port spanning-tree edges
  • Storm control configured on access ports (broadcast, multicast, unknown-unicast)
  • Unicast Reverse Path Forwarding (uRPF) or equivalent anti-spoofing enabled on customer-facing and uplink interfaces (BCP38 compliance)
  • Dynamic ARP Inspection (DAI) on access switches; IPv6 RA Guard and ND inspection on IPv6-capable networks
  • Unused routing protocols disabled; passive-interface defaults applied for routing on user-facing segments
  • Route filtering at administrative boundaries (max-prefix, prefix lists, route maps)
🚦

4.3 Data Plane

IT-NET-WI-03 →
  • Unused switch ports are administratively disabled and assigned to an unused VLAN
  • Port security configured on access ports (MAC limit, sticky learning, violation action of restrict or shutdown)
  • Native VLAN on trunk ports is a non-default VLAN ID, distinct from any user VLAN
  • Trunk ports allow only the VLANs they need to carry; default "allow all" trunk configurations are not used
  • DHCP snooping enabled with trusted-port designation only on DHCP-server-facing interfaces
  • Inter-VLAN routing filtered by ACL or firewall; flat L3 networks between user, server, and management segments are prohibited
  • Guest, IoT, OT, and high-risk segments are isolated; inter-segment traffic passes through a firewall
  • VLAN 1 is not used for production traffic; user VLANs start from a non-default ID per agency convention
📡

4.4 Wireless

IT-NET-WI-04 →
  • WPA3-Enterprise where supported; WPA2-Enterprise (802.1X with EAP-TLS or PEAP-MSCHAPv2) as the minimum for corporate SSIDs
  • WPA2/3-Personal acceptable only for guest networks with rotated PSK or vendor-provided guest-access flows
  • WEP, WPA-TKIP, and open authentication on corporate SSIDs are prohibited
  • Protected Management Frames (802.11w PMF) enabled — required on WPA3, enabled-where-supported on WPA2
  • Guest networks fully isolated from corporate networks (separate VLAN, firewall-filtered egress, no corporate-resource reachability)
  • Client isolation enabled on guest SSIDs
  • SSID broadcast disabled is not a security control — broadcast remains enabled; security is enforced by authentication, not obscurity
  • Rogue-AP detection and containment enabled where the controller supports it
  • WIDS/WIPS or equivalent threat-detection enabled where licensed
  • RF region, channel plan, and transmit power configured per site survey; no "max-power-everywhere" defaults
🛡️

4.5 Firewall

IT-NET-WI-05 →
  • Default-deny applied on all interfaces; explicit-allow rules required for permitted traffic
  • Egress filtering enforced on internet-facing interfaces; "any-any-outbound" rules are prohibited
  • Rule base is documented: every rule has a recorded purpose, owner, creation date, and expiry/review date
  • Rules without traffic for 90 days are flagged for review; rules without traffic for 180 days are removed unless justified
  • Geo-blocking applied where business needs do not include the blocked geography
  • IPS / IDS / antivirus engines enabled where licensed; signature updates current
  • TLS inspection policy documented; encrypted traffic without inspection is logged for visibility (banner certs, SNI logging)
  • Application-layer (Layer 7) controls enforced where the device supports them
  • Anti-spoofing, fragmentation protection, and reassembly enabled where the platform offers them
  • High-availability pair configured for Tier 0 and Tier 1 firewalls; failover tested at minimum annually
⚙️

4.6 Configuration Management & Change Control

IT-NET-WI-06 →
  • Every device has a documented golden configuration capturing the baseline state
  • Automated configuration backups daily for Tier 0/1 devices; weekly for Tier 2/3
  • Configuration changes pass through documented change control; emergency changes have an expedited path with post-hoc review
  • Drift detection automated; deviations from golden config are reported and remediated within defined SLAs
  • Firmware/OS versions tracked per device; vendor-supported versions only; security advisories tracked through patch management
  • Decommissioning follows GRC-SOP-05 §12: configuration backup retained, secrets revoked, certificates revoked, device wiped per data-classification destruction method, asset record updated to Decommissioned
  • Test/lab devices clearly identifiable and on isolated networks; no production credentials on test devices
  • Out-of-band management infrastructure (console servers, OOB switches) hardened to the same baseline
One requirement that's worth saying twice. Default-deny is not optional and is not a discussion. Any firewall with "any-any" allow rules at perimeter or between trust zones is in breach of this baseline, regardless of vendor or operational pressure. If business need cannot be met within default-deny, the path is a documented exception, not a permissive rule.
Section 5
Roles & Responsibilities

IT Operations Lead

Owns this SOP. Approves baseline changes and exceptions. Owns the patch/firmware lifecycle. Sets change-window cadence. Accountable for baseline conformance metrics.

Network Engineering

Implements the baseline. Maintains golden configurations. Executes change windows. Investigates and remediates drift. First-line responder for network incidents.

SOC

Consumes logs from network devices. Investigates anomalous authentication, configuration changes, and traffic patterns. Coordinates with Network Engineering on incident response.

GRC

Owns governance and audit evidence. Reviews baseline annually. Maintains exception register. Reports baseline conformance to Risk Committee quarterly.

Local IT (per agency)

Applies baseline to devices in their operating unit. Submits exception requests through their LCSO/CSLC. Coordinates change windows with central Network Engineering.

Architecture / Technical Authority

Adjudicates interpretation questions and edge cases. Approves design-level exceptions. Reviews new technology before adoption to ensure baseline applicability.

Separation of duties. The engineer who configures a device is not the same person who approves its move to production. Configuration changes pass through change control. This separation exists for audit defensibility and to catch errors that single-person review misses.

Section 7
Governance & Review

7.1 Baseline ownership and update authority

This SOP is owned by the IT Operations Lead with co-ownership by the Head of GRC. Material changes (new requirements, scope changes, framework-alignment changes) require sign-off from both. Minor clarifications and editorial improvements may be applied by the owner with notification to GRC.

7.2 Review cadence

  • Annual review — minimum, by the owner with GRC sign-off, on the anniversary of approval
  • Out-of-cycle review — triggered by: a material CIS benchmark update; a major incident involving network devices; introduction of new vendor or technology; a regulatory change affecting network security (e.g. NIS2 implementing-act change); an audit finding
  • Sub-WI review — each Work Instruction (IT-NET-WI-01 through 06) is reviewed independently and may update more frequently than the SOP

7.3 Exception handling

Where a device cannot meet a requirement in §4, the operating unit follows IT-GOV-SOP-01 (Baseline Exception & Risk Acceptance). Exceptions are time-bounded, recorded in GRC-REG-EXCEPTIONS, linked from the affected asset records in GRC-REG-ASSET, and reviewed annually. Permanent exceptions are reviewed annually for currency; an exception that persists for three years without remediation is escalated to the Risk Committee for review.

Common cases that warrant exceptions: vendor product limitations (no SNMPv3 support on legacy hardware), operational constraints (a device that must allow non-standard management for an integration), integration requirements (a tool that requires SNMPv2 read-only access). Acceptable mitigations include compensating controls (SNMP traffic restricted to a management VLAN, ACL-protected), monitoring uplift (enhanced log analysis for the affected device), and time-bounded remediation plans.

7.4 Compliance & audit

Baseline conformance is measured continuously through automated reconciliation (configuration management tooling, periodic CIS-CAT or equivalent scans) and reported monthly to IT Operations. Quarterly aggregate reports are presented to GRC and to the Risk Committee. Audit evidence includes: golden configurations, scan results, exception register entries, change-control records, and incident records related to network-baseline gaps.
