ID-PB-01: Suspected or Confirmed Account Compromise

Standardised decision and response framework for handling suspected or confirmed user account compromise — ensuring rapid containment, consistent escalation, and evidence preservation across global SOC operations.

🔐 Identity threats Tier 1 / 2 / 3 MDR integrated

👤 Author: Paulo Ferreira

📅 Published: 22 December 2025

🏷️ Category: Identity Threat

Active
Section 1
Purpose

Provide a standardised decision and response framework for handling suspected or confirmed user account compromise, ensuring rapid containment, consistent escalation, and evidence preservation across global SOC operations.

Section 2
Scope

✅ In scope

  • Cloud identity (Entra ID / Azure AD)
  • Email-initiated compromise
  • Credential theft
  • Token abuse
  • Session hijacking

❌ Out of scope

  • Service account compromise → ID-PB-05
  • Endpoint-only compromise with no identity impact
Section 3
Trigger Conditions

Initiate this playbook when any of the following occur:

  • MDR escalates account compromise suspicion
  • Phishing playbook confirms credential submission
  • Anomalous sign-in detected (geography, device, risk score)
  • User reports suspicious login activity
  • SOC identifies abnormal authentication behaviour
Section 4
Initial Validation (Decision Logic)

Confirm the alert is identity-related by checking for any of the following:

  • Successful sign-ins from unusual or unfamiliar locations?
  • MFA prompts generated without user-initiated action?
  • Legacy authentication protocols in use?
  • Token-based access occurring without a password entry event?

If NO to all → Route to appropriate playbook
If YES to any → Proceed with compromise assessment (Section 5)

Section 5
Compromise Assessment

Assess the likelihood of compromise. Any confirmed indicator should be treated as a compromised account:

Successful sign-in from unfamiliar IP address
Impossible travel detected
MFA bypass or fatigue attack confirmed
OAuth consent granted without user approval
Mailbox rules or forwarding created without business need
Section 6
Severity & Escalation Criteria

Escalate to High Severity if any of the following apply:

🔴 Privileged role assigned to the compromised account
🔴 Multiple compromise indicators present simultaneously
🔴 Lateral movement suspected or confirmed
🔴 MDR confirms active attacker control of the account

Notify immediately
🔐
Identity Team
Account containment support
👔
SOC Leadership
High severity awareness
🤝
MDR Partner
If not already engaged
Section 7
Response Strategy

The following actions must be orchestrated across the response team:

Active session containment — revoke all tokensRequired
Credential resetRequired
MFA enforcement and re-registrationRequired
Sign-in log reviewRequired
Persistence removal (rules, OAuth, delegates)Conditional
Scope expansion to related accountsConditional
Section 8
Runbook Invocation Matrix

This playbook does not execute actions directly — it invokes the following runbooks:

Revoke sessions / reset account ID-RB-01 › Review sign-in activity ID-RB-03 › Remove mailbox rules ID-RB-02 › Disable external forwarding EM-RB-07 › Endpoint follow-up investigation EP-RB-01 › OAuth app abuse investigation ID-RB-04 ›
Section 9
Communication & Coordination
  • Notify the affected user via an approved out-of-band communication channel
  • Coordinate password reset timing to avoid alerting the attacker prematurely
  • Brief IT support if hands-on user assistance is required for MFA re-registration
  • Ensure MDR ticket is aligned and updated with all containment actions
Section 10
Closure Criteria

The incident may be closed when all of the following are confirmed:

  • Credentials reset and MFA enforced and verified
  • All active sessions revoked
  • All persistence mechanisms removed
  • Sign-in activity reviewed and confirmed clean
  • Affected user confirms access fully restored
Section 11
Evidence to Capture
Sign-in logs
Risk detections from Entra ID
IP addresses and geolocations
Session revocation timestamps
User confirmation record
Section 12
Post-Incident Actions
  • Identify root cause (phishing, password spray, credential reuse)
  • Recommend user awareness follow-up if phishing was the vector
  • Tune identity detections based on indicators observed
  • Feed IOCs into threat hunting queries
