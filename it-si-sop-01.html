IT-SI-SOP-01: Security Infrastructure Baseline

The authoritative policy-level baseline for Web Application Firewalls (WAFs), load balancers, and Privileged Access Management (PAM) platforms across operating units. Defines shared baseline principles plus device-type-specific minimum security requirements. Operationalised by IT-SI-WI-01 (shared management plane) and IT-SI-WI-02 through 04 (device-type-specific). Vendor-agnostic, CIS/OWASP/NIST-aligned.

Section 1
Purpose

This SOP defines the minimum security baseline for security-infrastructure platforms: Web Application Firewalls (WAFs), load balancers and application delivery controllers, and Privileged Access Management (PAM) platforms. It establishes the controls every such platform must implement to be considered fit for production use, regardless of vendor, hosting model (on-premises, cloud-managed), or operating unit.

These platforms sit at three of the highest-value boundaries in the Institution's IT estate. A WAF protects web applications that often contain regulated or sensitive data; a load balancer terminates TLS for those same applications and is therefore in the cryptographic critical path; a PAM platform holds the credentials that grant administrative access to everything else. A weakness on any of them is uniquely impactful — and unlike network devices, which the network domain treats as a single conceptual class, these three platforms have distinct architectures and threat models that warrant device-type-specific guidance.

This SOP is operationalised by four Work Instructions: one shared management-plane WI (IT-SI-WI-01) and three device-type-specific WIs (IT-SI-WI-02 WAF; IT-SI-WI-03 Load Balancer; IT-SI-WI-04 PAM). See §4 for the requirements summary and the linked WIs for execution detail.

Section 2
Scope

2.1 In scope

  • WAFs (Web Application Firewalls) — on-premises appliances (F5 ASM, Imperva, FortiWeb, Barracuda) and cloud-managed WAFs (Azure Front Door / Application Gateway WAF, Cloudflare, AWS WAF). Includes virtual appliance instances.
  • Load balancers and application delivery controllers (ADCs) — F5 LTM, Citrix ADC, HAProxy Enterprise, NGINX Plus, A10, AVI Networks, and equivalents. Includes hardware appliances, virtual instances, and software-only deployments.
  • PAM platforms — CyberArk PAM, BeyondTrust, Delinea (Thycotic), HashiCorp Vault (as a credential broker), and equivalents. Covers the appliance and platform hardening aspects.
  • Devices in production, pre-production, disaster-recovery, and any environment carrying Institutional traffic or holding production credentials
  • Devices at any tier (Tier 0 / 1 / 2 / 3 per the existing tier model)

2.2 Out of scope

  • Reverse proxies and API gateways (standalone NGINX as RP, Kong, Apigee, Azure APIM, AWS API Gateway) — covered under a future Application Delivery sub-domain. Where a load balancer also performs reverse-proxy duties, this SOP's load-balancer requirements still apply to that aspect.
  • Cloud-WAF tenant-platform integration — Azure Defender for Cloud / Azure Policy / Log Analytics integration specifics are covered under the future Cloud Baselines domain. The WAF rule-set and operational-policy controls in this SOP apply regardless of where the WAF runs.
  • Identity governance for PAM — credential-vault rotation cadence, just-in-time access approval workflows, session-recording retention policy, and privileged-access governance are covered under a future Identity domain SOP. This SOP covers the platform hardening that protects the vault and brokering layer.
  • Network devices (switches, routers, traditional perimeter firewalls, wireless) — covered under IT-NET-SOP-01.
  • Endpoints and servers — covered under the existing Security Baselines domain.
  • SaaS-hosted security infrastructure where the vendor controls the security configuration entirely — these still appear in the Asset Register; the baseline applies to anything tenant-administered.
Section 3
Baseline Principles

The seven principles below shape every requirement in this baseline. Where an implementation choice arises, refer to these principles to resolve it consistently.

3.1 High-value targets need stronger baseline

These platforms hold or protect high-value assets (credentials, encrypted sessions, application traffic). The baseline applies more strictly than for general infrastructure: tighter management access, stronger authentication, more comprehensive logging, more frequent review.

3.2 Authentication strength matches what's protected

PAM access — and any administrative access to WAF or load-balancer platforms — requires the strongest authentication the Institution operates. NIST 800-63B AAL2 minimum for general administration; AAL3 where PAM grants access to Tier 0 systems.

3.3 Default-deny applies to everything, including the platforms themselves

Management interfaces accept only the protocols, sources, and accounts genuinely required. WAFs default to monitor-then-block, never permit-by-default. Load balancers expose only the virtual servers and management interfaces that have a documented purpose.

3.4 Cryptography in the critical path is not negotiable

Load balancers and WAFs that terminate TLS are part of the cryptographic critical path. Cipher suites, certificate validation, and key handling on these platforms protect every application behind them. Outdated TLS configuration here is a baseline violation, not a tuning question.

3.5 PAM has its own recovery path

If the PAM platform is unreachable, recovery cannot depend on credentials only retrievable from PAM. Every PAM deployment has a documented, tested, vault-independent break-glass procedure. This is the single biggest operational risk in PAM deployment and the most common implementation gap.

3.6 WAF tuning is continuous, not one-off

A WAF deployed in active blocking mode without ongoing tuning produces either too many false positives (operations disable it) or too many false negatives (security gap). The baseline requires an ongoing tuning programme, not a one-time configuration.

3.7 Comprehensive logging and audited privileged actions

Every platform forwards authentication, configuration changes, and security-relevant events to the central log destination. PAM session recordings are integrity-protected and retained per the Institution's retention policy. The audit trail must be sufficient to reconstruct who did what when, on demand.

Section 4
Minimum Security Requirements

The requirements below are mandatory for every in-scope platform. Each requirement domain is operationalised by a dedicated Work Instruction (linked at the right of each domain header). Where a platform cannot meet a requirement, a documented exception is required per §7.

🛡️

4.1 Common Management Plane & Logging (all platforms)

IT-SI-WI-01 →
  • Plaintext management protocols disabled (Telnet, HTTP, SNMPv1/v2c, FTP, TFTP); SSHv2, HTTPS TLS 1.2 minimum, SNMPv3 authPriv only
  • Administrative authentication via central directory (TACACS+, RADIUS, SAML, or Entra ID) with MFA required; PAM-administrator access requires AAL3 where the platform supports it
  • No shared accounts; the local emergency account is documented, vaulted (in a vault independent of this platform if this platform is the PAM itself), and used only with break-glass approval
  • Management-plane access restricted to a defined management network; bound to a dedicated management interface where the platform supports it
  • Idle session timeout ≤15 minutes for privileged sessions; AAL3-bound sessions follow more aggressive timeouts
  • Authorised-use banner displayed pre-authentication
  • NTP configured to two or more authoritative time sources with authentication
  • Syslog forwarding to two SIEM destinations; events include authentication (success and failure), configuration changes, and platform-specific security events
  • Certificate hygiene: organisation-CA-issued certificates for management interfaces (never self-signed); certificate inventory and renewal tracked centrally
  • Change-control workflow followed for every configuration change; emergency-change path exists with post-hoc review within 5 working days
🌐

4.2 WAF-specific requirements

IT-SI-WI-02 →
  • Active rule set covers the current OWASP Top 10; for WAFs supporting OWASP CRS, paranoia level 2 minimum (level 3 for high-sensitivity applications) after tuning
  • Every protected application has a documented WAF policy — rule set, exclusions, learning state, owner, last review date
  • Public-facing applications at Tier 0/1 are in active blocking mode within 90 days of WAF deployment; monitor-only is a transition state, not a permanent posture
  • False-positive triage workflow is documented and assigned; alerts review within agreed SLA (≤5 working days for Tier 0/1 applications)
  • Signature and rule-set updates applied within 30 days of vendor release; emergency signatures (active exploitation) applied within 7 days
  • TLS termination on the WAF uses approved cipher suites only (see §4.3); back-end re-encryption to the application server is required for Tier 0/1 applications
  • Bot management or rate-limiting controls active for public-facing applications; brute-force protection on authentication endpoints
  • WAF block events forwarded to SIEM with sufficient context for SOC investigation (rule ID, request metadata, source)
⚖️

4.3 Load balancer-specific requirements

IT-SI-WI-03 →
  • TLS 1.2 minimum on all virtual servers; TLS 1.3 preferred where back-end and clients support it; TLS 1.0 and 1.1 disabled
  • Cipher suites limited to AEAD ciphers (AES-GCM, ChaCha20-Poly1305); CBC-mode and RC4 disabled; ephemeral key exchange required (ECDHE preferred)
  • HSTS enabled on internet-facing virtual servers with a documented max-age (typically 1 year minimum); preload only after policy approval
  • Certificate management: tracked inventory with expiry alerting (≥30 days before expiry); auto-renewal where possible; certificate transparency logging awareness
  • Virtual server hygiene: every VS has a documented purpose, owner, last-used metric; orphaned VS removed within 90 days of no traffic
  • Health-check security: health-check endpoints do not expose application metadata; health checks reach the application via dedicated paths, not by faking application sessions
  • Connection rate limiting and SYN-flood protection enabled on internet-facing virtual servers
  • Source IP preservation (X-Forwarded-For with trusted-proxy validation, or transparent mode) so back-end logging retains real client IPs
  • HA pair configured for Tier 0 and Tier 1 load balancers; configuration sync verified; failover tested at minimum annually
  • Management interface and data-plane interfaces logically separated; management interface never reachable from data-plane VLANs
🔐

4.4 PAM platform-specific requirements

IT-SI-WI-04 →
  • Vault encryption: at rest using vendor-approved cryptography; HSM integration where supported; encryption keys not stored on the PAM appliance itself
  • Session broker hardening: jump-host configuration follows the same baseline as Tier 0 servers (the SB is a Tier 0 system regardless of what it brokers to)
  • Session recording integrity: recordings written to write-once or integrity-protected storage; tamper detection alerts to SOC
  • Just-in-time access enforcement: standing privileged access is the exception, not the default; documented approval workflow for time-bounded access grants
  • Identity-source integration: PAM authenticates users against the central directory (Entra ID, AD) with MFA; PAM does not maintain its own user database except for the documented break-glass account
  • High-availability pair configured for the PAM platform; configuration sync verified; failover tested at minimum annually
  • Break-glass procedure: documented, tested, vault-independent — recovery does not require credentials retrievable only from PAM. Tested annually under controlled conditions with results logged
  • Network exposure: PAM management plane and brokering interfaces on a dedicated management network; user-facing access endpoints reachable from administrative networks only
  • Privileged session events (start, end, command execution where captured, anomalies) forwarded to SIEM with retention matching session-recording retention
  • Backup of vault data (encrypted) included in the configuration-management WI's backup discipline; restore tested annually
One requirement that's worth saying twice. The single most common failure pattern in PAM deployment is a non-existent or untested break-glass procedure. When the PAM platform is unreachable and the only documented credentials are stored in it, recovery requires vendor intervention at best and a major incident at worst. Test the break-glass procedure annually. Test it before you need it.
Section 5
Roles & Responsibilities

IT Operations Lead

Owns this SOP. Approves baseline changes and exceptions. Owns the patch/firmware lifecycle. Sets change-window cadence. Accountable for baseline conformance metrics.

Network Engineering

Implements WAF and load-balancer hardening. Maintains golden configurations. Executes change windows. First-line responder for WAF/LB incidents.

Identity & Access Engineering

Implements PAM platform hardening. Maintains PAM configuration. Coordinates with Network Engineering on appliance-layer hardening. Owns the break-glass procedure.

Application Teams

Co-own WAF policy for their applications: rule-set selection, false-positive triage, allowlist requests, application-specific signature tuning.

SOC

Consumes logs from WAF (block events, anomalies), load balancers (anomalous traffic, TLS handshake failures), PAM (privileged session events, integrity anomalies). Investigates and escalates per the IRP.

GRC

Owns governance and audit evidence. Reviews baseline annually. Maintains exception register. Reports baseline conformance to the Risk Committee quarterly.

Local IT (per agency)

Applies baseline to platforms in their operating unit. Submits exception requests through their LCSO/CSLC. Coordinates change windows with central Network Engineering and IAE.

Architecture / Technical Authority

Adjudicates interpretation questions. Approves design-level exceptions. Reviews new platform adoption to confirm baseline applicability.

Separation of duties. The engineer who configures a WAF rule, load-balancer cipher list, or PAM access policy is not the same person who approves its move to production. Configuration changes pass through change control. For PAM specifically, the engineer with administrative access to the PAM platform is not the same person who approves a new privileged-access policy — this separation is foundational to PAM trustworthiness.

Section 7
Governance & Review

7.1 Baseline ownership and update authority

This SOP is owned by the IT Operations Lead with co-ownership by the Head of GRC and the Head of Identity & Access Engineering (for the PAM aspects). Material changes (new requirements, scope changes, framework-alignment changes) require sign-off from all three. Minor clarifications and editorial improvements may be applied by the owner with notification to GRC.

7.2 Review cadence

  • Annual review — minimum, by the owner with GRC sign-off, on the anniversary of approval
  • Out-of-cycle review — triggered by: a material OWASP Top 10 update; new vendor product adoption; a major incident involving these platforms; regulatory change affecting WAF or PAM (e.g. PCI DSS update, NIS2 implementing-act change); audit finding
  • Sub-WI review — each Work Instruction (IT-SI-WI-01 through 04) is reviewed independently and may update more frequently than the SOP, particularly WI-02 (WAF), which tracks OWASP and signature-vendor releases

7.3 Exception handling

Where a platform cannot meet a requirement in §4, the operating unit follows IT-GOV-SOP-01 (Baseline Exception & Risk Acceptance). Exceptions are time-bounded, recorded in GRC-REG-EXCEPTIONS, linked from the affected asset records in GRC-REG-ASSET, and reviewed annually. An exception that persists for three years without remediation is escalated to the Risk Committee.

Common cases that warrant exceptions: legacy vendor product limitations (older PAM appliance without HSM support), operational constraints (a WAF that must run in monitor-only mode pending application-team capacity for false-positive triage), integration requirements (a load balancer that must support a deprecated cipher suite for a legacy partner integration with a documented sunset date). Acceptable mitigations include compensating controls, monitoring uplift, and time-bounded remediation plans.

7.4 Compliance & audit

Baseline conformance is measured continuously through configuration-management tooling, periodic vendor-supplied scans (e.g. CIS-CAT, vendor-published audit scripts), and TLS configuration scanning (testssl.sh, SSL Labs, or vendor equivalents). Reports are produced monthly to IT Operations and quarterly to GRC and the Risk Committee. Audit evidence includes: golden configurations, scan results, exception register entries, change-control records, WAF tuning logs, PAM break-glass test records, and incident records related to security-infrastructure gaps.
