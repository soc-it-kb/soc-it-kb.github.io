Tier-1 Quick Checklist

ID-05: Token Theft / Session Hijacking

Use this checklist to triage a login where MFA was expected but not recorded. The key signal is successful authentication with no MFA challenge.

🔐 Identity Tier 1 Published 20 Dec 2025
🔑 What is token theft?
An attacker steals a login session token — usually via a phishing site that captures it in real time. With the token they can log in as the user without needing the password or MFA. This is why a successful login with no MFA challenge is a red flag.
🔔 When to use this checklist
Login alert without MFA challenge recorded / MDR escalation / impossible travel event where MFA was expected but not triggered.

Step 4 — Mandatory actions before escalating
Capture and attach to ticket
Source IP address
Device ID (from sign-in logs)
User-Agent string (browser / client used)
MFA result — challenged / bypassed / not applied
Attach full sign-in log entry to incident ticket
Do NOT reset credentials without Tier-2 approval — premature reset can alert the attacker
⚠️ Common Tier-1 mistakes to avoid
Assuming MFA always prevents compromise — token theft bypasses MFA entirely by stealing the post-MFA session
Closing the alert because the password was recently changed — token theft is independent of the password
Ignoring "token-based" login indicators in the sign-in logs — these appear as successful logins with no interactive MFA
Treating endpoint and identity as separate incidents — if a token was stolen, the endpoint may also be compromised
💡
Analyst Tip
If MFA is enabled and there is no MFA challenge recorded in the logs — assume token theft until disproven. A password reset alone will not help. Tokens must be explicitly revoked.
