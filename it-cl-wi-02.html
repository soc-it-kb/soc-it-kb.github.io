IT-CL-WI-02: M365 Tenant Hardening

Tier-aware implementation guidance for Microsoft 365 tenant hardening: Exchange Online anti-phishing, SPF / DKIM / DMARC, SharePoint / OneDrive sharing, Teams external access, Purview Information Protection and DLP, Defender for Office 365, Insider Risk Management, Customer Lockbox. Operationalises §4.2 of IT-CL-SOP-01.

Prerequisites

Before applying this WI:

  • WI-01 (Tenant Identity & Conditional Access) complete on this tenant — MFA, legacy-auth block, break-glass accounts in place
  • Tenant licensing tier confirmed and recorded in GRC-REG-ASSET (A1 / A3 / A5 / E1 / E3 / E5)
  • List of accepted email domains for the tenant available (every domain needs SPF / DKIM / DMARC)
  • External sharing policy decision made — what's the agency's posture on guest sharing, anonymous links, and federation? (the technical configuration follows the policy, not the other way around)
  • Data classification taxonomy available (typically Public / Internal / Confidential / Restricted) for sensitivity label deployment
  • Communication plan for users — external email banners, sharing policy changes, DLP enforcement notifications
  • For E5 tenants: list of pilot users for Defender for Office 365 features; Attack Simulator stakeholder agreement
Tier-aware throughout. Each step indicates the minimum tier at which it applies. Items marked All apply to every tenant; A3 / E3+ require Entra ID P1 / Purview / Defender features; E5 require Entra ID P2 / Defender for Office 365 Plan 2 / advanced Purview. For each tenant, document which steps applied and which were "feature unavailable at this licence tier."
Step-by-step instructions

The steps below cover the M365-tenant-specific controls required by IT-CL-SOP-01 §4.2.

1
Exchange Online anti-spam, anti-malware, anti-phishing baseline All
Default tenant protections enabled; spoof and phishing detection active for every mailbox.

Defender for Office 365 / Exchange Online Protection provides baseline anti-spam, anti-malware, and anti-phishing policies. Even A1 / E1 tenants get the basic policies.

  • Anti-spam policy — the default policy is acceptable for most tenants; review and confirm bulk-email threshold (typically BCL 7), action for spam (Junk Folder), and action for high-confidence spam (Quarantine)
  • Anti-malware policy — common-attachment filter enabled (blocks dangerous file types: .exe, .scr, .js, .vbs, .bat, .ps1, .msi, etc.); zero-hour auto-purge enabled
  • Anti-phishing policy — impersonation protection enabled, targeting key users (executives, finance, named high-risk roles), mailbox intelligence on; spoof intelligence on; first-contact safety tips on
  • External tagging — external email tagging enabled at organisation level (Set-ExternalInOutlook); messages from outside the organisation show a banner / tag in Outlook
  • Quarantine notifications — user-level quarantine notification policy enabled so users see their quarantined messages
  • Mailbox audit logging — on by default at tenant level; confirm enabled (Get-OrganizationConfig | fl AuditDisabled should show False)
Verify. Send test phishing-like message from external account; quarantined or junk-foldered with banner. Test malware attachment (EICAR test file or controlled lab); blocked. External tag visible on inbound external messages.
CIS M365 2.1.x · Defender for Office 365 baselines
2
SPF, DKIM, DMARC for every accepted domain All
Email authentication prevents spoofing of tenant domains by external attackers.

SPF, DKIM, DMARC are DNS-level controls that the receiving side uses to verify sender authenticity. Configuring them correctly is the largest single anti-spoofing control available.

  • SPF — TXT record per domain listing authorised senders, ending in -all (hard fail) or ~all (soft fail). For M365 the base: v=spf1 include:spf.protection.outlook.com -all. Add include records for other legitimate senders (marketing platforms, ticketing systems) that send as the domain.
  • DKIM — enable in the Defender portal (Policies & rules → Email authentication → DKIM) for each accepted domain; rotate the key pair annually; ensure the published CNAMEs point to Microsoft's selectors
  • DMARC — TXT record at _dmarc.<domain>; deploy progressively:
    • Phase 1: p=none; rua=mailto:dmarc-reports@<domain> (monitor only; collect reports for 30–60 days)
    • Phase 2: p=quarantine; pct=25 → pct=100 (gradual enforcement)
    • Phase 3: p=reject (target state)
  • DMARC report processing — either a third-party DMARC reporting service or in-house parsing of the rua reports; without this DMARC is opaque
  • Subdomain policy — sp=reject to prevent attackers from spoofing subdomains the tenant doesn't actively use
  • Reject parked / unused domains — for domains the organisation owns but doesn't send from, publish v=spf1 -all and DMARC p=reject immediately
Target: DMARC p=reject. p=none is a baseline starting point, not a destination. Each accepted domain progresses from p=none to p=quarantine to p=reject over a 90–180-day window with monitoring.
Verify. External tool (mxtoolbox or equivalent) shows valid SPF and DMARC records for every accepted domain. Test message from a spoofed external source rejected by DMARC. DMARC reports being ingested.
CIS M365 2.1.x · RFC 7489 (DMARC)
3
SharePoint and OneDrive external sharing restrictions All
External sharing is deliberate, not permissive; tenant-level controls prevent accidental data exposure.

SharePoint Online and OneDrive default to permissive external sharing. Tighten at tenant level so individual users do not accidentally over-share.

  • Tenant-level external sharing — SharePoint admin centre → Policies → Sharing. Default for most tenants: "New and existing guests" (not "Anyone"). For high-sensitivity tenants: "Existing guests" only. Reduce as needed per agency risk posture.
  • OneDrive sharing can be more restrictive than SharePoint (e.g. SharePoint = New and existing guests; OneDrive = Existing guests only); set the lower limit deliberately
  • Anonymous (Anyone) links — disabled by default; if enabled for a specific scenario, set link expiry (≤30 days) and link permissions (View, not Edit)
  • Link defaults — default sharing link type "Specific people" not "Anyone with the link"; default link permission View not Edit
  • Allow / block domains list — allow-list specific trusted external domains for sharing; block-list known competitors / high-risk domains. Allow-list mode is more restrictive than block-list and is preferred for sensitive tenants.
  • Guest access expiration — guest users automatically lose access to specific sites after a period (typically 60–180 days) unless reauthorised
  • External user re-authentication — external users must re-authenticate every 30–90 days to maintain access
  • Conditional Access for external users applies to SharePoint and OneDrive (per WI-01 CA-007); sensitive sites blocked for guests without explicit justification
Verify. Test sharing scenarios from a standard user account: external sharing to an approved domain works; anonymous link creation refused or constrained; sharing to a blocked domain refused. Guest user count and recent guest activity visible in admin centre.
CIS M365 7.2.x · "SharePoint external sharing"
4
Teams external access, federation, and meeting policies All
Teams external collaboration is allow-listed; anonymous meeting access controlled.

Teams external access (federation between organisations) and guest access (named external users in your tenant's Teams) are separate controls. Configure both.

  • External access (federation) — Teams admin centre → Users → External access. Default-deny pattern: "Allow only specific external domains" with an allow-list of partner-organisation tenants; or block known high-risk domains. Skype consumer federation: typically disabled.
  • Guest access — enabled but governed by WI-01 §8 (Entra guest invite policy). Guest restrictions on Teams resources: cannot make private calls, cannot share screen by default in meetings with guests present.
  • Meeting policies — default meeting policy and named-user policies for higher-risk roles:
    • Anonymous users can join meetings — restricted by default; enable per-meeting for legitimate external events only
    • Lobby for external attendees on by default; named lobby bypass for trusted partner tenants
    • Recording — allowed but transcripts default to off for sensitive groups; recordings expire after a documented retention
    • Live caption / transcription — consent-based; documented in the meeting policy
    • Whiteboard, chat retention, file sharing in chat — governed per policy and tenant data classification
  • Apps in Teams — Teams app permissions policy: block third-party apps by default; allow-list specific approved apps; review the catalogue quarterly
  • Channel email integration — disabled by default unless the agency has a documented use case
Verify. Attempt to invite a non-allow-listed external organisation to a Teams meeting; refused. Anonymous join attempt against a sensitive meeting hits the lobby. Third-party app installation by standard user fails.
CIS M365 8.x · "Teams external access"
5
Unified Audit Log and retention All
Tenant-level audit captures user and administrative actions across services for the maximum period the tier allows.

Unified Audit Log (UAL) captures activity across Exchange, SharePoint, OneDrive, Teams, Entra, Power Platform, and Defender. It is the single most important forensic source for cloud incidents.

  • Enable UAL at tenant level — Search-UnifiedAuditLog works when enabled; check in Compliance / Purview portal under Audit. Most tenants have it on by default but verify.
  • Retention — per tier:
    • A1 / E1 90 days standard retention
    • A3 / E3 90 days standard; Audit (Premium) add-on extends to 1 year
    • A5 / E5 1 year by default; 10-year retention available as add-on for specific event types
  • Audit log search — named SOC analysts have Audit Log Search role (or scoped equivalent via PIM); routine searches documented in SOC playbooks
  • Forwarding to central SIEM — via Microsoft Graph API (Office 365 Management Activity API), Azure Monitor, or vendor connector. Forward all audit records; do not pre-filter at the tenant side (the SIEM decides what to retain).
  • High-value events alerted in SIEM: Global Administrator role assignment, eDiscovery hold creation, mailbox permission change to high-privilege user, large-volume export, sharing-policy change, audit-log-disabled event (this last is critical — if it fires, the tenant is compromised)
Verify. UAL on. Test event (e.g. file-share, mail-send) appears in UAL within ~30 minutes. SIEM ingestion pipeline operational. Sample alert rules firing on test events.
CIS M365 1.x · "Unified Audit Log"
6
Purview Information Protection sensitivity labels A3 / E3+
Documents and emails are labelled with sensitivity; labels drive downstream controls (encryption, sharing restrictions, DLP).

Sensitivity labels are the cornerstone of Microsoft Information Protection (MIP). They classify content (manual or automatic) and apply protection policies (encryption, watermark, sharing restrictions, retention).

  • Label taxonomy — align to the organisation classification (e.g. Public / Internal / Confidential / Restricted). Keep the taxonomy short (≤6 labels at top level) to be usable.
  • Label configuration per level
    • Public — no encryption; visual marking optional
    • Internal — no encryption; "Internal Use Only" footer / watermark
    • Confidential — encryption applied; usage restrictions (block forwarding, block printing for some sub-labels); visible marking
    • Restricted — encryption + named-user rights only; download/copy restrictions; visible marking
  • Label policy publication — publish labels to users / groups; document the rollout plan; train users (most labels remain unused without training)
  • Default label — new documents and emails get a default label (typically Internal) unless the user chooses otherwise — reduces "Public by accident"
  • Mandatory labelling — users cannot save / send without selecting a label; deploy after a training period
  • Auto-labelling E5 — auto-apply labels based on content (sensitive info types like credit cards, government IDs, named data classes); start in simulation mode then enforce
  • Co-authoring on encrypted documents — enable for usability; verify the tenant configuration supports it
  • Label scope — Files & Emails, Groups & Sites (containers), Schematized data (Power BI) — configure per organisation need
Verify. Labels visible in Office desktop, web, and mobile. Default label applied to new documents. Encrypted label opens correctly for authorised users; access denied for non-authorised. Auto-labelling (E5) flags sample sensitive content.
CIS M365 6.x (Purview) · ISO 27001 A.5.13
7
Purview DLP policies A3 / E3+
Sensitive data leaving the tenant is detected; policy-tip and block actions reduce accidental exfiltration.

Data Loss Prevention policies detect sensitive content in motion (email, sharing, Teams messages) and at rest (SharePoint, OneDrive). Deploy progressively.

  • Data classes — identify what the agency actually handles: personal data (per applicable regulation), payment card data, health data, government ID numbers, financial account numbers, intellectual property. Use Microsoft's built-in Sensitive Information Types where they match.
  • Policy locations — Exchange Online, SharePoint, OneDrive, Teams chat & channel messages, Devices (DLP for Endpoint, requires Defender for Endpoint). Cover all locations from the start; partial coverage gets bypassed.
  • Rules per data class
    • Notify user with a policy tip (do this first, monitor for FP)
    • After tuning: block the action with override allowed (user can override with justification, logged)
    • For high-sensitivity: block without override
    • Notify SOC / GRC on policy match for high-sensitivity classes
  • Rollout pattern — Test mode (no user notification) → Test mode with notification → Enforce. Each phase 2–4 weeks.
  • Exclusions discipline — legitimate exclusions documented (e.g. legal team handling sensitive data by job nature); reviewed quarterly
  • Endpoint DLP E5 — extends DLP to client-side actions (USB copy, print, browser upload to non-corporate destinations); requires Defender for Endpoint
  • DLP alerts in central SIEM — cross-reference with sign-in and sharing logs to identify likely exfiltration patterns
Verify. Test message containing sample sensitive data (controlled test) triggers DLP policy tip / block as configured. DLP alerts visible to SOC. Override actions logged.
CIS M365 6.x · ISO 27018
8
Defender for Office 365 Plan 2 — Safe Links, Safe Attachments, Attack Simulator E5
Inline detonation of attachments and time-of-click URL rewriting catches what signature engines miss; user training measures human risk.

Defender for Office 365 Plan 2 (included in E5; available as add-on to E3) adds significant protections beyond the baseline Exchange Online anti-phishing.

  • Safe Attachments policy — sandbox detonation of email attachments before delivery; Dynamic Delivery mode (body delivered immediately, attachment replaced after scanning) preferred over Block (better user experience, equivalent security); cover all users
  • Safe Attachments for SharePoint, OneDrive, Teams — enabled; files identified as malicious are blocked from opening / download
  • Safe Links policy — URL rewriting in email and Teams; time-of-click verification against Microsoft's reputation database; URLs scanned at click, not just at send (defeats delayed weaponisation); apply to all users
  • Safe Links for Office apps — rewriting inside Word, Excel, PowerPoint, Outlook desktop and web
  • Attack Simulator — quarterly minimum; campaigns per role / department; payloads include credential harvesting, malware attachment, link-click; results feed user training tracking
  • Targeted threat protection — preset Strict policies for VIP / executive / finance users (more aggressive thresholds, more detonation, additional impersonation protection)
  • Threat Explorer — SOC has access via Defender portal; queries documented in SOC playbooks for the email-threat scenarios
Verify. Test email with EICAR or controlled-test attachment — held for Safe Attachments scan, marked safe / unsafe correctly. Test URL in email — rewritten with safelinks domain. Attack Simulator campaign run, results visible.
CIS M365 (E5) 2.x · "Defender for Office 365 P2"
9
Insider Risk Management and Communication Compliance E5
Anomalous insider activity surfaced; regulated communications monitored where legally required.

Insider Risk Management (IRM) detects risky activity from internal users (data exfiltration before resignation, privilege escalation, anomalous download patterns). Communication Compliance (CC) monitors regulated communications channels.

  • Insider Risk indicators — configure the activity signals: data exfiltration to USB, anomalous download from SharePoint, unusual file deletion, departing-employee patterns (high download volume in weeks before HR-flagged departure), privilege escalation events
  • IRM policies — start with Microsoft's preset templates (Data Theft by departing users; General data leaks; Risky browser usage). Tune per agency risk posture.
  • Reviewers — named individuals with the Insider Risk Management Investigator role; not the SOC at large (privacy concern); coordinate with HR and Legal
  • Privacy considerations — IRM displays pseudonymised user information by default; un-pseudonymisation requires named approver; document the approval workflow
  • Communication Compliance — where regulated (financial services, healthcare, government), monitor communications channels (email, Teams chat) for prohibited content; investigate alerts via a separate role (Reviewer); document the legal basis
Coordinate with HR and Legal. IRM and CC have legitimate workplace-monitoring concerns. The legal basis varies by jurisdiction and the agency's location. Engage HR and Legal before enabling; document the lawful-basis assessment and the data-handling policy.
Verify. IRM policy active; sample test activity (controlled) appears in the IRM dashboard. Reviewer role configured. Privacy / pseudonymisation defaults documented.
CIS M365 (E5) · Local-law compliance
10
Customer Lockbox and Privileged Access Management for Exchange E5
Microsoft engineer access to tenant data requires explicit approval; privileged Exchange actions require multi-party approval.

Two E5-only controls that meaningfully reduce the standing-trust footprint:

  • Customer Lockbox — when a Microsoft engineer needs access to tenant data to resolve a support case, the tenant administrator must approve; without approval, the engineer cannot proceed. Configure approval workflow with named approvers and SLA (typically 12 hours).
  • Privileged Access Management (PAM) for Exchange Online — high-impact Exchange administrative actions (e.g. mailbox export, change journaling, modify role assignment) require activation through a PAM request with approval. Configure for the Exchange role groups and the high-impact tasks.

These are separate from the broader Entra PIM (WI-01 Step 7); Exchange PAM is finer-grained to Exchange-specific operations.

Verify. Customer Lockbox enabled; test approval workflow with a Microsoft-initiated request (if available) or via the platform test. Exchange PAM operational; test a privileged Exchange action requiring approval.
CIS M365 (E5) · "Customer Lockbox", "PAM for Exchange"

Quantitative metric

Microsoft Secure Score for M365 against the tenant's licence tier. Targets per IT-CL-SOP-01 §7.3: A1/E1 reported only (limited surface); A3/E3 ≥70%; E5 ≥80%.

Rollback & troubleshooting

M365 changes propagate quickly; have rollback ready especially for sharing and DLP policy changes that affect end-user workflow.

Common scenarios

  • DMARC p=reject rejecting legitimate email — cause: an unlisted legitimate sender (marketing platform, ticketing system, newsletter service) not in SPF. Fix: revert to p=quarantine temporarily; identify the sender from DMARC reports; add to SPF; re-progress to p=reject.
  • Mass DLP false positives blocking legitimate business — cause: policy too broad; data class definition too loose (regex catches too much). Fix: switch policy to Test mode (no block) while tuning; refine the data class; refine the locations / users in scope; re-enable.
  • External sharing tightening broke partner collaboration — cause: a legitimate external collaborator domain not in the allow-list. Fix: add to allow-list with documented justification and quarterly review; do not relax tenant-wide setting.
  • Teams federation block isolated an agency from a partner — cause: partner's tenant not in allow-list after the federation change. Fix: add the partner tenant; verify cross-tenant access settings.
  • Sensitivity label adoption stuck at low usage — cause: labels exist but users don't apply them; auto-labelling not deployed; training gap. Fix: enable default label assignment, mandatory labelling after training; deploy auto-labelling for the highest-volume data classes.
  • Safe Attachments delaying critical delivery — cause: Dynamic Delivery not enabled; messages held during scan. Fix: switch to Dynamic Delivery mode (body delivered immediately).
  • Attack Simulator backlash — cause: simulation perceived as entrapment; users distrust security communications. Fix: coordinate with HR / internal comms before campaigns; brief leadership before high-visibility simulations; debrief constructively with simulated victims.

Rollback procedure

  1. Identify the breaking change — recent change record, DLP / audit log, user reports
  2. For DLP: switch the policy to Test mode (immediate, no end-user impact)
  3. For DMARC: lower the policy strength (p=reject → p=quarantine → p=none in extremis)
  4. For sharing / federation: targeted exception, not tenant-wide loosening
  5. Tune the underlying issue (data-class definition, allow-list entry, federation peer); re-test; re-enable
  6. Update change record; document root cause
Common errors & tips
  • DMARC at p=none is not a destination. A common gap is staying in monitor mode for years. Schedule the progression to p=reject as a tracked action.
  • DMARC reports need to be processed. Without ingesting rua reports, the p=none phase produces no insight. Either subscribe to a DMARC processor or build the parsing.
  • Parked domains are a free spoofing surface. Domains the organisation owns but doesn't send from need v=spf1 -all + DMARC p=reject from day one.
  • External sharing tenant defaults are permissive. Out of the box, SharePoint allows "Anyone with the link." Tighten before users discover it.
  • Anonymous Teams meeting access is sometimes legitimate. All-hands events, public webinars. Don't tenant-wide block; configure per-meeting allowances.
  • UAL has gaps. Some workloads have delayed ingestion (some Power Platform events; some Defender events). Document the known gaps so SOC investigations don't assume completeness.
  • Auto-labelling in simulation mode forever is a non-control. Move from simulation to enforce; revisit quarterly.
  • DLP without endpoint DLP misses the most likely exfiltration path. A user copying to USB or uploading to a personal cloud bypasses tenant-side DLP. Endpoint DLP (E5 + Defender for Endpoint) is where the egress actually happens.
  • Safe Attachments Block mode is poor UX. Dynamic Delivery has equivalent security with better experience; default to it.
  • Attack Simulator without training is just punishment. Every simulated phish links to a training module; users who fail get the training, not just a "gotcha."
  • IRM has legal complexity. Don't deploy without HR / Legal sign-off and a documented lawful-basis. The technical capability is well ahead of the policy clarity in most organisations.
  • Customer Lockbox approval SLA matters. If the approval workflow can't respond within 12 hours, Microsoft support cases stall. Confirm the SLA against named approvers.
